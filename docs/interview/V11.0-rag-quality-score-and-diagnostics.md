# V11.0 RAG Quality Score & Diagnostics 面试文档

## 1. 阶段摘要

V11.0 的阶段名称是：

```text
V11.0 RAG Quality Score & Diagnostics
```

中文可以表述为：

```text
RAG 质量评分与诊断
```

这一阶段的核心目标是：把已有的 RAG 检索评估能力产品化，变成用户能理解的综合评分、维度分、扣分原因和优化建议。

在 V11.0 之前，系统已经能完成：

```text
上传文档；
构建知识库；
选择知识库分组；
检索上下文；
生成回答；
展示引用来源；
接入本地 Ollama 和多模型 Provider；
执行 Agent Task 和 Tool Workflow。
```

但是此时还有一个关键问题：

```text
系统能回答，但用户不知道这次回答质量到底好不好；
如果回答不好，用户不知道问题出在知识库、检索、上下文、模型、引用还是配置。
```

V11.0 就是为了解决这个问题。

---

## 2. 为什么需要 RAG 质量评分

RAG 系统的质量不是只看“模型回答得像不像”。

一个回答质量不好，可能有多种原因：

```text
知识库没有相关文档；
选择了错误的 Collection；
embedding 模型不适合；
topK 太小导致召回不足；
topK 太大导致噪声太多；
上下文片段和问题不够相关；
LLM 没有严格依据引用回答；
引用来源缺失；
模型太慢；
切换 embedding 模型后没有重新索引。
```

因此，RAG 评估应该拆成多层：

```text
检索质量；
上下文质量；
回答质量；
引用质量。
```

V11.0 的目标不是引入一堆新指标，而是把已有指标和启发式规则组织成用户能理解的质量反馈系统。

---

## 3. 阶段核心目标

用户在 Ask 页面提问后，系统不仅返回回答和引用，还要返回：

```text
1. 综合评分；
2. 检索质量分；
3. 上下文质量分；
4. 回答质量分；
5. 引用质量分；
6. 当前评分模式；
7. 权重说明；
8. 主要扣分原因；
9. 可执行优化建议；
10. 技术详情中的原始指标。
```

页面展示示例：

```text
综合评分：82 / 100
质量等级：良好
评分模式：平衡模式

主要扣分原因：
- 检索召回不足；
- 引用片段不够集中。

优化建议：
- 补充相关文档；
- 缩小知识库分组范围；
- 提高 topK；
- 切换更适合中文的 embedding 模型；
- 对文档重新索引。
```

## 4. 核心设计一：RagQualityScore 数据结构

V11.0 新增评分结构：

```text
RagQualityScore
```

典型字段：

```text
overallScore
overallLevel
displayLevel
retrievalScore
contextScore
answerScore
citationScore
mode
displayMode
weights
diagnosis
suggestions
metricDetails
```

### 4.1 分数范围

```text
0 - 100
```

### 4.2 等级划分

建议等级：

```text
EXCELLENT -> 优秀
GOOD      -> 良好
FAIR      -> 一般
POOR      -> 较差
UNKNOWN   -> 暂无评分
```

### 4.3 四个维度分

#### retrievalScore：检索质量分

表示：

```text
系统是否找到了足够相关的内容。
```

#### contextScore：上下文质量分

表示：

```text
检索回来的上下文是否集中、相关、少噪声。
```

#### answerScore：回答质量分

表示：

```text
回答是否直接回应问题，是否大体基于上下文。
```

#### citationScore：引用质量分

表示：

```text
引用来源是否充足、可追溯，并且和回答有对应关系。
```

## 5. 核心设计二：评分模式与权重

V11.0 支持三种评分模式：

```text
BALANCED
PRECISE
COMPREHENSIVE
```

中文：

```text
平衡模式；
精准模式；
全面模式。
```

### 5.1 平衡模式

适合：

```text
普通个人知识库问答。
```

权重建议：

```text
retrievalScore: 0.30
contextScore: 0.25
answerScore: 0.25
citationScore: 0.20
```

### 5.2 精准模式

适合：

```text
合同；
技术文档；
论文；
规则类文档；
需要保守回答的场景。
```

权重建议：

```text
retrievalScore: 0.20
contextScore: 0.30
answerScore: 0.30
citationScore: 0.20
```

特点：

```text
更重视上下文精度、回答忠实度和引用准确性。
```

### 5.3 全面模式

适合：

```text
调研；
总结；
资料汇总；
头脑风暴。
```

权重建议：

```text
retrievalScore: 0.40
contextScore: 0.20
answerScore: 0.20
citationScore: 0.20
```

特点：

```text
更重视召回覆盖面。
```

## 6. 核心设计三：综合评分计算

综合评分公式：

```text
overallScore =
  retrievalScore * retrievalWeight
+ contextScore * contextWeight
+ answerScore * answerWeight
+ citationScore * citationWeight
```

最后结果限制在：

```text
0 - 100
```

并保留整数。

### 6.1 为什么要做加权评分

不同场景对质量的要求不同：

```text
合同问答更需要精准；
调研总结更需要全面；
普通问答需要平衡。
```

因此不能只有一套固定评分。通过评分模式和权重，可以让用户选择：

```text
我更看重精确性；
还是更看重覆盖面。
```

## 7. 核心设计四：指标映射

V11.0 不继续堆新指标，而是复用已有指标：

```text
Recall@K；
Precision@K；
HitRate@K；
MRR / RR@K；
NDCG@K；
ContextPrecision@K；
citations count；
final context count；
no-context reason；
retrieval scores；
latency；
token usage。
```

### 7.1 retrievalScore 映射

优先使用：

```text
Recall@K；
HitRate@K；
MRR；
NDCG@K。
```

如果没有标准答案，则采用启发式估算：

```text
finalContextCount > 0 加分；
top score 高加分；
topK 有足够候选加分；
no-context 直接低分；
collection 过滤后无上下文低分。
```

### 7.2 contextScore 映射

优先使用：

```text
ContextPrecision@K；
Precision@K。
```

没有标注时使用启发式：

```text
top results score 分布集中，加分；
final contexts 数量合理，加分；
检索结果过少，扣分；
结果过多且分数差异大，提示可能混入噪声；
query 与 snippet 重合度太低，扣分。
```

### 7.3 answerScore 映射

V11.0 默认不使用真实 LLM-as-a-judge。

启发式包括：

```text
answer 非空；
answer 不是 no-context fallback；
answer 与 question 有关键词相关性；
answer 包含结构化段落；
answer 没有在 citations 为空时生成确定性答案。
```

注意：

```text
V11.0 不声称这是严格事实一致性评估，而是启发式回答质量估算。
```

### 7.4 citationScore 映射

依据：

```text
citations count；
citations 是否来自 active document；
citations 是否来自当前 collection scope；
citations 是否包含 chunkId / documentId / score / snippet；
answer 有内容但 citations 为空，扣分；
citations 太少，扣分。
```

---

## 8. 核心设计五：缺失指标处理

V11.0 的重要原则是：

```text
不伪造缺失指标。
```

如果没有标准答案或人工标注，就不能假装有严格 Recall@K。

正确处理方式：

```text
缺少标准答案时，标记为 heuristic；
部分指标为 UNKNOWN；
在技术详情中说明当前采用启发式估算。
```

例如：

```text
当前缺少标准答案或人工标注，因此 Recall@K 采用启发式估算。
```

这样可以保持系统诚实，也避免指标看似精确但实际虚假。

## 9. 核心设计六：Diagnosis Engine

V11.0 新增诊断服务：

```text
RagQualityDiagnosisService
```

目标是把低分原因转成中文可解释诊断。

诊断结构：

```text
summary
issues[]
suggestions[]
```

issue 字段：

```text
code
title
description
severity
relatedMetric
scoreImpact
```

suggestion 字段：

```text
code
title
description
actionType
```

严重等级：

```text
INFO
WARNING
CRITICAL
```

## 10. 典型诊断规则

### 10.1 NO_CONTEXT

条件：

```text
finalContextCount = 0
```

中文解释：

```text
未检索到可用上下文。
```

建议：

```text
上传更多相关文档；
检查是否选择了错误的知识库分组；
提高 topK；
确认文档未被删除或未被过滤。
```

---

### 10.2 LOW_RETRIEVAL_RECALL

条件：

```text
retrievalScore < 60
```

中文解释：

```text
检索召回不足，系统可能没有找到足够相关的知识片段。
```

建议：

```text
补充相关文档；
调整知识库分组；
尝试全面模式；
提高 topK；
更换更适合中文的 embedding 模型；
对文档重新索引。
```

---

### 10.3 LOW_CONTEXT_PRECISION

条件：

```text
contextScore < 60 且 finalContextCount > 0
```

中文解释：

```text
检索结果中可能混入较多噪声片段。
```

建议：

```text
缩小 Collection 范围；
降低 topK；
后续开启 rerank；
优化 chunk 大小；
移除重复或低质量文档。
```

---

### 10.4 WEAK_CITATION

条件：

```text
citationScore < 60
```

中文解释：

```text
引用来源不足或引用信息不完整。
```

建议：

```text
确认回答使用了知识库上下文；
检查 citation 生成链路；
增加可引用片段数量；
避免无上下文生成确定性回答。
```

---

### 10.5 ANSWER_NOT_GROUNDED_RISK

条件：

```text
answer 有确定性内容，但 citations 为空或 finalContextCount = 0
```

中文解释：

```text
回答可能缺少知识库依据。
```

建议：

```text
要求模型仅基于引用来源回答；
无上下文时返回“不知道”；
补充相关文档。
```

---

### 10.6 EMBEDDING_MODEL_SWITCH_RISK

条件：

```text
embedding model / dimension 与当前索引不一致；
或 metadata 中存在 provider warning。
```

中文解释：

```text
当前 embedding 模型可能与已有索引不一致。
```

建议：

```text
对相关文档重新索引；
确认默认 embedding provider；
检查模型设置页面。
```

---

### 10.7 SLOW_RESPONSE

条件：

```text
latencyMs 超过阈值，例如 15s / 30s
```

中文解释：

```text
本次回答耗时较长。
```

建议：

```text
使用更小的 LLM；
降低 topK；
缩小知识库范围；
减少上下文长度；
使用本地性能更好的模型或云模型。
```

## 13. 为什么默认不做 LLM-as-a-judge

LLM-as-a-judge 可以评估回答质量，但 V11.0 默认不做，原因是：

```text
会引入真实外部 API 依赖；
会增加成本；
评估本身可能不稳定；
需要额外 prompt 和模型配置；
默认测试会变复杂；
本阶段目标是先做可解释、低依赖、产品化的评分系统。
```

未来可以在高级模式中接入 judge 模型，但不是 V11.0 的边界。

## 19. 面试高强度追问与回答

### 19.1 为什么要做 RAG 质量评分？

回答：

```text
因为 RAG 系统不是只要能生成回答就够了。回答不好可能是知识库缺内容、检索召回不足、上下文噪声过多、引用不足或模型没有严格依据上下文。RAG 质量评分可以把这些问题拆成检索质量、上下文质量、回答质量和引用质量，让用户知道问题出在哪里。
```

---

### 19.2 为什么不继续堆更多指标？

回答：

```text
因为指标越多不代表产品越好。现有的 Recall@K、Precision@K、MRR、NDCG、ContextPrecision 已经足够覆盖主要检索质量。V11.0 的重点是把指标产品化，变成综合评分、扣分原因和优化建议，而不是继续堆用户看不懂的指标。
```

---

### 19.3 没有标准答案时怎么计算 Recall@K？

回答：

```text
没有标准答案时不能伪造严格 Recall@K。系统会标记部分指标为启发式估算，比如根据 finalContextCount、top score、no-context reason、citation 数量等信号估计检索和上下文质量，并在技术详情中说明当前缺少标准答案或人工标注。
```

---

### 19.4 为什么默认不用 LLM-as-a-judge？

回答：

```text
LLM-as-a-judge 会引入额外模型调用、成本、网络依赖和不稳定性，也会让默认测试依赖外部 API。V11.0 的目标是先做可解释、低依赖的质量诊断。后续可以把 LLM judge 作为高级可选能力，而不是默认路径。
```

---

### 19.5 精准模式和全面模式有什么区别？

回答：

```text
精准模式更适合合同、技术文档、论文等场景，权重更偏向上下文精度、回答质量和引用质量；全面模式更适合调研和总结，权重更偏向检索召回，希望覆盖更多相关材料。
```

---

### 19.6 no-context 怎么处理？

回答：

```text
如果 finalContextCount 为 0，系统会返回低评分并触发 NO_CONTEXT 诊断，提示用户可能没有上传相关文档、选择了错误的知识库分组、topK 太低，或者文档被删除 / 过滤。这个场景下既有行为仍然是不调用 LLM，避免无依据生成。
```

---

### 19.7 评分会不会误导用户？

回答：

```text
所以 V11.0 把评分设计为可解释结构，不只是给一个数字。系统会显示维度分、权重、扣分原因和建议；缺失标准答案时也会说明当前是启发式估算，而不是严格人工标注评估。
```

## 21. 总结表达

V11.0 可以用一段话总结：

```text
在 V11.0 中，我将 RAG 评估能力产品化为 RAG Quality Score & Diagnostics。系统在 Ask 回答之外返回综合评分、检索质量分、上下文质量分、回答质量分和引用质量分，并支持平衡、精准、全面三种评分模式。

同时，我实现了 Diagnosis Engine，根据 no-context、低召回、低上下文精度、引用不足、无依据回答风险、响应耗时过长等情况生成中文扣分原因和优化建议。这个阶段的重点不是引入复杂 LLM judge，而是在默认低依赖的前提下，用已有指标和启发式信号构建可解释的 RAG 质量反馈系统，让用户知道答案是否可信，以及应该补文档、调范围、提高 topK、切换模型还是重新索引。
```
