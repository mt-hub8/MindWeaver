# V2.1 面试文档：文档上传与基础 Chunking

## 1. 本版本解决的问题

V2.1 解决 RAG 系统的第一步：知识文档如何进入系统并被切成可处理片段。上传 .txt/.md 后，系统保存 document 记录，并把文本切成 document_chunk，为后续 embedding 和检索做准备。

## 2. 为什么现在做这个版本

它依赖前面任务编排和 LLM 执行基础。要做 RAG，必须先有文档、chunk 和可查询文本片段。它为 V2.2 Mock Embedding & Vector Search 做准备。此时不做 PDF/Word/OCR，是为了先验证纯文本链路。

## 3. 核心需求边界

本版本支持 POST /documents、GET /documents/{id}、GET /documents/{id}/chunks，只支持 .txt / .md，按固定长度切分。不做 embedding、不做向量检索、不做 PDF/Word/OCR、不做 RAG Answer。

## 4. 方案选择

document 表保存文件级元数据和处理状态，document_chunk 表保存分块文本。上传后读取 UTF-8 文本，不保存原始文件到磁盘，直接把内容切分入库。

## 5. 替代方案与暂时不选的原因

替代方案一是直接把整篇文档送给 LLM。当前不选，因为文档可能超长，也不利于检索。

替代方案二是一开始支持 PDF/Word/OCR。当前不选，因为解析复杂度高，先跑通 txt/md 更稳。

## 6. 数据库设计

新增 document 表和 document_chunk 表。document 保存文件名、类型、大小、状态、chunkCount、errorMessage 等。document_chunk 保存 document_id、chunk_index、content、content_length。后续 V2.1.1 增加 offset、headingPath 和 chunkStrategy。

## 7. API 设计

POST /documents 上传 .txt/.md 文件。

GET /documents/{documentId} 查询文档处理状态。

GET /documents/{documentId}/chunks 查询 chunk 列表。

非法文件类型应返回错误，处理失败时 document status = FAILED。

## 8. 核心流程

用户上传文档
-> DocumentController
-> DocumentService 校验后缀
-> UTF-8 读取文本
-> 保存 document
-> 固定长度切分
-> 保存 document_chunk
-> status = CHUNKED
-> 查询 chunks

## 9. 测试方案

建议覆盖 .txt 上传、.md 上传、非法类型拒绝、空文件、长文本多 chunk、GET 文档和 chunks。SQL 验证 document.status、chunk_count 和 document_chunk 条数。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是边界控制。文档处理看似简单，但文件类型、编码、空内容、长文本和失败状态都要处理，否则后续 embedding 会吃到脏数据。

## 12. 本版本亮点

- 建立 RAG 前置文档入口。
- 将文档拆成可检索 chunk。
- 清晰区分 document 元数据和 chunk 内容。
- 暂不处理复杂格式，降低早期风险。

## 13. 简历 bullet

### 简洁版

实现文档上传与基础 chunking，支持 .txt/.md 文档入库并切分为 document_chunk，为后续 embedding 检索打基础。

### 强化版

为 RAG 前置链路设计 document/document_chunk 数据模型和上传处理流程，将原始 .txt/.md 文档转化为可检索文本片段，为后续向量化、TopK 检索和 Citation 奠定基础。

## 14. 面试高强度追问与回答

Q1：为什么 RAG 前要做 chunking？

A1：LLM 上下文有限，检索也需要较小的语义单元。chunking 把长文档变成可检索片段。

Q2：为什么只支持 txt/md？

A2：纯文本链路最简单稳定，适合先验证 RAG 数据流。PDF/Word/OCR 解析可以后续扩展。

Q3：为什么不保存原始文件？

A3：当前目标是文本处理和检索，保存 chunk 足够。后续如需下载原文件可接对象存储。

Q4：document 和 chunk 为什么分表？

A4：document 是文件级元数据，chunk 是检索单元，访问模式不同。

Q5：固定长度切分有什么问题？

A5：可能切断标题、段落和句子，影响 embedding 语义质量。V2.1.1 会优化。

Q6：上传失败如何处理？

A6：document 状态应标记 FAILED，并保存 errorMessage，便于查询和排查。

Q7：为什么先做文档再做 embedding？

A7：embedding 的输入是 chunk，没有稳定 chunk 就无法验证向量检索。

Q8：当前局限是什么？

A8：只支持基础文本格式和固定切分，没有语义边界、PDF、OCR 和向量化。
