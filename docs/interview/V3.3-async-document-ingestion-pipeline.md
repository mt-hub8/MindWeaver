# V3.3 Async Document Ingestion Pipeline 面试文档

## 1. 版本摘要

V3.3 的版本名称是：

```text
V3.3 Async Document Ingestion Pipeline
```

中文可以表述为：

```text
异步文档摄入流水线
```

这个版本的核心目标是把 V3.2 的同步文档上传摄入，升级为更接近生产系统的异步任务流程。

在 V3.2 中，用户已经可以完成：

```text
上传 .txt / .md / 文本型 .pdf
  -> 文本提取
  -> 创建 document
  -> chunking
  -> embedding
  -> VectorStore 写入
  -> Ask 页面提问
```

但 V3.2 是同步流程。同步流程的问题是：

```text
上传请求需要等待整个 ingestion 过程完成；
embedding 和 vector write 慢时，用户会感觉页面卡住；
失败时不容易知道失败发生在哪一步；
不利于后续支持大文件、多文件、重试、进度展示和后台任务处理。
```

V3.3 解决的是：

```text
上传文档后快速返回 taskId；
后台异步执行 chunking / embedding / vector write；
持久化 ingestion task 状态；
Documents 页面用中文展示处理进度；
失败时展示原因；
FAILED 任务支持最小重试。
```

最终用户路径变成：

```text
上传文档
  -> 立即看到“文档已提交，正在排队处理”
  -> 后台异步处理
  -> 页面显示“待处理 / 处理中 / 已完成 / 失败”
  -> 已完成后提示“可以去知识库问答页面提问”
```

这个版本的重点不只是“异步跑通”，而是让普通用户能看懂文档现在处理到哪一步、失败了怎么办、什么时候可以开始提问。

---

## 2. 为什么现在做 V3.3

V3.2 已经打通了 Upload-to-Ask 产品闭环：

```text
用户上传文档；
系统建立知识库索引；
用户基于文档提问；
系统返回 answer + citations。
```

但 V3.2 仍然是同步 ingestion。

同步 ingestion 在 MVP 阶段可以快速打通闭环，但不适合继续产品化。

因为真实文档摄入通常包含多个耗时步骤：

```text
文件校验；
文本提取；
文档切分；
embedding 生成；
向量库写入；
状态记录；
失败处理。
```

其中 embedding 和 vector write 可能较慢，PDF 文本提取也可能失败。

如果所有步骤都塞在一次 HTTP 请求中，会导致：

```text
请求时间长；
用户无进度感；
错误定位困难；
失败后无法重试；
后续无法支持更大的文档和更复杂的处理链路。
```

所以 V3.3 把文档摄入升级成异步任务，是从 demo 走向生产系统的必要步骤。

---

## 3. 本版本解决的问题

### 3.1 解决上传请求阻塞的问题

V3.2 中，上传接口需要等待 ingestion 完成后才返回。

V3.3 改成异步语义：

```http
POST /documents/upload
```

上传成功后快速返回：

```json
{
  "taskId": 1001,
  "documentId": 123,
  "filename": "example.pdf",
  "status": "PENDING",
  "displayStatus": "待处理",
  "displayMessage": "文档已提交，正在排队处理。"
}
```

这样上传请求只负责：

```text
文件校验；
文本提取；
创建 document；
创建 ingestion task；
发送异步消息；
返回 taskId。
```

耗时的 chunking、embedding 和 vector write 放到后台处理。

---

### 3.2 解决用户不知道处理进度的问题

异步任务如果只有 taskId，对普通用户仍然不友好。

所以 V3.3 不只是新增后台任务，还要提供中文状态展示。

状态包括：

```text
待处理；
处理中；
已完成；
失败。
```

步骤包括：

```text
文档已提交；
已读取文档文本；
正在切分文档内容；
正在生成文档向量；
正在写入知识库索引；
处理完成，可以开始提问；
处理失败，请查看原因。
```

这样用户不需要理解 embedding、VectorStore、MQ 等底层概念，也能知道系统正在做什么。

---

### 3.3 解决失败不可追踪的问题

V3.2 的同步上传失败时，只能看到一次请求错误。

V3.3 引入 ingestion task 后，可以持久化：

```text
status；
step；
errorCode；
errorMessage；
retryCount；
updatedAt；
completedAt。
```

失败后用户可以看到：

```text
处理失败：该 PDF 没有可提取文字。当前版本暂不支持扫描版 PDF 或图片 OCR。
错误码：PDF_NO_EXTRACTABLE_TEXT
```

开发者也能通过 taskId、errorCode、traceId 和日志定位问题。

---

### 3.4 解决失败后无法重试的问题

V3.3 新增最小重试能力：

```http
POST /documents/ingestions/{taskId}/retry
```

规则是：

```text
只允许 FAILED 状态重试；
非 FAILED 状态不可重试；
retryCount + 1；
最多重试 3 次；
重新发送异步消息；
不做复杂死信队列；
不做自动指数退避。
```

前端按钮使用中文：

```text
重新处理
```

这让失败任务具备最小可恢复能力。

## 4. 核心设计

### 4.1 Ingestion Task 模型

V3.3 新增 ingestion task 持久化模型，建议表名：

```text
document_ingestion_task
```

核心字段包括：

```text
id；
document_id；
filename；
content_type；
status；
step；
chunk_count；
embedding_count；
vector_write_count；
error_code；
error_message；
retry_count；
created_at；
updated_at；
completed_at。
```

这个表的作用是：

```text
记录每次文档摄入任务；
追踪当前处理状态；
记录处理阶段；
记录处理结果；
记录失败原因；
支持重试；
支持前端查询状态。
```

---

### 4.2 状态枚举

状态至少包含：

```text
PENDING：待处理；
PROCESSING：处理中；
COMPLETED：已完成；
FAILED：失败。
```

这些状态用于描述任务整体生命周期。

状态流转通常是：

```text
PENDING -> PROCESSING -> COMPLETED
PENDING -> PROCESSING -> FAILED
FAILED -> PENDING -> PROCESSING -> COMPLETED
FAILED -> PENDING -> PROCESSING -> FAILED
```

其中 FAILED 可以通过 retry 重新回到 PENDING。

---

### 4.3 步骤枚举

步骤用于描述任务当前处理到哪一阶段。

建议步骤包括：

```text
UPLOADED：文档已提交；
TEXT_EXTRACTED：已读取文档文本；
CHUNKING：正在切分文档内容；
EMBEDDING：正在生成文档向量；
VECTOR_WRITING：正在写入知识库索引；
COMPLETED：处理完成，可以开始提问；
FAILED：处理失败，请查看原因。
```

状态描述任务结果，步骤描述任务过程。

两者配合起来，前端可以展示更友好的进度信息。

---

### 4.4 中文展示字段

V3.3 的一个重要设计是：API 不只返回英文枚举，还返回中文展示字段。

例如：

```json
{
  "status": "PROCESSING",
  "displayStatus": "处理中",
  "step": "EMBEDDING",
  "displayStep": "正在生成文档向量",
  "displayMessage": "系统正在为文档片段生成向量，请稍候。"
}
```

这样做的价值是：

```text
英文 status / step 保留给开发者和测试；
中文 displayStatus / displayStep / displayMessage 直接服务前端；
前端不用重复写复杂映射；
页面对普通用户更友好。
```

这也是 V3.3 明确强调“中文友好、低门槛交互”的体现。

---

### 4.5 上传 API 异步化

V3.3 保留或改造：

```http
POST /documents/upload
```

但语义从同步完成变成异步提交。

上传请求阶段负责：

```text
文件校验；
文本提取；
创建 document；
保存 source text 或等价文本内容；
创建 ingestion task，状态为 PENDING；
发送 MQ message；
快速返回 taskId。
```

推荐返回 HTTP 状态：

```text
202 Accepted
```

它表示请求已接受，但后台处理尚未完成。

---

### 4.6 为什么上传阶段仍然提取文本

V3.3 推荐策略是：

```text
上传阶段完成文件校验和文本提取；
后台阶段完成 chunking、embedding、vector write。
```

这样做的原因是：

```text
避免引入复杂文件存储系统；
避免处理 Windows / Linux 文件路径兼容；
PDF 解析失败可以立即返回；
当前文件大小已有上限；
测试更容易控制。
```

如果上传阶段不提取文本，而是保存原始文件给后台处理，就需要额外解决：

```text
文件存储路径；
文件清理；
文件权限；
临时文件生命周期；
跨平台路径；
后台读取失败；
安全问题。
```

V3.3 的目标是异步 ingestion task，不是复杂文件存储系统。

---

### 4.7 后台异步处理

后台 consumer / handler 的处理流程是：

```text
加载 task；
校验状态；
status = PROCESSING；
step = CHUNKING；
执行 chunking；
更新 chunkCount；
step = EMBEDDING；
生成 embeddings；
更新 embeddingCount；
step = VECTOR_WRITING；
写入 VectorStore；
更新 vectorWriteCount；
status = COMPLETED；
step = COMPLETED；
写入 completedAt。
```

失败时：

```text
status = FAILED；
step = FAILED；
写入 errorCode；
写入 errorMessage；
记录日志；
前端可以查询失败原因。
```

---

### 4.8 MQ / 异步机制

V3.3 优先复用项目已有 RabbitMQ / MQ 基础设施。

新增 message 可以是：

```text
DocumentIngestionMessage
```

至少包含：

```text
taskId；
documentId。
```

默认测试不能依赖真实 RabbitMQ。

测试中应该使用：

```text
mock publisher；
直接调用 consumer handler；
mock embedding provider；
mock vector store。
```

这体现了生产逻辑和测试依赖的隔离。

## 6. 前端中文友好设计

V3.3 明确要求：所有前端交互页面必须中文优先、使用者门槛友好。

这不是小细节，而是产品化质量的一部分。

### 6.1 页面标题

推荐使用：

```text
文档管理；
上传文档；
知识库问答；
检索质量评估；
系统能力说明。
```

不要只写：

```text
Documents；
Upload；
Ask；
Evaluation。
```

必要英文术语可以放在括号里：

```text
文档片段（Chunk）；
检索引用（Citation）；
向量索引（Vector Index）。
```

---

### 6.2 按钮文案

使用中文动作：

```text
选择文件；
上传文档；
刷新状态；
查看文档片段；
重新处理；
去提问；
查看引用来源。
```

不要只用英文：

```text
Upload；
Refresh；
Retry；
Ask。
```

---

### 6.3 空状态

空状态不能只写：

```text
No documents.
```

应该写成：

```text
还没有上传任何文档。
请上传 .txt、.md 或文本型 PDF，系统会自动建立知识库索引。
处理完成后，你可以前往「知识库问答」页面提问。
```

这样普通用户知道下一步该做什么。

---

### 6.4 失败提示

失败提示要分两层：

用户层：

```text
处理失败：该 PDF 没有可提取文字。当前版本暂不支持扫描版 PDF 或图片 OCR。
```

技术层：

```text
错误码：PDF_NO_EXTRACTABLE_TEXT
任务 ID：1001
追踪 ID：xxx
```

这样既能降低用户门槛，又保留工程排查能力。

---

### 6.5 成功引导

任务完成后应该提示：

```text
处理完成，现在可以前往「知识库问答」页面提问。
```

并提供按钮：

```text
去提问
```

这让用户路径闭环更加清晰。

## 7. 关键取舍

### 7.1 为什么不做 WebSocket / SSE

WebSocket 或 Server-Sent Events 可以实现实时进度推送。

但 V3.3 只需要让用户知道任务状态，不需要毫秒级实时刷新。

所以本版本选择：

```text
页面轮询或手动刷新；
轮询间隔 2-5 秒；
不引入实时连接复杂度。
```

这符合当前边界。

---

### 7.2 为什么不做复杂进度百分比

复杂百分比需要更细粒度的阶段计数。

例如：

```text
总 chunk 数；
当前 embedding 到第几个 chunk；
向量写入进度；
失败重试进度。
```

如果处理过程并不线性，百分比反而容易误导用户。

V3.3 先展示阶段状态：

```text
正在切分；
正在生成向量；
正在写入知识库索引。
```

这已经足够友好。

---

### 7.3 为什么只做最小重试

完整重试系统可能包括：

```text
自动重试；
指数退避；
死信队列；
最大失败次数；
失败告警；
人工恢复；
幂等检查。
```

V3.3 只做 FAILED 手动重试，是为了控制复杂度。

这已经可以解决用户层面的基本恢复问题：

```text
失败后可以重新处理。
```

---

### 7.4 为什么不做文档删除

异步 ingestion 后，文档删除会更复杂，因为需要处理：

```text
document；
chunks；
embedding cache；
VectorStore vectors；
ingestion task；
处理中任务取消；
删除和后台任务并发；
失败回滚。
```

V3.3 不做删除，是合理边界。

---

### 7.5 为什么不做 workspace / collection

workspace / collection 是多租户或多知识库能力。

它会影响：

```text
上传；
检索；
VectorStore metadata filter；
Ask 页面选择范围；
权限；
evaluation dataset；
文档管理。
```

V3.3 只解决单知识库下的异步 ingestion。

## 8. 测试设计

### 8.1 默认测试原则

V3.3 仍然要求：

```powershell
.\mvnw.cmd test
```

默认测试不能依赖：

```text
Qdrant；
Docker；
Python worker；
真实模型下载；
外部 API；
外部网络；
OpenAI API key；
真实 RabbitMQ。
```

即使生产代码使用 RabbitMQ，测试也必须通过 mock publisher 或直接调用 handler 完成。

---

### 8.2 测试覆盖点

必须覆盖：

```text
上传后创建 ingestion task；
初始状态 PENDING；
message publisher 被调用；
task detail 查询；
最近 task 列表查询；
consumer / handler 成功处理；
PENDING -> PROCESSING -> COMPLETED；
step 正确更新；
chunkCount / embeddingCount / vectorWriteCount 更新；
失败时 status = FAILED；
errorCode / errorMessage 写入；
FAILED 状态可以 retry；
非 FAILED 状态不可 retry；
retryCount 增加；
最大重试次数限制；
retry 后 message publisher 被调用；
Documents 页面中文文案资源存在；
Ask 页面中文流程说明存在；
不破坏 V3.2 文件类型支持。
```

---

### 8.3 为什么不测试真实 RabbitMQ

真实 RabbitMQ 属于集成环境依赖。

如果默认测试依赖 RabbitMQ，会导致：

```text
本地环境不稳定；
CI 不稳定；
开发者需要额外启动 Docker；
测试速度变慢；
失败原因混杂。
```

所以默认测试只验证业务逻辑和消息发布调用。

真实 RabbitMQ 走 manual verification。

## 9. 当前局限

V3.3 有明确局限：

```text
不做多文件批量上传；
不做大文件分片上传；
不做 OCR；
不做图片识别；
不做 Word / Excel；
不做网页 URL；
不做复杂文件存储系统；
不做复杂进度百分比；
不做 WebSocket；
不做 Server-Sent Events；
不做用户登录；
不做权限；
不做 workspace / collection；
不做文档删除；
不做文档编辑；
不做向量删除一致性；
不做复杂死信队列；
不做自动指数退避重试；
不做新的检索算法；
不改 RAG Answer API；
不引入前端框架。
```

这些边界保证 V3.3 聚焦异步 ingestion pipeline。

## 10. 产品化意义

V3.3 的产品化意义非常明确：

```text
用户不再面对一个“上传后卡住”的系统，而是面对一个能展示处理状态的产品。
```

它让 Documents 页面从简单上传页面，变成了文档处理控制台的雏形。

用户可以看到：

```text
文档是否已提交；
是否正在处理；
是否处理完成；
失败原因是什么；
是否可以重新处理；
什么时候可以去 Ask 页面提问。
```

这让系统更接近真实企业应用。

## 13. 面试高强度追问与回答

### 13.1 V3.3 为什么要把同步 ingestion 改成异步？

回答：

```text
同步 ingestion 在 MVP 阶段可以快速打通上传到问答的闭环，但它不适合生产化。因为文本提取、chunking、embedding 和 VectorStore 写入都可能耗时，用户会等待一个很长的 HTTP 请求，也不知道现在处理到哪一步。

V3.3 改成异步后，上传接口快速返回 taskId，后台处理摄入流程，前端通过任务状态查询展示“待处理、处理中、已完成、失败”。这更符合企业系统的任务处理方式。
```

---

### 13.2 为什么上传阶段还要做文本提取，不把整个文件交给后台？

回答：

```text
因为当前版本不想引入复杂文件存储系统。如果上传阶段只保存原始文件，后台再解析，就要处理文件路径、临时文件清理、跨平台路径兼容和后台读取失败等问题。

V3.3 选择上传阶段完成文件校验和文本提取，然后保存 source text，后台负责 chunking、embedding 和 VectorStore 写入。这样既能让 PDF 解析错误尽早返回，也避免引入复杂文件存储。
```

---

### 13.3 为什么不做 WebSocket 实时进度？

回答：

```text
当前只需要让用户知道任务状态，不需要实时推送。轮询 2-5 秒已经足够满足“查看处理状态”的需求。

WebSocket 或 SSE 会引入连接管理、断线重连、前后端状态同步和测试复杂度。V3.3 先用轮询保持边界克制。
```

---

### 13.4 为什么不做进度百分比？

回答：

```text
文档摄入不是完全线性的流程。chunking、embedding、vector write 的耗时比例不固定。如果硬做百分比，可能误导用户。

所以我先展示阶段状态，比如“正在切分文档内容”“正在生成文档向量”“正在写入知识库索引”。这比不准确的百分比更可靠，也足够低门槛。
```

---

### 13.5 失败重试如何设计？

回答：

```text
V3.3 只做最小手动重试。只有 FAILED 状态的任务可以重试，重试时 retryCount 加一，状态重新变为 PENDING，并重新发送 MQ message。最大重试次数限制为 3 次。

暂时不做自动指数退避、死信队列或复杂失败恢复，因为这些属于更完整的生产可靠性版本。
```

---

### 13.6 如何保证默认测试不依赖 RabbitMQ？

回答：

```text
我把消息发布抽象出来，默认测试中使用 mock publisher 验证消息是否被发布。consumer 的处理逻辑通过 service 或 handler 直接调用，配合 mock embedding provider 和 mock vector store 测试状态流转。

这样可以验证业务逻辑，不需要启动真实 RabbitMQ、Qdrant 或 worker。
```

---

### 13.7 这个版本和 Task Orchestrator 有什么关系？

回答：

```text
V3.3 开始把项目重新拉回 Orchestrator 主线。文档上传不再只是一个同步接口，而是一个异步任务：提交任务、排队、后台处理、状态追踪、失败记录、重试。

这和 AI Task Orchestrator 的核心能力一致：把耗时 AI 工作流封装成可追踪、可恢复的任务。
```

---

### 13.8 为什么前端要中文友好？

回答：

```text
因为这个项目现在已经不是纯后端 demo，而是 AI Knowledge Assistant MVP。用户需要知道如何上传文档、当前处理到哪一步、失败后怎么办、什么时候可以提问。

如果页面只显示英文枚举和技术字段，比如 PENDING、EMBEDDING、VECTOR_WRITING，普通用户理解成本很高。所以 V3.3 在 API 和页面中都加入中文 displayStatus、displayStep 和 displayMessage。
```

---

### 13.9 如果 embedding 过程中失败，如何处理？

回答：

```text
consumer 捕获异常后，任务状态更新为 FAILED，step 更新为 FAILED，同时写入 errorCode 和 errorMessage。前端查询任务时可以看到中文失败原因和技术错误码。

不能静默吞异常，也不能让任务一直停在 PROCESSING。
```

---

### 13.10 为什么不做文档删除？

回答：

```text
文档删除在异步系统里会引入一致性问题。需要删除 document、chunks、VectorStore vectors，还要处理正在处理中的任务、失败任务和重试任务。

这已经是单独的数据生命周期管理问题。V3.3 聚焦异步摄入，不把删除混进来。
```
