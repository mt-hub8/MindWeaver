# V3.1 Productization Reset / AI Knowledge Assistant MVP 面试文档

## 1. 版本摘要

V3.1 的版本名称是：

```text
V3.1 Productization Reset / AI Knowledge Assistant MVP
```

中文可以表述为：

```text
产品化重置：AI 知识库问答 MVP
```

这个版本的核心目标不是继续新增 RAG 检索算法，而是把前面已经完成的大量底层工程能力重新组织成一个用户能理解、面试官能看懂、开发者能体验的产品形态。

在 V3.1 之前，项目已经完成了很多底层能力：

```text
RAG Answer API with Citations
Embedding Cache / Dedup
Embedding Cache Metrics
Qdrant Manual Verification
Qdrant Benchmark
Retrieval Evaluation
Two-stage Retrieval & Rerank
Hybrid Retrieval Fusion
Minimal Web UI
Manual Demo Golden Path
Unified API Error Response
```

这些能力从工程角度看很完整，但如果普通用户打开项目，可能仍然不知道：

```text
这个系统到底能做什么？
我应该从哪里开始？
它适合什么场景？
这些 RAG、rerank、hybrid、evaluation 对我有什么用？
为什么不用现成框架？
```

所以 V3.1 的任务是做一次 productization reset（产品化重置），把项目包装成：

```text
一个企业知识库 AI 问答与 RAG 质量调试系统。
```

用户打开首页后，应该能在 5 分钟内理解：

```text
这个系统是什么；
可以用来做什么；
如何提问；
如何查看文档数据；
如何理解检索质量；
已有底层能力分别解决什么问题。
```

---

## 2. 为什么现在做 V3.1

V3.1 之前，项目主线偏工程能力建设。

例如：

```text
V2.7：RAG Answer API with Citations
V2.8：RAG Retrieval Evaluation & Quality Benchmark
V2.9：RAG Two-stage Retrieval & Rerank
V3.0：RAG Hybrid Retrieval Fusion
```

这些版本证明系统具备很强的 RAG 工程深度。

但继续做底层功能会带来一个问题：

```text
技术能力越来越多，但用户价值越来越不直观。
```

如果面试官或 reviewer 只看到一堆技术名词：

```text
VectorStore
EmbeddingProvider
Reranker
Hybrid Retrieval
RRF
MRR
Recall@K
Qdrant Benchmark
```

他们可能会问：

```text
这些功能组合起来，到底是一个什么产品？
用户为什么要用？
为什么不用 LangChain / LlamaIndex / Spring AI / Dify？
```

V3.1 解决的就是这个问题。

它把项目从：

```text
一个底层 RAG 工程能力集合
```

重新包装成：

```text
一个可体验的 AI Knowledge Assistant MVP。
```

也就是说，V3.1 的重点不是“再加一个算法”，而是“让已有能力形成产品闭环”。

## 3. 本版本解决的问题

### 3.1 解决项目定位不清的问题

在 V3.1 之前，项目容易被误解成：

```text
自己实现了一套 RAG 框架。
```

这个定位不准确，也容易被追问：

```text
为什么不用现成框架？
```

V3.1 之后，项目定位被调整为：

```text
AI Task Orchestrator 是一个面向企业 AI 应用的知识库问答与 RAG 质量调试系统。用户可以摄入私有文档，在浏览器中提问，系统基于文档生成带引用的回答，并展示检索策略、引用来源和质量评估结果。
```

这个定位更清楚地说明：

```text
它不是替代 LangChain / LlamaIndex；
它是一个 AI-native backend / RAG quality debugging system；
它强调后端工程、质量评估、可解释性和产品化集成。
```

---

### 3.2 解决用户入口不清的问题

原来用户可能只能看到 `/rag-demo.html`。

这个页面能问答，但不像完整产品入口。

V3.1 新增或强化了以下入口：

```text
/ 或 /index.html：产品首页；
/ask.html：知识库问答页面；
/documents.html：文档只读入口；
/evaluation.html：检索质量评估入口；
```

这样用户不再需要先理解后端接口，而是可以从产品首页开始，按导航理解系统能力。

---

### 3.3 解决技术能力和用户价值脱节的问题

V3.1 把技术能力翻译成用户价值。

例如：

```text
Citations：帮助用户追踪答案来源，降低幻觉风险；
Embedding Cache：避免重复生成 embedding，节省时间和成本；
Retrieval Evaluation：用指标判断检索质量，而不是凭感觉；
Rerank：把更相关的 chunk 排到前面；
Hybrid Retrieval：同时兼顾语义相似和关键词精确匹配；
Unified Error Response：让错误可读、可追踪、可排查。
```

这一步对面试非常重要。

因为面试官不只关心你实现了什么类，也关心你是否知道这些能力为什么有用。

### 4.3 Ask 页面产品化

V3.1 把原来的 RAG demo 页面升级为更像产品的 Ask 页面。

Ask 页面应该包含：

```text
query 输入；
topK 输入；

generation metadata 展示；
```

它的产品语义是：

```text

```

这比“测试 RAG API 的 demo 页面”更容易理解。

---

### 4.4 Documents 只读入口

V3.1 新增 Documents 入口：

```text
/documents.html
```

它的目标不是完整文档管理系统，而是让用户知道：

```text
系统里有哪些文档；
每个文档有多少 chunks；
是否已有可检索内容；
文档数据是否为空。
```

如果已有 document / chunk 查询能力，就复用。

如果没有，则新增最小只读 API：

```http
GET /documents
GET /documents/{documentId}/chunks
```

这类 API 只读，不做上传、删除、编辑、权限或复杂分页。

这个入口非常关键，因为 RAG 系统的回答质量和文档数据直接相关。没有文档入口，用户不知道系统到底在检索什么。

---

### 4.5 Evaluation 入口

V3.1 新增 Evaluation 页面：

```text
/evaluation.html
```

它不是复杂 dashboard，而是一个解释和入口页。

页面需要说明：

```text
系统可以评估 RAG 检索质量；
支持哪些指标；
dense / rerank / hybrid 对比有什么意义；
报告输出在哪里；
如何手工运行 evaluation。
```

需要解释的指标包括：

```text
HitRate@K（命中率@K）
Recall@K（召回率@K）
Precision@K（精确率@K）
MRR / RR@K（平均倒数排名 / 倒数排名@K）
```

这能让用户理解：

```text
这个项目不只是能回答，还能判断检索质量好不好。
```

---

### 4.6 System Capabilities 展示

V3.1 需要把系统能力用用户语言讲出来。

例如：

```text
RAG Answer API with Citations：基于文档回答问题，并展示来源；
Embedding Cache：避免重复生成 embedding，节省成本；

Unified API Error Response：让错误可追踪、可排查。
```

这一步把技术能力转成了产品卖点。

## 5. 关键取舍

### 5.1 为什么不继续做新的 RAG 算法

V3.1 之前已经连续做了：

```text
evaluation；
rerank；
hybrid retrieval。
```

继续做 sparse vector、CrossEncoder、strategy profile 虽然有价值，但会让项目越来越偏底层，用户越来越难理解。

所以 V3.1 暂停底层能力堆叠，转向产品化组织。

这体现的是一个工程判断：

```text
当底层能力已经足够支撑 demo 时，下一步应该让用户理解和使用，而不是继续无限增加内部模块。
```

---

### 5.2 为什么不引入 React / Vue

V3.1 仍然不引入 React、Vue、npm 或前端构建系统。

原因是：

```text
当前目标是产品化包装，不是建设复杂前端工程；
原生 HTML / JS / CSS 足够；
引入前端框架会增加 package、构建、部署和测试复杂度；
会偏离后端项目主线。
```

这个选择保持了项目边界。

---

### 5.3 为什么 Documents 只做只读

V3.1 的 Documents 页面只做只读展示，不做上传、删除、编辑。

原因是：

```text
上传属于 ingestion 产品入口；
删除和编辑涉及数据一致性；
文档管理涉及权限、状态、重试、清理；
这些都属于更大的版本。
```

当前只需要让用户知道系统里有什么文档和 chunk。

这已经足够解决“系统在检索什么”的可见性问题。

---

### 5.4 为什么 Evaluation 不做 dashboard

Evaluation 页面不做复杂图表或在线运行按钮。

原因是：

```text
现有 evaluation 已经输出 JSON / Markdown report；
页面只需要解释指标和报告入口；
dashboard 会引入后端报告查询、前端图表、历史记录管理等复杂度。
```

所以 V3.1 先做轻量入口，不做完整 dashboard。

---

### 5.5 为什么不做登录和权限

V3.1 是本地 / 项目展示级 MVP，不是 SaaS 产品。

登录和权限会引入：

```text
用户模型；
认证；
授权；
session/token；
权限校验；
前端状态；
测试复杂度。
```

当前版本重点是产品叙事和入口组织，不做用户系统。

## 10. 面试高强度追问与回答

### 10.1 为什么突然做产品化重置？

回答：

```text
因为前面几个版本已经完成了 RAG 的核心工程能力，包括 answer API、citations、evaluation、rerank 和 hybrid retrieval。但我发现如果只继续堆底层模块，用户和面试官很难直观看懂这个项目能做什么。

所以 V3.1 我暂停新增底层算法，把已有能力组织成 AI Knowledge Assistant MVP，让用户打开首页就能理解系统定位，并通过 Ask、Documents、Evaluation 三个入口体验问答、查看文档数据和理解检索质量。
```

---

### 10.2 这个项目到底能做什么？

回答：

```text
它可以作为一个企业知识库 AI 问答与 RAG 质量调试系统。用户可以摄入私有文档，在浏览器中提问，系统基于文档生成带引用的回答。同时系统会展示检索来源、retrieval metadata、generation metadata，并提供 retrieval evaluation、rerank 和 hybrid retrieval 等能力来分析和优化检索质量。
```

---

### 10.3 为什么不用 LangChain / LlamaIndex / Spring AI？

回答：

```text
生产中我会优先考虑成熟框架，比如 LangChain、LlamaIndex、Spring AI 或 Qdrant 的现成功能。这个项目的目的不是替代它们，而是通过自己实现一个可控的 RAG / AI backend，理解这些框架背后的核心工程问题。

例如文档 chunk、embedding cache、VectorStore、citations、retrieval evaluation、rerank、hybrid search、错误响应和报告输出。即使用现成框架，工程师也需要理解这些抽象背后的边界、配置、评估和排查方式。
```

---

### 10.4 V3.1 和前面 RAG 版本有什么关系？

回答：

```text
前面的版本主要是在建设底层能力：V2.7 做 RAG Answer API，V2.8 做 retrieval evaluation，V2.9 做 rerank，V3.0 做 hybrid retrieval。

V3.1 则是把这些能力包装成产品入口。它没有替代前面的功能，而是把它们组织到首页、Ask 页面、Documents 页面和 Evaluation 页面中，让用户和面试官能看懂这些能力如何组成一个完整系统。
```

回答：

```text
当前 V3.1 更偏 AI Knowledge Assistant MVP，它把 RAG 能力产品化。Agent Orchestrator 可以建立在这个基础上，因为 Agent 执行任务时也需要可靠的知识检索、引用来源、检索质量评估和任务状态追踪。

也就是说，RAG 是 Agent 的知识层，后续可以把任务编排、工具调用、多步骤执行和成本控制接到这个基础上。
```

## 11. 后续演进路线

V3.1 之后，不建议马上继续做底层检索算法。更合理的路线有几个方向。

### 11.1 Ingestion UI / Document Upload

补文档上传入口：

```text
上传文档；
解析；
切分；
embedding；
写入 VectorStore；
查看状态；
失败重试。
```

这会让产品形成完整闭环：

```text
上传文档 -> 提问 -> 查看引用 -> 评估质量。
```

---

### 11.2 Evaluation Report Viewer

把 evaluation report 做成简单可浏览页面：

```text
查看最近报告；
查看 summary metrics；
查看 failed cases；
查看 dense / rerank / hybrid 对比。
```

不一定要复杂图表，但可以增强产品感。

---

### 11.3 Answer Quality Evaluation

在 retrieval evaluation 之后，可以做 answer quality evaluation：

```text
faithfulness（忠实度）；
answer relevance（答案相关性）；
citation correctness（引用正确性）；
hallucination rate（幻觉率）。
```

---

### 11.4 Agent Task Orchestration

回到项目名称里的 Orchestrator：

```text
用户提交任务；
系统拆解步骤；
检索相关知识；
调用工具；
执行任务；
追踪状态；
失败重试；
输出结果。
```

RAG 能力可以作为 Agent 的知识基础。

---

### 11.5 Role-based / Workspace-based Productization

如果继续产品化，可以做：

```text
workspace；
project；
document collection；
user role；
query history；
audit log。
```

但这些属于更后期能力。

## 12. 本版本在项目中的位置

当前版本链可以这样理解：

```text
V2.7：RAG Answer API with Citations
V2.8：RAG Retrieval Evaluation & Quality Benchmark
V2.9：RAG Two-stage Retrieval & Rerank
V3.0：RAG Hybrid Retrieval Fusion
V3.1：Productization Reset / AI Knowledge Assistant MVP
```

其中：

```text
V2.7-V3.0 解决底层 RAG 工程能力；
V3.1 解决产品入口和用户理解问题。
```

V3.1 是一个转折点。

它不是继续加深底层算法，而是让项目具备更完整的产品叙事。

## 13. 总结表达

这个版本可以用一段话总结：

```text
在 V3.1 中，我对 AI Task Orchestrator 做了一次产品化重置。前面几个版本已经完成了 RAG Answer API、citations、embedding cache、retrieval evaluation、rerank 和 hybrid retrieval 等底层能力，但这些能力如果缺少产品入口，用户和面试官很难直观看懂项目价值。

所以我在 V3.1 重写 README 第一屏，新增产品首页，并整理 Ask、Documents、Evaluation 等入口，把项目重新包装成 AI Knowledge Assistant MVP。用户可以打开首页理解系统定位，通过 Ask 页面进行知识库问答，通过 Documents 页面查看已摄入数据，通过 Evaluation 页面理解检索质量评估结果。

这个版本体现的是从底层工程实现到产品化交付的能力：不仅让系统能跑，还让用户知道它能做什么、怎么用、为什么有价值。
```
