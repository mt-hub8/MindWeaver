# 1. STAR 总览

## S：Situation，背景

项目在 V14-V17 后已经具备了比较完整的 RAG 工程底座：

```text
V14.0 Knowledge Base Health Report
  负责发现知识库检索质量问题，例如 Recall 低、ContextPrecision 低、跨集合污染、错误版本污染。

V15.0 Structured Chunking & Hybrid Retrieval Pipeline
  负责优化检索主链路，例如结构化 chunk、metadata pre-filter、Hybrid Search、RRF、rerank、parent context。

V16.0 Vector Index Isolation & Deduplication Hardening
  负责保证向量库底层不污染，例如 vector identity、幂等 upsert、generation、cleanup、audit。

V17.0 Query Understanding & Retrieval Routing
  负责让系统理解用户 query，并自动选择 collection / version / strategy / filter。
```

到这个阶段，系统已经更容易检索到正确上下文，但仍然存在 RAG 最后一公里问题：

```text
1. 检索结果正确，但 LLM 没有忠实使用上下文；
2. LLM 把上下文中没有的信息补出来；
3. 答案看起来有引用，但引用并不支撑对应结论；
4. 答案中的多个关键结论没有逐条证据；
5. final context 太长，模型被噪声带偏；
6. parent / adjacent chunk 回填后，直接命中 chunk 和扩展 chunk 没有区分；
7. citation 只引用 document，不引用 chunk / section，粒度太粗；
8. 上下文不足时，模型仍然硬答；
9. 用户不知道答案哪些内容可靠，哪些只是推断；
10. V14 虽然能评估 CitationAccuracy / Faithfulness，但 Ask 生成链路本身没有强约束。
```

因此，V18.0 需要把回答生成从：

```text
检索 chunks -> 拼 prompt -> LLM 自由回答
```

升级为：

```text
检索结果
  ↓
标准化 final context
  ↓
Grounded Answer Contract
  ↓
强引用 Prompt
  ↓
生成后引用校验
  ↓
未支持主张检测
  ↓
上下文不足拒答
  ↓
Answer Grounding Score
  ↓
Ask 页面展示可信状态
```

## T：Task，任务

V18.0 的任务是建立一个可信回答生成链路：

```text
1. 设计 Grounded Answer Contract，明确回答必须基于 final context；
2. 标准化 GroundedContextBundle，让每个 chunk 都有 citationKey；
3. 设计 GroundedAnswerPromptBuilder，强制关键结论引用；
4. 标准化 Citation Model，引用必须指向具体 chunk / section；
5. 实现 CitationVerificationService，校验引用是否存在、是否在 final context 中、是否符合 collection / version / status filter；
6. 实现 UnsupportedClaimDetector，识别没有资料支持的关键主张；
7. 实现 RefusalPolicyService，在上下文不足时拒答；
8. 实现 AnswerGroundingScore，对单次答案可信程度打分；
9. Ask 页面展示可信回答状态、引用校验、未支持主张、拒答原因；
10. 与 V14 Knowledge Health 联动，把更严格的 CitationAccuracy / Faithfulness / RefusalAccuracy 进入评测。
```

V18.0 设计并实现以下模块：

完成后，系统可以做到：

```text
1. 每个 final context chunk 都有稳定 citationKey；
2. 答案关键结论必须引用具体 chunk / section；
3. citation 必须来自 final context，不能引用不存在或被 filter 排除的 chunk；
4. 版本、配置项、API path、代码符号等高风险实体必须由引用 chunk 支撑；
5. 上下文不足时优先拒答，而不是硬生成；

7. 系统能标出未被资料支持的主张；
8. V14 的 Faithfulness、CitationAccuracy、RefusalAccuracy 可以使用更严格的校验结果；
9. 用户可以判断答案的哪些部分有证据，哪些部分需要复核。
```

# 2. 遇到的问题：为什么检索正确不等于回答可信

## 2.1 RAG 的三层风险

V18.0 关注的是 RAG 的回答层风险。RAG 系统不是只要检索到 chunk 就一定可信，它至少有三层风险：

```text
第一层：检索风险
正确资料没有召回，或召回了错误资料。

第二层：上下文组装风险
检索结果中有直接命中 chunk，也有 parent / adjacent 扩展 chunk。如果不区分来源，模型可能把扩展内容当成主证据。

第三层：生成风险
LLM 可能从上下文中抽取错误、过度推断、遗漏前提、编造结论，或者引用不支撑答案。
```

V14-V17 主要降低第一层和第二层风险，V18 专门处理第三层风险。

## 2.2 “有引用”不等于“引用有效”

很多 RAG Demo 会在答案后面放几个引用，例如：

```text
根据资料，系统支持多模型供应商配置。[1]
```

但真实问题是：

```text
[1] 是否真的在 final context 中？
[1] 是否来自正确 collection？
[1] 是否来自正确 version？
[1] 是否包含“多模型供应商配置”这件事？
[1] 是否只是语义相关，但不能支撑结论？
答案中其他句子是否没有引用？
```

所以 V18 的重点不是“展示引用”，而是“校验引用”。

## 2.3 上下文不足时硬答是更严重的问题

在知识库问答场景里，最危险的不是回答“不知道”，而是上下文不足时仍然给出确定性结论。

例如：

```text
用户问：V18 的最终评分公式是什么？

如果知识库里还没有 V18 文档，正确行为应该是：
根据当前资料无法确认。

错误行为是：
模型根据 V14 / V15 的评分公式编一个 V18 公式。
```

因此 V18 必须把拒答视为正常能力，而不是失败。

# 3. 参考了什么技术思路

## 3.1 Ragas：Faithfulness / Context Precision / Answer Relevancy

Ragas 将 RAG 评估拆分为多个组件指标，例如 Faithfulness（忠实性）、Context Precision（上下文精确率）、Context Recall（上下文召回）和 Answer Relevancy（回答相关性）。其中 Faithfulness 衡量回答是否与检索上下文事实一致。

对 V18 的启发：

```text
RAG 最终答案不能只看语义流畅度；
必须检查答案是否被 context 支持；
Faithfulness 和 CitationAccuracy 应该和检索指标分开计算。
```

V18 的实现选择：

```text
默认不调用 LLM Judge；
先用 citation verification 和 unsupported claim heuristic 做本地可测试版本；
后续可以接 LLM Judge 作为增强。
```

## 3.2 ARES：RAG 评估拆成 Context Relevance / Answer Faithfulness / Answer Relevance

ARES 论文将 RAG 评估拆成：

```text
context relevance
answer faithfulness
answer relevance
```

对 V18 的启发：

```text
检索上下文是否相关、答案是否忠实、答案是否回应问题，是三个不同维度。
```

V18 对应关系：

```text
Context relevance -> V14 / V15 的检索指标和 ContextPrecision
Answer faithfulness -> V18 的 UnsupportedClaimDetector / GroundingScore
Answer relevance -> V14 的 AnswerRelevance / V18 的 RefusalPolicy 和回答契约
```

## 3.3 Azure AI Search / Azure RAG Evaluators 的 Groundedness 思路

Azure 的 RAG 相关文档强调 grounding documents（接地资料）对回答一致性的作用，Azure RAG evaluators 也把 groundedness / retrieval / relevance 作为评估维度。

对 V18 的启发：

```text
答案必须和 grounding documents 保持一致；
如果输出偏离 source material，就是 groundedness 风险；
检索和生成需要分别观察。
```

V18 的产品化落地：

```text
GroundedContextBundle 作为 source material；
CitationVerificationService 校验答案是否引用 final context；
UnsupportedClaimDetector 检查答案是否偏离 source material。
```

## 3.4 LangChain / LlamaIndex RAG 引用模式

LangChain 和 LlamaIndex 的 RAG 示例中，通常会将检索到的 documents 作为 context 传入，并在回答中保留 source / metadata。

对 V18 的启发：

```text
RAG 应保留 source metadata；
回答引用不能只到 document 级别，最好能到 chunk / section；
context chunk 和 answer citation 之间需要稳定映射。
```

V18 的增强：

```text
不是只展示 source，而是建立 Citation Model；
每个 citationKey 映射到具体 chunkId；
生成后校验 citation 是否真的有效。
```

## 3.5 事实核验 / Claim Verification 思路

事实核验系统通常会将回答拆成 claims，再检查 claim 是否被 evidence 支持。

V18 借鉴这个思路，但做了工程取舍：

```text
不做复杂自然语言推理；
先识别高风险关键句；
对版本号、代码符号、配置项、API path、数字等实体做启发式校验；
无法判断时标记 UNKNOWN，而不是假装 VERIFIED。
```

## 3.6 Guardrail / Safe Completion 思路

生产系统常见的安全策略是：当置信度不足、证据不足、上下文为空时，不直接输出确定性答案。

V18 的对应设计是：

```text
RefusalPolicyService
```

它将拒答条件显式化：

```text
NO_CONTEXT
LOW_RETRIEVAL_CONFIDENCE
NO_VALID_CITATION
QUERY_AMBIGUOUS
VERSION_NOT_FOUND
COLLECTION_NOT_SELECTED
STRICT_MODE_NO_EVIDENCE
```

# 4. 核心优化一：Grounded Answer Contract

## 4.1 问题

原来的 RAG prompt 可能只是：

```text
请根据以下资料回答用户问题。
```

这种 prompt 太软，LLM 仍然可能：

```text
1. 使用上下文之外的常识；
2. 把推断说成事实；
3. 不引用关键结论；
4. 上下文不足时硬答；
5. 引用不存在的文档或章节。
```

## 4.2 优化动作

新增：

```text
GroundedAnswerContract
```

契约规则：

```text
1. 只能基于 final context 回答；
2. 不得使用上下文之外的事实；
3. 每个关键结论必须引用至少一个 chunk；
4. 如果 final context 不足以回答，必须拒答；
5. 如果只能部分回答，必须说明“只能根据当前资料确认以下部分”；
6. 不允许引用未进入 final context 的 chunk；
7. 不允许编造文档名、章节名、文件路径；
8. 对版本、接口、配置项、数字、结论类信息必须引用；
9. 对推断性内容必须标注“根据当前资料推断”；
10. 如果资料冲突，必须指出冲突，不能强行合并。
```

## 4.3 Contract Mode

新增：

```text
AnswerContractMode
```

包括：

```text
STRICT       严格模式
BALANCED     平衡模式
EXPLORATORY  探索模式
```

### STRICT 严格模式

适合：

```text
接口文档；
合同；
配置项；
版本文档；
财务 / 合规资料。
```

特点：

```text
所有关键结论必须有 citation；
上下文不足时拒答；
不允许无证据推断。
```

### BALANCED 平衡模式

默认模式。

特点：

```text
事实结论需要 citation；
允许少量摘要性表达；
不能脱离上下文。
```

### EXPLORATORY 探索模式

适合：

```text
学习资料；
头脑风暴；
宽泛总结；
研究辅助。
```

特点：

```text
允许基于上下文做轻度推断；
但必须标注“根据当前资料推断”。
```

## 4.4 工程权衡

### 为什么不只有一个模式

因为不同场景错误成本不同：

```text
接口文档和配置项错误成本高，必须严格；
学习总结允许一定推断；
普通问答需要在可用性和可靠性之间平衡。
```

所以 V18 采用三种 contract mode，而不是一个固定规则。

# 5. 核心优化二：Grounded Context Bundle

## 5.1 问题

之前很多 RAG 实现是：

```text
把 TopK chunk 直接拼成 prompt
```

这会导致：

```text
1. 哪个 chunk 对应哪个 citation 不清楚；
2. direct hit 和 expanded chunk 混在一起；
3. chunk 被截断后没有标记；
4. 同一 document 可能占满上下文；
5. citationKey 不稳定；
6. 生成后无法校验 citation 是否来自 final context。
```

## 5.2 优化动作

新增或增强：

```text
GroundedContextAssembler
```

输出：

```text
GroundedContextBundle
```

字段：

```text
contextId
query
chunks
citations
contextBudget
usedChars
usedTokensEstimate
truncated
warnings
```

ContextChunk 字段：

```text
chunkId
documentId
documentTitle
collectionId
version
docType
sectionPath
chunkType
text
rank
score
retrievalSource
isDirectHit
isExpanded
isParent
isAdjacent
citationKey
metadata
```

## 5.3 组装规则

```text
1. direct hit chunk 优先；
2. rerank 排名靠前优先；
3. filter 内 chunk 优先；
4. parent / adjacent chunk 可以进入 context，但必须标记 expanded；
5. 同一 document / section 不能过度占满上下文；
6. 去重相同 chunk / 相同 content_hash；
7. 超出 budget 时优先保留 direct hit；
8. final context 中每个 chunk 都必须有 citationKey；
9. citationKey 必须稳定映射到 chunkId；
10. chunk 被截断时必须标记 truncated=true。
```

## 5.4 工程权衡

### 为什么不直接用 retrieval result 拼 prompt

因为引用校验需要稳定映射：

```text
answer citation [1]
  ↓
context chunk [1]
  ↓
chunkId / documentId / sectionPath / version
```

如果没有 ContextBundle，生成后的 citation verification 没有可信基准。

# 6. 核心优化三：Grounded Answer Prompt Builder

## 6.1 问题

Prompt 是 RAG 回答可信度的第一道约束。如果 prompt 没有明确要求 citation 和拒答，模型就容易自由发挥。

## 6.2 优化动作

新增：

```text
GroundedAnswerPromptBuilder
```

Prompt 必须包含：

```text
1. 系统角色：你是基于资料回答的知识库助手；
2. 回答规则；
3. 用户问题；
4. 查询理解结果；
5. 资料上下文；
6. 输出格式要求。
```

回答规则示例：

```text
- 只能使用【资料】中的内容；
- 每个关键结论后必须给出引用，例如 [1]；
- 如果资料不足，回答“根据当前资料无法确认”；
- 不要编造文档、章节、配置项、接口路径；
- 不要把推断说成事实；
- 如果资料冲突，要指出冲突。
```

输出格式建议：

```text
直接回答：
...

依据：
- [1] ...
- [2] ...

不确定或缺失：
- ...
```

## 6.3 工程权衡

### Prompt 约束能不能完全防 hallucination

不能。

所以 V18 不只依赖 prompt，还增加：

```text
CitationVerificationService
UnsupportedClaimDetector
RefusalPolicyService
AnswerGroundingScore
```

Prompt 是前置约束，verification 是后置校验。

# 7. 核心优化四：Citation Model 标准化

## 7.1 问题

如果 citation 只是：

```text
source: document.pdf
```

粒度太粗，无法判断：

```text
答案中的这个结论到底由哪一段支持？
引用的是哪个 chunk？
引用是否来自正确 version？
引用是否在 final context 中？
```

## 7.2 优化动作

新增 Citation 模型。

字段：

```text
citationId
citationKey
answerSpan
chunkId
documentId
documentTitle
collectionId
version
sectionPath
quoteSnippet
supportsClaim
supportLevel
verificationStatus
warning
```

supportLevel：

```text
EXACT
PARTIAL
WEAK
UNSUPPORTED
UNKNOWN
```

verificationStatus：

```text
PENDING
VERIFIED
FAILED
HEURISTIC
```

## 7.3 硬规则

```text
citation 必须引用 final context 中存在的 chunk；
citationKey 必须和 GroundedContextBundle 中的 chunk 对应；
不允许 citation 指向不存在的 chunk；
不允许 citation 指向被 filter 排除的 chunk；
Ask 页面展示 citation 时，要能看到 documentTitle、sectionPath、chunk snippet。
```

# 8. 核心优化五：Citation Verification

## 8.1 问题

答案中出现 [1]，不代表 [1] 支持答案。

V18 要校验：

```text
引用是否存在；
引用是否在 final context 中；
引用是否符合 collection / version / status filter；
引用是否与 claim 有基本文本证据重合；
高风险实体是否在引用 chunk 中出现。
```

## 8.2 优化动作

新增：

```text
CitationVerificationService
```

校验规则：

### 1. CitationExists

答案中出现的：

```text
[1] / [2] / [3]
```

必须存在于 ContextBundle。

### 2. CitationInFinalContext

citation 指向的 chunk 必须在 final context 中。

### 3. CitationScopeValid

citation chunk 的：

```text
collection
version
status
```

必须符合 filter。

### 4. CitationSupportsClaim 基础版

第一版使用启发式：

```text
claim 中的关键词与 cited chunk 有重合；
version / config key / class name / API path 必须在 cited chunk 中出现；
数字 / 版本 / 路径类信息必须精确匹配；
如果 claim 引用 [1]，但 [1] chunk 没有相关关键词，则 supportLevel=WEAK 或 UNSUPPORTED。
```

### 5. MissingCitation

答案中关键句没有 citation。

### 6. InvalidCitation

答案引用了不存在的：

```text
[99]
```

### 7. NegativeCitation

引用来自：

```text
negative_doc_ids
wrong collection
wrong version
```

## 8.3 输出

```text
CitationVerificationResult
```

字段：

```text
valid
citationAccuracy
totalCitations
verifiedCitations
weakCitations
unsupportedCitations
missingCitationCount
invalidCitationCount
warnings
citationDetails
heuristic
```

## 8.4 工程权衡

### 为什么第一版不用 LLM Judge

原因：

```text
默认低依赖测试不应依赖 LLM；
LLM Judge 自身也可能误判；
本地 qwen2.5:7b 做 judge 稳定性有限；
云端 judge 会增加成本和隐私问题。
```

所以 V18 第一版采用：

```text
规则 + citation scope 校验 + 关键实体匹配 + unsupported claim heuristic
```

后续可以增加可选：

```text
LlmCitationJudge
```

但默认关闭。

# 9. 核心优化六：Unsupported Claim Detection

## 9.1 问题

即使 citation 格式存在，答案中仍可能有些句子没有任何证据。

例如：

```text
系统默认使用向量缓存，所以性能会提升 50%。[1]
```

如果 [1] 只提到了“向量缓存”，没有提到“50%”，那这个数字就是 unsupported claim。

## 9.2 优化动作

新增：

```text
UnsupportedClaimDetector
```

第一版不做复杂自然语言推理，只识别高风险关键句。

关键句识别规则：

```text
包含“支持”“实现”“可以”“必须”“默认”“不会”“一定”“禁止”等确定性表达；
包含版本号；
包含配置项；
包含 API path；
包含类名 / 方法名；
包含数字；
包含“原因是”“目标是”“作用是”等解释性结论。
```

检测规则：

```text
1. 关键句没有 citation -> MISSING_CITATION；
2. 关键句 citation 不存在 -> INVALID_CITATION；
3. 关键句中的 code symbol / config key / API path 未出现在引用 chunk -> UNSUPPORTED_SYMBOL；
4. 关键句中的 version 未出现在引用 chunk -> UNSUPPORTED_VERSION；
5. 关键句中的数字未出现在引用 chunk -> UNSUPPORTED_NUMBER；
6. no context 但有确定性回答 -> HALLUCINATION_RISK。
```

输出：

```text
UnsupportedClaimReport
```

字段：

```text
totalClaims
supportedClaims
unsupportedClaims
missingCitationClaims
hallucinationRisk
claimDetails
```

ClaimDetail：

```text
claimText
sentenceIndex
citationKeys
issueType
severity
reason
```

## 9.3 工程权衡

### 为什么不做完整 claim decomposition

完整 claim decomposition 需要：

```text
句法分析；
语义切分；
事实核验模型；
自然语言推理；
LLM judge。
```

这会把 V18 变成一个复杂事实核验平台。当前阶段只做高风险实体和确定性表达的启发式检测，更适合本地项目和低依赖测试。

# 10. 核心优化七：Refusal Policy

## 10.1 问题

传统 RAG 常见错误是：

```text
没有上下文也回答；
检索置信度低也回答；
没有有效引用也回答；
用户问题很模糊也回答。
```

这会造成看似自信但错误的答案。

## 10.2 优化动作

新增：

```text
RefusalPolicyService
```

触发条件：

```text
1. final context 为空；
2. retrieval score 低于阈值；
3. citation verification 全部 unsupported；
4. queryType=NO_ANSWER_RISK；
5. clarificationRequired=true；
6. required metadata filter 没命中任何 chunk；
7. STRICT 模式下没有可引用证据。
```

输出：

```text
RefusalDecision
```

字段：

```text
shouldRefuse
reasonCode
reasonText
suggestedAnswer
suggestedNextActions
```

reasonCode：

```text
NO_CONTEXT
LOW_RETRIEVAL_CONFIDENCE
NO_VALID_CITATION
QUERY_AMBIGUOUS
VERSION_NOT_FOUND
COLLECTION_NOT_SELECTED
STRICT_MODE_NO_EVIDENCE
```

拒答文案示例：

```text
根据当前知识库资料，无法确认这个问题的答案。建议选择更具体的知识库分组、版本，或补充相关文档后重试。
```

## 10.3 工程权衡

### 为什么拒答不是失败

因为在无答案场景里，正确拒答比胡编更有价值。

V18 把拒答纳入质量指标：

```text
RefusalAccuracy
```

No-answer case 正确拒答应得高分。

# 11. 核心优化八：Answer Grounding Score

## 11.1 问题

用户需要一个直观判断：

```text
这次回答可信不可信？
需要不需要复核？
主要问题是引用缺失，还是主张无支持？
```

## 11.2 优化动作

新增：

```text
AnswerGroundingScore
```

指标：

```text
CitationCoverage
CitationAccuracy
UnsupportedClaimRate
RefusalCorrectness
ContextUsage
GroundingScore
```

定义：

### CitationCoverage

```text
有 citation 的关键 claim 占比。
```

### CitationAccuracy

```text
有效 citation 占比。
```

### UnsupportedClaimRate

```text
无支持主张占比，越低越好。
```

### RefusalCorrectness

```text
无答案场景是否拒答。
```

### ContextUsage

```text
答案是否使用了 final context 中的主要证据。
```

## 11.3 评分公式

建议公式：

```text
groundingScore =
  CitationCoverage * 25
+ CitationAccuracy * 30
+ (1 - UnsupportedClaimRate) * 25
+ ContextUsage * 10
+ RefusalCorrectness * 10
```

范围：

```text
0-100
```

等级：

```text
90-100  可信
75-89   基本可信
60-74   需要复核
0-59    不可信
UNKNOWN 暂无评分
```

## 11.4 工程权衡

### 这个分数是不是事实真值

不是。

它是启发式可信评分，作用是：

```text
帮助用户发现引用缺失、引用无效、主张无支持、上下文不足等问题。
```

不能宣称等价于严格事实核验。

# 16. 工程权衡总结

## 16.1 Prompt 约束 vs 后置校验

只靠 prompt 不够，因为模型可能不遵守。

只做后置校验也不够，因为生成前没有约束，错误会更多。

V18 选择：

```text
前置 Grounded Answer Contract
  +
后置 CitationVerification / UnsupportedClaimDetection
```

## 16.2 规则校验 vs LLM Judge

### 规则校验

优点：

```text
低依赖；
可测试；
可解释；
不会增加模型成本；
适合版本号、配置项、API path、类名等精确实体。
```

缺点：

```text
复杂语义支持判断不够强；
不能完全理解隐含推理。
```

### LLM Judge

优点：

```text
能处理更复杂语义；
可以判断 paraphrase 和隐含支持关系。
```

缺点：

```text
依赖模型；
成本和延迟更高；
judge 也可能误判；
默认测试不可控。
```

V18 选择：

```text
默认规则 / 启发式；
预留 LLM Judge 接口；
默认关闭。
```

## 16.3 拒答 vs 用户体验

拒答会减少“看起来有用”的回答，但能显著降低错误答案风险。

V18 的策略：

```text
STRICT 模式：上下文不足必须拒答；
BALANCED 模式：可部分回答，但必须说明资料不足；
EXPLORATORY 模式：允许轻度推断，但必须标注推断。
```

## 16.4 citation 粒度：document vs chunk

document 级引用太粗，无法证明某个 claim。

V18 选择：

```text
chunk / section 级 citation。
```

原因：

```text
能定位具体证据；
能校验 version / section / chunk；
能在 UI 中展示 snippet；
更适合面试表达和产品可信度。
```

## 16.5 GroundingScore 是否应该直接作为真理分数

不应该。

它是：

```text
可信度启发式评分。
```

必须在文档中说明：

```text
不是严格事实真值；
用于提醒用户复核风险；
最终高风险场景仍需要人工确认。
```

# 19. 面试高强度追问与回答

## 19.1 V18 和普通“带引用 RAG”有什么区别？

回答：

```text
普通带引用 RAG 只是把 source 附在答案后面，不能保证 source 支撑结论。V18 不只是展示引用，而是建立 GroundedContextBundle、Citation Model 和 CitationVerificationService，要求 citation 必须来自 final context，并校验引用是否符合 collection、version、status filter，以及是否能基本支撑 claim。
```

## 19.2 为什么检索正确后还会 hallucination？

回答：

```text
因为 LLM 生成是概率过程，它可能忽略上下文、过度概括、补充上下文外知识，或者把多个 chunk 错误合并。检索正确只是降低风险，不等于答案一定忠实。V18 用回答契约、引用校验和未支持主张检测来约束生成层。
```

## 19.3 为什么不默认调用 LLM Judge？

回答：

```text
默认调用 LLM Judge 会引入模型依赖、成本、延迟和不确定性，而且本地小模型做 judge 稳定性有限。V18 第一版使用规则和启发式校验，保证默认测试低依赖、可解释、可复现。后续可以预留 LLM Judge 接口作为增强。
```

## 19.4 CitationVerification 怎么判断引用是否支持结论？

回答：

```text
第一版采用启发式：先检查 citation 是否存在、是否在 final context 中、是否符合 collection/version/status filter；然后对 claim 中的版本号、代码符号、配置项、API path、数字等高风险实体做精确匹配。如果 claim 引用的 chunk 完全没有相关关键词，则标记为 WEAK 或 UNSUPPORTED。
```

## 19.5 为什么要区分 direct hit chunk 和 expanded chunk？

回答：

```text
direct hit 是检索直接命中的证据，expanded chunk 是为了上下文完整性回填的 parent 或 adjacent 内容。二者的证据强度不同。如果不区分，系统可能把扩展内容当成核心证据，影响 citation verification 和 faithfulness 评估。
```

## 19.6 上下文不足时为什么要拒答？

回答：

```text
在知识库问答里，错误确定性答案比“不知道”更危险。如果 final context 为空、检索置信度低、没有有效 citation，系统应该拒答或部分回答，并说明资料不足。V18 把拒答纳入 RefusalAccuracy，正确拒答是质量提升，不是失败。
```

## 19.7 AnswerGroundingScore 是怎么计算的？

回答：

```text
它由 CitationCoverage、CitationAccuracy、UnsupportedClaimRate、ContextUsage 和 RefusalCorrectness 组成。大致公式是 CitationCoverage 占 25，CitationAccuracy 占 30，1-UnsupportedClaimRate 占 25，ContextUsage 占 10，RefusalCorrectness 占 10，最终归一化到 0-100。
```

## 19.8 这个评分是不是事实真值？

回答：

```text
不是。它是启发式可信评分，用来提示引用缺失、引用无效、未支持主张和上下文不足风险。它不能替代人工事实核验，尤其在高风险领域仍需要人工复核。
```

## 19.9 V18 和 V14 的区别是什么？

回答：

```text
V14 是知识库体检报告，主要评估检索和回答质量。V18 是把可信约束直接加入 Ask 生成链路，包括 Grounded Answer Contract、Citation Verification、Unsupported Claim Detection 和 Refusal Policy。V14 发现问题，V18 在生成阶段预防问题。
```

## 19.10 V18 和 V17 如何联动？

回答：

```text
V17 会抽取 versionHint、codeSymbols、configKeys 和 apiPaths。V18 会用这些信息做更严格的引用校验。例如用户问 V10.0，引用 chunk 必须匹配 V10.0；答案提到 LocalPythonLlmProvider，引用 chunk 必须出现该符号；答案提到 application-local-ai.properties，引用 chunk 必须包含相关配置项。
```

## 19.11 为什么 citation 要到 chunk / section 级别？

回答：

```text
document 级引用太粗，无法证明某个 claim。chunk / section 级引用可以定位具体证据，也便于校验 version、sectionPath、code symbol 和 config key。用户在 UI 上也能看到具体 snippet，而不是只看到一个文件名。
```

## 19.12 最大风险是什么，怎么控制？

回答：

```text
最大风险是启发式校验覆盖不全，可能误判复杂语义支持关系。控制方式是：无法判断时标记 UNKNOWN，不伪造 VERIFIED；关键实体做精确校验；保留人工复核提示；后续可选接入 LLM Judge，但默认关闭。
```

- 实现 UnsupportedClaimDetector 和 RefusalPolicyService，识别无引用关键主张、无资料支持的确定性表达，并在上下文不足、检索置信度低或无有效引用时触发拒答。

# 21. 最终总结

V18.0 可以这样总结：

```text
V18.0 解决的是 RAG 最终回答是否可信的问题。此前 V14-V17 已经让系统能够诊断知识库质量、优化检索链路、保证向量索引不污染，并理解用户问题如何检索。但即使检索正确，LLM 仍然可能脱离上下文发挥，或者给出看似有引用但实际不支撑结论的答案。

因此我设计了 Grounded Answer Contract，要求答案只能基于 final context，每个关键结论必须有引用，上下文不足时必须拒答。同时，我把 final context 标准化为 GroundedContextBundle，为每个 chunk 生成稳定 citationKey，并在生成后通过 CitationVerificationService 校验引用是否存在、是否来自 final context、是否符合 collection/version/status 过滤，以及是否能支撑对应 claim。

为了进一步降低 hallucination 风险，我实现了 UnsupportedClaimDetector 和 RefusalPolicyService，检测无引用关键主张、无资料支持的版本号、配置项、API path、代码符号和数字，并在证据不足时触发拒答。最终系统会输出 AnswerGroundingScore，并在 Ask 页面展示可信回答状态、引用校验结果和未支持主张。

和之前阶段相比，V18 的变化是：系统不再只是“检索后回答”，而是“基于证据回答，并能校验答案是否真的被证据支持”。
```
