# V0.9 面试文档：幂等与重复消费控制

## 1. 本版本解决的问题

V0.9 解决 MQ 重复投递或手动重复派发导致同一任务被重复执行的问题。RabbitMQ 不能保证业务 exactly-once，Consumer 必须在入口判断任务是否还能执行，避免重复调用 LLM、重复写结果和污染事件表。

## 2. 为什么现在做这个版本

它依赖 V0.8 的重试机制。重试和 Scheduler 会增加重复消息出现概率，因此必须补幂等。它为后续 Outbox at-least-once 投递和多 Consumer 并发执行做准备。此时不做分布式锁，是因为状态机入口保护更轻量。

## 3. 核心需求边界

本版本实现 tryStartTaskExecution：只有 PENDING / RETRY_PENDING 能进入 RUNNING。RUNNING / SUCCESS / FAILED / CANCELLED 收到重复消息直接忽略。不做 Redis 锁、不做乐观锁、不做 Outbox、不做 ack/nack 重构。

## 4. 方案选择

Consumer 执行前先调用 TaskService.tryStartTaskExecution。started=false 时直接 return，不执行 Mock LLM，也不写 SUCCESS/FAILED。后续 V2.2.1 会把这个入口升级为数据库原子 claim。

## 5. 替代方案与暂时不选的原因

替代方案一是相信 MQ 不重复。当前不选，因为消息系统常见语义是 at-least-once。

替代方案二是用 Redis 分布式锁。当前不选，因为它引入额外基础设施，且状态机已能解决早期重复执行问题。

## 6. 数据库设计

本版本未新增数据库结构，主要复用 task.status。status 本身成为幂等判断依据，只有允许执行的状态才能被更新为 RUNNING。

## 7. API 设计

新增或使用开发接口 POST /dev/tasks/{taskId}/dispatch 用于模拟重复投递。该接口仅用于本地验收，不是生产接口。

## 8. 核心流程

Consumer 收到 taskId
-> tryStartTaskExecution
-> 如果 PENDING/RETRY_PENDING，更新 RUNNING 并继续执行
-> 如果 RUNNING/SUCCESS/FAILED/CANCELLED，返回 false
-> started=false 时忽略消息

## 9. 测试方案

建议覆盖 SUCCESS 重复投递不执行、FAILED 重复投递不执行、RUNNING 重复投递不并发执行、RETRY_PENDING 仍可重试。SQL 验证 task_event 不出现 SUCCESS -> RUNNING 或 FAILED -> RUNNING。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是幂等不能只靠“代码看起来不会重复”。MQ、Scheduler、手动 dispatch、多实例 Consumer 都可能制造重复入口。后续需要数据库原子 claim 才能抵抗并发 race。

## 12. 本版本亮点

- 明确 Consumer 入口幂等规则。
- 避免终态任务被重复执行。
- 提供开发重复投递接口用于验收。
- 为 Outbox at-least-once 投递提供 Consumer 侧保护。

## 13. 简历 bullet

### 简洁版

实现 Consumer 入口幂等保护，限制只有 PENDING / RETRY_PENDING 任务可进入 RUNNING，避免 MQ 重复投递导致重复执行。

### 强化版

针对 RabbitMQ at-least-once 语义设计任务执行入口幂等控制，通过状态机约束重复消息处理路径，防止 SUCCESS/FAILED/CANCELLED 任务被重新执行或重复调用 LLM。

## 14. 面试高强度追问与回答

Q1：为什么 MQ 系统必须考虑幂等？

A1：消息可能重投、Consumer 可能崩溃重启，业务不能假设一条消息只处理一次。

Q2：tryStartTaskExecution 做了什么？

A2：它检查当前状态，只有 PENDING/RETRY_PENDING 可以进入 RUNNING，其他状态返回 false。

Q3：RUNNING 重复消息为什么要忽略？

A3：RUNNING 表示已有执行者在处理，再执行会造成并发重复调用模型。

Q4：SUCCESS 重复消息为什么危险？

A4：它可能覆盖已完成结果，写出 SUCCESS -> RUNNING 等脏事件。

Q5：为什么不靠前端避免重复点击？

A5：重复消息可能来自 MQ 和 Scheduler，不是前端能完全控制的问题。

Q6：这个版本是否强并发安全？

A6：应用层先查后改仍有 race 风险，后续 V2.2.1 用数据库原子 claim 加强。

Q7：为什么需要 /dev dispatch？

A7：它能本地模拟重复 MQ 投递，方便验证幂等逻辑。

Q8：当前局限是什么？

A8：它是应用层幂等，不是完整 exactly-once。需要配合原子 claim、Outbox 和 attempt 审计。

## 15. 后续演进路线

V0.9 支撑 V2.2.1 Atomic Claim 和 V2.2.2 Outbox。后续即使消息重复投递，Consumer 仍能通过状态 claim 防止重复执行。
