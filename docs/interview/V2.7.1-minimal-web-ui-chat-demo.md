# V2.7.1 Minimal Web UI / Chat Demo 面试文档

## 1. 版本摘要

V2.7.1 的版本名称是：

```text
V2.7.1 Minimal Web UI / Chat Demo
```

中文可以表述为：

```text
极简 RAG 问答演示页面
```

这个版本的核心目标不是建设完整前端系统，而是为已经完成的 `POST /rag/answers` 接口提供一个最小可体验入口。

在 V2.7 中，系统已经完成了 RAG Answer API（检索增强问答接口），能够完成：

```text
用户问题
  -> 生成 query embedding
  -> 调用 VectorStore 检索相关 chunk
  -> 构造 citations
  -> 构造 RAG prompt
  -> 调用 LLM 生成 answer
  -> 返回 answer / citations / metadata
```

但是如果只停留在 HTTP API 层面，用户或面试官需要通过 Postman、curl 或 HTTP 文件才能体验系统能力。V2.7.1 解决的就是这个体验门槛问题。

本版本新增了一个 Spring Boot static resources（静态资源）页面，使用户可以在浏览器中输入 query 和 topK，调用后端 RAG Answer API，并直接查看：

- answer（回答内容）；
- citations（引用来源）；
- retrieval metadata（检索元数据）；
- generation metadata（生成元数据）；
- loading（加载状态）；
- error response（错误响应）。

这个版本的重点是：

> 用最小前端改动，把后端 RAG 能力转化成一个可以直接演示的产品化入口。

---

## 2. 为什么现在做这个版本

V2.7 已经完成了 RAG Answer API with Citations，也就是说系统已经具备了核心 RAG 问答能力。

但是在 V2.7 之后，系统还有一个明显问题：

```text
能力已经存在，但体验门槛仍然偏高。
```

如果没有 Web UI，面试官或 reviewer 想体验项目，需要理解：

- 接口地址是什么；
- 请求方法是什么；
- 请求体怎么写；
- topK 是什么意思；
- citations 在响应体哪里；
- retrieval metadata 和 generation metadata 怎么看；
- 错误响应怎么看。

这会让项目停留在“工程接口可用”的阶段，而不是“用户可体验”的阶段。

所以 V2.7.1 选择做一个极简 Web UI，目的是：

```text
让项目从“可以用接口调用”升级为“可以在浏览器中直接体验”。
```

这一步对求职项目很重要。因为面试官不会总是有时间阅读完整后端源码，但如果可以直接打开浏览器看到 RAG 问答、引用和元数据，项目的可信度和展示效果会明显提升。

---

## 3. 本版本解决的问题

### 3.1 降低 RAG API 的体验门槛

原来的使用方式是：

```text
使用 Postman / curl / HTTP 文件
手动构造 JSON
调用 POST /rag/answers
查看原始 JSON 响应
```

V2.7.1 之后，使用方式变成：

```text
打开浏览器
输入问题
选择 topK
点击 Ask
查看 answer、citations 和 metadata
```

这让系统更接近真实产品体验。

---

### 3.2 让 citations 更容易被理解

RAG 系统最重要的能力之一不是“生成回答”，而是：

```text
回答是否有依据？
依据来自哪些文档？
命中了哪些 chunk？
检索分数是多少？
```

所以页面单独展示 citations，每条 citation 至少包含：

- sourceIndex；
- documentId；
- chunkId；
- score；
- contentSnippet。

这样用户可以看到：

```text
answer 不是凭空生成的，而是基于检索结果生成的。
```

这也能体现项目的可解释性。

---

### 3.3 让 metadata 成为工程证据

本项目不是简单调用 LLM API，而是一个 RAG / LLM orchestration 系统。

所以除了 answer，页面还展示：

- retrieval metadata（检索元数据）；
- generation metadata（生成元数据）。

这些 metadata 可以帮助说明：

- 本次检索使用了什么 topK；
- 实际返回了多少检索结果；
- 使用了什么 provider；
- 使用了什么 model；
- 检索和生成链路是否正常；
- 后续是否可以继续做性能分析、质量评估和可观测性建设。

这让项目从“能跑”进一步变成“能解释、能观察、能评估”。

## 4. 核心设计

### 4.1 页面入口设计

本版本使用 Spring Boot static resources 托管静态页面。

页面访问路径为：

```text
/rag-demo.html
```

而不是直接使用：

```text
/index.html
```

原因是：

```text
/rag-demo.html 的语义更明确；
它只代表 RAG demo，不代表完整产品首页；
未来如果要做真正首页，可以再单独设计 /index.html；
当前版本不应该过早占用首页语义。
```

---

### 4.2 文件结构设计

本版本采用轻量文件拆分：

```text
src/main/resources/static/rag-demo.html
src/main/resources/static/rag-demo.js
src/main/resources/static/rag-demo.css
```

这样做的原因是：

```text
HTML 负责页面结构；
JS 负责请求和渲染逻辑；
CSS 负责基础样式；
三者分离后更清楚，也仍然足够轻量。
```

没有引入复杂前端工程目录，也没有引入 npm 构建流程。

---

### 4.3 请求设计

页面点击 Ask 后，前端调用：

```http
POST /rag/answers
```

请求体格式：

```json
{
  "query": "问题内容",
  "topK": 5
}
```

这里使用同源相对路径：

```text
/rag/answers
```

原因是：

```text
静态页面由同一个 Spring Boot 服务托管；
后端 API 也在同一个 Spring Boot 服务中；
浏览器请求属于同源请求；
不需要新增 CORS 配置；
不会引入额外部署复杂度。
```

这符合当前版本的边界：只做最小可体验页面，不改后端部署架构。

---

### 4.4 响应展示设计

成功响应后，页面展示四类信息：

```text
answer
citations
retrieval metadata
generation metadata
```

其中 answer 是用户最关心的最终回答。

citations 是 RAG 的证据链。

metadata 是工程可观测信息。

这三类信息分别对应：

```text
answer：用户价值
citations：可解释性
metadata：工程证据
```

## 5. 关键取舍

### 5.1 为什么不引入 React / Vue

本版本没有引入 React、Vue、Vite、npm、TypeScript、Tailwind 等前端工程工具。

原因是：

```text
当前版本目标不是建设正式前端工程；
目标只是让 RAG Answer API 可以被浏览器体验；
原生 HTML / JS / CSS 已经足够；
引入前端框架会带来 package、构建、部署、测试等额外复杂度；
这会偏离版本目标。
```

面试时可以这样表达：

```text
我没有在这个阶段引入 React 或 Vue，因为 V2.7.1 的目标不是展示前端工程能力，而是用最小成本验证和演示后端 RAG 闭环。Spring Boot static resources 已经能满足这个目标，所以我选择保持边界克制。
```

---

### 5.2 为什么不做多轮对话

本版本没有做 conversationId、chat history、多轮上下文记忆。

原因是：

```text
V2.7 的后端语义是 single-turn RAG answer；
多轮对话需要新增 conversation、message、history、上下文压缩、引用继承等设计；
这些属于新的后端能力，不是 Web UI 展示层能力；
如果放进 V2.7.1，会扩大版本边界。
```

所以 V2.7.1 只展示现有单轮 RAG API，不改变后端语义。

---

### 5.3 为什么不做 streaming

本版本没有做 streaming（流式输出）。

原因是：

```text
当前 POST /rag/answers 是普通同步响应接口；
streaming 需要改造响应协议；
可能涉及 SSE、WebSocket 或 chunked response；
还会影响前端渲染、错误处理和测试方式；
这不是当前版本的目标。
```

streaming 可以作为后续独立版本做，例如：

```text
V2.8 RAG Streaming Answer API
```

但不应该混入 V2.7.1。

---

### 5.4 为什么不做文件上传 UI

本版本没有做文件上传、文档管理或 RAG ingestion 页面。

原因是：

```text
文件上传属于 RAG ingestion 链路；
RAG Answer API 属于 query 链路；
V2.7.1 只是 query 链路的可视化演示；
如果同时做上传，会把 ingestion 和 query 两条链路混在一起。
```

这会让版本边界变复杂，也不利于后续维护。

---

### 5.5 为什么 metadata 用 JSON pretty print 展示

metadata 没有做复杂卡片式 UI，而是用 JSON pretty print 的方式展示。

原因是：

```text
metadata 本质上是工程调试和观测信息；
字段可能随着后续版本变化；
JSON 展示方式更稳定；
前端不需要强绑定每一个字段；
reviewer 可以看到完整响应结构。
```

这是一种符合 demo 阶段的设计取舍。

## 6. 页面能力说明

### 6.1 输入区

页面提供：

- query 多行输入框；
- topK 数字输入框，默认值为 5；
- Ask 按钮。

query 用于输入用户问题。

topK 表示检索时最多返回多少个相关 chunk。

topK 默认设置为 5，是因为：

```text
topK=5 通常能在上下文覆盖和噪声控制之间取得比较好的平衡；
数量太少可能召回不足；
数量太多可能引入无关 chunk，影响 prompt 质量。
```

---

### 6.2 Loading 状态

点击 Ask 后，页面进入 loading 状态：

- 禁用 Ask 按钮；
- 显示加载文案；
- 请求结束后恢复按钮；
- 成功和失败都必须结束 loading。

这个设计解决的问题是：

```text
防止重复点击；
让用户知道请求正在处理；
避免页面无反馈导致用户误以为系统卡死。
```

---

### 6.3 Answer 展示区

Answer 区域展示后端返回的 `answer` 字段。

这是用户最直接感知到的结果。

页面不做 Markdown 渲染，只展示纯文本。

原因是：

```text
当前重点是 RAG 问答能力；
Markdown 渲染不是核心目标；
引入 Markdown 渲染库会增加依赖和安全边界；
纯文本展示足够满足 demo 需求。
```

---

### 6.4 Citations 展示区

Citations 区域按列表展示每条引用来源。

每条 citation 至少展示：

- sourceIndex；
- documentId；
- chunkId；
- score；
- contentSnippet。

这让用户可以直接看到：

```text
系统检索到了哪些文本；
每条文本的相关性分数是多少；
最终回答可以追溯到哪些 chunk。
```

如果 citations 为空，页面也不会报错，而是展示空状态。

这对应 V2.7 里的 no-context 场景。

---

### 6.5 Retrieval Metadata 展示区

Retrieval metadata 表示检索阶段的信息。

它可以帮助解释：

```text
这次检索使用了什么参数；
返回了多少结果；
使用了什么 embedding space；
检索链路是否符合预期。
```

这对调试 RAG 系统非常重要。

---

### 6.6 Generation Metadata 展示区

Generation metadata 表示生成阶段的信息。

它可以帮助解释：

```text
是否调用了 LLM；
使用了什么 LLM provider；
使用了什么 model；
生成链路是否正常。
```

在 no-context 场景下，如果系统不调用 LLM，generation metadata 也能辅助说明生成阶段的行为。

---

### 6.7 Error 展示区

错误区兼容 V2.6.4 统一 API 错误响应结构。

统一错误响应一般包含：

- timestamp；
- status；
- code；
- message；
- path；
- traceId。

页面重点展示：

- code；
- message；
- traceId。

这样设计的原因是：

```text
code 方便识别错误类型；
message 方便用户理解错误原因；
traceId 方便后端排查日志。
```

如果出现网络错误或非标准错误，页面展示 fallback 错误信息，避免页面直接崩溃。

本版本仍然要求：

```powershell

```

- 外部 API；
- 外部网络；
- Python worker；
- 真实模型下载；
- Qdrant；
- Docker；
- OpenAI API key。

这延续了前面版本的测试分层原则：

```text
默认测试必须低依赖、可重复、可在普通开发环境中稳定运行。
```

---

### 7.2 静态资源测试

如果本版本新增了测试，测试范围应该非常轻量，例如：

```text
验证 /rag-demo.html 可以访问；
或者验证 static/rag-demo.html 资源存在。
```

不应该引入：

- Selenium；
- Playwright；
- Cypress；
- npm test；
- 浏览器自动化测试。

原因是：

```text
当前版本不是前端工程化阶段；

只需要保证静态资源没有破坏后端测试即可。
```

## 11. 面试高强度追问与回答

### 11.1 你为什么要做这个 Web UI？

回答：

```text
因为 V2.7 已经完成了 RAG Answer API，后端链路已经能完成 query embedding、向量检索、citations 构造和 LLM generation。但如果只停留在 API 层，面试官或 reviewer 需要通过 Postman 或 curl 才能体验系统能力，门槛比较高。

所以 V2.7.1 我做了一个极简 Web Demo，让用户可以直接在浏览器输入问题，看到 answer、citations 和 metadata。这个版本的核心价值不是前端复杂度，而是把后端 RAG 能力变成可体验、可演示的产品入口。
```

---

### 11.2 为什么不用 React 或 Vue？

回答：

```text
我刻意没有引入 React 或 Vue。因为这个版本的目标不是建设前端工程，而是最小成本展示后端 RAG 闭环。

如果引入 React/Vue，就会带来 npm、构建流程、前端路由、依赖管理和部署复杂度。这些复杂度和当前版本目标不匹配。

Spring Boot static resources 已经足够托管一个极简 HTML/JS/CSS 页面，所以我选择用最小技术栈完成浏览器体验，保持项目边界克制。
```

---

### 11.3 为什么页面叫 `/rag-demo.html`，而不是 `/index.html`？

回答：

```text
因为当前页面只是 RAG 问答演示页，不是完整产品首页。使用 `/rag-demo.html` 可以让语义更明确，也避免过早占用未来首页设计。

如果后续项目发展出完整产品入口、文档管理页面、任务管理页面、模型选择页面，再设计 `/index.html` 会更合理。
```

---

### 11.4 为什么不做多轮对话？

回答：

```text
因为当前后端 `POST /rag/answers` 的语义是 single-turn RAG answer，也就是单轮问答。

多轮对话需要额外设计 conversationId、message history、上下文压缩、历史引用继承、记忆管理等能力。这些属于新的后端版本，不是 Web UI 展示层应该顺手做掉的东西。

V2.7.1 的边界是可视化已有 RAG Answer API，不改变后端语义。
```

---

### 11.5 为什么不做 streaming？

回答：

```text
因为 streaming 会改变接口响应协议。当前 `POST /rag/answers` 是同步 JSON 响应，如果做 streaming，可能需要引入 SSE、WebSocket 或 chunked response，同时前端也要处理增量渲染、错误中断和连接状态。

这已经不是 minimal web demo 的范围。当前版本先保证完整结果展示，后续可以把 streaming 作为独立版本演进。
```

---

### 11.6 为什么不做文件上传？

回答：

```text
文件上传属于 RAG ingestion 链路，而 V2.7.1 只关注 query 链路的可视化。

RAG 系统一般可以拆成两条链路：第一条是 ingestion，也就是文档上传、解析、切分、embedding、写入向量库；第二条是 query，也就是用户提问、检索、构造 prompt、生成回答。

本版本只把 query 链路可视化。如果把上传也做进来，会把两个版本目标混在一起，导致边界变大。
```

---

### 11.7 为什么要展示 citations？

回答：

```text
因为 citations 是 RAG 系统可解释性的核心。只展示 answer，用户无法判断答案是不是模型幻觉，也不知道回答依据了哪些文本。

展示 citations 后，用户可以看到每条引用的 documentId、chunkId、score 和 contentSnippet。这说明答案可以追溯到具体检索结果，也能帮助调试检索质量。
```

---

### 11.8 为什么要展示 metadata？

回答：

```text
metadata 是工程化 RAG 系统的重要证据。它可以帮助我们知道本次请求用了什么检索参数、返回了多少结果、使用了什么 provider/model，以及生成阶段的行为。

这对于调试、评估和后续可观测性建设都很重要。一个成熟的 RAG 系统不能只看最终 answer，也要能解释 retrieval 和 generation 两个阶段发生了什么。
```

---

### 11.9 这个版本体现了什么工程能力？

回答：

```text
这个版本体现的是工程边界控制和产品化意识。

我没有为了展示效果引入复杂前端框架，也没有顺手扩展多轮对话、上传、streaming 等能力，而是围绕一个明确目标：把已有 RAG Answer API 变成可体验入口。

同时页面兼容成功响应、空 citations、metadata 缺失、统一错误响应和网络错误，说明它不是只能跑 happy path，而是考虑了基本的交互鲁棒性。
```

---

### 11.10 如果面试官问“这不就是一个简单页面吗？”

回答：

```text
这个页面本身确实不复杂，复杂度不是它的目标。它的价值在于把前面多个后端版本沉淀出的能力整合成一个可体验入口。

这个页面背后连接的是 Document、Chunk、EmbeddingProvider、VectorStore、RAG Answer API、citations、LLM generation 和统一错误响应。V2.7.1 的意义是让这些后端能力可以被直接体验和验证，而不是停留在接口文档里。
```

## 13. 本版本在项目中的位置

项目版本链可以这样理解：

```text
V2.6.x：项目收敛、错误响应、Qdrant 验证、Local Embedding Worker、Benchmark、Embedding Cache、Cache Metrics
V2.7：RAG Answer API with Citations
V2.7.1：Minimal Web UI / Chat Demo
```

V2.7 是后端 RAG 闭环。

V2.7.1 是这个闭环的浏览器体验入口。

它不改变底层 RAG 架构，而是把已有能力展示出来。

所以它的位置是：

```text
backend RAG capability
  -> interactive demo surface
```

中文可以表述为：

```text
把后端 RAG 能力包装成一个最小可体验产品界面。
```

## 14. 总结表达

这个版本可以用一段话总结：

```text
在 V2.7 完成 RAG Answer API 后，我在 V2.7.1 中新增了一个极简 Web Demo，通过 Spring Boot static resources 托管 HTML/JS/CSS 页面，直接调用同源的 `POST /rag/answers` 接口。页面支持输入 query 和 topK，展示 answer、citations、retrieval metadata、generation metadata，并兼容 loading 和统一错误响应展示。

这个版本没有引入 React/Vue/npm，也没有扩展多轮对话、streaming、文件上传等能力，而是严格聚焦于把现有后端 RAG 能力变成可体验、可演示的产品入口。它体现了我对版本边界、工程复杂度和项目展示价值的控制。
```
