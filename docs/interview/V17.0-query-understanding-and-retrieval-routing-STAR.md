# V17.0 Query Understanding & Retrieval Routing 面试文档（STAR 版）

## 0. 阶段名称

英文阶段名：

```text
V17.0 Query Understanding & Retrieval Routing
```

中文阶段名：

```text
查询理解与检索路由
```

项目定位：

```text
MindWeaver / Personal AI Knowledge Workspace
本地个人 AI 知识工作台
```

一句话总结：

```text
V17.0 的目标是让系统在用户自然语言提问时，先理解问题类型、抽取版本 / 文档类型 / 代码符号 / 配置项 / API 路径等检索约束，再自动选择合适的检索策略和过滤条件，避免用户模糊提问时盲目全库搜索，从而减少跨集合污染、错误版本污染和检索策略不匹配问题。
```

```text

```

```text
V15.0 Structured Chunking & Hybrid Retrieval Pipeline

```

系统已经具备：

```text
1. 知识库体检报告；
2. Gold Test Set 评测；
3. Recall@K、MRR@K、NDCG@K、ContextPrecision@K 等指标；
4. CrossCollectionLeakRate 跨集合污染检测；
5. WrongVersionLeakRate 错误版本污染检测；

7. metadata pre-filter；
8. Hybrid Search；
9. RRF 融合；
```

但是，系统仍然有一个关键入口问题：

```text
用户提问时，系统还不知道“应该怎么搜”。
```

真实用户不会每次都精确选择：

```text
collection；
version；
doc_type；
status；
检索策略；
topK；
rerank；
context expansion。
```

用户通常会问：

```text
V10 的模型供应商配置支持什么？
最新方案里怎么处理向量重复？
LocalPythonLlmProvider 是干什么的？
application-local-ai.properties 里配置了什么？
V14 和 V15 的区别是什么？
总结一下知识库体检报告的价值。
这个为什么报错？
```

这些问题对应的检索策略完全不同。如果系统固定使用一种检索策略，例如：

```text
Vector Only + Top10
```

就会出现：

```text
1. 版本问题没有 version filter；
2. 最新方案问题召回 deprecated 文档；
3. 代码符号问题只靠向量检索，召回不稳定；
4. 配置项问题缺少 keyword / BM25；
5. 总结类问题 Recall 不够；
6. 对比类问题被单文档结果占满；
7. 模糊问题盲目全库搜索，造成跨集合污染；
8. 无答案问题仍然硬生成答案。
```

## T：Task，任务

V17.0 的任务是实现一个轻量但可解释的：

```text
Query Understanding Layer
查询理解层

Retrieval Routing Layer
检索路由层
```

目标是：

```text
1. 自动识别 query type；
2. 抽取 version、doc_type、collection、code symbol、config key、API path 等 metadata hints；
3. 生成轻量 query rewrite；
4. 根据 query type 自动选择 retrieval strategy；
5. 自动构建 RetrievalFilter；
6. 对模糊问题触发 clarification guard，避免全库盲搜；
7. 在 Ask 页面展示查询理解与检索路由过程；
8. 在 V14 Knowledge Health 中对比启用查询理解前后的指标变化；
9. 保持默认低依赖，不默认调用云端 LLM 或复杂 Agent planner。
```

V17.0 设计并实现：

## R：Result，结果

V17.0 完成后，系统能够：

```text
1. 自动识别版本限定、最新版本、代码符号、配置项、API 路径、总结、对比、模糊问题；
2. 自动抽取 version / doc_type / codeSymbols / configKeys / apiPaths；
3. 代码符号和配置项问题自动路由到 Hybrid + RRF + Rerank；
4. 版本问题自动启用 version filter；
5. 最新方案问题默认过滤 deprecated；
6. 总结类问题启用更高 Recall 和 context expansion；
7. 模糊问题不再默认全库搜索，而是提示用户选择 collection；
8. Ask 页面能解释系统为什么这样搜；
9. V14 Knowledge Health 能量化启用 Query Understanding 后 Recall、MRR、ContextPrecision、CrossCollectionLeakRate、WrongVersionLeakRate 是否改善。
```

# 2. 遇到的问题：为什么需要 Query Understanding

## 2.1 固定检索策略的问题

如果所有问题都走：

```text
Vector Only + Top10
```

会有明显缺陷。

### 版本问题

用户问：

```text
V10.0 的模型供应商配置支持哪些能力？
```

如果没有识别出：

```text
version = V10.0
```

系统可能召回 V11、V12、V14 的相似文档，导致错误版本污染。

### 最新问题

用户问：

```text
最新方案是怎么处理向量库污染的？
```

系统应该默认过滤：

```text
DEPRECATED
TRASHED
PURGED
```

否则可能引用旧方案。

### 代码符号问题

用户问：

```text
LocalPythonLlmProvider 是做什么的？
```

向量检索可能觉得“本地模型调用”“Python Worker”“模型供应商”都语义相关，但未必精确命中类名。这个问题更适合：

```text
BM25 / keyword + vector hybrid
```

### 配置项问题

用户问：

```text
application-local-ai.properties 里配置了什么？
```

这类问题需要精确命中配置文件名和 key，BM25 / keyword 比纯向量更可靠。

### 总结问题

用户问：

```text
总结一下 V14 知识库体检报告的价值。
```

这需要更高 Recall 和更多上下文，不适合只取 top5 短 chunk。

### 模糊问题

用户问：

```text
这个怎么优化？
```

如果系统没有上下文，却全库搜索，很容易召回错误项目文档。

## 2.2 用户不应该手动理解所有检索参数

用户不应该被迫理解：

```text
Vector Only；
BM25 Only；
Hybrid；
RRF；
Rerank；
Parent Context；
TopK；
metadata filter；
collection_id；
version；
doc_type。
```

这些应该由系统根据 query type 自动选择。

V17 的本质是：

```text
把检索策略选择从用户手里收回来，变成系统自动决策，并且把决策过程解释给用户。
```

# 3. 参考了什么技术思路

## 3.1 Azure AI Search / Google Vertex AI Search 的启发

Azure AI Search 和 Google Vertex AI Search 这类产品中，检索不是单纯把 query 丢进向量库，而是会结合：

```text
full-text search；
vector search；
hybrid search；
semantic ranking；
metadata filtering；
query options；
filter expressions。
```

对 V17 的启发是：

```text
检索前需要理解 query，并选择不同搜索路径。
```

尤其是：

```text
关键词明显的问题不能只做 vector；
metadata 明确的问题必须进 filter；
需要排序质量的问题可以加 rerank。
```

## 3.2 LangChain / LlamaIndex Query Routing 的启发

LangChain 和 LlamaIndex 都有 query routing / router retriever 的思路：根据用户问题选择不同 retriever 或 index。

对 V17 的启发是：

```text
不同问题应该路由到不同 retrieval pipeline，而不是固定一种检索策略。
```

但 V17 做了工程取舍：

```text
不默认使用 LLM router；
先做 heuristic / rule-based router；
保留后续 LLM Query Planner 接口。
```

原因：

```text
本地小模型 qwen2.5:7b 做 planner 稳定性有限；
默认测试不能依赖真实 LLM；
规则型 query understanding 更可控、可解释、可测试。
```

## 3.3 Self-Query Retriever 思路

Self-Query Retriever 的核心思想是：从自然语言 query 中抽取结构化过滤条件，然后用于检索。

对 V17 的启发是：

```text
用户说“V10.0”“最新”“API 文档”“配置项”，这些都应该变成结构化 filter。
```

V17 的实现更轻量：

```text
不用 LLM 生成 filter；
用规则抽取 version、doc_type、codeSymbols、configKeys、apiPaths；
用户手动 filter 优先于自动推断。
```

## 3.4 RAG Fusion / Multi-query Retrieval 的启发

RAG Fusion 和 multi-query retrieval 思路强调：同一个问题可以改写成多个查询，从不同角度召回。

对 V17 的启发是：

```text
一个 query 可以拆出 semanticQuery、keywordQuery、symbolQuery、versionAwareQuery。
```

但 V17 不做复杂 query decomposition，只做轻量 rewrite：

```text
保留原 query；
生成 normalizedQuery；
生成 keywordQuery；
生成 semanticQuery；
如果有符号，保留 symbolQuery；
如果有版本，保留 versionAwareQuery。
```

## 3.5 信息检索中的 Query Classification

传统搜索系统会做 query classification，例如：

```text
navigational；
informational；
transactional；
entity query；
code query；
error query；
freshness query。
```

V17 对应的项目内 query type 是：

```text
VERSION_SPECIFIC；
LATEST_VERSION；
CODE_SYMBOL；
CONFIG_KEY；
API_PATH；
SUMMARY；
MULTI_DOC_COMPARE；
AMBIGUOUS；
NO_ANSWER_RISK。
```

这使系统能根据 query type 决定检索策略。

## 3.6 工程化检索系统中的 Guardrail 思路

很多生产系统不会对所有模糊 query 盲目搜索，而会做：

```text
clarification；
safe fallback；
no-answer；
low confidence warning。
```

对 V17 的启发是：

```text
当 query 太短、没有 collection、没有实体、文档量很大时，系统应该提示用户补充范围，而不是全库搜索。
```

# 4. 核心优化一：Query Type 识别

## 4.1 问题

系统必须先知道用户问的是什么类型的问题。

不同 query type 对应不同检索策略：

```text
版本问题 -> version filter
最新问题 -> status=ACTIVE
代码符号 -> keyword + vector hybrid
配置项 -> keyword boost
总结类 -> higher recall + parent context
模糊问题 -> clarification
无答案风险 -> stronger refusal
```

## 4.2 设计

新增：

```text
QueryType
QueryUnderstandingService
```

QueryType：

```text
SINGLE_DOC_FACT          单文档事实问题
MULTI_DOC_COMPARE        多文档对比问题
VERSION_SPECIFIC         版本限定问题
LATEST_VERSION           最新版本问题
CODE_SYMBOL              类名 / 方法名 / 代码符号问题
CONFIG_KEY               配置项问题
API_PATH                 API 路径问题
METADATA_FILTER          元数据过滤问题
SUMMARY                  总结类问题
TROUBLESHOOTING          故障排查问题
NO_ANSWER_RISK           无答案风险问题
AMBIGUOUS                模糊问题
GENERAL_CHAT             普通闲聊
```

输出：

```text
QueryUnderstandingResult
```

字段：

```text
originalQuery
normalizedQuery
queryType
confidence
collectionHint
versionHint
docTypeHint
sourceHint
statusHint
tags
entities
codeSymbols
configKeys
apiPaths
timeHints
requiresHybrid
requiresRerank
requiresContextExpansion
requiresClarification
noAnswerRisk
warnings
```

## 4.3 第一版识别规则

```text
包含 V10.0 / v10 / 版本 / 第几版
-> VERSION_SPECIFIC

包含 最新 / 当前 / active / deprecated
-> LATEST_VERSION

包含 LocalPythonLlmProvider / RagQualityScore / VectorIdentityService
-> CODE_SYMBOL

包含 application.properties / app.xxx / yml / yaml
-> CONFIG_KEY

包含 /api/xxx / GET / POST / PUT / DELETE
-> API_PATH

包含 对比 / 区别 / 相比 / 哪个更好
-> MULTI_DOC_COMPARE

包含 总结 / 概括 / 归纳 / 梳理
-> SUMMARY

query 太短或没有实体
-> AMBIGUOUS
```

## 4.4 工程权衡

### 为什么不用 LLM 做 query type 分类

因为：

```text
默认测试不能依赖 LLM；
本地 qwen2.5:7b 做 planner 可能不稳定；
规则型分类可解释、可测试、成本低；
当前 query 类型相对明确，规则足够覆盖第一版。
```

后续可以预留：

```text
LlmQueryPlanner
```

但默认不开启。

# 5. 核心优化二：Entity / Metadata Hint 抽取

## 5.1 问题

用户 query 中经常隐含检索约束：

```text
V10.0；
最新；
接口文档；
README；
LocalPythonLlmProvider；
application-local-ai.properties；
GET /documents/upload。
```

如果不抽取，系统只能做裸检索。

## 5.2 设计

新增：

```text
QueryMetadataExtractor
```

抽取：

```text
versionHint
docTypeHint
collectionHint
statusHint
codeSymbols
configKeys
apiPaths
```

## 5.3 抽取规则

### versionHint

```text
V10.0
v10
版本10
最新版本
deprecated
active
```

### docTypeHint

```text
README
manual
API 文档
面试文档
设计文档
需求文档
论文
代码文档
```

### statusHint

```text
latest -> ACTIVE
deprecated -> DEPRECATED
draft -> DRAFT
```

### codeSymbols

识别：

```text
PascalCase
camelCase
Provider / Service / Controller / Repository / DTO / Entity / Config 后缀
```

示例：

```text
LocalPythonLlmProvider
RagQualityScore
VectorIdentityService
```

### configKeys

识别：

```text
app.xxx
spring.xxx
application.properties
application-local-ai.properties
yaml/yml key
```

### apiPaths

识别：

```text
/documents/upload
/rag/evaluation/runs
GET /xxx
POST /xxx
PUT /xxx
DELETE /xxx
```

## 5.4 优先级规则

```text
用户手动选择 filter > 自动推断 filter
```

原因：

```text
用户显式选择 collection / version / doc_type 时，系统不能擅自覆盖；
自动推断只做补充。
```

# 6. 核心优化三：Retrieval Routing Policy

## 6.1 问题

识别 query type 后，还需要决定怎么搜。

不能所有问题都走同一条 pipeline。

## 6.2 设计

新增：

```text
RetrievalRoutingPolicyService
```

输入：

```text
QueryUnderstandingResult
UserSelectedFilters
SystemRetrievalConfig
```

输出：

```text
RetrievalRoutingDecision
```

字段：

```text
strategy
filter
vectorTopK
keywordTopK
finalTopK
rerankTopN
contextExpansion
scoringProfile
clarificationRequired
clarificationQuestion
routingReasons
warnings
```

## 6.3 路由规则

### CODE_SYMBOL / CONFIG_KEY / API_PATH

策略：

```text
HYBRID_RRF_RERANK
```

配置：

```text
keywordTopK=50
vectorTopK=30
finalTopK=8
rerank=true
contextExpansion=ADJACENT
```

原因：

```text
精确符号需要 keyword / BM25；
语义解释需要 vector；
rerank 用于把真正相关的符号上下文排前。
```

### VERSION_SPECIFIC

必须启用：

```text
version filter
```

如果 query 看起来像版本问题但缺少 versionHint：

```text
requiresClarification=true
```

策略：

```text
HYBRID_RRF
```

### LATEST_VERSION

filter：

```text
status=ACTIVE
includeDeprecated=false
```

策略：

```text
HYBRID_RRF_RERANK
```

### SUMMARY

策略：

```text
HYBRID_RRF_RERANK_PARENT_CONTEXT
```

配置：

```text
vectorTopK=80
keywordTopK=40
finalTopK=12
contextExpansion=PARENT_AND_ADJACENT
```

原因：

```text
总结类问题需要更高 Recall 和上下文完整度。
```

### MULTI_DOC_COMPARE

策略：

```text
HYBRID_RRF_RERANK
```

配置：

```text
vectorTopK=80
keywordTopK=50
finalTopK=10
```

额外规则：

```text
保留多个 document 的结果，不要让单个 document 占满 topK。
```

### SINGLE_DOC_FACT

如果有明显 symbol / keyword：

```text
HYBRID_RRF
```

否则：

```text
VECTOR_WITH_METADATA_FILTER
```

### AMBIGUOUS

如果没有 collection 且全局文档很多：

```text
clarificationRequired=true
```

不要默认全库检索。

### NO_ANSWER_RISK

提高拒答倾向：

```text
如果上下文没有依据，请明确说明未找到依据。
```

## 6.4 工程权衡

### 为什么路由策略先做规则

因为：

```text
规则更稳定；
便于测试；
能解释 routingReasons；
本地运行无成本；
适合项目当前阶段。
```

### 为什么不做复杂 planner

复杂 planner 容易演变成 Agent：

```text
多步规划；
工具调用；
多轮推理；
状态管理。
```

这不是 V17 目标。

# 7. 核心优化四：Query Rewrite 轻量版

## 7.1 问题

中文 query 和文档表达经常不一致。

例如：

```text
用户问：V10 的模型供应商能干啥？
文档写：V10.0 Model Provider Settings 支持多模型供应商配置、API Key 管理、默认模型选择和连通性测试。
```

如果直接用原 query，召回可能不稳定。

## 7.2 设计

新增：

```text
QueryRewriteService
```

输出：

```text
rewrittenQueries
```

字段：

```text
originalQuery
normalizedQuery
keywordQuery
semanticQuery
symbolQuery
versionAwareQuery
```

## 7.3 示例

原 query：

```text
V10 的模型供应商能干啥？
```

normalizedQuery：

```text
V10.0 的模型供应商配置支持哪些能力？
```

keywordQuery：

```text
V10.0 模型供应商 API Key 默认模型 连通性测试
```

semanticQuery：

```text
模型供应商配置支持的功能能力
```

## 7.4 关键约束

```text
不默认调用 LLM；
使用规则和同义词表；
保留原 query；
rewrite 不能删除版本、类名、配置项等关键信息；
diagnostics 中展示 rewrite。
```

# 8. 核心优化五：Clarification Guard 防盲搜

## 8.1 问题

用户可能问：

```text
这个怎么做？
怎么优化？
有什么问题？
下一步？
```

如果系统没有上下文，却全库搜索，很容易召回错误项目文档。

## 8.2 设计

新增：

```text
QueryClarificationGuard
```

触发条件：

```text
query 太短；
没有 collection；
没有实体；
文档总量超过阈值；
collection 数量超过阈值；
queryType=AMBIGUOUS；
routing confidence < threshold。
```

配置：

```properties
app.query-understanding.clarification-enabled=true
app.query-understanding.min-confidence=0.55
app.query-understanding.max-global-search-documents=10000
```

行为：

```text
返回 clarificationRequired=true；
不执行 RAG 或只执行保守检索；
页面提示用户选择 collection / version / doc_type。
```

提示示例：

```text
当前问题比较模糊，建议先选择知识库分组，避免跨项目文档混入。
```

## 8.3 工程权衡

### 为什么不默认全库搜索

因为：


# 12. 工程权衡总结

## 12.1 规则型 Query Understanding vs LLM Query Planner

### 规则型

优点：

```text
稳定；
可解释；
可测试；
低成本；
不依赖外部服务；
适合本地项目默认行为。
```

缺点：

```text
覆盖有限；
复杂语义理解弱；
需要维护规则。
```

### LLM Query Planner

优点：

```text
理解能力更强；
可处理复杂自然语言；
能做更复杂 query decomposition。
```

缺点：

```text
依赖模型；
本地小模型稳定性不一定够；
测试复杂；
可能生成不可控 filter；
成本和延迟更高。
```

V17 选择：

```text
默认规则型；
预留 LLM Query Planner 接口；
后续可选启用。
```

## 12.2 防盲搜 vs 用户体验

防盲搜会让用户多一步选择 collection，但好处是：

```text
减少跨集合污染；
降低错误答案；
提高用户对检索范围的感知。
```

V17 的策略：

```text
低置信度 + 大文档量 + 没有 collection 时触发 clarification；
如果用户已经选择 collection，就尽量继续检索。
```

## 12.3 Query Rewrite 的边界

V17 只做轻量 rewrite，不做复杂 query decomposition。

原因：

```text
复杂 query decomposition 可能变成 Agent planning；
容易扩大阶段范围；
第一版目标是提升检索路由，不是做多步推理。
```

## 12.4 自动 filter 的风险

自动抽取 version / doc_type 有误时，可能误过滤正确文档。

所以规则：

```text
用户手动 filter 优先；
自动 filter 要记录 diagnostics；
confidence 低时不强行过滤；
必要时要求用户澄清。
```

# 15. 面试高强度追问与回答

## 15.1 V17 和 V15 的区别是什么？

回答：

```text
V15 优化的是检索主链路，包括结构化 chunk、metadata filter、hybrid retrieval、RRF、rerank 和 context expansion。V17 优化的是查询入口层，也就是用户提问后，系统如何判断 query type、抽取 metadata hints，并选择 V15 中合适的检索策略。
```

## 15.2 为什么需要 Query Understanding？

回答：

```text
因为不同 query 需要不同检索策略。版本问题需要 version filter，代码符号问题需要 BM25 / keyword，最新方案问题要过滤 deprecated，总结类问题要提高 Recall 并回填上下文。没有 Query Understanding，系统只能固定 topK 向量检索，很容易混召回。
```

## 15.3 为什么不直接用 LLM 做 Query Planner？

回答：

```text
默认用 LLM planner 会增加模型依赖、延迟和不确定性。本项目强调本地低依赖和可测试，所以 V17 第一版用 heuristic / rule-based query understanding，保留 LLM Query Planner 接口作为后续增强。
```

## 15.4 如何识别代码符号问题？

回答：

```text
通过 PascalCase、camelCase 以及 Provider、Service、Controller、Repository、DTO、Entity、Config 等后缀识别代码符号。例如 LocalPythonLlmProvider、VectorIdentityService、RagQualityScore。识别后路由到 Hybrid + RRF + Rerank，因为它需要 keyword 精确命中和 vector 语义解释。
```

## 15.5 如何处理版本问题？

回答：

```text
QueryMetadataExtractor 会从 query 中抽取 V10.0、v10、版本10 等 versionHint。RoutingPolicy 会把 versionHint 写入 RetrievalFilter。对于 LATEST_VERSION，会默认 status=ACTIVE、includeDeprecated=false，避免旧版本文档污染答案。
```

## 15.6 为什么模糊问题不直接全库搜索？

回答：

```text
万级文档下，全库搜索很容易跨 collection 召回语义相似但错误的内容。如果 query 太短、没有实体、没有 collection 且置信度低，系统应该触发 clarification，提示用户选择知识库分组，而不是给出看似确定但错误的答案。
```

## 15.7 Query Rewrite 为什么不能删除版本号和类名？

回答：

```text
版本号、类名、配置项是检索约束，不是普通词。如果 rewrite 删除这些信息，就会导致 version filter 失效或 keyword 检索失效。因此 V17 的 rewrite 只做轻量归一化和关键词补充，必须保留 version、code symbol、config key、API path。
```

## 15.8 如何证明 Query Understanding 有效果？

回答：

```text
通过 V14 Knowledge Health 对比。baseline 使用原 query + 固定 retrieval strategy，candidate 使用 QueryUnderstanding + RetrievalRouting。比较 Recall@K、MRR@K、ContextPrecision@K、CrossCollectionLeakRate、WrongVersionLeakRate、CitationAccuracy 和 latency。如果启用 query understanding 后污染率下降、MRR 提升，就说明路由有效。
```

## 15.9 如果自动抽取错了怎么办？

回答：

```text
第一，用户手动 filter 优先级最高；第二，自动抽取有 confidence；第三，低置信度时不会强行过滤，而是触发 clarification；第四，QueryUnderstandingDiagnostics 会展示抽取结果，方便用户和开发者发现错误。
```

## 15.10 V17 和 Agent Planner 有什么区别？

回答：

```text
Agent Planner 通常涉及多步任务规划、工具调用和状态管理。V17 不是 Agent，它只做 RAG 查询入口层的 query type 识别、metadata 抽取、query rewrite 和 retrieval routing，目标是选择正确检索链路，不做复杂任务执行。
```

## 15.11 为什么要把 Query Understanding 结果展示给用户？

回答：

```text
因为 RAG 错误很多时候来自检索范围和策略错误。如果用户能看到系统识别出的 query type、version、doc_type、strategy 和 filter，就能判断系统有没有理解错问题。这也提高了产品可解释性。
```

## 15.12 这个阶段的最大风险是什么？

回答：

```text
最大风险是规则覆盖不全或错误抽取导致误过滤。为控制风险，V17 使用 confidence、manual filter priority、clarification guard 和 diagnostics。低置信度不强搜，用户显式选择不被覆盖，所有路由决策都可解释。
```

# 17. 最终总结

V17.0 可以这样总结：

```text
V17.0 解决的是 RAG 查询入口层的问题。前面 V14-V16 已经让系统具备知识库质量诊断、结构化检索和向量索引防污染能力，但用户自然语言提问时，系统仍然需要判断应该怎么搜。

因此我设计了 Query Understanding Layer，通过规则和启发式识别 query type，抽取 version、doc_type、code symbol、config key、API path 等 metadata hints，并根据 query type 路由到不同检索策略。例如版本问题启用 version filter，最新问题过滤 deprecated，代码符号和配置项问题启用 Hybrid + RRF + Rerank，总结类问题提高 Recall 并启用 context expansion，模糊问题则触发 clarification，避免全库盲搜。

这个阶段的重点不是做复杂 Agent，而是把用户问题转换成可解释、可测试、可诊断的检索决策。最终系统不再是固定向量 TopK，而是能够根据问题类型自动选择检索路径，并通过 V14 Knowledge Health 量化验证查询理解是否真的提升了召回、排序和污染控制。
```
