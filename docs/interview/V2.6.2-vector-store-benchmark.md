# V2.6.2 Vector Search Benchmark: Exact vs Qdrant ANN

## 1. 版本摘要

V2.6.2 的目标是建立 `VectorStore Benchmark`（向量存储评估）框架，用同一套 query-evidence benchmark（查询-证据评测集）对比两个 `VectorStore` 实现：

- baseline（基线）：`ExactCosineVectorStore`
- candidate（候选）：`QdrantVectorStore` 或测试中的 fake candidate

本版本完成的核心能力是：

- 复用已有 `retrieval-corpus-v1.md` 和 `retrieval-benchmark-v1.json`；
- 复用已有 evidence marker 到 expected chunk 的映射逻辑；
- 复用 `RetrievalEvaluationService`，不重新实现 Recall@K（召回率@K）、Precision@K（精确率@K）、MRR（平均倒数排名）、NDCG@K（归一化折损累计增益@K）等指标；
- 输出 baseline summary metrics、candidate summary metrics 和 delta metrics；
- 支持 per-case result（单 case 结果）和多个 topK values；
- 支持 latency（延迟）字段记录，但默认测试中的 fake latency 不代表真实 Qdrant 性能；
- 默认 Maven test 不启动 Docker、不连接真实 Qdrant、不访问外部网络；
- 为后续真实 Qdrant external benchmark、VIP 检索策略、Rerank（重排）和 Hybrid Search（混合检索）打基础。

本版本不是性能结论版本，不声称 Qdrant 一定提升 Recall@K、NDCG@K 或 latency。

---

## 2. 本版本解决的问题

V2.6.1 已经在 `VectorStore` 抽象下接入了 `QdrantVectorStore`。但是，仅仅接入 Qdrant 还不能说明它在本项目的 RAG 检索链路中更有效。

真正需要回答的问题是：

> 在同一套 corpus、同一组 benchmark queries、同一组 expectedEvidenceIds 和同一套 metrics 下，`QdrantVectorStore` 相比 `ExactCosineVectorStore` 的检索质量和延迟表现如何？

如果没有 V2.6.2，只能说：

```text
我接入了 Qdrant。
```

有了 V2.6.2，才能进一步说：

```text
我设计了 exact baseline 与 candidate VectorStore 的对比评估框架，可以用固定 query-evidence benchmark 评估不同向量检索后端在 Recall@K、NDCG@K、MRR 和 latency 上的差异。
```

这说明项目从“工具接入”进入了“检索系统评估闭环”。

Qdrant 官方也提供 ANN Recall 相关教程，强调可以用 exact search 结果作为对照，衡量 ANN（近似最近邻检索）结果对 true nearest neighbors（真实最近邻）的覆盖程度。参考：

- https://qdrant.tech/documentation/tutorials-search-engineering/ann-recall/

## 3. 为什么现在做 V2.6.2

V2.6.2 的位置刚好在 Qdrant 接入之后、检索策略和重排之前。

前置能力已经具备：

| 版本 | 前置能力 |
|---|---|
| V2.4 | Retrieval Evaluation Harness |
| V2.4.1 | Precision@K、NDCG@K 等指标补齐 |
| V2.4.2 | 固定 benchmark seed |
| V2.4.3 | Fixed vs Adaptive Chunking Comparison |
| V2.4.4 | Benchmark Runner / Evidence Mapper |
| V2.5 | EmbeddingProvider 抽象 |
| V2.5.1 | Embedding Provider Benchmark Comparison |
| V2.5.2 | Local Embedding Worker |
| V2.6 | VectorStore 抽象 + Exact baseline |
| V2.6.1 | QdrantVectorStore 接入 |

因此现在可以做：

```text
ExactCosineVectorStore baseline
        ↓
QdrantVectorStore candidate
        ↓
同一套 benchmark 对比
```

如果跳过 V2.6.2 直接做 V2.7 Retrieval Policy / VIP Search，会有几个问题：

1. 不知道 Qdrant 与 exact search 的检索质量差异；
2. 不知道 ANN 是否带来 Recall@K 损失；
3. 不知道是否值得给 VIP 用户更高 topK 或更强检索策略；
4. 不知道是否需要 payload index（元数据索引）；
5. 不知道后续 Rerank 是补救召回问题，还是提升排序质量。

所以 V2.6.2 是从“接入能力”到“评估能力”的必要过渡。

## 4. 方案选择

本版本选择：

> 默认 fake candidate benchmark + optional external Qdrant benchmark。

也就是：

```text
默认 mvn test：
  使用 ExactCosineVectorStore + fake candidate
  验证 benchmark 框架、指标、delta、per-case result、latency 字段
  不连接真实 Qdrant

手工 external benchmark：
  本地启动 Qdrant
  显式配置 qdrant provider
  对比 ExactCosineVectorStore 和 QdrantVectorStore
```

### 4.1 为什么不默认连接真实 Qdrant

默认 CI / Maven test 应该稳定、快速、可重复。如果默认测试依赖真实 Qdrant，就会受到以下因素影响：

- Docker 是否启动；
- 本地端口是否冲突；
- Qdrant collection 是否存在；
- 网络和外部服务是否稳定；
- 测试环境是否具备 Qdrant 数据。

因此，默认测试只验证 benchmark pipeline（评估管线）是否正确，真实 Qdrant benchmark 留给手工 external benchmark。

### 4.2 为什么不只做 fake benchmark

如果只做 fake benchmark，就只能验证框架，不能为真实 Qdrant 评估预留入口。V2.6.1 已经接入 Qdrant，V2.6.2 应该至少在架构上支持 external Qdrant candidate。

### 4.3 为什么不做 Testcontainers

Testcontainers 可以后续引入，但当前版本不做。原因是：

- 会新增依赖；
- 要求 Docker 环境；
- 增加测试耗时；
- 当前版本重点是 benchmark harness，不是容器化集成测试。

### 4.4 为什么不做 payload index benchmark

Qdrant 的 payload index 可以提升过滤查询性能，但这属于过滤性能优化变量。Qdrant 官方文档说明 payload index 可以帮助更高效地按 payload filter 搜索。参考：

- https://qdrant.tech/documentation/manage-data/indexing/

V2.6.2 只做基础 exact vs candidate benchmark，不把 payload index、HNSW 参数调优、Rerank、Hybrid Search 混在一起。

## 5. 核心架构设计

V2.6.2 的核心链路是：

```text
retrieval-corpus-v1.md
    ↓
retrieval-benchmark-v1.json
    ↓
BenchmarkEvidenceMapper
    ↓
expectedEvidenceIds -> expectedChunkIds
    ↓
EmbeddingProvider / MockEmbeddingClient
    ↓
baseline VectorStore: ExactCosineVectorStore
candidate VectorStore: fake candidate / optional QdrantVectorStore
    ↓
RetrievalEvaluationService.evaluate()
    ↓
baseline summary / candidate summary / delta metrics
```

关键原则：

1. 同一套 corpus；
2. 同一套 benchmark cases；
3. 同一套 topKValues；
4. 同一套 expectedEvidenceIds；
5. 同一套 RetrievalEvaluationService；
6. 不重新实现指标。

只有这样，baseline 和 candidate 的对比才有可归因性。

---

## 6. 核心类设计

### 6.1 `VectorStoreBenchmarkRunner`

职责：

- 编排 baseline 和 candidate 两套检索；
- 使用同一份 benchmark dataset；
- 调用 `RetrievalEvaluationService`；
- 输出 baseline summary、candidate summary、delta metrics、per-case results 和 latency 字段。

它不是生产 API，不直接暴露给普通用户。

### 6.2 `BenchmarkEvidenceMapper`

职责：

- 解析 corpus 中的 `[EVIDENCE:id]` marker；
- 将 `expectedEvidenceIds` 映射为当前 chunking / vector store 下的 `expectedChunkIds`；
- 处理重复 marker、缺失 marker 等边界问题。

这个类延续了 V2.4.2 / V2.4.4 的设计：

> benchmark seed 应该基于稳定的 evidence id，而不是运行时生成的 chunkId。

### 6.3 `RetrievalBenchmarkDataset` / `RetrievalBenchmarkCase`

职责：

- 表示从 `retrieval-benchmark-v1.json` 加载出的 benchmark 数据；
- 保存每个 query、expectedEvidenceIds、topKValues 等信息。

### 6.4 `LatencyMeasuringVectorStore`

职责：

- 包装已有 `VectorStore`；
- 统计 search latency；
- 为 benchmark response 提供 latency 字段。

注意：

> 默认测试中的 fake latency 只能验证字段结构和计算逻辑，不能代表真实 Qdrant 性能。

### 6.5 `VectorStoreBenchmarkResponse`

职责：

- 保存完整 benchmark 对比结果；
- 包括 baseline、candidate、metricDeltas、searchLatencyDeltaNanos 等。

### 6.6 `VectorStoreBenchmarkSideResult`

职责：

- 表示某一侧 VectorStore 的评估结果；
- 包括 vectorStoreName、caseCount、topKValues、summary、cases、totalSearchLatencyNanos、searchCount 等。

### 6.7 `VectorStoreMetricDelta`

职责：

- 计算并保存 candidate - baseline 的指标差异；
- 支持每个 K 值的 Recall@K、Precision@K、HitRate@K、MRR、NDCG@K、ContextPrecision@K delta。

### 6.8 `ReversedRankingInMemoryVectorStore`

职责：

- 测试用 fake candidate；
- 模拟 candidate 排序与 baseline 不同；
- 不连接真实 Qdrant；
- 用于验证 delta metrics、per-case results、latency 字段和 benchmark pipeline。

## 7. 数据结构设计

### 7.1 `VectorStoreBenchmarkSideResult`

核心字段：

```text
vectorStoreName
caseCount
topKValues
summary
cases
totalSearchLatencyNanos
searchCount
```

含义：

- `vectorStoreName`：baseline 或 candidate 名称；
- `caseCount`：benchmark case 数量；
- `topKValues`：本次评估的 K 值列表；
- `summary`：复用 `RetrievalEvaluationSummaryResponse`；
- `cases`：复用 `RetrievalEvaluationCaseResultResponse`；
- `totalSearchLatencyNanos`：累计 search latency；
- `searchCount`：search 调用次数。

### 7.2 `VectorStoreMetricDelta`

delta 语义必须是：

```text
candidate - baseline
```

字段包括：

```text
k
recallAtKDelta
precisionAtKDelta
hitRateAtKDelta
mrrDelta
ndcgAtKDelta
contextPrecisionAtKDelta
```

解释：

- delta > 0：candidate 在该指标上高于 baseline；
- delta = 0：两者相同；
- delta < 0：candidate 在该指标上低于 baseline。

### 7.3 `VectorStoreBenchmarkResponse`

核心字段：

```text
datasetId
baseline
candidate
metricDeltas
searchLatencyDeltaNanos
```

其中：

```text
searchLatencyDeltaNanos = candidate - baseline
```

同样不能把默认 fake latency 当成真实 Qdrant 性能。

## 18. 面试高强度追问与回答

### Q1：为什么接入 Qdrant 后还要做 benchmark？

A1：因为接入工具不等于效果更好。Qdrant 解决规模、服务化和 ANN 检索问题，但 ANN 可能用召回换速度。必须用固定 benchmark 对比 Recall@K、NDCG@K、MRR 和 latency，才能判断是否值得引入。

### Q2：为什么 exact search 是 baseline？

A2：exact search 在小数据集下可以作为 correctness baseline。Qdrant ANN 的目标是降低延迟和提升规模能力，但它的结果需要和 exact search 对比，才能知道是否有召回损失。

### Q3：为什么默认测试不连接真实 Qdrant？

A3：默认测试要稳定、快速、可重复。真实 Qdrant 属于外部服务，容易受 Docker、端口、网络、环境影响。默认测试验证 benchmark pipeline，真实 Qdrant 结果放到 optional external benchmark。

### Q4：为什么不直接写“Qdrant 提升 xx%”？

A4：没有真实 benchmark 数据不能写。V2.6.2 的核心是建立评估链路，而不是编造结果。只有在同一 corpus、同一 queries、同一 topK、同一 metrics 下跑出结果，才能写具体提升或下降。

### Q5：为什么不在这个版本做 Rerank？

A5：Rerank 是二阶段排序，VectorStore 是候选召回。两者混在一起后，无法判断指标变化来自 Qdrant 召回还是 rerank 排序。因此必须分版本做。

### Q6：为什么不做 Hybrid Search？

A6：Hybrid Search 会引入 BM25、dense/sparse 分数融合、权重调节等新变量。V2.6.2 只比较 exact vector search 和 candidate vector search，不混入关键词检索变量。

### Q7：如果 Qdrant Recall@K 下降怎么办？

A7：可以提高 topK、调整 HNSW 参数、启用 exact 模式做对照、引入 rerank，或为 VIP 用户提供更高质量策略。但第一步是通过 benchmark 定位问题。

### Q8：为什么 latency 不能只看单元测试？

A8：单元测试里的 fake candidate latency 不代表真实 Qdrant。真实 latency 受 Docker、网络、collection size、payload filter、索引参数影响。默认测试只能验证字段和计算逻辑，真实性能必须用 external benchmark。

### Q9：为什么不新增 benchmark result 表？

A9：V2.6.2 的核心是建立 benchmark harness，不是做评估结果平台。如果现在设计持久化表，会引入数据库迁移、查询接口、清理策略等额外问题。持久化可以作为后续独立版本。

### Q10：如何保证 baseline 和 candidate 对比公平？

A10：必须使用同一套 corpus、同一套 benchmark cases、同一套 expectedEvidenceIds、同一套 topKValues、同一套 RetrievalEvaluationService。只有 candidate VectorStore 不同，指标变化才可归因。

## 21. 本版本在项目中的位置

V2.6.2 是 RAG 检索基础设施阶段的评估闭环版本。

它的位置是：

```text
V2.4.x：Retrieval Evaluation
    ↓
V2.5.x：EmbeddingProvider
    ↓
V2.6：VectorStore 抽象
    ↓
V2.6.1：Qdrant Integration
    ↓
V2.6.2：Vector Search Benchmark: Exact vs Qdrant ANN
    ↓
V2.7：Retrieval Policy & VIP Search
    ↓
V2.8：Rerank
    ↓
V2.9：Generation Evaluation
```

本版本最大的价值不是“跑了一个测试”，而是：

> 把外部 Vector DB 接入纳入可量化、可回归、可对比的检索评估体系，为后续检索策略、VIP 通道和商业化能力提供工程依据。
