# V1.3 面试文档：Model Router

## 1. 本版本解决的问题

V1.3 解决用户请求模型和实际执行模型耦合的问题。用户可以传 requestedModel，但系统需要通过 ModelRouter 选择实际 selected model，并保存 requestedModel 与 llmModel，支持 fallback 和后续策略路由。

## 2. 为什么现在做这个版本

它依赖 LlmClient、Prompt Template 和 usage metadata。只有执行链路能传 model 并保存结果后，路由才有意义。它为成本、质量、延迟、VIP 路由和多 Provider 做准备。此时不做真实多 Provider，是因为仍在 Mock 阶段。

## 3. 核心需求边界

本版本支持 requested_model、ModelRouter、mock-llm/mock-fast/mock-smart 和 unknown fallback。不做真实 OpenAI/Claude/Qwen、不做成本优化算法、不做负载感知路由、不做 KV Cache-aware Scheduling。

## 4. 方案选择

CreateTaskRequest 增加 model，TaskEntity 保存 requestedModel。TaskExecutionService 执行时调用 ModelRouter.route，Prompt Template 变量和 LlmRequest.model 使用 selectedModel。未知模型 fallback 到 mock-llm，不让任务直接失败。

## 5. 替代方案与暂时不选的原因

替代方案一是用户传什么就执行什么。当前不选，因为系统需要控制可用模型和 fallback。

替代方案二是硬编码 mock-llm。当前不选，因为后续无法做模型策略和用户分层。

## 6. 数据库设计

task 表新增 requested_model。llm_model 已保存实际执行模型。两者分离后可以知道用户想用什么、系统实际用了什么。后续可扩展 provider、route_reason、policy_version。

## 7. API 设计

POST /tasks 可传 model，例如 mock-fast。GET /tasks/{id} 返回 requestedModel 和 llmModel。未知模型请求仍可成功，但 llmModel fallback 为 mock-llm。

## 8. 核心流程

用户 POST /tasks 带 model
-> 保存 requestedModel
-> 执行时 ModelRouter.route
-> selectedModel 写入 Prompt variables
-> LlmRequest.model = selectedModel
-> MockLlmClient 返回 model
-> 保存 llmModel

## 9. 测试方案

建议覆盖 model 为空返回 mock-llm、mock-fast/mock-smart 原样返回、unknown fallback、GET 返回 requestedModel 与 llmModel。失败任务仍应走重试。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是要区分 requestedModel 和 llmModel。前者是用户意图，后者是系统决策结果。混在一起会让路由策略和审计失真。

## 12. 本版本亮点

- 引入模型路由抽象。
- 支持未知模型 fallback，提升容错。
- requestedModel / llmModel 解耦。
- 为成本、质量、延迟和 VIP 路由打基础。

## 13. 简历 bullet

### 简洁版

实现 ModelRouter，支持创建任务时传入 requestedModel，并将实际执行模型 llmModel 持久化。

### 强化版

设计模型路由层，将用户请求模型与系统实际执行模型解耦，支持 mock-llm/mock-fast/mock-smart 及未知模型 fallback，为后续多 Provider、成本策略和 VIP 路由奠定基础。

## 14. 面试高强度追问与回答

Q1：为什么 requestedModel 和 llmModel 要分开？

A1：用户请求不一定等于系统实际选择。分开保存能审计 fallback 和策略决策。

Q2：为什么 unknown 不直接失败？

A2：早期系统更适合容错 fallback，避免用户传错模型导致任务失败。生产中可按策略决定。

Q3：ModelRouter 现在做了什么？

A3：它根据 requestedModel 在支持列表中选择模型，空值或未知值回退 mock-llm。

Q4：为什么不做真实成本路由？

A4：还没有真实 Provider、真实价格和效果指标，先搭建路由接口更稳。

Q5：Prompt Template 为什么要用 selectedModel？

A5：模型信息可能影响 prompt 内容，应该使用实际执行模型而非用户原始输入。

Q6：后续如何支持 VIP？

A6：ModelRouter 可以读取用户等级，为 VIP 选择更强模型、更大 topK 或更高优先级。

Q7：如何评价模型路由效果？

A7：需要结合成本、latency、成功率和质量评测，当前版本只做机制闭环。

Q8：当前局限是什么？

A8：只有 Mock 模型，没有多 Provider、负载感知和真实策略。
