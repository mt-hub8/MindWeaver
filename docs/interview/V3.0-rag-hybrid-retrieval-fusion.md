# V3.0 RAG Hybrid Retrieval Fusion 面试文档

## 1. 版本摘要

V3.0 的版本名称是：

```text
V3.0 RAG Hybrid Retrieval Fusion
```

中文可以表述为：

```text
RAG 混合检索与结果融合
```

这个版本的核心目标是：在已有 dense vector retrieval（稠密向量检索）、rerank（重排序）和 evaluation benchmark（评估基准）基础上，引入 app-layer hybrid retrieval（应用层混合检索）能力。

在 V2.9 中，系统已经支持：

```text
query
  -> dense vector search candidateTopK
  -> rerank
  -> finalTopK
  -> citations
  -> prompt
  -> answer
```

V3.0 将检索链路进一步升级为：

```text
query
  -> dense vector retrieval
  -> lexical keyword retrieval
  -> RRF fusion
  -> optional rerank
  -> finalTopK
  -> citations
  -> prompt
  -> answer
```

也就是说，本版本把单一路径的 dense retrieval 扩展为两路召回：

```text
第一路：dense vector retrieval，负责语义相似召回；
第二路：lexical keyword retrieval，负责关键词、类名、配置项、错误码等精确匹配；
然后通过 RRF（Reciprocal Rank Fusion，倒数排名融合）合并两路结果；
最后可选经过 V2.9 的 reranker，再选择 finalTopK 进入 RAG prompt。
```

本版本没有直接引入 Qdrant sparse vector、Elasticsearch、OpenSearch 或真正 BM25 index，而是先做应用层 hybrid retrieval 架构。这样可以避免修改 Qdrant collection schema、避免新增 sparse embedding worker、避免重建数据，同时保持默认测试低依赖。

---

## 2. 为什么现在做 V3.0

V2.8 做了 RAG Retrieval Evaluation & Quality Benchmark（RAG 检索质量评估与基准测试），让系统可以量化检索质量。

V2.9 做了 Two-Stage Retrieval & Rerank（两阶段检索与重排序），让系统可以先扩大候选召回，再对候选结果重新排序。

但 V2.9 仍然主要依赖 dense vector retrieval。dense retrieval 擅长语义相似，但在很多工程型 query 上并不总是最稳。

例如用户问：

```text
Embedding Cache 的 cache key 是什么？
GlobalExceptionHandler 返回哪些字段？
candidateTopK 和 finalTopK 有什么区别？
QdrantVectorStore 是什么？
traceId 在错误响应中有什么作用？
```

这些问题包含很多：

```text
类名；
配置项；
字段名；
版本名；
错误码；
精确技术术语；
英文标识符。
```

这类信息有时 lexical keyword retrieval（关键词检索）更稳，因为它直接关注词项匹配。

所以 V3.0 的合理演进是：

```text
V2.8：先能评估检索质量；
V2.9：再做 rerank 优化排序；
V3.0：进一步引入 dense + lexical hybrid retrieval，提升召回覆盖。
```

这让项目从单一路径检索演进到多路召回与结果融合。

---

## 3. 本版本解决的问题

### 3.1 解决 dense retrieval 对精确术语不稳定的问题

Dense vector retrieval 的优势是语义相似。

例如：

```text
用户问“系统如何避免重复生成 embedding？”
文档中写“Embedding Cache 使用 chunkHash + provider + model + dimension 作为 cache key”
```

即使 query 和文档不完全同词，dense retrieval 也可能召回相关 chunk。

但如果用户问的是：

```text
candidateTopK
finalTopK
GlobalExceptionHandler
BusinessException
traceId
QdrantVectorStore
```

这类精确技术词时，keyword matching 往往更直接。

V3.0 引入 lexical retriever 后，可以增强这类查询的召回能力。

---

### 3.2 解决 rerank 无法挽救未召回结果的问题

V2.9 的 rerank 有一个前提：

```text
相关 chunk 必须先出现在 candidateTopK 里。
```

如果 dense retrieval 第一阶段没有召回到相关 chunk，reranker 再强也无法处理它。

Hybrid retrieval 的作用是增加候选来源：

```text
dense retrieval 找语义相似；
lexical retrieval 找关键词精确匹配；
fusion 把两路结果合并；
rerank 再进一步排序。
```

这可以提高候选池的覆盖面。

---

### 3.3 解决检索优化只有单一路径的问题

没有 hybrid 之前，系统优化主要围绕：

```text
embedding model；
topK；
rerank；
chunking；
```

V3.0 新增了一条新的优化维度：

```text
lexical retrieval + fusion。
```

这让后续可以比较：

```text
dense-only；
dense + rerank；
hybrid；
hybrid + rerank。
```

并继续用 V2.8/V2.9 的 evaluation framework 对比指标变化。

## 4. 核心设计

### 4.1 App-layer Hybrid Retrieval（应用层混合检索）

V3.0 选择做 app-layer hybrid retrieval，而不是直接做 Qdrant sparse vector 或 Elasticsearch。

应用层 hybrid 的流程是：

```text
query
  -> dense retriever
  -> lexical retriever
  -> fusion ranker
  -> optional reranker
  -> final results
```

这种设计的好处是：

```text
不需要修改 Qdrant collection schema；
不需要新增 sparse embedding；
不需要重建历史向量；
不依赖 Elasticsearch / OpenSearch；
不影响默认测试；
更容易在当前项目中快速验证 hybrid 架构。
```

它的定位是先把 hybrid retrieval 的工程骨架打通，后续再考虑更强的 sparse retrieval 或 BM25 检索实现。

---

### 4.2 Dense Retriever

Dense retriever 复用现有 VectorStore search。

它的职责是：

```text
输入 query embedding；
调用 VectorStore；
返回 dense candidates；
每个 candidate 包含 denseRank、denseScore、chunkId、documentId、contentSnippet 等信息。
```

Dense retrieval 的优势是：

```text
语义召回能力强；
不要求 query 和 chunk 完全同词；
适合自然语言问题。
```

局限是：

```text
对精确字段名、类名、配置项、错误码等词项匹配不一定稳定；
相关 chunk 可能语义相似度不高而被漏召回。
```

---

### 4.3 LexicalRetriever 抽象

V3.0 新增 LexicalRetriever 抽象。

它的职责是：

```text
输入 query；
基于关键词 / token overlap 从已有 chunks 中召回 lexical candidates；
输出 lexical rank 和 lexical score。
```

输入至少包含：

```text
query；
lexicalTopK。
```

输出的 lexical candidate 至少包含：

```text
rank；
documentId / documentTitle；
chunkId；
content / contentSnippet；
lexicalScore。
```

抽象出来的好处是：

```text
当前可以使用 SimpleLexicalRetriever；
后续可以替换为 BM25Retriever；
也可以替换为 ElasticsearchRetriever；
还可以替换为 QdrantSparseRetriever；
RAG 主流程不需要绑定具体实现。
```

---

### 4.4 SimpleLexicalRetriever

V3.0 默认实现一个低依赖的 SimpleLexicalRetriever。

它的基本思路是：

```text
对 query 做简单 tokenization；
对 chunk content 做简单 tokenization；
计算 query tokens 与 chunk tokens 的 overlap；
根据命中 token 数、term frequency 或 contains 命中情况计算 lexicalScore；
按 lexicalScore 排序；
返回 lexicalTopK。
```

它的优点是：

```text
无外部依赖；
不需要搜索引擎；
不需要改数据库 schema；
不需要 Python worker；
不需要模型下载；
分数可解释；
测试稳定。
```

它的局限是：

```text
不是真正 BM25；
没有倒排索引；
数据量大时效率有限；
中文分词能力有限；
同义词和语义泛化能力弱；
主要用于验证 hybrid retrieval 的工程链路。
```

面试时要强调：

```text
SimpleLexicalRetriever 不是最终搜索引擎方案，而是一个低依赖、可测试、可替换的 lexical retrieval baseline。
```

---

### 4.5 FusionRanker 与 RRF Fusion

V3.0 新增 FusionRanker 或等价融合组件。

本版本优先实现：

```text
RRF（Reciprocal Rank Fusion，倒数排名融合）
```

RRF 的核心思想是：

```text
一个 chunk 在某一路结果中排名越靠前，贡献分越高；
如果同一个 chunk 在 dense 和 lexical 两路都出现，则两路贡献分相加；
最终按 fusionScore 排序。
```

公式可以理解为：

```text
fusionScore = Σ 1 / (k + rank_i)
```

其中：

```text
rank_i 是 chunk 在第 i 路检索结果中的排名；
k 是平滑常数，例如 60。
```

RRF 的优点是：

```text
不需要把 denseScore 和 lexicalScore 强行归一化到同一尺度；
只依赖排名；
适合融合不同检索来源；
实现简单、稳定、可测试。
```

融合后的结果应尽量保留：

```text
denseRank；
lexicalRank；
denseScore；
lexicalScore；
fusionScore；
denseHit；
lexicalHit。
```

这些字段有助于解释某个 chunk 为什么被排到前面。

## 5. RAG Answer API 集成

### 5.1 默认 hybrid disabled

V3.0 的默认配置应该是：

```properties
rag.hybrid.enabled=false
```

默认关闭 hybrid 时，系统保持 V2.9 行为不变：

```text
dense retrieval
  -> optional rerank
  -> finalTopK
```

这样可以保证兼容性，也避免对已有测试和演示造成影响。

---

### 5.2 hybrid enabled

当配置为：

```properties
rag.hybrid.enabled=true
```

系统执行：

```text
query
  -> dense retrieval denseTopK
  -> lexical retrieval lexicalTopK
  -> RRF fusion
  -> optional rerank
  -> finalTopK
  -> citations
  -> prompt
  -> answer
```

如果 rerank 关闭：

```text
dense + lexical -> fusion -> finalTopK
```

如果 rerank 开启：

```text
dense + lexical -> fusion -> rerank -> finalTopK
```

这说明 V3.0 没有替代 V2.9，而是和 V2.9 形成组合：

```text
hybrid 负责扩大和融合候选来源；
rerank 负责进一步排序候选。
```

---

### 5.3 不改变必填请求字段

V3.0 不改变 `POST /rag/answers` 的必填请求字段。

原因是：

```text
hybrid retrieval 是后端检索策略；
不应该要求前端和调用方每次都传复杂策略参数；
现有 Web UI 不需要改；
默认关闭时保持行为不变；
策略切换通过配置完成。
```

这符合项目一直保持的兼容性原则。

## 6. Retrieval Metadata 设计

V3.0 需要增强 retrieval metadata，让 hybrid 过程可解释。

建议 metadata 至少包含：

```text
hybridEnabled；
denseTopK；
lexicalTopK；
finalTopK；
fusionStrategy；
denseCandidateCount；
lexicalCandidateCount；
fusedCandidateCount；
hybridLatencyMs。
```

如果实现支持，citations 或 retrieved items 可以包含：

```text
denseRank；
lexicalRank；
denseScore；
lexicalScore；
fusionScore；
denseHit；
lexicalHit。
```

这些字段可以回答：

```text
这个结果是 dense 命中的，还是 lexical 命中的？
它在 dense 结果里排第几？
它在 lexical 结果里排第几？
它是否两路都命中？
fusion 后为什么排到前面？
```

这对调试和面试展示都很重要。

## 7. Evaluation 对比设计

V3.0 必须复用 V2.8/V2.9 的 evaluation framework。

推荐对比模式：

```text
dense-only baseline；
hybrid；
dense + rerank；
hybrid + rerank。
```

至少必须支持：

```text
dense-only vs hybrid。
```

对比报告应包含：

```text
baseline summary metrics；
hybrid summary metrics；
delta metrics；
improved cases；
regressed cases；
missed cases。
```

指标继续沿用：

```text
HitRate@K（命中率@K）；
Recall@K（召回率@K）；
Precision@K（精确率@K）；
RR@K（倒数排名@K）；
MRR（平均倒数排名）。
```

重点仍然是：

```text
不承诺 hybrid 一定提升；
但必须能真实记录指标变化。
```

如果 hybrid 提升 MRR，说明相关结果排序更靠前。

如果 hybrid 提升 Recall@K，说明找回了更多 expected items。

如果 hybrid 降低 Precision@K，说明可能引入了更多噪声。

这就是评估框架的价值。

## 9. 关键取舍

### 9.1 为什么不直接做 Qdrant sparse vector

Qdrant sparse vector 可以实现更正式的 sparse retrieval，但会引入：

```text
collection schema 修改；
sparse vector 写入；
已有数据重建；
sparse embedding provider；
索引配置；
更多 manual verification；
更复杂的测试边界。
```

当前项目更适合先做 app-layer hybrid retrieval，把架构链路打通。

后续如果要接 Qdrant sparse vector，可以在 V3.1 或 V3.2 单独做。

---

### 9.2 为什么不接 Elasticsearch / OpenSearch

Elasticsearch / OpenSearch 可以提供成熟的 BM25 和倒排索引能力。

但接入它们会引入：

```text
新基础设施；
docker compose 修改；
索引 schema；
数据同步；
MySQL / Qdrant / Elasticsearch 多存储一致性；
CI 和 manual 验证复杂度。
```

V3.0 先做低依赖 lexical retriever，是为了控制工程边界。

---

### 9.3 为什么不做真正 BM25

BM25 比 simple token overlap 更成熟，但也需要更完整的倒排索引、文档频率统计、长度归一化等设计。

V3.0 不做真正 BM25，是因为本版本目标不是建立完整搜索引擎，而是建立 hybrid retrieval 架构：

```text
lexical retriever abstraction；
dense + lexical candidates；
fusion；
evaluation comparison。
```

后续可以把 SimpleLexicalRetriever 替换成 BM25Retriever。

---

### 9.4 为什么不改 Web UI

V3.0 不改 Web UI。

原因是：

```text
hybrid 是后端检索策略；
现有 /rag-demo.html 已经可以展示 answer、citations 和 metadata；
metadata 中新增 hybrid 字段后，页面可以通过原有 metadata 区域看到信息；
不需要新增前端开关。
```

这避免把前端改动混入检索能力版本。

## 10. 当前局限

V3.0 有明确局限：

```text
SimpleLexicalRetriever 不是 BM25；
没有倒排索引；
大规模数据下 lexical scan 可能效率有限；
不做 Qdrant sparse vector；
不接 Elasticsearch / OpenSearch；
不做 Web UI hybrid 开关；
不做 dashboard；
不评估 answer quality；
不做 LLM-as-judge；
不保证 hybrid 一定提升所有指标。
```

这些不是缺陷，而是刻意控制边界。

V3.0 解决的是：

```text
让 RAG 系统支持 dense + lexical 多路召回、结果融合和质量对比。
```

## 11. 产品化意义

V3.0 的产品化意义是：

```text
让 RAG 系统从单一 dense retrieval 进入 hybrid retrieval 架构。
```

版本链可以这样理解：

```text
V2.7：RAG 能回答；
V2.8：RAG 检索质量能评估；
V2.9：RAG 能 rerank 优化排序；
```

这已经接近真实企业 RAG 系统的检索演进路线。

```text
dense retrieval；
keyword retrieval；
fusion；
rerank；
evaluation。
```

V3.0 就是在项目里建立这条工程主线。

## 13. 面试高强度追问与回答

### 13.1 为什么要做 Hybrid Retrieval？

回答：

```text
Dense retrieval 擅长语义相似，但对类名、配置项、错误码、字段名、版本号等精确术语不一定稳定。Lexical retrieval 擅长关键词匹配，可以补充 dense retrieval 的不足。

所以我在 V3.0 引入 app-layer hybrid retrieval，同时召回 dense candidates 和 lexical candidates，再用 RRF fusion 合并结果，最后可选 rerank。这让系统既能处理自然语言语义问题，也能更好地处理工程技术术语问题。
```

---

### 13.2 Hybrid Retrieval 和 V2.9 Rerank 有什么区别？

回答：

```text
Rerank 是对已经召回的候选结果重新排序，它不能处理“相关 chunk 没有被召回”的问题。

Hybrid Retrieval 是增加候选来源。Dense retrieval 找语义相似结果，lexical retrieval 找关键词匹配结果，然后 fusion 合并两路候选。

简单说，hybrid 解决召回来源问题，rerank 解决候选排序问题。
```

---

### 13.3 为什么不直接用 Qdrant sparse vector？

回答：

```text
Qdrant sparse vector 是更正式的 sparse retrieval 方案，但它需要修改 collection schema、写入 sparse vectors、可能重建已有数据，还要新增 sparse embedding provider 和 manual verification。

当前版本的目标是先打通 hybrid retrieval 架构，所以我选择 app-layer hybrid，不改变 Qdrant schema，也不引入新的 worker。后续如果架构稳定，可以单独升级到 Qdrant sparse vector。
```

---

### 13.4 为什么不接 Elasticsearch？

回答：

```text
Elasticsearch 可以提供成熟的 BM25 和倒排索引能力，但它会引入新的基础设施、索引 schema、数据同步和运维复杂度。

当前项目已经有 MySQL、Qdrant、Embedding Worker 和 Spring Boot。如果 V3.0 再引入 Elasticsearch，版本边界会过大。所以我先做低依赖 SimpleLexicalRetriever 和 RRF fusion，验证 hybrid 架构价值。
```

---

### 13.5 SimpleLexicalRetriever 是不是太简单？

回答：

```text
它确实不是完整搜索引擎，也不等价于 BM25。它的定位是低依赖 lexical retrieval baseline，用来打通 LexicalRetriever 抽象、hybrid retrieval 流程、RRF fusion 和 evaluation 对比。

它的价值是工程骨架，而不是最终检索算法。后续可以替换为 BM25Retriever、ElasticsearchRetriever 或 QdrantSparseRetriever。
```

---

### 13.6 RRF 是什么？

回答：

```text
RRF 全称是 Reciprocal Rank Fusion，中文可以叫倒数排名融合。它根据结果在不同检索列表中的排名来计算融合分数。

某个 chunk 在 dense 结果中排名越靠前，得分越高；在 lexical 结果中排名越靠前，也得分越高。如果同一个 chunk 在两路都靠前，分数会叠加，因此更容易排到最终结果前面。

它的优点是不需要把 denseScore 和 lexicalScore 强行归一化到同一尺度。
```

---

### 13.7 为什么不用简单加权 denseScore + lexicalScore？

回答：

```text
因为 denseScore 和 lexicalScore 通常不在同一个尺度上。denseScore 可能是 cosine similarity，lexicalScore 可能是 token overlap，它们直接相加需要复杂归一化。

RRF 基于排名而不是原始分数，更适合融合不同检索来源。
```

---

### 13.8 Hybrid 一定能提升指标吗？

回答：

```text
不一定。Hybrid 可能提升 Recall@K，因为它增加了关键词召回来源；也可能降低 Precision@K，因为它引入了更多候选噪声。

所以我复用 V2.8/V2.9 的 evaluation framework，对比 dense-only 和 hybrid 的 HitRate@K、Recall@K、Precision@K 和 MRR。重点不是主观宣称提升，而是用报告真实记录 improved / regressed cases。
```

---

### 13.9 hybrid 和 rerank 同时启用时顺序是什么？

回答：

```text
顺序是先 hybrid，再 rerank。

具体来说，先分别执行 dense retrieval 和 lexical retrieval，然后用 RRF fusion 合并候选结果。如果 rerank 开启，再对 fused candidates 进行 rerank，最后取 finalTopK 构造 citations 和 prompt。

也就是：dense + lexical -> fusion -> optional rerank -> finalTopK。
```

---

### 13.10 为什么默认关闭 hybrid？

回答：

```text
为了保证兼容性和可控性。V2.9 已经有稳定的 dense + optional rerank 行为，如果默认开启 hybrid，会改变现有 RAG Answer API 的检索结果和测试预期。

所以 hybrid 默认关闭，只有显式配置启用时才走 dense + lexical + fusion。
```

---

### 13.11 metadata 为什么要暴露 hybrid 信息？

回答：

```text
Hybrid retrieval 会合并多个来源。如果不暴露 metadata，用户无法知道某个 citation 是 dense 命中的、lexical 命中的，还是两路都命中后被 fusion 提上来的。

通过 hybridEnabled、denseRank、lexicalRank、denseScore、lexicalScore、fusionScore、denseHit、lexicalHit 等字段，可以让检索链路更可解释，也方便调试。
```

---

### 13.12 这个版本体现什么工程能力？

回答：

```text
这个版本体现的是检索系统架构能力。它不是简单调一个向量库接口，而是把 RAG 检索拆成 dense retrieval、lexical retrieval、fusion、optional rerank 和 evaluation 对比几个模块。

同时我没有直接引入重基础设施，而是先用 app-layer hybrid 方式控制复杂度，保持默认测试低依赖，并用 benchmark 验证 dense-only 和 hybrid 的质量差异。
```

## 16. 总结表达

这个版本可以用一段话总结：

```text
在 V3.0 中，我将 RAG 检索链路从单一路径 dense vector retrieval 升级为 app-layer hybrid retrieval。系统会同时执行 dense vector retrieval 和 lexical keyword retrieval，再通过 RRF（Reciprocal Rank Fusion，倒数排名融合）对两路结果去重合并，最后可选经过 V2.9 的 reranker，选择 finalTopK 构造 citations 和 RAG prompt。

我没有一开始引入 Qdrant sparse vector、Elasticsearch 或真正 BM25，而是先抽象 LexicalRetriever 和 FusionRanker，使用低依赖 SimpleLexicalRetriever 与 RRF fusion 打通 hybrid 架构，同时复用 V2.8/V2.9 的 evaluation benchmark 对比 dense-only 和 hybrid 的 HitRate@K、Recall@K、Precision@K、MRR 变化。

这个版本体现的是 RAG 检索系统从单路向量召回，向多路召回、结果融合、可解释 metadata 和质量评估闭环的演进。
```
