# V4.0 Knowledge Base Lifecycle Management 面试文档

## 1. 阶段摘要

V4.0 的阶段名称是：

```text
V4.0 Knowledge Base Lifecycle Management
```

中文可以表述为：

```text
知识库生命周期管理
```

这个阶段的核心目标是：让 AI Knowledge Assistant 不只是能上传文档、处理文档、基于文档问答，还能管理文档的生命周期。

在 V3.x 阶段，系统已经具备：

```text
文档上传；
文本提取；
异步摄入；
处理状态追踪；
事件时间线；
摄入性能分析；
Ask 页面知识库问答；
citations 引用来源；
rerank；
hybrid retrieval；
```

但此时系统还有一个明显缺口：

```text
文档上传后如何下线？
文档过期后如何不再参与问答？
文档内容变化后如何重新建立索引？
重新索引后旧片段如何避免继续被检索？
```

V4.0 解决的就是这些问题。

本阶段把文档从“只能上传和问答”推进到“可删除、可重新索引、可过滤旧版本、可解释生命周期状态”的知识库管理能力。

---

## 2. V4.0 包含的 Batch

V4.0 不再拆成多个小版本文档，而是作为一个大阶段推进，内部包含 3 个 Batch。

### Batch 1：Document Soft Delete & Ask Filter

目标：

```text
支持文档软删除，并确保已删除文档不会再进入 Ask / RAG 的最终上下文。
```

核心能力：

```text
DocumentStatus：ACTIVE / DELETED；
DELETE /documents/{documentId}；
GET /documents 返回 status / displayStatus；
Documents 页面中文删除入口；
Ask / RAG 检索过滤 DELETED 文档；
citations 不包含已删除文档；
hybrid / rerank 不绕过删除过滤。
```

---

### Batch 2：Document Re-index / Re-ingestion

目标：

```text
支持对已启用文档重新建立索引，并确保重新索引后只使用最新有效 chunks。
```

核心能力：

```text
POST /documents/{documentId}/reindex；
复用 source text；
chunk generation / active status；
old chunks 标记 inactive / superseded；
new chunks 成为 current active chunks；
Ask / RAG 过滤旧 chunks；
reindex task / event timeline；
Documents 页面中文“重新建立索引”入口。
```

---

### Batch 3：Lifecycle UI + Events + Tests + Manual 收敛

目标：

```text
把软删除、重新索引、生命周期状态、事件时间线、处理分析入口统一成完整中文友好的知识库生命周期管理体验。
```

核心能力：

```text
Documents 页面生命周期管理收敛；
Ask 页面生命周期过滤说明；
删除 / reindex 事件中文展示；
analytics 入口保留；
manual 文档补齐；
阶段级回归测试；
V4.0 阶段总验收。
```

## 3. 为什么要做 V4.0

一个知识库问答系统如果只能上传文档和提问，还不够完整。

真实使用时，用户一定会遇到：

```text
上传错了文档；
文档过期了；
文档内容更新了；
旧文档不应该再参与问答；
重新处理失败后不能影响旧知识库；
向量库里残留旧向量；
Ask 引用了已经删除或过期的内容。
```

如果不解决这些问题，系统会出现严重的产品风险：

```text
用户删除了文档，但系统仍然引用它；
文档重新索引后，旧 chunks 仍然进入回答；
向量库中旧 vectors 没有物理删除，导致搜索污染；
用户不知道当前文档是否可用于问答。
```

V4.0 的目标是建立明确的 lifecycle boundary（生命周期边界）：

```text
只有 ACTIVE 文档可以参与问答；
只有 current active chunks 可以进入 final context；
已删除文档不会再进入 citations；
旧版本 chunks 不会再进入 prompt；
即使底层向量未物理删除，也通过 application-level filter 过滤。
```

这让系统从“能用”进一步走向“可管理”。

## 4. 核心设计一：文档软删除

### 4.1 为什么选择软删除

V4.0 选择 soft delete（软删除），而不是 physical delete（物理删除）。

软删除的语义是：

```text
不删除 document 记录；
不删除 chunks；
不删除 embedding cache；
不删除 VectorStore 中的 vectors；
只把 document.status 改成 DELETED；
Ask / RAG 层过滤该文档。
```

这样做的原因是：

```text
保留历史记录；
避免误删不可恢复；
避免同时处理 MySQL、VectorStore、embedding cache 的复杂一致性；
降低本阶段实现风险；
后续可以单独做物理清理或回收站。
```

物理删除看似简单，但实际上涉及：

```text
document 表；
chunk 表；
embedding cache；
VectorStore vectors；
event timeline；
analytics history；
正在处理中的 ingestion task；
失败任务；
重试任务。
```

所以 V4.0 先做软删除是合理边界。

---

### 4.2 文档状态模型

V4.0 引入文档状态：

```text
ACTIVE：已启用；
DELETED：已删除。
```

默认新上传文档为：

```text
ACTIVE
```

删除后：

```text
status = DELETED；
deletedAt = now；
updatedAt = now。
```

前端展示：

```text
ACTIVE -> 已启用；
DELETED -> 已删除。
```

页面不能只展示 ACTIVE / DELETED 这种英文枚举，而要显示中文状态，让普通用户看懂文档是否还参与问答。

---

### 4.3 删除 API

V4.0 新增或改造：

```http
DELETE /documents/{documentId}
```

语义：

```text
软删除文档；
不物理删除底层数据；
幂等处理已删除文档；
不存在 document 返回统一错误响应。
```

成功响应包含：

```text
documentId；
status；
displayStatus；
message；
deletedAt。
```

示例语义：

```text
文档已删除，后续知识库问答不会再使用该文档。
```

## 5. 核心设计二：Ask / RAG 生命周期过滤

### 5.1 为什么必须做 application-level filter

V4.0 不物理删除 VectorStore 中的 vectors。

这意味着：

```text
已删除文档对应的 vectors 仍然可能被 VectorStore 检索出来。
```

如果只做 document 表软删除，但 RAG 检索不做过滤，就会出现：

```text
用户删除了文档，但 Ask 仍然引用该文档。
```

这是严重错误。

因此，V4.0 必须在应用层统一过滤：

```text
document.status 必须是 ACTIVE；
chunk 必须是当前有效 chunk；
否则不能进入 final context / citations / prompt。
```

这就是 application-level lifecycle filter（应用层生命周期过滤）。

---

### 5.2 过滤位置

过滤必须发生在：

```text
finalTopK；
citations；
prompt context；
retrieval metadata final results。
```

也就是说，即使 VectorStore search 返回了已删除文档或旧 chunks，也不能进入最终答案上下文。

过滤必须覆盖：

```text
dense retrieval；
lexical retrieval；
hybrid fusion；
rerank；
citations；
prompt construction。
```

如果只在某一路过滤，另一条路径仍然可能绕过。

---

### 5.3 被删除文档的行为

删除后：

```text
GET /documents 可以继续看到该文档状态为“已删除”；
Ask 不再使用该文档；
citations 不再引用该文档；
VectorStore 中旧 vectors 可以暂时保留；
历史事件和 analytics 仍然可以保留。
```

这既保证用户体验，也避免物理删除复杂度。

---

## 6. 核心设计三：文档重新索引

### 6.1 为什么需要重新索引

文档重新索引用于处理：

```text
chunking 策略变了；
embedding model 变了；
VectorStore 写入策略变了；
旧索引质量不好；
文档处理逻辑升级；
需要重新生成 chunks / embeddings / vectors。
```

如果没有 reindex，用户只能重新上传文件，体验差，也不利于管理同一个 document 的生命周期。

V4.0 新增：

```http
POST /documents/{documentId}/reindex
```

中文页面按钮：

```text
重新建立索引
```

---

### 6.2 source text 复用

重新索引必须有原始文本来源。

V4.0 要求上传阶段保存 source text 或等价原始文本字段。

支持来源：

```text
.txt 读取后的文本；
.md 读取后的文本；
文本型 PDF 提取后的文本。
```

如果旧文档缺少 source text，reindex 必须返回清晰错误：

```text
当前文档缺少原始文本，无法重新建立索引。请重新上传文档。
```

不能要求用户每次 reindex 都重新上传文件。

---

### 6.3 reindex 的文档状态规则

只允许 ACTIVE 文档重新索引。

规则：

```text
ACTIVE 文档：可以 reindex；
DELETED 文档：不能 reindex；
source text 缺失：不能 reindex；
已有 PENDING / PROCESSING reindex task：拒绝重复提交。
```

对应中文提示：

```text
当前文档已删除，不能重新索引。
当前文档缺少原始文本，无法重新索引。
该文档已有重新索引任务正在处理。
```

---

## 7. 核心设计四：chunk generation / active filtering

### 7.1 为什么需要 chunk 版本或状态

重新索引会生成新的 chunks 和 vectors。

但 V4.0 不物理删除旧 chunks 和 old vectors。

因此必须解决：

```text
旧 chunks 如何不再进入 Ask？
新 chunks 如何成为当前有效版本？
VectorStore 返回旧 vectors 时如何过滤？
```

V4.0 的设计是给 chunk 增加 lifecycle 标识：

```text
chunk_status：ACTIVE / SUPERSEDED；
generation；
document_revision；
active boolean。
```

具体字段可以因项目而异，但必须满足：

```text
当前有效 chunks 可识别；
旧 chunks 可识别；
Ask / RAG 只使用当前有效 chunks。
```

---

### 7.2 current generation 设计

document 可以维护：

```text
currentGeneration；
reindexCount；
lastReindexedAt。
```

reindex 成功后：

```text
currentGeneration + 1；
reindexCount + 1；
lastReindexedAt = now；
新 chunks 绑定 currentGeneration；
旧 chunks 标记为 inactive / superseded。
```

这样系统能判断：

```text
哪个 chunk 是当前有效版本；
哪个 chunk 是旧版本。
```

---

### 7.3 reindex 的一致性策略

推荐安全策略是：

```text
先生成新 chunks；
生成 embeddings；
写入 VectorStore；
全部成功后再切换 currentGeneration；
旧 chunks 标记为 SUPERSEDED；
新 chunks 标记为 ACTIVE。
```

这样如果 reindex 失败，旧索引仍然可以继续用于问答。

这比一开始就禁用旧 chunks 更安全。

如果 reindex 失败，页面应该说明：

```text
重新索引失败，系统会保留旧索引继续用于问答。
```

这体现了可用性优先的设计。

## 8. 核心设计五：生命周期事件

V4.0 扩展 event timeline，使删除和重新索引也能被追踪。

### 8.1 删除事件

建议事件：

```text
DOCUMENT_DELETE_REQUESTED；
DOCUMENT_DELETED。
```

中文展示：

```text
用户请求删除文档；
文档已删除，后续不会再用于知识库问答。
```

---

### 8.2 reindex 事件

建议事件：

```text
DOCUMENT_REINDEX_REQUESTED；
DOCUMENT_REINDEX_QUEUED；
DOCUMENT_REINDEX_STARTED；
DOCUMENT_REINDEX_COMPLETED；
DOCUMENT_REINDEX_FAILED。
```

中文展示：

```text
用户请求重新建立索引；
重新索引任务已进入处理队列；
开始重新切分文档内容；
重新索引完成，新的文档片段已可用于问答；
重新索引失败，系统将保留旧索引继续用于问答。
```

这样用户和开发者都能在处理记录中看到生命周期操作历史。

## 12. 关键取舍

### 12.1 为什么不做物理删除

物理删除涉及多个存储：

```text
MySQL document；
MySQL chunks；
embedding cache；
VectorStore vectors；
event timeline；
analytics history。
```

如果只删一部分，会造成数据不一致。

V4.0 选择软删除 + 应用层过滤，是为了：

```text
快速实现生命周期语义；
保留历史；
避免复杂一致性；
降低误删风险；
为后续物理清理留接口。
```

后续可以单独做 vector cleanup 或 hard delete。

---

### 12.2 为什么旧 vectors 仍然保留

因为本阶段不做 VectorStore delete-by-id。

旧 vectors 留在 VectorStore 中，但通过应用层过滤排除。

这要求：

```text
VectorStore 返回候选后，系统必须回查 document / chunk 状态；
只有 ACTIVE document + current active chunk 才能进入 final context。
```

这是一种合理的中间方案。

---

### 12.3 为什么不做恢复文档

恢复文档会引入：

```text
恢复 DELETED 文档；
恢复旧 chunks；
检查 old vectors 是否仍然可用；
和 reindex 状态冲突；
和删除事件语义冲突；
前端确认逻辑。
```

这超出 V4.0。

---

### 12.4 为什么不做版本 diff

reindex 后可能有多个 generation，但 V4.0 不做复杂版本比较。

原因是：

```text
当前目标是保证 Ask 只使用当前版本；
不是做文档版本管理系统。
```

版本 diff 可以后续作为独立功能。

---

### 12.5 为什么不做 workspace / collection

workspace / collection 会影响：

```text
上传；
文档管理；
Ask 检索范围；
VectorStore metadata filter；
权限；
analytics；
lifecycle 操作。
```

V4.0 先解决单知识库生命周期。

## 14. 当前局限

V4.0 明确不做：

```text
不做物理删除；
不做 VectorStore delete-by-id；
不做删除 chunks；
不做恢复文档；
不做文档版本 diff；
不做文档内容在线编辑；
不做批量删除；
不做批量重新索引；
不做 workspace / collection；
不做权限；
不做新的上传格式；
不做新的检索算法；
不改 RAG Answer API 请求结构；
不引入前端框架。
```

这些边界保证 V4.0 聚焦知识库生命周期核心能力。


用户可以：

```text

确保已删除文档不再参与问答；
```

这让 AI Knowledge Assistant 更接近真实知识库产品。

```text
跨 dense / lexical / hybrid / rerank 的统一过滤；
```

## 18. 面试高强度追问与回答

### 18.1 为什么选择软删除，而不是直接删除数据？

回答：

```text
因为文档删除涉及多处数据：document、chunks、embedding cache、VectorStore vectors、event timeline 和 analytics history。如果直接物理删除，很容易出现部分删除成功、部分删除失败，导致数据不一致。

所以 V4.0 先做软删除，把 document 状态标记为 DELETED，并在 RAG retrieval 层过滤它。这样既保留历史记录，又能保证已删除文档不会再参与问答。
```

---

### 18.2 如果 VectorStore 里旧向量还在，为什么 Ask 不会引用它？

回答：

```text
因为我在 retrieval service 层做了 application-level filter。VectorStore 返回 candidates 后，系统会根据 documentId 和 chunkId 回查数据库状态，只保留 ACTIVE document 和 current active chunks。即使旧 vectors 仍在向量库中，也不会进入 finalTopK、citations 或 prompt。
```

---

### 18.3 为什么过滤要覆盖 hybrid 和 rerank？

回答：

```text
因为系统已经支持 dense retrieval、lexical retrieval、hybrid fusion 和 rerank。如果只在某一路过滤，其他路径可能把已删除文档或旧 chunks 重新带入最终结果。

所以 lifecycle filter 必须放在统一的 retrieval candidate validation 层，保证所有路径进入 final context 前都经过状态校验。
```

---

### 18.4 reindex 为什么需要 chunk generation？

回答：

```text
重新索引会生成新的 chunks 和 vectors，但旧 chunks 和 old vectors 暂时不会物理删除。为了区分当前有效版本和旧版本，需要 generation 或 active status。

这样 Ask 只使用 currentGeneration 的 active chunks，旧 chunks 即使被 VectorStore 检索出来，也会被过滤掉。
```

---

### 18.5 reindex 失败时如何保证系统可用？

回答：

```text
推荐策略是先生成新 chunks、生成 embeddings、写入 VectorStore，全部成功后再切换 currentGeneration，并把旧 chunks 标记为 inactive。如果中途失败，旧 active chunks 仍然可用，文档不会进入无可用索引的状态。
```

---

### 18.6 为什么 DELETED 文档不能 reindex？

回答：

```text
DELETED 的语义是该文档已经下线，不再用于问答。如果允许 DELETED 文档 reindex，会让生命周期语义变得混乱。恢复文档应该是单独的能力，而不是通过 reindex 隐式恢复。
```

---

### 18.7 为什么不做 VectorStore delete-by-id？

回答：

```text
VectorStore 物理删除会引入额外一致性问题，比如 MySQL 状态更新成功但向量删除失败，或者向量删除成功但 chunks 状态更新失败。V4.0 先用软删除和应用层过滤保证用户语义正确。后续可以单独做向量清理任务。
```

---

### 18.8 source text 为什么重要？

回答：

```text
reindex 需要重新 chunking 和 embedding。如果没有保存 source text，就必须要求用户重新上传文件，这不是真正的 reindex。因此 V4.0 要求上传时保存提取后的原始文本，后续可以复用它重新建立索引。
```

---

### 18.9 这个版本和 V3.3 / V3.4 有什么关系？

回答：

```text
V3.3 提供异步 ingestion task，V3.4 提供事件时间线。V4.0 的 reindex 复用这些能力，把重新索引也建模成异步任务，并记录 reindex requested、queued、started、completed、failed 等事件。这样 lifecycle 操作和文档摄入使用同一套任务与可观测性基础设施。
```

---

### 18.10 这个版本体现什么工程能力？

回答：

```text
它体现的是数据生命周期管理和 RAG 检索一致性设计。RAG 系统不只是把文档写进向量库，还要处理文档删除、重新索引、旧版本过滤和向量残留问题。V4.0 通过软删除、chunk generation、current active filtering 和统一 retrieval filter，保证问答结果只基于当前有效知识。
```
