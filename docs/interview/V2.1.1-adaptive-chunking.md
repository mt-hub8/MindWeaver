# V2.1.1 面试文档：Adaptive Chunking

## 1. 本版本解决的问题

V2.1.1 解决固定长度切分容易切断语义边界的问题。RAG 检索质量高度依赖 chunk 质量，如果 chunk 在标题、段落或句子中间被切断，embedding 表达会变差。Adaptive Chunking 用递归语义边界和 overlap 提升 chunk 可用性。

## 2. 为什么现在做这个版本

它依赖 V2.1 已经有 document 和 document_chunk。只有基础上传和切分跑通后，才需要优化 chunk 质量。它为 V2.2 embedding 检索、V2.3 citation 和 V2.4 evaluation 做准备。此时不做真实语义模型切分，是为了保持本地可控。

## 3. 核心需求边界

本版本新增 DocumentChunker 和 DocumentChunkResult，使用 RECURSIVE_TEXT 策略、chunkSizeChars、overlap、headingPath、startOffset、endOffset、chunkStrategy。不做 embedding、不做 parent-child chunking、不做 late chunking、不做 PDF 表格专项解析。

## 4. 方案选择

按 Markdown 标题、空行、换行、中文句号问号感叹号、英文标点、逗号分号和固定长度兜底递归切分。相邻 chunk 保留 overlap。Markdown 标题路径保存到 headingPath，offset 记录原文位置。

## 5. 替代方案与暂时不选的原因

替代方案一是继续固定长度切分。当前不选，因为会破坏语义边界，影响检索。

替代方案二是用 LLM 或 NLP 模型做语义切分。当前不选，因为成本高、依赖外部模型，且 baseline 还没建立。

## 6. 数据库设计

document_chunk 表新增 chunk_strategy、start_offset、end_offset、heading_path。chunk_strategy 标识切分策略，offset 支持 citation 定位，headingPath 支持上下文解释和结果展示。

## 7. API 设计

GET /documents/{documentId}/chunks 返回 chunkStrategy、startOffset、endOffset、headingPath。上传接口不变，内部切分策略升级。

## 8. 核心流程

上传文档
-> DocumentService 读取文本
-> DocumentChunker.chunk
-> 递归寻找语义边界
-> 添加 overlap
-> 生成 DocumentChunkResult
-> 保存 document_chunk metadata
-> 查询 chunks 返回 metadata

## 9. 测试方案

建议覆盖短文本一个 chunk、长文本多个 chunk、chunkIndex 递增、overlap 生效、Markdown 标题写入 headingPath、offset 合法、空文本返回空列表。不要过度断言精确文本。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是既要尽量不超过 chunkSize，又要保留语义边界和 overlap，同时避免死循环。offset 和 headingPath 也要尽量对应原文，便于后续 citation。

## 12. 本版本亮点

- 从固定长度升级为递归语义边界切分。
- 保存 headingPath 和 offset，为 citation 做准备。
- overlap 减少跨 chunk 语义丢失。
- 为检索评估提供 chunk metadata。

## 13. 简历 bullet

### 简洁版

实现 Adaptive Chunking，支持按标题、段落、句子和标点递归切分，并保存 headingPath、offset 和 chunkStrategy。

### 强化版

优化 RAG 文档切分策略，从固定长度升级为递归语义边界切分与 overlap，并持久化 headingPath、start/end offset 和 chunkStrategy，为向量检索、Citation 和 Chunking Evaluation 打基础。

## 14. 面试高强度追问与回答

Q1：固定长度 chunking 为什么不够？

A1：它可能切断标题、段落和句子，导致 chunk 缺少完整语义，embedding 检索效果变差。

Q2：为什么按标题优先？

A2：标题通常代表语义层级，保留标题路径能帮助理解 chunk 所属上下文。

Q3：overlap 有什么作用？

A3：它减少相邻 chunk 边界处的信息丢失，让跨段内容仍有上下文。

Q4：overlap 有什么风险？

A4：会增加存储和检索冗余，也可能重复召回相似 chunk，需要控制大小。

Q5：offset 为什么重要？

A5：Citation 需要定位答案依据来自原文哪里，offset 是最基础的定位信息。

Q6：为什么不做模型语义切分？

A6：当前目标是工程 baseline，模型切分会引入成本和不确定性，先用可解释规则更稳。

Q7：如何评估 chunking 效果？

A7：后续构造 query-evidence 集，计算 Recall@K、MRR、Context Precision 等指标。

Q8：当前局限是什么？

A8：规则切分不能真正理解语义，复杂表格、代码、PDF 布局仍需要专项策略。
