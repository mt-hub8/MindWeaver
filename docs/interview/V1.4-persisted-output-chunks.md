# V1.4 面试文档：持久化输出片段

## 1. 本版本解决的问题

V1.4 解决完整 resultContent 不适合增量展示和后续 streaming 演进的问题。系统在 LLM 成功后把输出拆成 task_output_chunk 片段保存，并提供 GET /tasks/{taskId}/output-chunks 查询，形成持久化增量输出 baseline。

## 2. 为什么现在做这个版本

它依赖 LLM 结果持久化和任务执行链路。只有完整输出已保存后，才有必要拆分片段。它为 SSE/WebSocket、前端 polling 和长输出分页展示做准备。此时不做真正 streaming，是为了先完成可持久化、可查询的输出片段。

## 3. 核心需求边界

本版本新增 task_output_chunk 表、TaskOutputChunkService 和查询接口。不做 SSE、不做 WebSocket、不做真实 Provider streaming、不做前端实时订阅。保持克制是为了避免通信协议复杂化。

## 4. 方案选择

LLM 成功后，TaskExecutionService 在确认任务仍 RUNNING 时先保存 chunks，再 markTaskSucceeded。chunkIndex 从 0 递增，content 保存片段。最终 resultContent 仍保留完整输出。

## 5. 替代方案与暂时不选的原因

替代方案一是只保存 resultContent。当前不选，因为不利于增量展示和长文本处理。

替代方案二是直接上 SSE/WebSocket。当前不选，因为真实 streaming 涉及连接管理、断线恢复和 Provider 支持，早期先持久化更稳。

## 6. 数据库设计

新增 task_output_chunk 表，字段包括 id、task_id、chunk_index、content、created_at。task_id + chunk_index 支持按任务顺序读取。后续可增加 tokenOffset、finishReason、providerChunkId 等字段。

## 7. API 设计

Method: GET

Path: /tasks/{taskId}/output-chunks

Response: 返回 chunk 列表，包含 id、taskId、chunkIndex、content、createdAt。该接口用于本地和前端轮询式增量读取。

## 8. 核心流程

LlmClient 返回 content
-> 检查任务仍 RUNNING 且未取消
-> TaskOutputChunkService.saveChunks
-> 按固定长度拆分
-> 保存 task_output_chunk
-> markTaskSucceeded 保存完整 resultContent
-> 查询 chunks

## 9. 测试方案

建议覆盖正常任务生成多个 chunk、chunkIndex 从 0 递增、拼接内容接近 resultContent、失败任务不写成功 chunks、取消/超时后不写 chunks。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是写 chunks 的时机。任务如果已经取消或超时，不能再写成功输出片段。另一个难点是保持 chunk 顺序稳定，方便客户端拼接。

## 12. 本版本亮点

- 在不引入 SSE/WebSocket 的情况下实现增量输出雏形。
- 保留完整 resultContent 与 chunk 列表两种读取方式。
- 为长输出展示、轮询和后续 streaming 升级铺路。
- 写入前保留取消/超时保护。

## 13. 简历 bullet

### 简洁版

实现任务输出片段持久化，支持将 LLM 输出拆分为 task_output_chunk 并通过查询接口按顺序返回。

### 强化版

设计 persisted output chunks 作为 streaming baseline，在保留完整 resultContent 的同时持久化有序输出片段，为后续 polling、SSE/WebSocket 和长文本结果展示奠定基础。

## 14. 面试高强度追问与回答

Q1：为什么不直接做 SSE？

A1：SSE 需要连接生命周期、断线恢复和实时 Provider 支持。持久化 chunks 更适合作为第一步。

Q2：为什么还保留 resultContent？

A2：完整结果便于一次性查询和审计，chunks 适合增量展示，两者服务不同场景。

Q3：chunkIndex 为什么重要？

A3：它保证客户端能按顺序重建输出，不依赖 createdAt 的精度和排序偶然性。

Q4：失败任务写 chunks 吗？

A4：不写成功 chunks。失败时应保留 errorMessage 和 metadata。

Q5：取消后为什么不能写 chunks？

A5：用户已经取消任务，再写输出会破坏取消语义。

Q6：重试成功后 chunks 怎么处理？

A6：当前可追加或后续清理覆盖。更严格方案可在 attempt 维度保存 chunks。

Q7：如何升级为 streaming？

A7：真实 Provider streaming 时，每个增量片段实时写 task_output_chunk，并通过 polling/SSE/WebSocket 推送。

Q8：当前局限是什么？

A8：它不是真实时流，只是成功后拆分持久化，不能降低首 token 延迟。
