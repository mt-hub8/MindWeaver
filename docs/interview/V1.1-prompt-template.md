# V1.1 面试文档：Prompt Template

## 1. 本版本解决的问题

V1.1 解决 TaskExecutionService 直接把用户 prompt 传给 LLM 的问题。真实 AI 系统需要系统指令、变量渲染、模板追踪和可演进 prompt。Prompt Template 让最终 renderedPrompt 可控、可追踪、可复用。

## 2. 为什么现在做这个版本

它依赖 V1.0 LlmClient 和 V1.0.4 结果保存。只有模型调用链路存在后，才需要管理输入 prompt。它为 RAG 拼接 context、Prompt A/B、模型路由变量和后续 Evaluation 做准备。此时不做 CRUD API，是为了先验证默认模板执行闭环。

## 3. 核心需求边界

本版本新增 prompt_template 表、默认模板、PromptTemplateRenderer、rendered_prompt 和 prompt_template_code。不做模板管理后台、不做复杂模板引擎、不做 Prompt 版本发布系统。

## 4. 方案选择

默认模板 default_task_prompt 使用 {{prompt}}、{{taskId}}、{{model}} 等变量。PromptTemplateRenderer 用简单正则替换变量，缺失变量抛错。TaskExecutionService 渲染后把 renderedPrompt 传给 LlmClient，并保存实际使用模板。

## 5. 替代方案与暂时不选的原因

替代方案一是直接拼接字符串。当前不选，因为难以追踪模板来源，也不利于后续版本化。

替代方案二是引入完整模板引擎。当前不选，因为变量替换场景简单，引入第三方引擎会增加复杂度。

## 6. 数据库设计

新增 prompt_template 表，字段包括 template_code、template_name、template_content、enabled、created_at、updated_at。task 表新增 rendered_prompt 和 prompt_template_code，用于记录每次任务实际使用的 prompt 和模板编码。

## 7. API 设计

本版本主要是内部执行能力，未新增 Prompt Template 用户 API。GET /tasks/{taskId} 返回 renderedPrompt 和 promptTemplateCode。

## 8. 核心流程

TaskExecutionService 选定模型
-> 查询 default_task_prompt
-> 构造 prompt/taskId/model variables
-> PromptTemplateRenderer.render
-> LlmRequest.prompt = renderedPrompt
-> 成功后保存 renderedPrompt / promptTemplateCode

## 9. 测试方案

建议覆盖正常变量替换、多变量、变量空格、缺失变量抛错、空模板抛错、任务执行成功后返回 renderedPrompt。异常路径包括默认模板不存在和模板变量缺失。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是 prompt 是 AI 行为的重要输入，必须可追踪。缺失变量不能静默留空，否则模型输入可能错误但系统难以发现。

## 12. 本版本亮点

- 把用户输入和最终 LLM prompt 解耦。
- 保存 renderedPrompt 和 promptTemplateCode，支持审计。
- 简单渲染器满足早期需求，避免过度设计。
- 为 RAG context 拼接和 Prompt 评估打基础。

## 13. 简历 bullet

### 简洁版

实现 Prompt Template 数据模型和渲染器，支持默认模板渲染并保存 renderedPrompt 与 promptTemplateCode。

### 强化版

为 LLM 执行链路引入 Prompt Template 层，通过模板变量渲染、缺失变量校验和 renderedPrompt 持久化，实现模型输入可追踪、可演进的 Prompt 管理基础。

## 14. 面试高强度追问与回答

Q1：为什么 Prompt 需要模板化？

A1：真实任务不只是用户输入，还需要系统指令、上下文和模型变量。模板化能统一管理这些结构。

Q2：为什么保存 renderedPrompt？

A2：它是模型实际看到的输入，排查输出质量时必须能回溯。

Q3：缺失变量为什么要报错？

A3：静默替换会制造错误 prompt，模型可能输出异常但难以定位。

Q4：为什么不用第三方模板引擎？

A4：当前只需要简单变量替换，正则实现更轻量，后续复杂化再引入。

Q5：enabled 字段有什么用？

A5：支持禁用模板，后续可以做模板切换和版本管理。

Q6：templateCode 有什么价值？

A6：它是稳定引用标识，任务保存后能知道使用了哪套模板。

Q7：如何支持 RAG？

A7：后续可在模板变量中加入 context、citations、question 等字段。

Q8：当前局限是什么？

A8：没有 CRUD、版本发布、灰度和复杂控制流，只是默认模板执行闭环。
