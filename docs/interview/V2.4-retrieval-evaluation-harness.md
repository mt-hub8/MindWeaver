# V2.4 Retrieval Evaluation Harness 面试文档

## 0. 版本摘要

V2.4 新增 Retrieval Evaluation Harness，支持用户传入 query-evidence benchmark cases，复用现有 document search / embedding search，对每个 case 执行 TopK retrieval，计算 Recall@K、HitRate@K、MRR / RR@K 和 ContextPrecision@K，并返回 per-case result 与 summary metrics。

这一版不是继续堆新的 RAG 功能，而是给 RAG retrieval 增加可评估能力。V2.2 证明系统可以检索 TopK chunks，V2.3 证明系统可以基于 TopK chunks 生成 answer 和 citations，V2.4 则开始回答一个更工程化的问题：检索质量到底怎么样，后续升级有没有变好。

必须明确当前边界：

- 当前仍使用 Mock Embedding；
- 当前指标数值不代表真实语义检索效果；
- 当前没有真实 query-evidence benchmark 数据集；
- 当前没有持久化 evaluation result；
- 当前没有 Vector DB、Rerank、Hybrid Search；
- 当前不是完整 production-grade RAG evaluation platform。

V2.4 的价值是建立 evaluation framework、metric calculation 和 search -> evaluate 的工程链路，为 V2.5 Real Embedding Provider、V2.6 Vector DB Selection、V2.7 Retrieval Policy & VIP Search 做准备。

## 1. 本版本解决的问题

V2.3 已经能跑通 RAG Answer baseline：

```text
query
-> retrieve TopK chunks
-> build context prompt
-> call MockLlmClient
-> return answer + chunk-level citations
```

但这并不等于检索质量可控。V2.3 之后仍然存在几个问题：

- retrieved chunks 是否真的相关，不知道；
- topK = 3 / 5 / 10 应该怎么选，不知道；
- Adaptive Chunking 是否真的优于 Fixed Chunking，还不能量化；
- 后续换真实 embedding 后怎么判断是否变好，还没有统一评估标准；
- 后续接 Vector DB / Rerank 后怎么比较效果，还没有 benchmark；
- 当前只能说“能检索”，还不能说“能评估检索质量”。

V2.4 的目标是用 query + expectedChunkIds 构造 ground truth，复用同一套 search pipeline，然后用指标计算 retrieval quality。

一句话概括三个阶段的区别：

```text
V2.2 解决“能不能检索”
V2.3 解决“能不能基于检索回答”
V2.4 解决“怎么评估检索质量”
```

这一步非常关键。没有 evaluation harness，后续接入真实 embedding、rerank 或 vector DB 时，只能凭主观感觉判断“好像更准了”。有了 evaluation harness，后续每次检索升级都可以用同一套 benchmark 做对比。

## 2. 为什么现在做 V2.4

V2.4 依赖前面几层能力：

- V2.1 有文档上传和基础 chunk；
- V2.1.1 有 Adaptive Chunking 和 chunk metadata；
- V2.2 有 Mock Embedding & Vector Search；
- V2.3 有 RAG Answer with Citation；
- V2.2.x 有可靠性硬化、API 边界和测试基线。

现在系统已经有了可被评估的 retrieval pipeline，所以 evaluation harness 有了被评估对象。

为什么不更早做？因为没有 search pipeline 前，evaluation 只能空转。你可以先写指标公式，但没有真实的 retrieved chunks，也无法验证 search -> evaluate 的工程链路。

为什么不更晚做？如果先接真实 embedding、Vector DB、Rerank，再补 evaluation，就无法客观判断这些升级是否有效。比如接入 BGE 后结果变了，到底是变好还是变坏？如果没有 benchmark，只能靠人工看几个 case。

所以合理顺序是：

```text
先建立 evaluation harness
-> 再接真实 embedding
-> 再比较 chunking / model / topK / rerank / vector DB 策略
```

这也是面试里很能体现工程成熟度的一点：不是只会堆功能，而是知道什么时候应该补评测体系。

## 3. 方案选择

当前方案是同步 API：

```http
POST /evaluations/retrieval
```

它是 request-driven evaluation，不持久化 result，不新增数据库表，不新增 Flyway。

### 方案 A：只写单元测试，不暴露 API

优点：

- 简单；
- 不增加用户接口；
- 适合纯内部验证。

缺点：

- 不方便手工验收；
- 不方便展示给面试官；
- 不方便后续接 benchmark tool / dashboard；
- 不容易让用户提交自己的 query-evidence cases。

这个方案适合底层指标函数，但不足以作为 retrieval evaluation harness。

### 方案 B：同步 evaluation API，不持久化

优点：

- 边界小；
- 易验收；
- 不污染数据库；
- 可以通过 API 直接展示；
- 可以让用户传入 benchmark cases；
- 后续可以升级为持久化 evaluation run。

缺点：

- 评测历史不会保存；
- 不能直接做趋势图；
- 不适合长时间大规模 benchmark；
- 每次都需要重新提交 cases。

这是当前版本选择的方案，因为它最适合 V2.4 的目标：建立评测链路，而不是一次性做完整评测平台。

### 方案 C：持久化 evaluation run / cases / result

优点：

- 可以保存历史；
- 可以对比不同版本；
- 可以做 dashboard 和报告；
- 可以支持企业客户的长期质量追踪。

缺点：

- 需要新增多张表；
- 需要设计 run / case / result schema；
- 需要考虑权限、租户、数据保留；
- 会把 V2.4 变大；
- 当前还没接真实 embedding，不应该过早持久化。

后续做稳定 benchmark 和 dashboard 时，这个方案会变得更有价值。

### 方案 D：异步 evaluation job

优点：

- 适合大规模评测；
- 可以复用任务系统；
- 可以后台运行；
- 可以记录执行状态和失败原因。

缺点：

- 需要 taskType；
- 需要 evaluation result 持久化；
- 需要状态追踪；
- 需要和 task_attempt / outbox / retry 结合；
- 会引入更多调度复杂度。

当前暂时不选。V2.4 先做同步 baseline，等 benchmark 规模变大后再考虑异步化。

## 4. API 设计

### 4.1 POST /evaluations/retrieval

接口：

```http
POST /evaluations/retrieval
```

请求 DTO：`RetrievalEvaluationRequest`

字段：

- `documentId`：可选，限定文档范围；
- `topKValues`：可选，默认 `[1, 3, 5]`；
- `cases`：必填，评测 case 列表。

case DTO：`RetrievalEvaluationCaseRequest`

字段：

- `caseId`：可选，用于标识 case；
- `query`：必填，评测问题；
- `expectedChunkIds`：必填，期望命中的相关 chunk ids。

请求示例：

```json
{
  "documentId": 1,
  "topKValues": [1, 3, 5],
  "cases": [
    {
      "caseId": "outbox-001",
      "query": "为什么这个项目要使用 Transactional Outbox？",
      "expectedChunkIds": [12, 13]
    },
    {
      "caseId": "attempt-001",
      "query": "task_attempt 的作用是什么？",
      "expectedChunkIds": [21]
    }
  ]
}
```

参数规则：

- `cases` 不能为空；
- 每个 case 的 `query` 不能为空；
- 每个 case 的 `expectedChunkIds` 不能为空；
- `expectedChunkIds` 会去重；
- `topKValues` 为空时使用 `[1, 3, 5]`；
- `topKValues` 中 `<= 0` 或 `null` 会被过滤；
- `K > 20` 会截断为 20；
- K 会去重并升序排序；
- 如果全部 K 无效，回退默认 `[1, 3, 5]`。

### 4.2 Response

响应 DTO：`RetrievalEvaluationResponse`

字段：

- `documentId`；
- `caseCount`；
- `topKValues`；
- `summary`；
- `cases`。

`summary` 类型是 `List<RetrievalEvaluationSummaryResponse>`，每个 K 包含：

- `k`；
- `recallAtK`；
- `hitRateAtK`；
- `mrr`；
- `contextPrecisionAtK`。

`cases` 类型是 `List<RetrievalEvaluationCaseResultResponse>`，每个 case result 包含：

- `caseId`；
- `query`；
- `expectedChunkIds`；
- `retrievedChunks`；
- `metrics`。

`retrievedChunks` 类型是 `List<RetrievedChunkEvaluationResponse>`，字段包括：

- `rank`；
- `chunkId`；
- `documentId`；
- `score`；
- `relevant`；
- `headingPath`；
- `contentPreview`。

`metrics` 类型是 `List<RetrievalMetricAtKResponse>`，字段包括：

- `k`；
- `recallAtK`；
- `hitRateAtK`；
- `mrr`；
- `contextPrecisionAtK`。

响应示例：

```json
{
  "documentId": 1,
  "caseCount": 2,
  "topKValues": [1, 3, 5],
  "summary": [
    {
      "k": 1,

  ],
  "cases": [
    {
      "caseId": "outbox-001",

      "expectedChunkIds": [12, 13],

        {
          "rank": 1,

          "documentId": 1,
          "score": 0.91,
          "relevant": true,

          "contentPreview": "Transactional Outbox 用于解决数据库提交和 MQ 投递之间的 dual write problem..."
        }
      ],
      "metrics": [
        {
          "k": 1,
          "recallAtK": 0.5,
          "hitRateAtK": 1.0,
          "mrr": 1.0,
          "contextPrecisionAtK": 1.0
        }
      ]
    }
  ]
}
```

上面的数值只是说明 response 形态，不代表当前项目的真实检索效果。

## 5. 指标设计

### 5.1 Recall@K

定义：

```text
Recall@K = TopK 中命中的 expected relevant chunks 数 / expected relevant chunks 总数
```

示例：

```text
expectedChunkIds = [12, 13]
Top5 = [8, 12, 20, 13, 30]

Recall@5 = 2 / 2 = 1.0
Recall@3 = 1 / 2 = 0.5
```

Recall@K 适合衡量：应该找回来的证据，有多少被找回来了。

在 RAG 中，Recall 很重要。如果关键证据没被召回，后面的 LLM 再强也没有依据回答。

### 5.2 HitRate@K

定义：

```text
TopK 中是否至少命中一个 relevant chunk
```

命中则为 `1.0`，未命中则为 `0.0`。

HitRate@K 适合衡量：用户问一个问题时，系统是否至少给 LLM 提供了一条有效证据。

它比 Recall 更粗，但对产品体验很直观。很多问答场景里，只要 TopK 里有一条核心证据，LLM 就可能答出关键点；如果一条都没有，回答质量通常会明显下降。

### 5.3 MRR / RR@K

MRR 是 Mean Reciprocal Rank。单个 case 中使用 RR，也就是第一个 relevant chunk 的排名倒数。

示例：

```text
Top5 = [8, 12, 20, 13, 30]
第一个 relevant chunk 是 rank 2

RR = 1 / 2 = 0.5
```

如果 TopK 内没有 relevant chunk，则 `RR@K = 0.0`。

本项目在 per-K metrics 中计算的是当前 K 范围下的 RR@K，summary 中对所有 cases 的 RR@K 求平均，因此 response 字段命名为 `mrr`。

MRR 适合衡量：相关证据是否排在前面。

这对 RAG 很重要。即使 TopK 最终包含了 relevant chunk，如果它排得很靠后，也可能因为 context budget、截断或 LLM 注意力分散而影响答案质量。

### 5.4 ContextPrecision@K

定义：

```text
ContextPrecision@K = TopK 中 relevant chunk 数 / actualReturnedCountWithinK
```

如果实际返回结果少于 K，分母使用实际返回数量；如果实际返回数量为 0，则 ContextPrecision@K 为 `0.0`。

ContextPrecision@K 适合衡量：塞给 LLM 的上下文里，有多少是真正有用的。

指标之间需要组合看：

- Recall 高但 Context Precision 低：说明找回了正确证据，但混入很多噪声；
- Context Precision 高但 Recall 低：说明上下文比较干净，但可能漏掉关键证据；
- MRR 高：说明第一条 relevant evidence 排得靠前；
- HitRate 高：说明大多数问题至少能找回一条相关证据。

RAG 质量不能只看单个指标。一个好的检索系统应该尽量兼顾召回、排序和上下文纯度。

## 6. 核心流程

V2.4 的核心流程：

```text
POST /evaluations/retrieval
-> RetrievalEvaluationController 接收请求
-> RetrievalEvaluationService 校验 request
-> 标准化 topKValues
-> maxK = max(topKValues)
-> 每个 case 调用一次现有 DocumentEmbeddingService.search(...)
-> 获取 Top maxK retrieved chunks
-> 标记每个 chunk 是否 relevant
-> RetrievalMetricsCalculator 对每个 K 计算 metrics
-> 聚合 summary metrics
-> 返回 RetrievalEvaluationResponse
```

重点设计：

- 复用现有 search service；
- 不重新实现 embedding generation；
- 不重新实现 cosine similarity；
- 不重新实现 vector parsing；
- 不绕过现有 search pipeline；
- 每个 case 只用 maxK 查询一次；
- 各个 K 的指标都从同一批 retrieved chunks 截取 TopK 计算；
- metrics calculator 和 service 解耦。

每个 case 只查询一次 maxK 很重要。假设 `topKValues = [1, 3, 5]`，service 只会用 `topK = 5` 调用一次 search，然后分别用前 1、前 3、前 5 计算指标。这避免了对每个 K 重复检索，也保证不同 K 指标来自同一排序结果。

## 10. 工程权衡

### 10.1 为什么做 request-driven evaluation，而不是持久化 evaluation run

当前目标是验证评测链路，而不是搭建完整 evaluation platform。

Request-driven evaluation 的好处是边界小：用户提交 cases，系统即时返回 per-case 和 summary metrics，不需要新增 run / case / result 表，也不需要设计历史查询、权限和报表。

等后续 benchmark 结构稳定、真实 embedding 接入、企业用户需要历史报告时，再设计持久化 evaluation run 会更自然。

### 10.2 为什么用 expectedChunkIds，而不是 LLM-as-a-judge

`expectedChunkIds` 是确定性 ground truth。它让指标计算稳定、可重复，也适合单元测试。

LLM-as-a-judge 可以评估 answer quality 或 citation support，但它有成本高、输出不完全稳定、judge prompt 难设计、不同模型判断标准不一致等问题。

所以 V2.4 先用 expectedChunkIds 做 retrieval evaluation。后续做 answer evaluation 或 citation accuracy 时，可以再引入 LLM-as-a-judge，但那应该是独立版本。

### 10.3 为什么现在不做 answer evaluation

V2.4 关注 retrieval。

RAG answer 质量由 retrieval 和 generation 共同决定。如果回答不好，可能是没召回证据，也可能是 LLM 没用好证据，还可能是 prompt 设计问题。

当前先把 retrieval 评估拆出来，能降低问题复杂度。只有先知道证据有没有被找回来，后续评估 answer 才更有意义。

### 10.4 为什么现在不做 citation accuracy

当前 citation 是 chunk-level。Citation Accuracy 要验证 answer 中的 claim 是否真的被 cited chunk 支撑，这需要更复杂的数据集和评估逻辑。

比如答案一句话可能综合多个 chunk，也可能是对 chunk 的概括。判断它是否被 citation 支撑，需要 evidence alignment 或人工标注。

所以 V2.4 不做 citation accuracy，而是先做 retrieval metrics。后续可以在 V2.9 或类似阶段单独做 citation evaluation。

### 10.5 为什么仍然用 Mock Embedding

V2.4 的目标是 evaluation harness，不是证明当前检索效果优秀。

Mock Embedding 的指标数值不能代表真实语义效果，但它可以验证：

- API 是否能接收 benchmark cases；
- search pipeline 是否被复用；
- metrics 是否计算正确；
- per-case / summary response 是否合理；
- 后续换真实 embedding 后是否能复用同一套 harness。

这就是 baseline 的意义。

### 10.6 为什么不接 Rerank / Vector DB

没有 evaluation 前，接 rerank 或 vector DB 只能主观判断效果。

比如接了 rerank 后，返回顺序变了，怎么知道变好还是变坏？接了 Qdrant 后，召回是否下降？如果没有 benchmark，就没有客观答案。

所以先建立评测基线，再做 rerank、真实 embedding、vector DB，顺序更合理。

### 10.7 为什么每个 case 用 maxK 查询一次

如果 `topKValues = [1, 3, 5]`，每个 K 都调用一次 search，会重复计算 embedding 和排序，也可能引入不必要的差异。

当前做法是每个 case 使用 `maxK` 查询一次，然后从同一批检索结果里截取 Top1、Top3、Top5 计算指标。

这样有三个好处：

- 避免重复检索；
- 降低计算成本；
- 保证不同 K 的指标来自同一次排序结果。

### 10.8 为什么 ContextPrecision 使用 actualReturnedCount 作为分母

当实际返回结果少于 K 时，用 K 做分母会人为拉低指标。

例如请求 Top5，但系统只返回 2 条，其中 1 条 relevant。如果用 5 做分母，ContextPrecision 是 0.2；如果用实际返回 2 条做分母，ContextPrecision 是 0.5。后者更能反映实际塞给 LLM 的上下文相关比例。

如果 actualReturnedCount 为 0，则 ContextPrecision 为 0。

## 11. 产品化 / VIP / 商业化思考

## 15. 面试高强度追问与回答

### Q1：为什么 RAG 需要 evaluation？

RAG 的效果不只取决于 LLM，也取决于检索有没有把正确证据放进上下文。如果检索不到关键 chunk，LLM 再强也只能猜。

Evaluation 的价值是把“看起来能检索”变成“可以量化检索质量”。通过 Recall@K、MRR、Context Precision 等指标，可以判断相关证据是否被召回、是否排得靠前、上下文里噪声多不多。

在这个项目中，V2.4 建立了 evaluation harness，为后续真实 embedding、chunking 策略、rerank 和 vector DB 的选型提供统一 benchmark。

### Q2：为什么 V2.4 不直接接真实 embedding？

因为没有 evaluation harness 时，接真实 embedding 也很难客观判断效果。

如果先接 BGE 或 OpenAI Embedding，然后感觉结果变好了，这仍然是主观判断。V2.4 先建立 query-evidence benchmark 的接口和指标计算，后续再接真实 embedding，就能用同一套 cases 评估不同模型。

当前仍使用 Mock Embedding，所以指标不代表真实语义效果。重点是把评价框架搭起来。

### Q3：Recall@K 是什么？

Recall@K 表示 TopK 检索结果中命中的 expected relevant chunks 占所有 expected relevant chunks 的比例。

比如 expected 是 `[12, 13]`，Top5 是 `[8, 12, 20, 13, 30]`，命中 2 个，所以 Recall@5 是 1.0。如果 Top3 是 `[8, 12, 20]`，只命中 1 个，所以 Recall@3 是 0.5。

它衡量的是“应该找回来的证据找回来多少”。在 RAG 中，Recall 是底线指标，因为没召回证据就很难回答正确。

### Q4：HitRate@K 和 Recall@K 有什么区别？

HitRate@K 只关心 TopK 里是否至少命中一个 relevant chunk，命中就是 1，没命中就是 0。

Recall@K 关心命中了多少个 relevant chunks。比如 expected 有 3 个，TopK 命中 1 个，HitRate 是 1.0，但 Recall 只有 1/3。

HitRate 更适合衡量“有没有至少一条有效证据”，Recall 更适合衡量“证据召回是否完整”。两者要一起看。

### Q5：MRR 衡量什么？

MRR 衡量第一个 relevant chunk 排得有多靠前。

如果第一个 relevant chunk 排第 1，RR 是 1.0；排第 2，RR 是 0.5；排第 5，RR 是 0.2；没有命中就是 0。

RAG 中排序很重要，因为排前面的 chunk 更容易被模型关注，也更不容易被 context 截断。MRR 高说明相关证据更早出现。

### Q6：Context Precision 衡量什么？

Context Precision 衡量 TopK context 中有多少比例是真正 relevant 的。

它关注的是上下文噪声。如果 TopK 里混入很多无关 chunk，LLM 可能被干扰，即使正确证据也在里面，回答质量仍然可能下降。

当前项目中，如果实际返回结果少于 K，分母用 actualReturnedCount，这样更能反映实际传给 LLM 的上下文纯度。

### Q7：Recall 高但 Context Precision 低说明什么？

说明正确证据找回来了，但同时混入了很多噪声。

这种情况 LLM 有机会答对，但上下文中无关内容太多，可能导致模型注意力分散、引用错误或回答不够稳定。

后续可以通过 rerank、提高 chunk 质量、降低 topK、引入 hybrid search 或 query rewrite 来改善。

### Q8：Context Precision 高但 Recall 低说明什么？

说明返回的上下文比较干净，但漏掉了部分关键证据。

如果问题只需要一个证据，影响可能不大；但如果答案需要多个证据综合，Recall 低就会导致回答不完整。

这种情况可能需要提高 topK、改进 embedding、优化 chunking，或者引入 multi-query retrieval。

### Q9：为什么用 expectedChunkIds 作为 ground truth？

因为 expectedChunkIds 是确定性的，适合做 retrieval evaluation。

它明确告诉系统：对于这个 query，哪些 chunk 是相关证据。这样 Recall、HitRate、MRR、ContextPrecision 都能稳定计算，不依赖 LLM 的判断。

这对于单元测试和后续 benchmark 都很重要。LLM-as-a-judge 可以做补充，但不适合作为第一版 retrieval ground truth。

### Q10：expectedChunkIds 怎么构造？

最简单方式是人工标注。开发者或业务专家先准备 query，然后在文档 chunks 中标出能回答该 query 的 chunk ids。

后续可以结合工具辅助：比如搜索候选 chunks，再由人工确认；或者根据历史问答日志构造 query-evidence 数据集。

高质量 expectedChunkIds 是 evaluation 的核心。如果 ground truth 本身不准，指标就没有意义。

### Q11：为什么不用 LLM-as-a-judge？

LLM-as-a-judge 有价值，但它不是 V2.4 的优先选择。

它会引入额外成本、调用延迟、judge prompt 设计、模型偏差和稳定性问题。不同 judge 模型可能给出不同评分，同一个模型在不同 prompt 下也可能不稳定。

V2.4 关注 retrieval，expectedChunkIds 更直接、更确定。后续做 answer evaluation 或 citation accuracy 时，可以考虑引入 LLM-as-a-judge。

### Q12：为什么不持久化 evaluation result？

当前版本目标是同步 evaluation baseline，不是完整评测平台。

如果持久化 result，就要设计 evaluation_run、case、result 表，还要考虑历史查询、版本对比、权限、清理和 dashboard。这会让 V2.4 变大。

现在先通过 API 返回结果，验证评测链路。等 benchmark 稳定后，再持久化会更合理。

### Q13：为什么每个 case 用 maxK 查询一次？

因为不同 K 的指标应该来自同一次排序结果。

如果 topKValues 是 `[1, 3, 5]`，只需要查 Top5，然后截取 Top1、Top3、Top5 计算指标。这样避免重复检索，也避免不同查询之间可能出现的不一致。

这也是更高效的做法，尤其后续接真实 embedding 或 vector DB 时，重复查询成本会更明显。

### Q14：Mock Embedding 下的指标有什么局限？

Mock Embedding 是 deterministic 的工程模拟，不代表真实语义理解能力。

所以当前指标只能验证 evaluation harness 和 metric calculation 是通的，不能证明检索效果好。比如 Recall@K 高低更多反映 Mock 特征和测试数据的关系，而不是模型真实语义能力。

后续接真实 embedding 后，才适合用这套 harness 做真实效果对比。

### Q15：后续接真实 embedding 后如何比较模型？

可以用同一套 query-evidence benchmark，对不同 embedding provider / model 跑相同 cases。

比较维度包括 Recall@K、MRR、ContextPrecision、latency、成本、维度、最大输入长度、中文效果、部署复杂度。

只有在同一数据集和同一指标下比较，模型选型才有工程依据。

### Q16：如何比较 Fixed Chunking 和 Adaptive Chunking？

需要让同一批文档分别用 Fixed Chunking 和 Adaptive Chunking 生成 chunks，然后基于相同 query-evidence 评测集跑 retrieval。

比较 Recall@K、MRR、ContextPrecision，看 Adaptive 是否更容易保留语义边界、是否减少噪声、是否让相关证据排得更靠前。

当前项目还没有真实 Fixed vs Adaptive 实验报告，所以不能说 Adaptive 已经量化优于 Fixed。V2.4 只是为这个对比提供工具。

### Q17：如何比较 topK = 3 / 5 / 10？

可以在同一次 maxK 检索结果上分别计算不同 K 的指标。

通常 topK 增大会提高 Recall，但可能降低 ContextPrecision，因为更多 chunk 会带来更多噪声。topK 太小可能漏证据，topK 太大可能干扰 LLM。

所以 topK 选择要看 Recall 和 ContextPrecision 的平衡，也要考虑 LLM context budget 和成本。

### Q18：如果企业用户要评估自己的知识库质量，产品怎么设计？

可以让企业上传 query-evidence benchmark，系统定期运行 evaluation，输出哪些问题召回差、哪些文档缺证据、哪些 chunking 策略效果不好。

还可以按知识库、文档类型、部门、模型版本生成报告，帮助企业维护知识库质量。

这个能力可以从当前 request-driven API 演进为持久化 evaluation run 和 dashboard。

### Q19：VIP 用户在 retrieval evaluation 上能获得什么？

VIP 可以获得更大规模的 benchmark、更高 topK、更强 embedding、更复杂 rerank、定期自动评测、策略 A/B test、质量报告和低质量检索告警。

企业版还可以支持不同知识库隔离、权限控制、部门级报表、SLA 和成本分析。

这些当前没有实现，但 V2.4 的 evaluation harness 是起点。

### Q20：如果数据量扩大 10 倍，evaluation 的瓶颈在哪里？

瓶颈首先在 search pipeline。当前底层仍是 Mock Embedding + MySQL TEXT vector + Java in-memory exact scan，不适合大规模向量检索。

evaluation 本身也会放大检索成本，因为每个 case 都要跑一次 retrieval。case 数越多，成本越高。

后续需要 Vector DB、ANN index、批量 evaluation、异步 job、结果缓存和持久化报告。V2.4 先建立指标和 API，后续再扩展规模。
