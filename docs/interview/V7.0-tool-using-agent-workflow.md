# V7.0 Tool-Using Agent Workflow 面试文档

## 1. 阶段摘要

V7.0 的阶段名称是：

```text
V7.0 Tool-Using Agent Workflow
```

中文可以表述为：

```text
可调用工具的 Agent 工作流
```

这个阶段的核心目标是：把 V6.0 的 Agent Task 从“一次 RAG 检索 + 一次 LLM 生成”升级为“可以执行固定工具步骤的受控 Agent Workflow”。

V6.0 的执行链路是：

```text
用户提交任务
-> RAG 检索知识库上下文
-> 调用 LLM
-> 生成结构化结果
-> 保存 result / citations / metadata / events
```

V7.0 的执行链路升级为：

```text
用户提交任务
-> 创建固定 step plan
-> 执行 knowledge_search 工具
-> 保存工具输入 / 输出
-> 执行 context_summary 工具
-> 保存工具输入 / 输出
-> 基于工具结果调用 LLM 生成 final report
-> 保存最终结果 / citations / metadata / events
```

这个阶段的重点不是做“无限自主 Agent”，而是做一个**可控、可追踪、可测试、可解释**的 tool-using workflow（工具调用工作流）。

V7.0 明确不做复杂 ReAct loop、多 Agent、自主规划、shell、代码执行、浏览器操作和外部联网工具。它只做安全内置工具和固定步骤计划。

---

## 2. V7.0 在项目中的位置

当前项目已经完成：

```text
V3.1：AI Knowledge Assistant 产品化；
V3.2：文档上传到问答闭环；
V3.3：异步文档摄入；
V3.4：摄入事件时间线；
V3.5：文档处理分析；
V4.0：知识库生命周期管理；
V5.0：知识库分组与范围检索；
V6.0：AI Runtime 与 Agent Task 异步编排；
V7.0：可调用工具的 Agent 工作流。
```

其中：

```text
V6.0 让系统可以异步执行 AI Task；
V7.0 让 AI Task 可以按步骤调用工具。
```

这是从：

```text
LLM workflow backend
```

进入：

```text
Agent workflow backend
```

的关键阶段。

---

## 3. 为什么要做 V7.0

V6.0 已经支持 Agent Task，但本质还是一次性流程：

```text
检索上下文 -> 生成结果
```

这更像一个异步 RAG report generator（异步 RAG 报告生成器）。

真正的 Agent Infra 至少要具备：

```text
工具注册；
工具选择或计划；
工具输入；
工具输出；
步骤状态；
工具失败；
工具结果汇总；
最终报告生成；
执行过程可追踪。
```

V7.0 先做最小可控版本：

```text
固定工具；
固定步骤；
无自主循环；
每步可记录；
每步可测试；
失败可定位。
```

这样既有 Agent 的形态，又不会失控膨胀。

## 4. 核心设计一：Tool Registry 工具注册中心

### 4.1 AgentTool 抽象

V7.0 新增 AgentTool 抽象。

语义可以概括为：

```text
toolName；
displayName；
description；
inputSchema；
outputSchema；
execute(input, context) -> ToolExecutionResult。
```

每个工具至少需要描述：

```text
技术名称；
中文展示名；
中文说明；
输入结构；
输出结构；
是否启用。
```

例如：

```text
toolName = knowledge_search
displayName = 检索知识库
description = 基于任务目标和知识库范围检索相关文档片段
```

---

### 4.2 AgentToolRegistry

V7.0 新增 AgentToolRegistry。

职责是：

```text
注册内置工具；
根据 toolName 查找工具；
返回可用工具列表；
为前端展示工具说明；
避免工具逻辑散落在 AgentTaskService 中。
```

这样 Agent Task Service 只负责编排，不直接关心每个工具的内部实现。

---

### 4.3 GET /agent/tools

新增工具列表 API：

```http
GET /agent/tools
```

返回：

```text
toolName；
displayName；
description；
inputSchema；
outputSchema；
enabled。
```

前端可以展示：

```text
可用工具；
检索知识库；
总结检索结果；
分析知识库分组。
```

## 5. 核心设计二：内置安全工具

V7.0 只实现安全的内置工具。

### 5.1 knowledge_search：检索知识库

中文名：

```text
检索知识库
```

用途：

```text
基于 task objective 和 collectionId 检索相关知识库片段。
```

输入：

```text
query；
collectionId，可选；
topK。
```

输出：

```text
matchedChunks；
citations；
retrievalMetadata；
noContextReason。
```

关键要求：

```text
复用 V5.0 scoped retrieval；
继续应用 V4.0 lifecycle filter；
不允许已删除文档、旧版本 chunk、collection 外文档进入结果。
```

这个工具是整个 Agent Workflow 的知识入口。

---

### 5.2 context_summary：总结检索结果

中文名：

```text
总结检索结果
```

用途：

```text
把 knowledge_search 返回的片段整理成结构化摘要。
```

输入：

```text
matchedChunks；
taskObjective。
```

输出：

```text
summary；
keyPoints；
limitations。
```

实现可以使用 LLM Provider，也可以在默认测试中使用 deterministic mock summary（确定性模拟摘要）。

关键边界：

```text
默认测试不能调用真实 Python LLM；
真实 LLM 只能通过 profile / manual 验证启用。
```

---

### 5.3 collection_overview：分析知识库分组（可选）

中文名：

```text
分析知识库分组
```

用途：

```text
统计指定 collection 的文档数量、可用文档数量、chunk 数量、最近更新时间。
```

输入：

```text
collectionId。
```

输出：

```text
collectionName；
documentCount；
activeDocumentCount；
chunkCount；
lastUpdatedAt；
warnings。
```

这个工具适合帮助 Agent 在执行前理解当前知识库范围。

## 6. 核心设计三：工具安全边界

V7.0 明确禁止危险工具：

```text
shell command；
code execution；
database write；
file delete；
external web browsing；
email；
calendar；
payment；
network scan；
browser automation。
```

原因是这些工具涉及：

```text
权限；
安全；
副作用；
沙箱；
审计；
用户授权；
失败补偿。
```

当前阶段只允许读取系统内已有知识库数据，并生成摘要或报告。

这保证 Agent Workflow 可控。

## 7. 核心设计四：Agent Task Step

### 7.1 为什么需要 Step 表

V6.0 只有任务级状态和事件。

V7.0 引入工具执行后，需要记录每一步：

```text
第几步；
执行什么工具；
输入是什么；
输出是什么；
状态是什么；
耗时多久；
是否失败；
错误是什么。
```

因此新增 agent_task_step 或等价结构。

---

### 7.2 agent_task_step 字段设计

建议字段：

```text
id；
task_id；
step_order；
step_type；
tool_name；
title；
display_title；
status；
input_json；
output_json；
error_code；
error_message；
trace_id；
started_at；
completed_at；
duration_ms；
created_at；
updated_at。
```

step_type 可以包括：

```text
TOOL_CALL；
LLM_GENERATION；
FINAL_REPORT。
```

status 可以包括：

```text
PENDING：待执行；
RUNNING：执行中；
COMPLETED：已完成；
FAILED：执行失败；
SKIPPED：已跳过。
```

前端必须显示中文状态，而不是只显示英文枚举。

---

## 8. 核心设计五：固定 Step Plan

V7.0 不做自主 planner。

它使用固定步骤计划：

```text
Step 1：knowledge_search，检索知识库；
Step 2：context_summary，总结检索结果；
Step 3：final_report，生成最终报告。
```

为什么选择固定 plan：

```text
更可控；
更容易测试；
不会无限循环；
不会乱选工具；
不会扩大安全边界；
适合 MVP。
```

这也符合当前阶段目标：先把工具执行链路跑通，而不是追求复杂智能。

---

## 9. 核心设计六：Agent Workflow 执行流程

V7.0 的执行流程是：

```text
load task
-> create fixed step plan
-> run Step 1 knowledge_search
-> save step input / output
-> run Step 2 context_summary
-> save step input / output
-> run Step 3 final_report
-> call LLM Provider
-> save final result
-> save citations
-> save metadata
-> task COMPLETED
```

失败时：

```text
当前 step = FAILED；
task = FAILED；
记录 TASK_FAILED event；
保存 errorCode；
保存 errorMessage；
保存 traceId；
页面展示中文错误。
```

这个流程让 Agent Task 具备“过程可解释性”。

## 10. 核心设计七：Step Events

除了 step 表，V7.0 继续记录 agent_task_event。

新增或补齐事件：

```text
STEP_PLAN_CREATED；
TOOL_EXECUTION_STARTED；
TOOL_EXECUTION_COMPLETED；
TOOL_EXECUTION_FAILED；
FINAL_REPORT_STARTED；
FINAL_REPORT_COMPLETED；
FINAL_REPORT_FAILED。
```

中文 displayMessage 示例：

```text
已生成任务执行计划。
开始执行工具：检索知识库。
工具执行完成：检索知识库。
工具执行失败：检索知识库。
开始生成最终报告。
最终报告生成完成。
最终报告生成失败。
```

event 更适合时间线展示，step 更适合结构化步骤展示。

## 11. 核心设计八：工具输入输出保存

每个 step 都需要保存：

```text
input_json；
output_json。
```

用途：

```text
调试工具输入；
查看工具输出；
复盘任务执行；
定位失败原因；
面试展示 Agent 可观测性。
```

但需要控制大小：

```text
不保存超大全文；
matchedChunks 保存 snippet / citation 信息；
metadata 可以截断；
完整 prompt 不直接塞满页面。
```

前端可以展示：

```text
查看工具输入；
查看工具输出；
查看技术详情。
```

## 12. 核心设计九：no-context 行为

如果 knowledge_search 没有可用上下文，推荐流程是：

```text
Step 1 COMPLETED，但 output 标记 noContext；
Step 2 SKIPPED 或 COMPLETED，说明无可总结内容；
Step 3 COMPLETED，生成 deterministic final result；
不调用 LLM；
task COMPLETED；
result = 当前范围内没有可用于执行任务的知识库内容。
```

为什么 no-context 不调用 LLM：

```text
避免幻觉；
节省模型成本；
让任务结果诚实；
更容易测试。
```

这与 V6.0 的 no-context 设计保持一致。

## 13. 核心设计十：Final Report 生成

最终报告基于工具结果生成。

System prompt 语义：

```text
你是一个企业知识库任务执行助手。你需要基于工具检索和总结得到的结果完成任务。不要使用工具结果之外的信息编造内容。如果信息不足，需要明确说明。
```

User prompt 包含：

```text
task objective；
scope；
knowledge_search result；
context_summary result；
citations；
output format instruction。
```

输出格式建议：

```text
任务结论；
关键依据；
风险 / 不确定性；
下一步建议；
引用来源。
```

这样最终报告不是直接基于原始文档，而是基于工具执行结果。

---

## 14. 核心设计十一：Citations 贯通

knowledge_search 生成的 citations 必须贯通到最终任务结果。

要求：

```text
final task detail 能看到 citations；
citations 不包含 DELETED document；
citations 不包含 old / superseded chunks；
citations 不包含 collection 外文档；
citations sourceIndex 和最终报告引用索引一致或可解释。
```

这保证 Agent Task 的输出仍然可溯源。

---

## 15. 核心设计十二：Filter Consistency

V7.0 最重要的工程约束之一：

```text
工具执行不能绕过 V5.0 Collection filter 和 V4.0 Lifecycle filter。
```

knowledge_search 必须复用已有 scoped retrieval。

最终进入工具结果、citations、final report prompt 的内容必须同时满足：

```text
属于指定 Collection，或全库模式；
document 为 ACTIVE；
chunk 为 current active；
不是 DELETED document；
不是 old / superseded chunk。
```

否则 Agent Tool 会绕开之前所有知识库治理能力。

## 16. 核心设计十三：失败处理

V7.0 需要处理：

```text
tool not found；
tool input invalid；
knowledge_search failed；
context_summary failed；
final report LLM failed；
serialization failed；
timeout；
provider unavailable。
```

建议错误码：

```text
AGENT_TOOL_NOT_FOUND；
AGENT_TOOL_INPUT_INVALID；
AGENT_TOOL_EXECUTION_FAILED；
AGENT_STEP_FAILED；
AGENT_FINAL_REPORT_FAILED；
AGENT_WORKFLOW_FAILED。
```

失败时必须：

```text
step = FAILED；
task = FAILED；
保存 errorCode / errorMessage / traceId；
记录 TOOL_EXECUTION_FAILED 或 FINAL_REPORT_FAILED；
页面展示中文错误。
```

不能让任务一直卡在 RUNNING。

## 20. 关键取舍

### 20.1 为什么不做 ReAct loop

ReAct loop 会引入：

```text
模型自主选择工具；
多轮 thought-action-observation；
循环终止条件；
工具调用失败恢复；
上下文压缩；
成本控制；
安全边界。
```

这会显著扩大工程复杂度。

V7.0 先做固定 step plan，是为了把工具执行、步骤状态、工具输入输出、最终报告这些基础设施打牢。

---

### 20.2 为什么不做自主 planner

自主 planner 需要让 LLM 生成计划。

这会带来：

```text
计划不可控；
测试不稳定；
工具选择不确定；
失败边界复杂；
prompt 依赖严重。
```

V7.0 先使用固定 plan，保证系统行为稳定。

---

### 20.3 为什么不做 shell / code execution 工具

这类工具有明显安全风险：

```text
执行任意命令；
访问文件系统；
破坏数据；
泄露信息；
需要沙箱。
```

当前项目还没有权限、沙箱和审计系统，所以不能做。

---

### 20.4 为什么保留工具输入输出

因为 Agent 系统如果只保存最终结果，很难排查：

```text
工具输入错了；
检索范围错了；
工具输出为空；
总结阶段失败；
最终报告幻觉。
```

保存 step input / output 可以让每一步可复盘。

---

### 20.5 为什么 no-context 不失败

no-context 不一定是系统错误。

它可能只是：

```text
当前 collection 为空；
没有 active document；
没有 current chunk；
问题和知识库无关。
```

所以推荐 task COMPLETED，并明确说明没有可用上下文，而不是标记 FAILED。

## 26. 面试高强度追问与回答

### 26.1 V7.0 和 V6.0 有什么区别？

回答：

```text
V6.0 的 Agent Task 是一次 RAG 检索加一次 LLM 生成。V7.0 在此基础上引入 Tool Registry 和 Agent Step，把任务拆成固定步骤：先调用 knowledge_search 工具检索知识库，再调用 context_summary 工具总结上下文，最后基于工具结果生成最终报告。每一步都有输入、输出、状态、耗时和错误。
```

---

### 26.2 为什么不做 ReAct？

回答：

```text
ReAct 会让模型自主决定工具调用，并产生多轮 thought-action-observation 循环。这需要处理循环终止、工具安全、成本控制、上下文压缩和失败恢复。V7.0 的目标是先把工具注册、步骤执行、输入输出记录和最终报告链路做稳，所以使用固定 step plan。
```

---

### 26.3 Tool Registry 有什么作用？

回答：

```text
Tool Registry 负责集中管理系统支持的内置工具，包括 toolName、displayName、description、inputSchema 和 execute 逻辑。这样 Agent Task 不需要硬编码所有工具实现，前端也可以通过 GET /agent/tools 展示当前可用工具。
```

---

### 26.4 knowledge_search 工具为什么重要？

回答：

```text
knowledge_search 是 Agent Workflow 的知识入口。它复用已有 scoped retrieval，基于 task objective 和 collectionId 检索相关 chunks，并返回 matchedChunks、citations 和 retrievalMetadata。后续 summary 和 final report 都基于这个工具结果生成。
```

---

### 26.5 工具调用如何保证不绕过生命周期过滤？

回答：

```text
knowledge_search 不是重新写一套检索，而是复用 V5.0 scoped retrieval 和 V4.0 lifecycle filter。候选结果必须同时满足属于指定 Collection、document ACTIVE、chunk current active。这样已删除文档、旧版本 chunks 和 collection 外文档都不会进入工具输出。
```

---

### 26.6 为什么保存 step input / output？

回答：

```text
Agent 系统如果只保存最终结果，很难排查问题。保存每一步 input 和 output，可以看到工具拿到了什么输入、返回了什么结果、哪一步失败、失败原因是什么。这是 Agent Workflow 可观测性的基础。
```

---

### 26.7 no-context 为什么不调用 LLM？

回答：

```text
如果 knowledge_search 没有找到可用上下文，继续调用 LLM 只会增加幻觉风险和成本。V7.0 的设计是 no-context 时不调用 LLM，直接返回确定性结果，说明当前范围内没有可用于执行任务的知识库内容。
```

---

### 26.8 context_summary 是不是也要调用 LLM？

回答：

```text
可以调用 LLM，但默认测试不能依赖真实 LLM。实现上可以在 mock provider 下使用确定性摘要，在 local-ai profile 下通过 LLM Provider 生成更自然的摘要。关键是 provider 抽象和测试隔离。
```

---

### 26.9 为什么工具范围这么保守？

回答：

```text
因为 shell、代码执行、浏览器、外部联网工具都有安全和副作用问题，需要权限、沙箱、审计和失败补偿。当前阶段先做只读的知识库工具，把 Agent Workflow 基础设施搭好。
```

---

### 26.10 这个阶段体现什么工程能力？

回答：

```text
V7.0 体现的是 Agent Infra 的基础能力：工具注册、工具 schema、固定执行计划、步骤级状态机、工具输入输出持久化、工具事件时间线、工具结果生成最终报告，以及和 RAG 范围过滤、文档生命周期过滤的一致性集成。
```
