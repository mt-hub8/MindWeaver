# V2.2.x Production Hardening 面试文档

## 0. 阶段摘要

V2.2.x 是在 V2.2 Mock Embedding & Vector Search 之后插入的可靠性硬化阶段。它没有继续堆 RAG Answer / Agent Runtime，而是回到任务编排底座，补齐并发正确性、消息一致性、执行审计、测试证据和可观测性。

一句话定位：

把 AI Task Orchestrator 从“功能能跑的异步 AI 任务系统”升级为“具备生产级任务编排雏形的可靠后端系统”。

这个阶段的重点不是新增用户可见的 AI 功能，而是让已有的异步任务系统更经得住重复消息、并发 Consumer、MQ 投递失败、重试派发、多次执行审计等后端工程问题。

## 1. 为什么插入 V2.2.x

原本下一步可以做 V2.3 RAG Answer with Citation，把 V2.2 的 TopK chunk 检索结果拼进 Prompt，再调用 MockLlmClient 生成 answer 和 citations。但如果底层任务系统仍然存在可靠性缺口，继续往上堆 RAG 功能会让问题更难定位。

当时底层仍然存在这些风险：

- Consumer 并发重复执行风险；
- createTask 与 RabbitMQ 投递的 dual write problem；
- Retry Scheduler 多实例重复派发风险；
- task 表覆盖每次执行 metadata；
- 缺少 task_attempt 执行证据链；
- 缺少系统化测试和 CI；
- 缺少可观测性指标。

所以先插入 Production Hardening，再继续做 RAG Answer。这个决策体现的是工程优先级：在 AI 应用里，模型调用只是一个环节，任务可靠性、状态一致性、失败恢复、审计证据和可测试性才是平台底座。

### 为什么“先查后改”不安全

原先任务执行入口如果是：

```text
find task
if status in PENDING / RETRY_PENDING
set status = RUNNING
save
```

在单实例、低并发、本地测试里通常能跑通，但它不是强并发安全的。两个 Consumer 或两条重复 MQ 消息可能同时读到同一个 task 还是 PENDING，然后都认为自己可以执行，最终导致同一个任务被重复执行。

这类问题不能只靠应用层 if 判断解决，因为判断和写入之间不是一个不可分割的原子操作。

### 为什么使用数据库原子 update

V2.2.1 把入口升级为数据库条件更新：

```text
update task
set status = RUNNING,
    timeout_at = ?
where id = ?
  and status in (PENDING, RETRY_PENDING)
```

只有数据库返回 `updated == 1` 的执行者才真正 claim 成功，可以继续执行 LLM / Mock LLM。返回 `updated == 0` 说明任务已经被其他执行者 claim、取消、完成、失败或状态不允许执行，当前 Consumer 必须直接跳过。

这个设计把并发仲裁交给数据库，避免重复 MQ 消息或多 Consumer 同时执行同一个任务。它不是全链路 exactly-once，但它把最关键的执行入口变成了单点原子门闩。

## 3. V2.2.2 Reliable Dispatch & Scheduler Safety

本版本做了：

- task_outbox 表；
- Transactional Outbox Pattern；
- createTask 不直接发送 RabbitMQ；
- createTask 只写 task + task_event + task_outbox；
- Outbox Dispatcher 可靠投递 MQ；
- Outbox claim；
- 发送成功标记 SENT；
- 发送失败标记 FAILED 并重试；
- Retry Scheduler 派发安全化。

### 为什么数据库事务管不了 RabbitMQ

如果 createTask 里直接做：

```text
insert task
insert task_event
send RabbitMQ
```

会出现 dual write problem：

- 数据库事务回滚，但 MQ 已经发出；
- 数据库提交成功，但 MQ 发送失败，任务停留在 PENDING，永远没人执行。

数据库事务不能天然覆盖 RabbitMQ。除非引入分布式事务，但这会显著增加复杂度、性能成本和运维难度，对当前阶段不合适。

### 什么是 Transactional Outbox Pattern

V2.2.2 使用 Transactional Outbox Pattern：createTask 不再直接发 MQ，而是在同一个数据库事务中写入：

```text
task
task_event
task_outbox
```

事务提交后，由 Outbox Dispatcher 定时扫描 `task_outbox`，先原子 claim outbox，再发送 RabbitMQ。发送成功标记 SENT，发送失败标记 FAILED 并设置下一次重试时间。

这样，业务数据和“待发送消息”先在同一个数据库事务中达成一致，MQ 投递变成可重试的后台过程。

### Outbox 如何保证多实例安全

Outbox Dispatcher 不直接拿到记录就发送，而是先做数据库原子 claim：

```text
PENDING / FAILED -> PROCESSING
```

只有 claim 成功的 dispatcher 实例才允许投递 MQ。其他实例 claim 失败后跳过，从而避免多实例 dispatcher 重复处理同一条 outbox。

Retry Scheduler 也不再直接发送 MQ，而是将到期 retry 任务安全地写入 outbox，让所有 dispatch 都统一走同一套可靠派发链路。

## 4. V2.2.3 Execution Attempt & Observability

本版本做了：

- task_attempt 表；
- TaskAttemptEntity / Repository / Status；
- claim 成功后创建 RUNNING attempt；
- 执行成功后 attempt -> SUCCESS；
- 执行失败后 attempt -> FAILED；
- attempt 保存 provider、model、prompt、token、latency、error、startedAt、finishedAt；
- task 表继续保留最终摘要；
- 最小结构化日志；
- 可用时基础 metrics。

### 为什么 task 表不能承载所有执行历史

task 表适合保存“当前状态”和“最终摘要”，例如：

- 当前 status；
- 最终 resultContent；
- 最终 errorMessage；
- 最终 llmModel；
- 最终 token usage；
- retryCount。

但如果一个任务经历多次 retry，task 表上的这些字段会被覆盖。比如：

```text
Attempt 1: mock-fast timeout
Attempt 2: mock-smart failed
Attempt 3: mock-smart success
```

如果所有 metadata 都写在 task 表里，最后只能看到 Attempt 3 的结果，Attempt 1 和 Attempt 2 的模型、错误、延迟、token 都会丢失。

### task_attempt 的价值

task_attempt 记录每次执行尝试，提供执行审计证据链。它可以支撑：

- 故障排查：是哪一次 attempt 失败、错误是什么；
- 成本分析：每次 attempt 消耗多少 token；
- 模型路由分析：不同模型在不同任务上的表现；
- SLA 诊断：延迟发生在哪一次调用；
- retry 审计：任务为什么重试、重试几次、最终是否成功。

task 表继续作为查询任务详情的快速摘要表，task_attempt 作为历史明细表，两者职责分离。

## 5. 核心工程权衡

### 为什么先做 reliability，再做 RAG

RAG Answer 是上层能力，依赖底层任务系统可靠执行。如果任务可能重复执行、消息可能丢失、重试证据无法追踪，那么 RAG 结果再漂亮也难以稳定运营。先做 reliability，可以让后续 RAG Answer、Citation、Evaluation 都建立在更可靠的任务底座上。

### 为什么 atomic claim 放在 outbox 之前

Atomic claim 解决的是 Consumer 执行入口的并发正确性。即使后续 Outbox 做到了可靠投递，MQ 仍然是 at-least-once 语义，重复消息仍然可能发生。所以必须先保证 Consumer 收到重复消息时不会重复执行任务。

### 为什么 outbox 放在 task_attempt 之前

Outbox 解决的是任务创建和消息投递之间的一致性。如果消息本身可能丢失，attempt 就没有机会被创建。先保证任务能可靠派发，再记录每次执行 attempt，链路顺序更自然。

### 为什么 task 表暂时保留 metadata 字段

task 表上的 LLM metadata 已经被 GET /tasks/{taskId} 使用。直接删除会破坏 API 兼容性。当前选择是保留 task 表作为最终摘要，新增 task_attempt 作为历史明细。后续如果要做 API 分层，可以再增加 attempts 查询接口，而不是破坏现有接口。

### 为什么不引入 Kafka / 分布式锁 / Debezium

当前项目目标是构建可靠任务编排底座，而不是过早引入复杂基础设施。Kafka、分布式锁、Debezium 都能解决某些生产问题，但会带来更高部署和运维复杂度。当前用 MySQL 条件更新 + RabbitMQ + Outbox 已经能覆盖这个阶段的核心一致性问题。

### 为什么不一口气做完整 observability 平台

完整 Prometheus / Grafana / tracing / dashboard 是后续生产化方向。本阶段先做结构化日志和基础可观测字段，确保 taskId、attemptId、attemptNo、provider、model、latency、error 可以被日志检索和问题复盘使用。

### 为什么不同时做 Agent Runtime

Agent Runtime 会引入 tool calling、planning、memory、multi-step execution 等复杂链路。如果任务编排底座还没有可靠派发、幂等、审计和测试证据，Agent 层会把问题放大。先硬化底座，再做 Agent，是更稳的路线。

## 6. 数据库设计总结

- `task`：保存当前任务状态和最终摘要，适合 GET /tasks/{taskId} 快速返回；
- `task_event`：保存生命周期事件，例如 TASK_CREATED、STATUS_CHANGED；
- `task_outbox`：保存待投递消息，实现可靠派发和 MQ 投递重试；
- `task_attempt`：保存每次执行尝试，包括模型、provider、prompt、token、latency、error、startedAt、finishedAt；
- `task_output_chunk`：保存任务输出片段，为 persisted output chunks 和后续 streaming/polling 打基础；
- `document` / `document_chunk` / `document_chunk_embedding`：保存 RAG 前置检索链路，包括文档、chunk、embedding 和 search metadata。

这些表不是重复设计，而是职责拆分：

- task 关注“当前是什么状态”；
- task_event 关注“生命周期发生过什么”；
- task_outbox 关注“哪些消息需要可靠发出去”；
- task_attempt 关注“每次执行尝试具体发生了什么”；
- document 相关表关注“知识如何被切分和检索”。

## 10. 面试高强度追问与回答

### Q1：为什么不直接做 V2.3 RAG Answer，而先做 Production Hardening？

A1：

RAG Answer 是上层能力，它依赖任务创建、异步调度、检索、LLM 调用和结果保存这些底层链路。如果底层还存在重复执行、消息丢失、重试不可追踪等问题，继续做 RAG Answer 会让系统看起来功能更完整，但可靠性更差。

在这个项目里，V2.2 已经跑通了 Mock Embedding 和 Vector Search，但任务系统仍然有并发 claim、dual write、retry dispatch、attempt audit 等工程缺口。先做 Production Hardening，是为了让后续 RAG Answer 有稳定的执行底座。

当前局限是这还不是完整生产平台，但它已经把关键可靠性问题拆开并逐步解决。后续做 V2.3 时，RAG Answer 可以复用已有的 outbox、atomic claim、attempt 记录和测试基线。

### Q2：Atomic Claim 解决了什么问题？

A2：

Atomic Claim 解决的是多个 Consumer 或重复 MQ 消息同时执行同一个 task 的问题。原来的 find-then-save 在并发下可能让两个执行者同时读到 PENDING，然后都进入执行。

项目里通过数据库条件 update 做 claim：只有 status 仍然是 PENDING 或 RETRY_PENDING 时，才能更新为 RUNNING。数据库返回 updated == 1 的执行者继续执行；updated == 0 的执行者直接跳过。

它解决的是入口并发正确性。局限是它不能保证整个系统 exactly-once，因为 MQ 和 Outbox 仍然可能重复投递，所以 Consumer 幂等仍然必须保留。

### Q3：为什么应用层 synchronized 不能解决这个问题？

A3：

`synchronized` 只在单 JVM 内有效。真实部署里可能有多个应用实例、多个 Consumer、多个进程，甚至重启后锁状态也会消失。RabbitMQ 的重复消息可能被不同实例消费，应用层锁无法覆盖这种分布式场景。

数据库是所有实例共享的状态源，把 claim 放到数据库条件 update 中，才能跨实例生效。这个项目选择数据库原子更新，是因为它简单、直接、和当前 MySQL 技术栈匹配。

后续如果规模更大，可以考虑分布式锁、乐观锁 version 或任务分片，但当前阶段数据库原子 claim 足够覆盖核心风险。

### Q4：为什么数据库原子 update 比 find-then-save 更可靠？

A4：

find-then-save 包含两个步骤：先读，再写。两个执行者可能在读的时候都看到 PENDING，然后都继续执行。数据库原子 update 把“判断状态”和“修改状态”合并成一个不可分割的 SQL 操作。

在项目里，claim SQL 的 where 条件包含 taskId 和允许的状态。只有当前状态仍然符合条件时才会更新成功。并发场景下，第一个更新成功后，第二个再执行时 where 条件已经不满足，updated rows 会是 0。

这能让数据库承担并发仲裁，而不是依赖应用层判断。

### Q5：DB 成功但 MQ 失败怎么办？

A5：

这就是 dual write problem。如果 task 已经插入成功，但 RabbitMQ 发送失败，任务会停在 PENDING，没人消费。V2.2.2 通过 Transactional Outbox 解决这个问题。

createTask 不直接发 MQ，而是在同一个数据库事务中写 task、task_event、task_outbox。只要数据库提交成功，就一定有 outbox 记录。之后 Outbox Dispatcher 负责发送 MQ，失败就标记 FAILED 并延迟重试。

这样 MQ 失败不会导致任务永久丢失，只会变成 outbox 重试。

### Q6：什么是 Transactional Outbox Pattern？

A6：

Transactional Outbox 是一种避免数据库和消息队列双写不一致的模式。核心思想是：业务数据和待发送消息先写入同一个数据库事务，事务提交后，再由后台 dispatcher 从 outbox 表读取并投递消息。

在项目中，createTask 写入 task、task_event 和 task_outbox。Outbox Dispatcher 扫描 PENDING / FAILED 的 outbox，claim 成 PROCESSING，发送 RabbitMQ，成功后标记 SENT，失败后标记 FAILED 并设置 next_retry_at。

它不需要分布式事务，复杂度比 XA 低，也更符合大多数后端系统的实际工程做法。

### Q7：Outbox 是否能保证 exactly-once？

A7：

不能。Outbox 更接近 at-least-once。它能保证消息不会因为 MQ 临时失败而永久丢失，但仍然可能因为 dispatcher 重试、网络超时、确认丢失等情况导致重复投递。

所以 Outbox 必须和 Consumer 幂等配合。在项目里，Consumer 入口有 atomic claim，即使同一个 taskId 被重复投递，也只有一个执行者能把任务从 PENDING / RETRY_PENDING claim 到 RUNNING。

这就是工程权衡：不用承诺 exactly-once，而是通过 at-least-once + idempotent consumer 达到业务可接受的一致性。

### Q8：为什么 Outbox 仍然需要 Consumer 幂等？

A8：

因为 Outbox 只负责可靠投递，不保证消息只投递一次。比如 dispatcher 发出 MQ 后，在标记 SENT 前进程崩溃，下一次扫描可能再次发送同一条 outbox。RabbitMQ 本身也可能出现重复投递。

Consumer 幂等是最后一道防线。项目中的 `tryStartTaskExecution` 使用数据库 atomic claim，确保重复消息不会导致重复执行。

换句话说，Outbox 解决“消息别丢”，Consumer 幂等解决“重复消息别重复执行”。

### Q9：Retry Scheduler 多实例下有什么风险？

A9：

如果多个 Retry Scheduler 实例同时扫描到同一个 RETRY_PENDING 任务，并都直接发送 MQ，就会造成短时间重复投递。虽然 Consumer claim 能防止重复执行，但上游重复消息会增加系统噪音和资源消耗。

V2.2.2 改造后，Retry Scheduler 不再直接发 MQ，而是先对 retry task 做原子 reserve，成功后写 task_outbox。最终仍然由 Outbox Dispatcher 统一派发。

这让首次派发和 retry 派发走同一套可靠投递机制，减少特殊路径。

### Q10：task_event 和 task_attempt 有什么区别？

A10：

task_event 记录任务生命周期事件，例如创建、状态变化。它回答的是“任务状态什么时候从哪里变到哪里”。

task_attempt 记录每次执行尝试的细节，例如 attemptNo、provider、model、renderedPrompt、token、latency、error、startedAt、finishedAt。它回答的是“某一次执行尝试具体发生了什么”。

两者互补：task_event 更像生命周期日志，task_attempt 更像执行审计明细。

### Q11：为什么 task 表不直接保存所有 attempt 信息？

A11：

task 表是当前状态和最终摘要表。如果把所有 attempt 信息都塞进 task 表，会导致结构膨胀、字段重复、历史覆盖和查询混乱。

多次 retry 时，每次模型、prompt、token、latency、error 都可能不同。用 task_attempt 独立建表，可以自然表达一对多关系，也方便后续按 taskId 查询历史。

task 表保留最终摘要，task_attempt 保存历史明细，是更清晰的职责划分。

### Q12：为什么不删除 task 表上的 LLM metadata 字段？

A12：

因为这些字段已经是现有 GET /tasks/{taskId} 响应的一部分，删除会破坏 API 兼容性。当前选择是保留 task 表字段作为最终摘要，同时新增 task_attempt 保存每次执行明细。

这样既不破坏已有接口，又为后续审计和成本分析打基础。未来可以新增 attempts 查询 API，再逐步把展示层拆得更清楚。

### Q13：这套方案扩大 10 倍数据量后瓶颈在哪里？

A13：

可能的瓶颈包括：outbox 扫描压力、task_attempt 历史表增长、数据库条件 update 竞争、Retry Scheduler 扫描效率、RabbitMQ 队列堆积，以及后续向量检索的内存 exact scan。

当前方案适合项目阶段的 baseline。扩大 10 倍后，可以考虑 outbox 分片、批量 claim、归档历史 attempt、增加索引、消费者水平扩展、引入真正的向量数据库和更完整的监控告警。

但在没有真实流量和评测数据前，不应该提前引入过重架构。

### Q14：如果商业化，普通用户和 VIP 用户的 retry / dispatch / observability 可以怎么分层？

A14：

普通用户可以使用默认 retry 次数、默认 topK、默认 dispatch 优先级和基础日志。VIP 用户可以配置更高 maxRetry、更大的 topK、更高优先级队列、更详细 attempt 记录、更大的 context budget 和更完整的成本统计。

在当前项目里，task_attempt、outbox、model router、metadata 字段已经为这种分层打了基础。后续可以在任务创建时引入 user tier，再影响 retry policy、model selection、dispatch priority 和 observability detail。

局限是当前还没有用户系统和 quota/cost 模块，所以这只是后续商业化路线。

### Q15：下一步为什么可以继续做 V2.3 RAG Answer？

A15：

因为 V2.2 已经有文档上传、Adaptive Chunking、Mock Embedding、TopK vector search；V2.2.x 又补齐了任务可靠性、Outbox 派发、Atomic Claim、attempt 审计和测试基线。

下一步 V2.3 可以把 TopK chunks 拼成 context prompt，调用 MockLlmClient，返回 answer + citations。底层执行链路已经有足够的可靠性和审计能力，后续出了问题也更容易定位是检索问题、prompt 问题、LLM 调用问题，还是任务调度问题。

这就是先补底座再加功能的价值。
