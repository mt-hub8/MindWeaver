# V2.2.5 - V2.2.7 Project Surface & Hardening Polish 面试文档

## 0. 阶段摘要

V2.2.5 到 V2.2.7 是在 V2.2.x Production Hardening 之后、V2.3 RAG Answer 之前插入的“项目展示面与可靠性 polish”阶段。

这个阶段没有继续堆 RAG / Agent 功能，而是解决三个更基础的问题：

1. README 主体与真实架构不一致；
2. `task_attempt` 已有数据但缺少展示 API；
3. Outbox Dispatcher 的事务边界和最小可观测性还不够清晰。

一句话定位：

这三个版本把项目从“代码能力已经演进，但展示和边界不统一”的状态，进一步整理成“README、API、Outbox、Attempt、Metrics 都能支撑面试展示”的状态。

## 1. 为什么不直接做 V2.3

原本下一步可以做 V2.3 RAG Answer with Citation，但如果 README 主流程还停留在旧的 direct RabbitMQ dispatch，`PATCH /tasks/{taskId}/status` 还像正式 API，`task_attempt` 没有查询 API，那么继续做 V2.3 会让项目展示更混乱。

所以 V2.2.5 到 V2.2.7 的顺序是有意安排的：

- 先统一 README，让项目入口能准确描述当前真实架构；
- 再补 `task_attempt` 展示 API，让执行审计能力可被现场验证；
- 再 polish outbox transaction boundary 和 metrics，让可靠派发链路更像一个可解释的后端系统；
- 最后再进入 V2.3 RAG Answer。

这个阶段体现的是工程作品集意识：不是功能越多越好，而是每一层能力都要能被解释、被验证、被维护。

## 2. V2.2.5 README Architecture Alignment

### 2.1 解决的问题

README 前半部分仍然像旧流程：

```text
POST /tasks
-> task 入库
-> 直接发送 RabbitMQ
-> Consumer 执行
```

但真实代码已经演进到：

```text
POST /tasks
-> task + task_event + task_outbox
-> Outbox Dispatcher
-> RabbitMQ
-> Consumer
-> atomic task claim
-> task_attempt
-> LLM execution pipeline
-> output chunks
-> final task status
```

这种不一致会让项目看起来像是“代码做了很多，但文档还停在旧阶段”。对面试项目来说，这会削弱可信度。

### 2.2 为什么这个问题严重

README 是面试官最先看的入口。它不是一个附属说明，而是项目架构的门面。

如果 README 写的是 direct RabbitMQ dispatch，但代码已经升级为 Transactional Outbox + Outbox Dispatcher + Atomic Claim + Task Attempt，面试官会产生两个疑问：

- 你是否真的理解当前系统演进到了哪里；
- 你的文档是否可信，还是只是功能堆叠后的临时摘要。

对于后端工程项目来说，README 和代码不一致是架构表达问题，不是简单的排版问题。

### 2.3 修复内容

V2.2.5 做的事情是把 README 主体对齐当前真实代码阶段：

- 重构 README 主流程；
- 将 Transactional Outbox、Atomic Claim、Task Attempt 放到核心流程；
- 明确 `createTask` 不再直接发送 RabbitMQ；
- 说明 RabbitMQ 投递由 Outbox Dispatcher 负责；
- 说明 Consumer 入口通过 atomic claim 防止重复执行；
- 修正 Mock Embedding / Semantic Search 已实现边界；
- 将 Real Embedding / Vector DB / RAG Answer / Citation / Evaluation 归入未实现；
- 将 `PATCH /tasks/{taskId}/status` 标记为 dev/internal；
- 保留本地启动命令、测试命令、Docker / RabbitMQ 信息。

### 2.4 面试表达

如果面试官问“为什么要专门修 README”，可以这样回答：

README 是项目的第一层架构表达。这个项目已经从最早的 `task -> MQ -> Consumer` 演进到了 Transactional Outbox、Atomic Claim 和 Task Attempt，如果 README 仍然写旧流程，面试官无法快速理解我真正完成的可靠性设计。修 README 不是文档小事，而是在把代码能力转化成可展示、可验证、可沟通的工程资产。

它也体现了一个后端项目的作品集意识：不只是把功能做出来，还要让系统边界、真实能力、未实现内容和后续路线都能被准确表达。

## 3. V2.2.6 Task Attempt Read API & Dev Status Endpoint

### 3.1 解决的问题

V2.2.3 已经引入了 `task_attempt` 表和写入逻辑，但如果没有查询 API，面试时只能口头说“我记录了每次 attempt”，不能现场展示每次执行尝试的状态、模型、token、latency 和错误信息。

V2.2.6 把 `task_attempt` 从数据库设计变成可展示的执行审计能力。

### 3.2 新增 API

新增正式用户 API：

```text
GET /tasks/{taskId}/attempts
```

当前返回的 `TaskAttemptResponse` 字段包括：

- `attemptId`
- `taskId`
- `attemptNo`
- `status`
- `workerId`
- `llmProvider`
- `llmModel`
- `promptTemplateCode`
- `promptTokenCount`
- `completionTokenCount`
- `totalTokenCount`
- `llmLatencyMs`
- `errorMessage`
- `startedAt`
- `finishedAt`
- `createdAt`
- `updatedAt`

attempts 按 `attemptNo` 升序返回。任务存在但没有 attempts 时返回空数组；任务不存在时按当前实现返回 404。

### 3.3 为什么不返回 renderedPrompt

`renderedPrompt` 当前没有默认返回，原因是：

- prompt 可能很长；
- 后续接真实 LLM 后，prompt 可能包含用户输入或敏感信息；
- attempts API 第一版主要展示执行审计 metadata；
- 如果后续确实需要查看完整 prompt，可以设计单独的 detail API 或 admin-only API，并加权限与脱敏。

这个取舍说明 attempts API 的第一目标是“审计可见”，不是“把所有执行上下文都暴露出去”。

### 3.4 Dev Status Endpoint

历史接口：

```text
PATCH /tasks/{taskId}/status
```

不应该作为正式用户 API。原因是外部用户不能随便把任务状态改成 `RUNNING`、`SUCCESS`、`FAILED` 或其他内部状态。

任务状态应该由系统内部驱动：

- Consumer 驱动 `RUNNING` / `SUCCESS` / `RETRY_PENDING` / `FAILED`；
- Timeout Scheduler 驱动超时失败；
- Cancel API 驱动 `CANCELLED`；
- Dev/Internal API 只用于本地调试。

V2.2.6 将路径迁移为：

```text
PATCH /dev/tasks/{taskId}/status
```

并通过 `dev` profile 控制 `DevTaskController`。

### 3.5 面试表达

如果面试官问“task_attempt 有什么价值”，可以这样回答：

`task` 表保存的是当前状态和最终摘要，但一次任务可能经历多次重试。如果只看 `task`，前几次失败、使用的模型、token、latency 和错误原因都会被后续结果覆盖。`task_attempt` 把每次执行尝试单独记录下来，再通过 `GET /tasks/{taskId}/attempts` 暴露出来，就能支撑 retry 审计、故障排查、模型路由分析和后续成本统计。

如果面试官问“为什么 dev/internal API 不能混进正式 API”，可以这样回答：

状态修改接口是很危险的调试能力。正式用户 API 应该表达业务意图，比如 create、cancel、query，而不是允许外部直接篡改内部状态。把它迁移到 `/dev` 路径并用 `dev` profile 控制，是为了明确生产边界。

## 4. V2.2.7 Outbox Boundary & Minimal Metrics Polish

### 4.1 解决的问题

Transactional Outbox 已经引入，但 dispatcher 的事务边界需要更清晰。

理想处理流程是：

```text
事务 1：claim outbox -> PROCESSING
事务外：send RabbitMQ
事务 2：mark SENT / FAILED
```

不应该把 RabbitMQ send 放在长数据库事务里。RabbitMQ 是外部 I/O，可能变慢、阻塞或失败。如果它被包在数据库事务中，会延长 DB 锁持有时间，也会让事务边界混入不可控外部系统。

### 4.2 Outbox 三阶段处理

V2.2.7 将 outbox dispatch 拆成三个阶段：

1. claim 阶段：短事务，原子更新 outbox 状态为 `PROCESSING`；
2. send 阶段：事务外调用 RabbitMQ；
3. mark result 阶段：短事务，成功标记 `SENT`，失败标记 `FAILED` 并设置 retry 信息。

当前代码中：

- `TaskOutboxDispatcherScheduler` 负责定时扫描和流程编排；
- `TaskOutboxDispatchService#claimDueOutbox` 是短事务；
- `TaskOutboxDispatchService#sendOutboxMessage` 不带事务；
- `TaskOutboxDispatchService#markSent` 是短事务；
- `TaskOutboxDispatchService#markFailed` 是短事务。

### 4.3 stale PROCESSING 恢复

如果 dispatcher claim 成功后进程崩溃，outbox 可能停留在 `PROCESSING`。

为了避免永久卡死，V2.2.7 支持 stale `PROCESSING` outbox 被重新 claim。当前语义是：

```text
status = PROCESSING
and lockedAt < staleThreshold
```

满足条件后，后续 dispatcher 可以重新 claim 这条 outbox，重新发送 RabbitMQ 或进入失败重试流程。

这解决的是“dispatcher 崩溃后恢复处理”的问题，而不是 exactly-once。

### 4.4 这是否保证 exactly-once

不保证 exactly-once。

Outbox + RabbitMQ 通常提供 at-least-once 语义。也就是说，消息可能被重复投递，但不应该丢。

因此仍然需要 Consumer 端 atomic task claim：

- outbox 解决可靠投递；
- RabbitMQ 承载异步消息；
- atomic claim 防止同一个 task 被重复执行；
- task_attempt 记录每次真正执行尝试。

这几个能力组合起来，才构成当前项目的可靠任务系统 baseline。

### 4.5 Minimal Metrics

当前真实代码已接入的 metrics 是 outbox 相关的四个 counter：

- `outbox.claim.success.count`
- `outbox.claim.conflict.count`
- `outbox.dispatch.success.count`
- `outbox.dispatch.failed.count`

它们通过 `MetricsRecorder` 和 Micrometer `MeterRegistry` 记录。

当前没有把以下指标写成已实现：

- `task.execution.success.count`
- `task.execution.failed.count`
- `task.claim.success.count`
- `task.claim.conflict.count`
- `llm.execution.latency`
- `llm.tokens.total`

这些可以作为后续 observability 扩展方向，但不能在当前文档中当作已实现能力。

### 4.6 面试表达

如果面试官问“为什么 Outbox Dispatcher 事务边界重要”，可以这样回答：

Outbox 的核心是把业务数据和待投递消息放进同一个数据库事务，但真正发送 MQ 不应该在这个事务里。正确边界是先短事务 claim，再事务外发送 MQ，最后短事务标记结果。这样可以避免外部 I/O 拉长数据库事务，也能让失败和恢复语义更清楚。

如果面试官问“metrics 为什么重要”，可以这样回答：

可靠性机制必须能被观测。Outbox claim 成功/冲突、dispatch 成功/失败这些指标可以帮助判断 dispatcher 是否正常工作，是否有大量发送失败，是否存在竞争或卡住。当前只是 baseline metrics，还不是完整 observability platform；后续可以接 Prometheus / Grafana、增加 latency、队列深度和业务维度指标。

## 6. 工程权衡

为什么先修 README，再做 V2.3：

如果项目入口还描述旧架构，继续堆 RAG 会让展示更乱。先修 README 是为了让面试官能沿着正确主线理解项目。

为什么 attempts API 不返回 `renderedPrompt`：

`renderedPrompt` 可能很长，也可能包含敏感用户输入。第一版 attempts API 只展示 metadata，更符合审计和安全边界。

为什么不直接删除 dev status 能力，而是迁移到 `/dev`：

这个接口对本地验证状态机仍有价值，但不应该混在正式用户 API 里。迁移到 `/dev` 并通过 dev profile 控制，是保留调试能力和控制风险之间的折中。

为什么 outbox 仍然不是 exactly-once：

Outbox 解决的是可靠投递和 dual write problem，不保证消息只投递一次。重复投递仍然可能发生，所以必须配合 Consumer 端 atomic claim。

为什么 metrics 只做 baseline，不做 Prometheus / Grafana：

当前目标是让关键 outbox 事件可计数，先建立观测点。Prometheus / Grafana 涉及部署、采集、面板和告警策略，不应该在这个版本扩展过深。

为什么没有在这个阶段做 RAG Answer：

RAG Answer 依赖检索、Prompt 构造和答案引用，但项目展示面、API 边界和可靠派发还需要收口。先把底座讲清楚，再做答案生成，会更稳。

## 9. 面试高强度追问与回答

### Q1：为什么你在 V2.3 前先修 README？

README 是项目的第一入口。当前代码已经演进到 outbox、atomic claim、task_attempt，但如果 README 还写 direct RabbitMQ dispatch，面试官会先看到一个过时架构。继续做 V2.3 只会让功能更多、表达更乱。所以我先把 README 对齐真实架构，让项目主线可信，再继续做 RAG Answer。

### Q2：README 和代码不一致为什么是严重问题？

因为 README 是架构契约的一部分。代码可以很复杂，但面试官首先需要从 README 判断项目是否有清晰边界。如果文档和代码不一致，会让人怀疑项目是临时堆出来的，而不是持续演进的工程系统。

### Q3：为什么 createTask 不应该直接发送 RabbitMQ？

因为数据库事务不能覆盖 RabbitMQ。直接发送会产生 dual write problem：DB 回滚但 MQ 已发出，或 DB 提交成功但 MQ 发送失败。Transactional Outbox 把 task、task_event、task_outbox 放在同一 DB 事务里，再由 dispatcher 异步投递 MQ。

### Q4：Transactional Outbox、Atomic Claim、Task Attempt 分别解决什么问题？

Transactional Outbox 解决可靠投递和 DB/MQ 一致性边界；Atomic Claim 解决重复消息和多 Consumer 下的重复执行风险；Task Attempt 解决多次 retry 的执行历史被最终 task 摘要覆盖的问题。三者分别对应派发、执行入口和执行审计。

### Q5：task_event 和 task_attempt 有什么区别？

`task_event` 记录生命周期事件，例如任务创建、状态变化。它回答“任务状态是如何变化的”。`task_attempt` 记录每次实际执行尝试，包括 worker、model、token、latency、error。它回答“每次执行到底发生了什么”。两者互补，不应该互相替代。

### Q6：为什么需要 GET /tasks/{taskId}/attempts？

没有这个 API，`task_attempt` 只是一张数据库表，面试时很难展示。暴露 attempts API 后，可以直接看到每次 retry 的执行记录、模型信息、token 和错误信息，这让执行审计能力可验证。

### Q7：为什么 attempts API 不默认返回 renderedPrompt？

`renderedPrompt` 可能很长，也可能包含用户输入或敏感内容。attempts API 第一版定位是执行审计 metadata，不是 prompt 全量审计。后续如果需要查看 prompt，应通过 admin/detail API，并加权限和脱敏。

### Q8：为什么 PATCH /tasks/{taskId}/status 不应该是正式用户 API？

正式用户 API 应该表达业务动作，比如创建、取消、查询。直接改状态是内部调试能力，外部用户不应该能把任务随便改成 SUCCESS、FAILED 或 RUNNING。否则状态机和执行链路都会被绕过。

### Q9：为什么迁移到 /dev/tasks/{taskId}/status，而不是直接删除？

开发阶段仍需要手动验证状态机和边界场景。完全删除会降低本地调试效率；迁移到 `/dev` 并使用 dev profile，可以保留调试能力，同时明确它不是正式用户 API。

### Q10：为什么 MQ send 不应该放在数据库事务里？

MQ send 是外部 I/O，可能慢、失败或阻塞。如果放在数据库事务中，会延长事务时间和锁持有时间，也让数据库事务受外部系统影响。更好的做法是短事务 claim，事务外 send，再短事务 mark result。

### Q11：Outbox 是否能保证 exactly-once？

不能。Outbox 更接近 at-least-once 投递语义，保证消息尽量不丢，但可能重复。exactly-once 在跨 DB 和 MQ 的场景里很难真正保证，通常靠幂等消费和业务去重来实现效果上的安全。

### Q12：为什么 outbox 仍然需要 consumer atomic claim？

因为 outbox 可能重复投递，RabbitMQ 也可能重复投递。Consumer 如果没有 atomic claim，同一个 task 可能被多个执行者同时执行。Atomic Claim 用数据库条件更新保证只有一个执行者能把任务从 PENDING/RETRY_PENDING 改成 RUNNING。

### Q13：什么是 stale PROCESSING outbox，如何恢复？

dispatcher claim outbox 后会把状态改成 PROCESSING。如果它在发送 MQ 前后崩溃，这条 outbox 可能一直停在 PROCESSING。stale lock 通过 lockedAt 判断是否过期，过期后允许新的 dispatcher 重新 claim，从而恢复处理。

### Q14：metrics 对这个项目的价值是什么？

metrics 让可靠性机制可观察。比如 outbox claim conflict 增多，可能说明多实例竞争；dispatch failed 增多，可能说明 RabbitMQ 不稳定。没有 metrics，系统只能靠日志和人工猜测。当前 metrics 是 baseline，为后续监控平台打点。

### Q15：为什么当前 metrics 只是 baseline？

当前只记录 outbox claim / dispatch 的四个 counter，还没有完整 latency、队列深度、业务维度、告警规则和 dashboard。它能支撑早期可观测性，但还不是完整 observability platform。

### Q16：做完 V2.2.5 ~ V2.2.7 后，为什么可以继续 V2.3？

因为项目入口已经对齐真实架构，attempt 能展示，dev API 边界清楚，outbox 事务边界更合理，关键 outbox metrics 已有 baseline。此时继续做 RAG Answer，不会在一个混乱的底座上继续堆功能。
