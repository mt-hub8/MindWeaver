# V0.7 面试文档：失败处理

## 1. 本版本解决的问题

V0.7 解决任务执行失败后“系统不能只靠日志知道失败”的问题。AI 任务可能因为模型调用、网络、输入或内部异常失败，失败状态和原因必须持久化到 task 表，并通过查询接口返回，方便用户和开发者判断任务为什么没有成功。

## 2. 为什么现在做这个版本

它依赖 V0.6 已经有 Consumer 执行主链路。只有先跑通成功路径，才能补失败路径。它为 V0.8 重试、V1.x LLM 错误处理和后续 task_attempt 审计做准备。此时不做重试，是为了先明确失败终态和错误记录。

## 3. 核心需求边界

本版本增加 error_message、markTaskFailed，并在 prompt 包含 fail / 失败时模拟异常。不做自动重试、不做死信队列、不做告警系统。保持克制是为了把“失败可见”先做扎实。

## 4. 方案选择

采用 task.error_message 保存失败原因，状态流转 RUNNING -> FAILED，错误信息为空时保存默认文案，过长时截断。失败逻辑集中在 TaskService.markTaskFailed，避免执行层直接散落写状态。

## 5. 替代方案与暂时不选的原因

替代方案一是只打印 error 日志。当前不选，因为日志不适合作为用户查询结果，也可能被轮转清理。

替代方案二是失败后立即重试。当前不选，因为还没有 retry_count、next_retry_at 和 Scheduler，直接重试会让失败语义混乱。

## 6. 数据库设计

新增 task.error_message 字段，类型可为空 TEXT。它保存最近一次失败原因，后续 task_attempt 可以保存每次尝试的错误。字段用 TEXT 是为了容纳异常信息，同时业务层截断避免过长内容污染响应。

## 7. API 设计

GET /tasks/{taskId} 返回 errorMessage。创建和执行接口不变。失败任务查询时 status = FAILED，errorMessage 提供可读原因。

## 8. 核心流程

Consumer 执行任务
-> TaskExecutionService 捕获异常
-> TaskService.markTaskFailed
-> 状态机校验 RUNNING -> FAILED
-> 保存 errorMessage
-> 写 STATUS_CHANGED
-> 查询接口返回失败原因

## 9. 测试方案

建议覆盖正常任务 SUCCESS、prompt 包含 fail / 失败时 FAILED、errorMessage 不为空、超长错误截断、非法状态不能标记失败。SQL 可查询 task.error_message 和 task_event。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是失败状态不能只存在执行线程内存中。异常被 catch 后必须落库，并且不能绕过状态机。错误信息还要控制长度，避免数据库和 API 响应被异常堆栈撑爆。

## 12. 本版本亮点

- 将失败从日志提升为可查询业务状态。
- 集中封装 markTaskFailed，保持状态机约束。
- 为重试和执行审计提供失败原因基础。
- 通过模拟失败规则支持本地稳定验收。

## 13. 简历 bullet

### 简洁版

实现任务失败处理与 errorMessage 持久化，支持执行异常后的 FAILED 状态和失败原因查询。

### 强化版

为异步 AI 任务执行链路补齐失败持久化能力，通过 markTaskFailed 集中处理 RUNNING -> FAILED 流转、错误原因规范化和事件记录，为重试与执行审计奠定基础。

## 14. 面试高强度追问与回答

Q1：为什么不能只打日志？

A1：日志不是业务结果，用户和前端无法稳定查询。失败原因进入 task 表后，API 和 SQL 都能直接定位问题。

Q2：errorMessage 为什么要截断？

A2：异常可能包含很长堆栈或外部响应，直接保存会影响数据库和接口性能。截断能保留核心信息并控制风险。

Q3：为什么失败要走状态机？

A3：只有 RUNNING 等合法状态才能失败。终态任务不能被重复改成 FAILED，否则会破坏生命周期语义。

Q4：为什么模拟 fail / 失败？

A4：它提供稳定、本地可复现的失败场景，便于验收失败处理，不依赖外部 LLM。

Q5：失败后为什么不立即重试？

A5：重试需要次数、延迟和调度机制。V0.7 先把失败记录清楚，V0.8 再引入重试。

Q6：FAILED 是不是终态？

A6：在基础状态机中 FAILED 是终态，表示任务已经结束。后续重试会通过 RETRY_PENDING 在失败前拦截。

Q7：错误信息为空怎么办？

A7：保存默认“未知错误”，保证查询接口不会返回无意义空值。

Q8：当前局限是什么？

A8：task 表只能保存最近错误，不保存每次尝试历史。后续 task_attempt 会补齐多次执行证据。
