# V3.2 Document Ingestion UI / Upload-to-Ask Product Loop 面试文档

## 1. 版本摘要

V3.2 的版本名称是：

```text
V3.2 Document Ingestion UI / Upload-to-Ask Product Loop
```

中文可以表述为：

```text
文档摄入页面：从上传文档到知识库问答的产品闭环
```

这个版本的核心目标是补齐 AI Knowledge Assistant 的关键产品闭环：

```text
上传文档
  -> 文本提取
  -> 创建 document
  -> 文本切分 chunks
  -> 生成 embeddings
  -> 写入 VectorStore
  -> Documents 页面查看状态
  -> Ask 页面提问
  -> 返回 answer + citations
```

在 V3.1 中，项目完成了产品化重置：用户打开首页后可以理解 Ask、Documents、Evaluation 这些入口分别做什么。

但 V3.1 仍然有一个关键断点：

```text
用户知道系统可以做知识库问答，但还不能方便地把自己的文档放进去。
```

V3.2 解决的就是这个问题。

本版本支持用户在 Documents 页面上传：

```text
.txt
.md
.pdf
```

其中 PDF 支持边界是：

```text
支持文本型 PDF 的文本提取；
不支持扫描版 PDF OCR；
不支持图片识别；
不保证复杂表格结构还原；
不支持加密 PDF。
```

V3.2 的重点不是继续做新的 RAG 检索算法，而是让系统从“预置数据 demo”升级为“用户可以上传自己的文档并提问”的真实产品闭环。

---

## 2. 为什么现在做 V3.2

V3.1 已经把项目包装成 AI Knowledge Assistant MVP，让用户知道系统能做什么。

但是一个知识库问答系统最核心的用户路径应该是：

```text
我有一份文档；
我上传进去；
我问文档相关问题；
系统基于文档回答；
我能看到答案来源。
```

如果没有文档上传入口，项目仍然像一个只能依赖预置 demo 数据的系统。

V3.2 的价值是把系统从：

```text
可以问已有数据
```

推进到：

```text
用户可以上传自己的文档并提问
```

这一步非常重要，因为它把 AI Knowledge Assistant 从“演示系统”推进到“可实际使用的 MVP”。

---

## 3. 本版本解决的问题

### 3.1 解决用户无法导入自己文档的问题

在 V3.2 之前，用户如果想让系统回答自己的文档内容，需要走手工 ingestion 路径或依赖已有 demo 数据。

这对普通用户不友好。

V3.2 新增 Documents 页面上传入口和最小上传 API，让用户可以直接上传文档。

这让产品路径变成：

```text
Documents 页面上传文件；
上传成功后看到 document 和 chunks；
回到 Ask 页面提问；
通过 citations 追踪答案来源。
```

这比只提供后端接口更接近真实产品体验。

---

### 3.2 解决知识库问答缺少完整闭环的问题

知识库问答产品不只是 Ask 页面。

真正完整闭环是：

```text
数据进入系统；
系统处理数据；
用户可以查看数据状态；
用户基于数据提问；
系统给出答案和引用。
```

V3.2 补齐的是入口链路中的“数据进入系统”。

没有这一步，RAG Answer API、citations、rerank、hybrid retrieval 和 evaluation 都只能围绕已有数据工作。

---

### 3.3 解决 PDF 这种常见文件格式的产品缺口

只支持 `.txt` 和 `.md` 对工程实现更简单，但从用户视角看不够真实。

企业知识库里大量文档是 PDF，例如：

```text
产品手册；
技术方案；
合同说明；
论文；
制度文档；
培训材料；
项目文档导出文件。
```

所以 V3.2 支持文本型 PDF 是合理的产品要求。

同时，本版本明确不做 OCR，避免把边界扩大成文档智能识别系统。

## 4. 核心设计

### 4.1 上传 API 设计

V3.2 新增最小上传接口：

```http
POST /documents/upload
Content-Type: multipart/form-data
```

字段：

```text
file
```

支持类型：

```text
.txt
.md
.pdf
```

上传成功后返回 ingestion summary，例如：

```json
{
  "documentId": 123,
  "title": "example.pdf",
  "status": "COMPLETED",
  "chunkCount": 12,
  "embeddingCount": 12,
  "vectorWriteCount": 12
}
```

这个 response 的意义是：

```text
documentId：新创建的文档 ID；
title / filename：用户上传的文件名；
status：本次摄入是否完成；
chunkCount：切分了多少 chunks；
embeddingCount：生成了多少 embeddings；
vectorWriteCount：写入了多少向量。
```

这些字段让用户和开发者知道 ingestion 是否真正完成，而不是只看到“上传成功”。

---

### 4.2 文件校验设计

上传前后端需要做基础校验：

```text
文件不能为空；
文件类型必须是 .txt / .md / .pdf；
文件大小不能超过限制；
PDF 必须能提取出文本；
不支持的类型返回统一错误响应；
解析失败返回统一错误响应。
```

文件大小建议通过配置控制，例如 2MB 或项目中实际设置的限制。

这样做的原因是：

```text
同步 ingestion 不适合处理超大文件；
大文件会影响请求延迟；
PDF 解析和 embedding 生成都可能比较慢；
先限制大小可以保证 MVP 稳定。
```

---

### 4.3 文本提取抽象

V3.2 建议抽象出 DocumentTextExtractor 或等价组件。

不同文件类型使用不同 extractor：

```text
TxtTextExtractor
MarkdownTextExtractor
PdfTextExtractor
```

职责是：

```text
输入 uploaded file；
输出纯文本 content；
无法提取时抛出清晰异常。
```

这样做的价值是：

```text
文件解析逻辑和 ingestion 主流程解耦；
后续可以扩展 Word、HTML、URL 等格式；
测试更容易覆盖；
错误边界更清楚。
```

---

### 4.4 PDF 文本提取设计

PDF 只支持文本型 PDF。

也就是：

```text
PDF 内部本身包含可复制文本；
通过 PDF 文本提取库可以读出文字。
```

不支持：

```text
扫描版 PDF；
图片型 PDF；
OCR；
图片识别；
复杂表格结构还原；
加密 PDF。
```

如果 PDF 提取后文本为空，应该返回清晰错误，例如：

```text
PDF has no extractable text. Scanned PDFs/OCR are not supported in this version.
```

这条边界非常重要。

因为 OCR 会引入完全不同的系统复杂度，例如：

```text
图像渲染；
OCR 模型；
版面分析；
表格识别；
图片质量处理；
多语言识别；
性能和成本问题。
```

V3.2 是 Upload-to-Ask 产品闭环，不是 OCR 系统。

---

### 4.5 同步 ingestion 流程

V3.2 采用同步 ingestion。

流程是：

```text
file validation
  -> text extraction
  -> create document
  -> split text into chunks
  -> persist chunks
  -> generate embeddings
  -> write vectors to VectorStore
  -> return ingestion summary
```

优点：

```text
实现简单；
产品闭环直观；
用户上传后立即知道是否完成；
不需要引入任务状态机；
不需要接 RabbitMQ；
默认测试更容易控制。
```

局限：

```text
不适合超大文件；
请求耗时可能较长；
失败重试能力有限；
无法展示精细进度。
```

这些局限可以后续通过异步 ingestion 版本解决。

---

### 4.6 复用现有链路

V3.2 不应该重新实现 embedding、chunk、VectorStore 写入逻辑。

正确做法是复用已有组件：

```text
document / chunk 存储能力；
chunking 逻辑；
EmbeddingProvider；
Embedding Cache / Dedup；
VectorStore 写入能力；
统一错误响应。
```

这体现的是系统复用能力，而不是复制代码。

如果文档 chunk embedding 已经接入 embedding cache，那么上传文档时也应该尽量复用该机制：

```text
相同 chunk content 在相同 provider / model / dimension 下，不重复生成 embedding。
```

## 6. 关键取舍

### 6.1 为什么做同步 ingestion，而不是异步

异步 ingestion 更像生产级设计：

```text
上传后创建 task；
后台 worker 处理；
状态 PENDING / PROCESSING / COMPLETED / FAILED；
支持重试；
支持进度展示；
支持取消。
```

但它会引入大量复杂度。

V3.2 的目标是打通产品闭环，不是重建任务编排系统。

所以本版本选择同步 ingestion，并通过文件类型和大小限制控制风险。

后续可以演进到：

```text
V3.3 Async Document Ingestion Pipeline
```

---

### 6.2 为什么支持 PDF，但不支持 OCR

PDF 是常见知识库文件类型，所以支持 PDF 是合理的产品需求。

但 OCR 是另一类问题。

OCR 需要处理：

```text
扫描图像；
文字识别；
版面分析；
表格还原；
图片质量；
识别准确率；
额外模型或服务；
性能和成本。
```

如果把 OCR 放进 V3.2，会让版本从“上传文档到提问”变成“文档智能解析系统”。

所以 V3.2 只支持文本型 PDF。

---

### 6.3 为什么不支持 Word / Excel

Word 和 Excel 也常见，但它们会引入：

```text
Apache POI / Tika 等解析依赖；
表格结构；
样式；
批注；
图片；
多 sheet；
复杂格式处理。
```

当前版本先支持 `.txt / .md / .pdf`，已经足够覆盖大量演示和知识库场景。

---

### 6.4 为什么不做文档删除和编辑

删除和编辑涉及数据一致性：

```text
删除 document；
删除 chunks；
删除 embeddings；
删除 VectorStore 中的 vectors；
处理失败回滚；
避免残留向量；
更新 evaluation 数据。
```

这已经超出 V3.2 的产品闭环目标。

当前只做上传和只读查看。

---

### 6.5 为什么不做文档范围过滤

Ask 页面默认全库检索。

文档范围过滤虽然有用，但需要增加：

```text
documentId 参数；
collection / workspace 概念；
前端选择器；
后端过滤条件；
VectorStore metadata filter；
测试覆盖。
```

这些可以后续作为独立版本处理。

## 11. 面试高强度追问与回答

### 11.1 V3.2 解决了什么核心问题？

回答：

```text
V3.1 解决了用户理解产品的问题，但用户还不能方便地把自己的文档放进系统。V3.2 补齐了 Upload-to-Ask 闭环：用户可以上传 txt、markdown 或文本型 PDF，后端完成文本提取、chunking、embedding 和 VectorStore 写入，然后用户可以在 Ask 页面基于上传文档提问并查看 citations。
```

---

### 11.2 为什么支持 PDF？

回答：

```text
因为 PDF 是企业知识库中非常常见的文件格式，比如产品手册、论文、制度文档、技术方案和导出文档。如果只支持 txt 和 markdown，产品体验会偏 demo。

所以 V3.2 支持文本型 PDF 的文本提取，但明确不支持扫描版 OCR，避免版本边界变成文档智能识别系统。
```

---

### 11.3 为什么不支持扫描版 PDF OCR？

回答：

```text
OCR 是另一类复杂问题。它涉及图像渲染、文字识别、版面分析、表格还原、图片质量和模型成本。如果放进 V3.2，会让版本从“上传文档到问答闭环”扩展成“文档智能解析系统”。

当前版本只支持 PDF 内部已有可复制文本的文本型 PDF。如果提取文本为空，会返回清晰错误。
```

---

### 11.4 为什么用同步 ingestion，而不是异步？

回答：

```text
同步 ingestion 更适合当前 MVP 阶段。它实现简单，用户上传后可以立即知道是否成功，也不需要引入任务状态机、RabbitMQ、进度追踪和失败重试。

我通过限制文件类型和文件大小控制同步链路的风险。后续如果要支持大文件、多文件、失败重试和进度展示，可以单独演进为异步 ingestion pipeline。
```

---

### 11.5 上传后具体发生了什么？

回答：

```text
上传后，系统先校验文件类型和大小，然后根据类型选择对应的 text extractor。txt 和 markdown 按 UTF-8 读取，PDF 使用文本提取库读取可复制文本。提取出文本后，系统创建 document，切分 chunks，生成 embeddings，写入 VectorStore，最后返回 documentId、chunkCount、embeddingCount 和 vectorWriteCount。
```

---

### 11.6 为什么不做 Word / Excel？

回答：

```text
Word 和 Excel 解析会引入更多格式复杂度，比如样式、表格、多 sheet、批注和图片，也通常需要 Apache POI 或 Tika 等依赖。

V3.2 的目标是打通 Upload-to-Ask 产品闭环，不是支持所有文档格式。txt、markdown 和文本型 PDF 已经覆盖了最小可用场景。
```

---

### 11.7 如果 embedding 或 vector write 失败怎么办？

回答：

```text
不能静默成功。上传接口需要返回清晰错误，并走统一 API 错误响应结构，包括 code、message 和 traceId。因为如果向量没有写入成功，Ask 页面就无法检索到内容。这个错误必须暴露给用户和开发者。
```

---

### 11.8 为什么不做文档删除？

回答：

```text
删除不只是删 document 表记录，还要处理 chunks、embedding cache、VectorStore 中的向量，以及 MySQL 和向量库的一致性。如果删除中途失败，还要处理残留数据。

这已经是一个独立的数据生命周期管理问题。V3.2 只做上传和只读查看，先完成产品闭环。
```

---

### 11.9 这个版本体现什么工程能力？

回答：

```text
它体现的是产品闭环和工程复用能力。不是重新写一条孤立上传逻辑，而是把文件校验、文本提取、document/chunk 存储、embedding、VectorStore 写入、Documents 页面和 Ask 页面串起来。

同时我明确控制边界：支持文本型 PDF，但不做 OCR；做同步 ingestion，但不做任务状态机；默认测试用 mock embedding 和 mock vector store，不依赖外部组件。
```

---

### 11.10 和前面 RAG / hybrid / evaluation 版本是什么关系？

回答：

```text
前面的版本解决的是 RAG 的底层能力，比如检索、引用、评估、rerank 和 hybrid search。V3.2 解决的是用户如何把自己的数据放进系统。

上传后的文档会进入同一套 document/chunk/embedding/vector store 链路，后续 Ask 页面仍然可以使用 citations、rerank、hybrid retrieval 和 evaluation 能力。
```

## 14. 总结表达

这个版本可以用一段话总结：

```text
在 V3.2 中，我为 AI Knowledge Assistant MVP 补齐了 Upload-to-Ask 产品闭环。用户可以在 Documents 页面上传 txt、markdown 或文本型 PDF，后端会完成文件校验、文本提取、document 创建、chunking、embedding 生成和 VectorStore 写入，并返回 chunkCount、embeddingCount、vectorWriteCount 等 ingestion summary。随后用户可以在 Ask 页面基于刚上传的文档提问，并通过 citations 查看答案来源。
```
