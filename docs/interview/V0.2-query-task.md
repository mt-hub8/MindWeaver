# V0.2 面试文档：查询任务详情

## 1. 本版本解决的问题

V0.2 解决异步任务系统中“客户端如何知道任务进展”的问题。创建任务只返回 taskId，并不代表任务完成，因此必须提供 GET /tasks/{taskId} 查询接口，让前端、测试工具或其他系统能通过 taskId 获取 prompt、status、时间字段以及后续扩展的错误、结果和元数据。

## 2. 为什么现在做这个版本

它依赖 V0.1 的 taskId 和 task 表。没有查询接口，异步系统就无法闭环，用户只能提交任务却不知道任务是否成功。它为后续前端轮询、状态机验证、失败排查、重试观测和结果展示做准备。此时不做列表、分页、筛选，是因为单任务查询已经足以验证核心生命周期。

## 3. 核心需求边界

本版本只做 GET /tasks/{taskId} 查询单个任务详情。不做任务列表、不做分页筛选、不做权限过滤、不做复杂搜索。不提前做这些能力，是为了优先稳定最常用的任务详情读取路径。

## 4. 方案选择

查询接口通过 TaskController 调用 TaskService，再由 Repository 按 id 查询 task。响应使用 DTO，而不是直接返回 Entity，避免数据库结构和 HTTP API 强绑定。后续即使 task 表新增字段，也可以有选择地暴露给调用方。

## 5. 替代方案与暂时不选的原因

替代方案一是创建任务后同步阻塞直到完成。当前不选，因为这违背异步任务系统设计，长任务会拖垮 HTTP 体验。

替代方案二是只通过日志排查任务状态。当前不选，因为日志不是稳定 API，无法支撑前端展示和自动化验收。

## 6. 数据库设计

本版本未新增数据库结构，主要复用已有 task 表。查询能力依赖 task.id 主键，后续可扩展 user_id、status、created_at 索引用于列表和筛选。

## 7. API 设计

Method: GET

Path: /tasks/{taskId}

Request: path variable taskId。

Response: 返回任务详情 DTO，包含 id、prompt、status、createdAt、updatedAt 等字段。任务不存在时应返回 404。查询和创建分离，是为了符合异步任务“提交后轮询”的使用方式。

## 8. 核心流程

用户请求
-> TaskController 接收 taskId
-> TaskService 查询任务
-> TaskRepository.findById
-> 转换为 TaskDetailResponse
-> 返回任务详情

## 9. 测试方案

建议覆盖已存在 task 查询成功、未知 taskId 返回 404、响应字段不泄漏内部 Entity 细节。API 手动测试可先 POST /tasks 创建，再 GET /tasks/{taskId} 查询。边界条件包括非法 id、空数据库和重复查询。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点在于 API 稳定性。查询接口会随着系统能力增加暴露越来越多字段，如 errorMessage、retryCount、resultContent、llm usage。需要通过 DTO 控制兼容性，而不是把 Entity 直接作为响应。

## 12. 本版本亮点

- 补齐异步任务的可观测入口。
- 支持前端和 Apifox 通过 taskId 轮询任务状态。
- 使用响应 DTO 隔离数据库模型和 API 合约。
- 为后续错误、重试、结果展示提供统一读取路径。

## 13. 简历 bullet

实现任务详情查询接口，支持基于 taskId 获取异步任务状态，为前端轮询和任务追踪提供基础能力。

设计异步任务查询 API，通过 TaskDetailResponse 隔离持久化模型与外部契约，支撑任务生命周期状态、错误信息、LLM 结果和执行元数据的持续演进。

## 14. 面试高强度追问与回答

Q1：为什么异步系统必须有查询接口？

A1：异步系统提交请求后不会立即返回最终结果，客户端需要通过查询接口获取任务状态和结果。没有查询接口，用户只能盲等或查日志，系统不可用。

Q2：为什么查询和创建分离？

A2：创建表示接收任务，查询表示观察状态。分离后 HTTP 请求可以快速返回，也允许任务执行时间远长于一次请求生命周期。

Q3：前端如何使用这个接口？

A3：前端可以创建任务后保存 taskId，按固定间隔轮询 GET /tasks/{taskId}，直到状态进入 SUCCESS、FAILED 或 CANCELLED。

Q4：为什么不直接返回 Entity？

A4：Entity 是数据库映射，不适合作为外部 API。DTO 可以控制字段暴露、命名和兼容性，避免数据库调整影响客户端。

Q5：任务不存在怎么处理？

A5：应该返回 404，明确表示资源不存在。不要返回空对象，否则调用方难以区分任务不存在和任务尚未完成。

Q6：后续如何做任务列表？

A6：可以增加 GET /tasks，支持分页、status 筛选、用户维度过滤和 createdAt 排序。需要配合索引和权限校验。

Q7：查询接口会不会压力很大？

A7：轮询会带来读压力，后续可以通过分页、缓存、长轮询、SSE 或 WebSocket 优化。当前阶段先用简单查询验证闭环。

Q8：V0.2 的局限是什么？

A8：它只支持单任务查询，不支持列表和实时推送。但这足够支撑早期状态验证和后续能力验收。
