# V3.4 Ingestion Event Timeline & Debug Trace 面试文档

## 1. 版本摘要

V3.4 的版本名称是：

```text
V3.4 Ingestion Event Timeline & Debug Trace
```

中文可以表述为：

```text
文档摄入事件时间线与调试追踪
```

这个版本的核心目标是：在 V3.3 异步文档摄入流水线基础上，为每一次文档摄入任务补充完整事件历史。

V3.3 已经实现了：

```text
上传文档
  -> 创建 ingestion task
  -> 异步处理
  -> PENDING / PROCESSING / COMPLETED / FAILED 状态
  -> 中文状态展示
  -> FAILED 最小重试
```

但 V3.3 主要回答的是：

```text
这个任务现在是什么状态？
```

V3.4 要进一步回答：

```text
这个任务经历了哪些步骤？
每一步什么时候发生？
每一步耗时多久？
哪一步失败？
失败错误码是什么？
失败原因是什么？
是否发生过重试？
```

因此，V3.4 把 ingestion task 从“只有当前状态”升级为“可追踪完整处理历史”。

---

## 2. 为什么现在做 V3.4

异步系统只记录当前状态是不够的。

例如一个文档处理失败，如果系统只显示：

```text
FAILED
处理失败
```

用户和开发者仍然不知道：

```text
是 PDF 文本提取失败？
是 chunking 失败？
是 embedding provider 失败？
是 VectorStore 写入失败？
是哪一步耗时特别长？
是否重试过？
重试后有没有再次进入队列？
```

真实企业系统中，只要引入异步任务，就必须考虑可观测性和可调试性。

V3.4 的价值是：

```text
为异步文档摄入任务建立事件时间线，让用户能看懂处理过程，让开发者能定位失败和性能瓶颈。
```

这一步让项目从“异步任务能跑”进一步升级到“异步任务可解释、可排查、可复盘”。

---

## 3. 本版本解决的问题

### 3.1 解决异步任务过程不可见的问题

V3.3 里，用户可以看到任务状态：

```text
待处理；
处理中；
已完成；
失败。
```

但“处理中”这个状态太粗。

V3.4 增加事件时间线后，用户可以看到：

```text
文档已提交；
任务已进入处理队列；
开始处理文档；
正在切分文档内容；
文档片段生成完成；
正在生成文档向量；
文档向量生成完成；
正在写入知识库索引；
处理完成，可以开始提问。
```

这让用户知道系统不是卡住，而是在执行具体步骤。

---

### 3.2 解决失败位置不清的问题

如果 ingestion 失败，V3.4 会记录失败事件：

```text
TASK_FAILED
failedStep
errorCode
errorMessage
traceId
```

例如：

```text
处理失败：生成文档向量时出现错误。
错误码：EMBEDDING_PROVIDER_ERROR
追踪 ID：xxx
```

这样可以明确定位：

```text
失败发生在 embedding 阶段，而不是文档解析或向量写入阶段。
```

---

### 3.3 解决步骤耗时不可分析的问题

V3.4 要记录关键步骤的耗时：

```text
chunking duration；
embedding duration；
vector write duration；
total duration。
```

这能帮助开发者判断瓶颈在哪里。

例如：

```text
chunking 20ms；
embedding 3200ms；
vector write 800ms。
```

可以得出：

```text
主要耗时在 embedding 阶段。
```

后续如果要做优化，就知道应该优先优化 embedding provider、batch embedding 或缓存命中率。

---

### 3.4 解决重试过程不可追踪的问题

V3.3 支持 FAILED 任务的最小重试。

V3.4 进一步记录：

```text
TASK_RETRY_REQUESTED；
TASK_RETRY_QUEUED。
```

这样用户和开发者可以看到：

```text
用户什么时候点击了重新处理；
任务是否重新进入队列；
重试后是否成功或再次失败。
```

这让重试行为也进入可审计、可复盘范围。

## 4. 核心设计

### 4.1 Ingestion Event 表

V3.4 新增 ingestion event 持久化模型，建议表名：

```text
document_ingestion_event
```

核心字段包括：

```text
id；
task_id；
event_type；
step；
status；
message；
display_message；
duration_ms；
metadata_json；
error_code；
error_message；
trace_id；
created_at。
```

字段含义：

```text
task_id：事件归属的 ingestion task；
event_type：事件类型，例如 TASK_CREATED、EMBEDDING_COMPLETED；
step：事件发生在哪个处理步骤；
status：该事件的结果状态；
message：技术消息；
display_message：中文用户提示；
duration_ms：该步骤耗时；
metadata_json：扩展信息，例如 filename、chunkCount、embeddingCount；
error_code：失败错误码；
error_message：失败原因；
trace_id：链路排查 ID；
created_at：事件发生时间。
```

这个表的作用是保存任务生命周期中的每个关键节点。

---

### 4.2 Event Type（事件类型）设计

V3.4 的事件类型可以覆盖完整主链路：

```text
TASK_CREATED：任务已创建；
TASK_QUEUED：任务已进入队列；
TASK_STARTED：开始处理；
TEXT_EXTRACTED：文本已提取；
CHUNKING_STARTED：开始切分；
CHUNKING_COMPLETED：切分完成；
EMBEDDING_STARTED：开始生成向量；
EMBEDDING_COMPLETED：向量生成完成；
VECTOR_WRITE_STARTED：开始写入向量索引；
VECTOR_WRITE_COMPLETED：向量写入完成；
TASK_COMPLETED：任务完成；
TASK_FAILED：任务失败；
TASK_RETRY_REQUESTED：用户请求重试；
TASK_RETRY_QUEUED：重试已进入队列。
```

这套事件类型把 ingestion task 的生命周期拆成可观察节点。

---

### 4.3 Event Recorder 抽象

V3.4 不应该让 Controller、Consumer、Service 到处直接调用 repository 保存事件。

更好的做法是新增集中组件：

```text
DocumentIngestionEventRecorder
```

它的职责是：

```text
统一记录事件；
统一生成中文 displayMessage；
统一记录 durationMs；
统一写入 metadata_json；
统一处理 errorCode / errorMessage / traceId；
避免事件保存逻辑散落在业务代码里。
```

这样后续如果要修改事件格式、中文文案或 metadata 结构，只需要集中调整 Event Recorder。

这体现了工程中的单一职责和可维护性。

---

### 4.4 上传阶段事件记录

在上传阶段，需要记录：

```text
TASK_CREATED；
TEXT_EXTRACTED；
TASK_QUEUED。
```

对应语义：

```text
TASK_CREATED：系统已创建文档处理任务；
TEXT_EXTRACTED：系统已读取文档文本；
TASK_QUEUED：任务已进入后台处理队列。
```

中文提示示例：

```text
文档已提交，系统已创建处理任务。
已读取文档文本，正在准备建立知识库索引。
文档已进入处理队列，请稍候。
```

如果消息发布失败，则不应该出现“任务创建成功但没有消息、没有失败事件”的悬挂状态。

应该记录失败事件，并把任务置为 FAILED 或返回统一错误响应。

---

### 4.5 后台处理阶段事件记录

后台 handler / consumer 需要记录：

```text
TASK_STARTED；
CHUNKING_STARTED；
CHUNKING_COMPLETED；
EMBEDDING_STARTED；
EMBEDDING_COMPLETED；
VECTOR_WRITE_STARTED；
VECTOR_WRITE_COMPLETED；
TASK_COMPLETED。
```

每个 completed 事件应该尽量记录 durationMs。

例如：

```text
CHUNKING_COMPLETED durationMs=35 metadata={"chunkCount":18}
EMBEDDING_COMPLETED durationMs=3200 metadata={"embeddingCount":18}
VECTOR_WRITE_COMPLETED durationMs=850 metadata={"vectorWriteCount":18}
```

这样可以分析每一步耗时。

---

### 4.6 失败事件记录

任何 ingestion 处理异常，都必须记录：

```text
TASK_FAILED
```

失败事件至少包含：

```text
failedStep；
errorCode；
errorMessage；
traceId；
exceptionClass，如实现支持；
```

中文提示应该面向用户：

```text
处理失败：生成文档向量时出现错误。你可以稍后点击「重新处理」。
```

技术字段应该面向开发者：

```text
errorCode=EMBEDDING_PROVIDER_ERROR
traceId=xxx
step=EMBEDDING
exceptionClass=...
```

不能只写日志，不写 event。

也不能让任务一直停留在 PROCESSING。

---

### 4.7 Retry 事件记录

V3.4 在 retry 流程里记录：

```text
TASK_RETRY_REQUESTED；
TASK_RETRY_QUEUED。
```

对应语义：

```text
用户已请求重新处理该文档；
重试任务已进入处理队列。
```

这让失败后的恢复行为也可以被追踪。

## 5. API 设计

### 5.1 查询事件时间线

新增 API：

```http
GET /documents/ingestions/{taskId}/events
```

返回结构：

```json
{
  "taskId": 1001,
  "events": [
    {
      "eventId": 1,
      "eventType": "TASK_CREATED",
      "step": "UPLOADED",
      "status": "COMPLETED",
      "displayMessage": "文档已提交，系统已创建处理任务。",
      "message": "Task created",
      "durationMs": null,
      "metadata": {
        "filename": "example.pdf"
      },
      "errorCode": null,
      "errorMessage": null,
      "traceId": null,
      "createdAt": "2026-07-02T10:01:21"
    }
  ]
}
```

事件按 createdAt 升序返回，形成时间线。

如果 taskId 不存在，返回统一 API 错误响应，而不是返回空数组伪装成功。

---

### 5.2 Task Detail 可选增强

`GET /documents/ingestions/{taskId}` 可以轻量增加：

```text
latestEventMessage；
latestEventAt。
```

这样任务列表可以直接显示最近一条事件。

但 V3.4 的核心 API 仍然是：

```http
GET /documents/ingestions/{taskId}/events
```

## 7. 关键取舍

### 7.1 为什么不直接接 OpenTelemetry

OpenTelemetry、Jaeger、Zipkin 是更完整的分布式 tracing 方案。

但 V3.4 的目标不是建设分布式 tracing 系统，而是让 ingestion task 的业务处理过程可见。

当前系统只需要：

```text
任务事件；
步骤耗时；
失败原因；
前端时间线展示。
```

如果直接接 OpenTelemetry，会引入：

```text
新的依赖；
trace exporter；
collector；
部署配置；
学习成本；
更多环境要求。
```

这会超出当前版本边界。

---

### 7.2 为什么不接 Prometheus / Grafana

Prometheus / Grafana 适合指标监控和趋势展示。

但 V3.4 要解决的是单个任务的处理历史。

两者关注点不同：

```text
Prometheus / Grafana：系统整体指标趋势；
Ingestion Event Timeline：单个任务的生命周期详情。
```

所以本版本先做业务事件时间线。

---

### 7.3 为什么不做复杂图表

复杂图表不是当前必要能力。

用户最需要知道的是：

```text
发生了什么；
什么时候发生；
耗时多久；
失败在哪里；
下一步怎么办。
```

简单的中文时间线比复杂图表更直接。

---

### 7.4 为什么不做全文日志系统

日志系统和事件时间线不同。

日志是开发者排查底层细节的原始材料。

事件时间线是面向业务任务的结构化记录。

V3.4 不是把所有日志搬到页面，而是记录关键业务事件。

## 13. 面试高强度追问与回答

### 13.1 V3.4 为什么要做事件时间线？

回答：

```text
V3.3 已经有异步 ingestion task 和状态追踪，但只看当前状态还不够。如果任务失败或很慢，开发者需要知道它具体卡在哪一步。V3.4 通过 ingestion event timeline 记录任务创建、排队、开始处理、chunking、embedding、vector write、完成、失败和重试等事件，并记录 durationMs、errorCode 和 metadata。这样可以复盘整个任务生命周期。
```

---

### 13.2 status 和 event 有什么区别？

回答：

```text
status 表示任务当前状态，比如 PENDING、PROCESSING、COMPLETED、FAILED。event 表示任务历史上发生过的关键事件，比如 TASK_CREATED、CHUNKING_COMPLETED、EMBEDDING_COMPLETED。

status 是当前结果，event 是历史过程。只有 status 很难排查问题，有 event timeline 才能知道任务具体经历了哪些步骤。
```

---

### 13.3 为什么要设计 Event Recorder？

回答：

```text
如果在 controller、consumer、service 中到处直接保存 event，事件记录逻辑会非常分散，后续修改 displayMessage、metadata 或 error 结构会很困难。

所以我设计 Event Recorder 作为集中入口，统一负责记录事件、生成中文提示、写入 durationMs、metadata、errorCode、traceId。这样事件逻辑更清晰，也更容易测试。
```

---

### 13.4 durationMs 有什么价值？

回答：

```text
durationMs 可以帮助定位性能瓶颈。比如一个任务总耗时 5 秒，其中 chunking 只花了 30ms，vector write 花了 800ms，但 embedding 花了 4 秒，那么优化方向很可能是 embedding provider、batch embedding 或 embedding cache，而不是 chunking。
```

---

### 13.5 为什么不直接接 OpenTelemetry？

回答：

```text
OpenTelemetry 是通用分布式 tracing 方案，适合跨服务链路追踪。但 V3.4 当前要解决的是业务任务级别的处理历史。用户和开发者需要看到这个文档摄入任务经历了哪些业务步骤，以及每一步是否成功。

所以我先做业务事件时间线。后续如果服务拆分变多，再接 OpenTelemetry 会更合理。
```

---

### 13.6 失败事件如何设计？

回答：

```text
任何 ingestion 异常都不能只写日志，也不能让任务停在 PROCESSING。系统需要记录 TASK_FAILED 事件，更新 task 状态为 FAILED，并写入 failedStep、errorCode、errorMessage 和 traceId。前端展示中文失败原因，开发者可以查看技术详情。
```

---

### 13.7 retry 为什么也要记录事件？

回答：

```text
重试是任务生命周期的一部分。如果不记录 retry 事件，就无法知道任务是第一次处理失败，还是用户后来重新处理过。V3.4 记录 TASK_RETRY_REQUESTED 和 TASK_RETRY_QUEUED，让恢复行为也可追踪。
```

---

### 13.8 为什么前端要显示中文时间线，而不是直接显示 eventType？

回答：

```text
eventType 是给开发者和测试看的，比如 EMBEDDING_COMPLETED、VECTOR_WRITE_COMPLETED。普通用户更需要看到“正在生成文档向量”“正在写入知识库索引”这种中文解释。

所以 API 和页面都要区分 displayMessage 和技术字段，做到用户友好和工程可排查并存。
```

---

### 13.9 事件时间线和日志有什么区别？

回答：

```text
日志是开发者排查系统细节的原始信息，通常非结构化、数量大。事件时间线是围绕某个业务任务抽象出来的结构化关键事件。

V3.4 不是把所有日志展示出来，而是记录和展示 ingestion task 的关键业务步骤。
```

---

### 13.10 这个版本体现什么工程能力？

回答：

```text
它体现的是异步任务系统的可观测性设计。不是只让任务能异步执行，而是让任务可以被追踪、被解释、被调试。包括事件建模、事件持久化、Event Recorder 抽象、步骤耗时、失败定位、重试追踪和中文用户提示。
```

## 16. 总结表达

这个版本可以用一段话总结：

```text
在 V3.4 中，我在 V3.3 异步文档摄入流水线基础上新增了 ingestion event timeline。系统会记录文档摄入任务从创建、排队、开始处理、chunking、embedding、VectorStore 写入，到完成、失败和重试的关键事件，并为每个事件保存 eventType、step、displayMessage、durationMs、metadata、errorCode、errorMessage 和 traceId。

Documents 页面可以用中文展示“处理记录 / 任务时间线”，让普通用户知道文档当前经历了哪些步骤，也让开发者能够通过技术详情定位失败位置和性能瓶颈。这个版本没有接 OpenTelemetry、Prometheus 或复杂 tracing 系统，而是先实现业务任务级别的结构化事件追踪，体现了异步 AI 任务系统的可观测性、可调试性和产品化交互能力。
```
