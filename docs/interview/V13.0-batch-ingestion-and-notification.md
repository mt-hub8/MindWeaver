# V13.0 Batch Ingestion & Notification 面试文档

## 1. 阶段摘要

V13.0 的阶段名称是：

```text
V13.0 Batch Ingestion & Notification
```

中文可以表述为：

```text
批量文档导入与通知中心
```

这一阶段的核心目标是：在本地个人知识库场景下，实现可控、可恢复、可观察的批量文档导入能力，并配套站内通知中心。

项目当前定位是：

```text
Personal AI Knowledge Workspace
个人 AI 知识工作台
```

V13.0 不是要把系统做成云端高并发文件处理平台，而是解决个人知识库真实使用中的一个核心问题：

```text
用户不可能每次只上传一个文件。
当用户一次导入几十个甚至上百个文档时，系统需要知道：
- 这是哪个批次；
- 每个文件处理到哪一步；
- 哪些成功；
- 哪些失败；
- 哪些重复被跳过；
- 是否可以重试失败文件；
- 是否可以取消剩余任务；
- 完成后如何提醒用户。
```

---

## 2. 为什么需要 V13.0

在 V13.0 之前，项目已经具备了：

```text
单文件上传；
异步文档摄入；
文档处理事件时间线；
文档重新索引；
知识库分组；
范围检索；
RAG Answer；
RAG 质量评分；
垃圾箱与本地存储管理；
Agent Task；
Tool-Using Agent Workflow；
多模型供应商配置。
```

但是单文件上传无法支撑真实知识库构建场景。

个人用户可能会导入：

```text
课程资料；
项目文档；
论文 PDF；
Markdown 笔记；
面试资料；
产品说明书；
代码文档；
历史报告。
```

如果没有批量导入系统，只支持 `POST /documents/upload`，用户体验会很差，而且系统无法统计批量任务进度、失败原因和重复文件。

V13.0 的意义是把“上传文档”升级为“批量构建知识库”的基础能力。

---

## 3. 核心设计原则

V13.0 的设计原则是：

```text
批量上传不是简单 multiple file upload；
必须有 batch / item 建模；
单文件处理必须复用已有 ingestion pipeline；
去重必须先做确定性去重；
通知先做站内通知；
并发必须适合本地机器；
默认测试必须低依赖。
```

### 3.1 为什么不是简单 multiple files

简单的：

```html
<input type="file" multiple>
```

只能让前端一次选择多个文件，但无法表达：

```text
批次状态；
每个文件状态；
失败重试；
重复跳过；
取消剩余任务；
导入完成通知；
批量导入报告。
```

因此 V13.0 采用：

```text
UploadBatch
UploadBatchItem
DocumentIngestionTask
Notification
```

这种结构化模型。

### 5.2 item 与 ingestion task 的关系

V13.0 不重新写文件解析、chunk、embedding、vector write。

正确关系是：

```text
UploadBatchItem
  -> DocumentIngestionTask
      -> existing document ingestion pipeline
```

也就是：

```text
Batch 管理多个文件；
Item 记录单个文件状态；
单个文件仍走原有 ingestion pipeline。
```

这样可以复用已有的文档处理、事件时间线、失败记录和重新索引能力。

## 6. 核心设计三：批量上传 API

新增 API：

```http
POST /documents/batches/upload
```

请求类型：

```text
multipart/form-data
```

字段：

```text
files：多文件；
collectionId：可选；
batchName：可选；
duplicatePolicy：可选。
```

### 6.1 duplicatePolicy

建议枚举：

```text
SKIP
IMPORT_ANYWAY
USE_EXISTING
```

默认：

```text
SKIP
```

### 6.2 行为流程

批量上传流程：

```text
1. 创建 upload_batch；
2. 对每个文件计算 fileHash；
3. 将文件保存到 staging area；
4. 为每个文件创建 upload_batch_item；
5. 检查文件级重复；
6. 非重复文件进入已有 document ingestion pipeline；
7. 每个文件创建或关联 document_ingestion_task；
8. 返回 batchId 和初始统计。
```

响应示例：

```json
{
  "batchId": 1,
  "status": "QUEUED",
  "totalCount": 10,
  "queuedCount": 8,
  "duplicateCount": 2,
  "message": "已创建批量导入任务"
}
```

## 7. 核心设计四：查询、重试和取消 API

建议 API：

```http
GET /documents/batches
GET /documents/batches/{batchId}
GET /documents/batches/{batchId}/items
POST /documents/batches/{batchId}/retry-failed
POST /documents/batches/{batchId}/cancel
```

### 7.1 retry-failed

只重试：

```text
FAILED item
```

不重试：

```text
COMPLETED；
SKIPPED_DUPLICATE_FILE；
SKIPPED_DUPLICATE_TEXT；
CANCELED。
```

限制：

```text
retryCount 超过 maxRetryCount 后不再重试；
重试时复用 staging_file_path；
如果 staging 文件不存在，返回清晰错误。
```

### 7.2 cancel batch

取消语义是：

```text
停止剩余处理，不回滚已成功导入的文档。
```

行为：

```text
batch 标记 CANCEL_REQUESTED；
PENDING / QUEUED item 标记 CANCELED；
PROCESSING item 不强杀，允许当前文件完成或失败；
不删除已经成功导入的文档；
不删除已经完成的 item。
```

页面必须说明：

```text
取消批量导入不会删除已经成功导入的文档。
```

## 8. 核心设计五：去重策略

V13.0 只做确定性去重。

### 8.1 文件级去重

计算：

```text
fileHash = sha256(file bytes)
```

如果已有 ACTIVE 文档拥有相同 fileHash：

```text
duplicatePolicy = SKIP：
  item 标记 SKIPPED_DUPLICATE_FILE；
  记录 duplicateOfDocumentId；
  不重复创建 document；
  不重复 ingestion；
  batch duplicateCount +1。
```

```text
duplicatePolicy = USE_EXISTING：
  关联已有文档；
  可将已有文档加入目标 Collection；
  不重新解析和 embedding。
```

```text
duplicatePolicy = IMPORT_ANYWAY：
  允许创建新 document；
  记录 duplicate warning。
```

如果重复文档在 TRASHED：

```text
不自动恢复；
提示“相同文件位于垃圾箱，可先恢复或选择重新导入”。
```

如果重复文档已 PURGED：

```text
视为不存在，允许重新导入。
```

### 8.2 文本级去重

文本提取后计算：

```text
textHash = sha256(normalized extracted text)
```

如果不同文件但提取文本相同：

```text
默认 SKIP；
item 标记 SKIPPED_DUPLICATE_TEXT；
记录 duplicateOfDocumentId；
不重复 embedding；
不重复写 vector。
```

文本级去重发生在解析之后、embedding 之前。

### 8.3 Chunk 级去重

复用已有 embedding cache：

```text
chunkHash + provider + model + dimension
```

V13.0 不重新实现 chunk 级去重。

### 8.4 为什么不做语义近似去重

V13.0 不做：

```text
SimHash；
MinHash；
embedding similarity dedup；
自动合并近似文档。
```

原因：

```text
近似去重可能误判；
个人知识库中文档可能内容相似但用途不同；
第一版应优先做确定性、可解释、不误删的数据去重。
```

## 9. 核心设计六：staging area

批量上传必须有 staging area。

建议配置：

```properties
app.batch-ingestion.staging-dir=data/staging/batches
```

原因：

```text
批量导入过程中，文件处理可能失败；
失败文件需要支持重试；
如果只把文件存在临时内存或临时目录，重试时可能找不到原始文件。
```

staging 策略：

```text
上传后先保存到 staging；
成功导入后可以清理 staging；
失败时保留 staging 以便重试；
永久取消或清理批次时再清理 staging。
```

## 10. 核心设计七：并发控制

V13.0 是本地个人知识库，不是云端高并发系统。

建议配置：

```properties
app.batch-ingestion.max-files-per-batch=100
app.batch-ingestion.document-parse-concurrency=2
app.batch-ingestion.embedding-concurrency=1
app.batch-ingestion.max-retry-count=2
app.batch-ingestion.enabled=true
```

### 10.1 为什么并发要保守

本地机器资源有限：

```text
文档解析吃 CPU；
PDF 解析可能占内存；
embedding 可能占 GPU / CPU；
向量写入可能占 I/O；
LLM 不应参与批量导入并发。
```

过高并发会导致：

```text
内存飙升；
Ollama / Python Worker 超时；
文档处理失败；
用户电脑卡死。
```

因此第一版保守并发更合理。

## 19. 简历 bullet（中文）

批量导入方向：

```text
- 设计并实现批量文档导入模块，新增 UploadBatch / UploadBatchItem 模型，将多文件上传拆解为可观察、可重试、可取消的文件级处理任务，并复用已有单文件 ingestion pipeline。
```

去重方向：

```text
- 实现文件级 SHA-256 去重与文本级 normalized textHash 去重，避免重复文件和重复内容反复进入 embedding 与向量写入流程，同时复用已有 chunk 级 embedding cache。
```

通知方向：

```text
- 新增站内通知中心，在批量导入完成、部分失败、失败或取消时生成通知，并支持未读数量、标记已读和跳转批次详情。
```

可靠性方向：

```text
- 为批量导入设计失败重试、取消剩余任务、staging 文件保留和批次统计机制，保证单个文件失败不会影响整个批次处理。
```

## 20. 面试高强度追问与回答

### 20.1 为什么批量上传不能只是 multiple files？

回答：

```text
multiple files 只能解决一次选择多个文件的问题，但无法表达批次状态、每个文件状态、失败重试、重复跳过、取消剩余任务和完成通知。真实产品需要 UploadBatch 和 UploadBatchItem 来管理批次和文件项。
```

### 20.2 为什么必须复用已有 ingestion pipeline？

回答：

```text
已有单文件 pipeline 已经包含解析、chunk、embedding、vector write、事件记录和错误处理。如果批量导入重新写一套，会产生重复逻辑和代码债。V13.0 只负责批次管理和文件分发，单文件处理继续复用已有 pipeline。
```

### 20.3 为什么先做文件级和文本级去重，不做语义去重？

回答：

```text
文件级和文本级去重是确定性的，不容易误判。语义近似去重可能把相似但用途不同的文档误跳过，影响用户知识库完整性。因此第一版先做可解释、低风险的去重。
```

### 20.4 取消批量导入为什么不回滚已成功文件？

回答：

```text
本地个人知识库的取消语义应该是停止剩余处理，而不是事务级回滚。已经成功导入的文档可能已经建立索引并可用于问答，强行回滚会增加复杂度，也可能误删用户数据。
```

### 20.5 为什么不做邮件通知？

回答：

```text
邮件通知需要 SMTP、邮箱授权码、失败重试和隐私提示，会引入新的凭据管理问题。V13.0 先做站内通知中心，满足批量任务完成提醒。邮件可以后续作为 Notification Provider 扩展。
```

## 22. 总结表达

V13.0 可以用一段话总结：

```text
在 V13.0 中，我将单文件上传扩展为批量文档导入系统，设计了 UploadBatch 和 UploadBatchItem 两级模型，使用户可以一次上传多个文档，并查看批次总进度、单文件状态、失败原因和重复跳过结果。批量导入并没有重写文档处理链路，而是复用已有单文件 ingestion pipeline，从而避免重复实现解析、chunk、embedding 和向量写入逻辑。

同时，我实现了文件级 SHA-256 去重、文本级 textHash 去重和已有 embedding cache 复用，并新增站内通知中心，在批量导入完成、部分失败、失败或取消时提醒用户。V13.0 的价值在于让个人知识库从单文档导入升级为可观察、可恢复、可重试的资料集导入能力。
```
