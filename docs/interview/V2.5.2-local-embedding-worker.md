# V2.5.2 Local Embedding Worker 面试文档

## 0. 版本摘要

V2.5.2 新增 Local Embedding Worker 接入能力。Java 侧通过 `LocalEmbeddingWorkerProvider` 调用本地 Python FastAPI Worker，由 Worker 加载 `sentence-transformers` 模型并生成 embedding。

一句话定位：

```text
V2.5.2 把 EmbeddingProvider 从“云端 API 可接入”扩展到“本地开源模型服务可接入”。
```

本版本边界：

- 不做 Vector DB；
- 不做 Rerank；
- 不做 Hybrid Search；
- 不做 Evaluation Persistence；
- 不做 Docker Compose 集成；
- 不做 GPU 调度；
- 不做 worker 注册中心；
- 不做生产级模型服务治理；
- 默认 provider 仍然是 `mock`；
- 默认 Maven test 不启动 Python Worker；
- 默认 Maven test 不下载模型；
- 默认 Maven test 不访问外部网络；
- 默认 Maven test 不需要 Python 环境；
- 不新增生产 API；
- 不新增数据库表；
- 不新增 Flyway；
- 不修改 README。

## 1. 本版本解决的问题

V2.5 已经支持 OpenAI-compatible 云端 embedding provider，但云端 API 会带来 API key、费用、外部网络、企业文档出公网、合规和私有化部署等问题。

V2.5.2 解决的是：本地 embedding 模型如何以 worker 方式接入 Java 后端，同时继续复用 `EmbeddingProvider` 抽象。Java 后端负责业务链路、metadata 管理和结果校验，Python Worker 负责模型加载和向量推理。

本版本也解决了测试边界问题：默认测试验证 Java provider 逻辑，不启动 FastAPI，不下载 Hugging Face 模型，不访问外部网络，也不要求本机具备 Python 环境。

## 2. 为什么现在做 V2.5.2

版本关系如下：

```text
V2.5   EmbeddingProvider 抽象和 OpenAI-compatible Provider
V2.5.1 Provider Benchmark Comparison
V2.5.2 Local Embedding Worker
```

有了 `EmbeddingProvider` 抽象后，`local-worker` 只是新的 provider 实现，不需要改 `DocumentEmbeddingService` 的核心逻辑。文档 embedding、query embedding、metadata 保存和 search 候选过滤仍然走同一条 Java 链路。

有了 V2.5.1 的 benchmark comparison 后，未来可以用同一套 benchmark 对比 `mock`、`openai`、`local-worker` 等 provider 的检索指标。V2.5.2 先完成接入能力，不编造真实模型效果，也不声称 local-worker 已经带来 Recall@K、Precision@K、NDCG@K 或 MRR 提升。

## 3. 核心架构设计

整体链路：

```text
DocumentEmbeddingService
-> EmbeddingProvider
   -> MockEmbeddingClient
   -> OpenAiCompatibleEmbeddingProvider
   -> LocalEmbeddingWorkerProvider
      -> LocalEmbeddingWorkerClient
         -> RestClientLocalEmbeddingWorkerClient
            -> Python FastAPI Worker
               -> sentence-transformers model
```

`LocalEmbeddingWorkerProvider` 实现 `EmbeddingProvider`，负责 single input、batch input、调用 client、把 worker response 转成统一的 `EmbeddingResponse`，并处理 empty data、empty vector、batch size mismatch、dimension mismatch 和 client exception。真实代码中，`dimension` 会被校验；`provider` / `model` 作为 metadata 回传，并在 worker 未返回时按当前 provider 和配置 model 归一化。

`LocalEmbeddingWorkerClient` 是 Java 侧调用本地 worker 的客户端接口。它把 HTTP 细节从 provider 中拆出去，测试时可以替换为 fake client，因此 `LocalEmbeddingWorkerProviderTest` 不需要启动真实 worker。

`RestClientLocalEmbeddingWorkerClient` 是真实 HTTP 调用实现，使用 Spring `RestClient`，请求路径为 `{baseUrl}/embeddings`，实际 `.uri("/embeddings")` 会拼到配置的 `baseUrl` 后面。

`LocalEmbeddingWorkerRequest` / `LocalEmbeddingWorkerResponse` 是 Java 与 Python Worker 之间的协议对象。请求包含 `model` 和 `input`，响应包含 `provider`、`model`、`dimension` 和 `data`。

Python FastAPI Worker 提供 `POST /embeddings`，使用 `SentenceTransformer` 加载模型并调用 `model.encode(...)` 生成向量。

## 4. Java 侧配置设计

默认 provider 仍然是：

```properties
app.embedding.provider=mock
```

启用本地 worker 时使用：

```properties
app.embedding.provider=local-worker
app.embedding.local-worker.base-url=http://127.0.0.1:8001
app.embedding.local-worker.model=sentence-transformers/all-MiniLM-L6-v2
app.embedding.local-worker.dimension=384
app.embedding.local-worker.timeout-ms=10000
```

`EmbeddingProperties` 中新增 `LocalWorker` 配置，`EmbeddingProviderConfiguration` 支持 `app.embedding.provider=local-worker`。只有显式配置为 `local-worker` 时，active `EmbeddingProvider` 才会选择 `LocalEmbeddingWorkerProvider`。

这意味着 mock provider 不会因为本地 worker 不存在而失败，openai provider 也不受 local-worker 配置影响。worker 不可用时，local-worker provider 会失败并给出明确错误，而不是静默降级成错误向量。

## 5. Python Worker 接口设计

接口：

```text
POST /embeddings
```

请求示例：

```json
{
  "model": "sentence-transformers/all-MiniLM-L6-v2",
  "input": ["What is retrieval evaluation?"]
}
```

响应示例：

```json
{
  "provider": "local-worker",
  "model": "sentence-transformers/all-MiniLM-L6-v2",
  "dimension": 384,
  "data": [
    {
      "index": 0,
      "embedding": [0.1, 0.2, 0.3]
    }
  ]
}
```

Python Worker 的 `input` 支持单条字符串或多条字符串。Java 侧的 `LocalEmbeddingWorkerRequest` 当前使用 `List<String>`，所以 Java 的 single input 会被 provider 包装成只含一个元素的 batch 请求。

`embedding` 是 `List[float]`，`dimension` 由向量长度决定，batch 返回顺序通过 `index` 表达。Java provider 会按 `index` 排序，确保返回结果与输入顺序对应。

Worker 负责模型加载和推理，Java 侧负责调用、校验和业务集成。

## 6. 为什么 Java 不直接跑 embedding 模型

Java 后端更适合负责业务 API、任务编排、数据库访问、文档管理、检索流程、evaluation 和 provider metadata 管理。

Python 更适合负责模型加载、`sentence-transformers`、Hugging Face 模型生态、CPU/GPU 推理和本地模型实验。

使用 Python Worker 的好处是：

- Java 服务和模型推理解耦；
- 模型可独立替换；
- Worker 可独立部署；
- 后续可以接 BGE、E5、Jina 或本地私有模型；
- 架构上更接近企业里的异构服务形态。

如果把模型直接塞进 Java 服务，Java 进程会同时承担业务服务和模型推理，依赖、资源和故障边界都会变重。本版本选择 worker 方式，是为了让 Java 保持稳定的业务编排职责，让模型侧独立演进。

## 7. 为什么默认测试不启动 Python Worker

默认测试不启动 Python Worker，原因是：

- 不要求本机安装 Python；
- 不要求下载 Hugging Face 模型；
- 不依赖外部网络；
- 不依赖 CPU/GPU 推理速度；
- 不让 CI 因模型下载失败而失败；
- 不让普通单元测试依赖外部进程；
- 默认测试应该验证 Java provider 逻辑，而不是验证第三方模型可用性。

当前做法是：Java 测试使用 fake `LocalEmbeddingWorkerClient`，不启动 FastAPI，不下载模型，不访问网络，不需要 Python 环境。

真实模型可用性应该放到手工验证、专门的 integration test 或后续 benchmark 流程里，而不是放进默认 `mvn test`。

## 8. provider / model / dimension 为什么仍然重要

embedding vector 属于特定向量空间。`local-worker` 生成的向量不能和 `mock` 或 `openai` 的向量混在同一次 cosine similarity 中。

原因是：

- provider 不同；
- model 不同；
- dimension 可能不同；
- 即使 dimension 相同，向量空间也可能不同；
- 混用会导致检索结果没有语义意义。

当前项目做法是：

- worker response 返回 `provider` / `model` / `dimension`；
- `LocalEmbeddingWorkerProvider` 校验 response dimension 和配置 dimension；
- `DocumentEmbeddingService` 保存 `embeddingProvider`、`embeddingModel`、`vectorDimension` 和 `distanceMetric`；
- search 查询候选时按 provider / model 过滤；
- search 计算前继续过滤 dimension 不一致的向量；
- dimension mismatch 直接失败，不静默写入错误向量。

这里的核心面试表达是：embedding metadata 不是展示字段，而是检索正确性的隔离条件。

## 9. 错误处理设计

本版本处理的错误包括：

- worker 返回 empty data；
- worker 返回 empty vector；
- batch response 数量不匹配；
- worker response dimension 和配置 dimension 不一致；
- 单个 vector 长度和配置 dimension 不一致；
- client exception；
- worker error 通过 HTTP/client 异常传递到 Java provider。

这些错误不能被吞掉。embedding 是检索链路的基础，如果向量不完整、维度不一致或 provider 返回异常，应该明确失败，避免把错误向量写入检索空间。

`LocalEmbeddingWorkerProvider` 对非 `EmbeddingProviderException` 的异常会包装成 `EmbeddingProviderException("Local embedding worker request failed", exception)`，这样上层看到的是 embedding provider 层面的明确失败。

## 12. 为什么不现在做 Docker Compose

Docker Compose 可以让 Java、数据库、RabbitMQ、Python Worker 一起启动，但当前版本先验证 provider 接入边界。

现在不做 Docker Compose 的原因是：

- 会扩大版本边界；
- 需要处理模型缓存目录；
- 需要处理镜像构建时间；
- 需要处理 worker health check；
- 需要处理启动顺序；
- 当前重点是 provider abstraction，不是部署编排。

后续可以单独做 Local Worker Dockerization、docker-compose 集成、worker health check 和模型缓存挂载。

## 13. 为什么不做 Vector DB / Rerank / Hybrid Search

V2.5.2 改变的是 embedding 生成方式。

Vector DB 改变的是向量存储和召回方式。Rerank 改变的是排序阶段。Hybrid Search 改变的是检索策略。

如果同时做这些能力，会导致效果不可归因：到底是 local-worker 变好了，还是 Vector DB、Rerank、Hybrid Search 改变了结果，无法清楚解释。

因此本版本后置这些内容：

- V2.6 Vector DB Selection；
- V2.8 Rerank；
- 后续 Hybrid Search。

## 17. 面试高强度追问与回答

### Q1：为什么要做 Local Embedding Worker？

因为 V2.5 已经证明 Java 后端可以通过 `EmbeddingProvider` 接入云端 OpenAI-compatible provider，但真实企业场景不一定允许文档出公网，也可能希望降低调用成本或接入私有模型。V2.5.2 用 `LocalEmbeddingWorkerProvider` 把本地模型服务接入同一套 provider 抽象，让 Java 业务链路不需要关心模型是在云端 API 里，还是在本地 Python Worker 里。

当前版本只解决接入边界，不声明本地模型效果优于云端模型。真实效果需要后续通过 V2.5.1 的 benchmark comparison 手工验证。

### Q2：为什么 Java 不直接跑 embedding 模型？

Java 服务在这个项目里承担业务 API、任务编排、数据库访问、文档管理、检索和 evaluation。embedding 模型加载、Hugging Face 生态和 CPU/GPU 推理更适合放在 Python 侧。

如果 Java 直接跑模型，业务进程会被模型依赖、内存占用、推理速度和硬件差异拖重。拆成 Python FastAPI Worker 后，Java 只通过 `LocalEmbeddingWorkerClient` 调用模型服务，模型侧可以独立替换、部署和扩容。

### Q3：为什么选择 Python FastAPI Worker？

FastAPI 适合作为轻量模型服务入口，能自然接收 JSON request body，并返回结构化响应。Python 生态里 `sentence-transformers` 使用成熟，接入 Hugging Face 模型也直接。

在本项目中，Worker 只提供 `POST /embeddings`，协议简单：输入 `model` 和 `input`，输出 `provider`、`model`、`dimension` 和 `data`。这让 Java 侧协议稳定，后续替换模型时不需要改业务链路。

### Q4：local-worker 和 openai provider 的区别是什么？

`OpenAiCompatibleEmbeddingProvider` 调用 OpenAI-compatible HTTP API，通常依赖外部服务、API key 和网络。`LocalEmbeddingWorkerProvider` 调用本地 Python FastAPI Worker，由本机或内网 worker 加载 `sentence-transformers` 模型。

两者都实现 `EmbeddingProvider`，所以 `DocumentEmbeddingService` 不需要为它们写两套逻辑。差异被封装在 provider 和 client 内部。

### Q5：为什么默认 provider 仍然是 mock？

默认 `mock` 是为了保证本地开发和 CI 稳定。mock 不需要 API key、不需要 Python、不需要模型下载、不访问外部网络，也不会因为第三方服务不可用导致默认测试失败。

`local-worker` 是可选 provider，只有配置 `app.embedding.provider=local-worker` 时才会启用。

### Q6：为什么默认测试不启动 Python Worker？

默认单元测试应该验证 Java provider 的业务逻辑，而不是验证外部进程管理。启动 Python Worker 会引入 Python 环境、端口占用、模型加载、网络下载和推理耗时等不稳定因素。

因此 `LocalEmbeddingWorkerProviderTest` 使用 fake `LocalEmbeddingWorkerClient`，直接模拟 worker response 和异常。

### Q7：为什么默认测试不下载模型？

模型下载依赖 Hugging Face 网络、缓存目录和本机环境，CI 上很容易因为网络或限流失败。模型下载耗时也不可控。

V2.5.2 的默认测试目标是验证 Java provider 对 single input、batch input、metadata、dimension mismatch 和异常的处理。真实模型下载和效果验证应放到手工验证或专门 benchmark 里。

### Q8：LocalEmbeddingWorkerClient 为什么要抽象出来？

因为 `LocalEmbeddingWorkerProvider` 的核心职责是实现 provider 语义，不应该把 HTTP 细节写死在 provider 里。抽象出 `LocalEmbeddingWorkerClient` 后，真实环境可以使用 `RestClientLocalEmbeddingWorkerClient`，测试环境可以用 fake client。

这个设计让测试能覆盖 provider 的错误处理和响应转换，同时不访问真实网络。

### Q9：RestClientLocalEmbeddingWorkerClient 负责什么？

`RestClientLocalEmbeddingWorkerClient` 负责真实 HTTP 调用。它根据 `app.embedding.local-worker.base-url` 构造 `RestClient`，设置连接 timeout，并向 `/embeddings` 发送 POST 请求。

它不负责业务判断，也不负责向量空间隔离。它只是协议客户端，调用后返回 `LocalEmbeddingWorkerResponse`。

### Q10：provider / model / dimension 在 local-worker 中有什么作用？

它们用来标识向量属于哪个向量空间。`provider=local-worker`、`model=sentence-transformers/all-MiniLM-L6-v2`、`dimension=384` 共同说明这批向量的来源和形状。

项目中 `DocumentEmbeddingService` 会保存 embedding provider、model、dimension 和 distance metric。search 时按 provider / model 查候选，并过滤 dimension 不一致的向量，避免把不同向量空间混在一起做 cosine similarity。

### Q11：如果 local-worker 返回 dimension 和配置不一致怎么办？

当前 `LocalEmbeddingWorkerProvider` 会直接抛出 `EmbeddingProviderException`。它会校验 response 的 `dimension` 和配置的 `dimension`，也会校验每个 vector 的长度。

这是必要的，因为 dimension mismatch 说明 worker 返回的数据不符合当前 Java 配置。如果继续写入，会污染检索空间。

### Q12：batch embedding 要注意什么？

batch embedding 要保证输入数量和输出数量一致，还要保证每个输出能对应回原始输入。当前 worker response 的 `data` 中有 `index`，Java provider 会按 `index` 排序后返回结果。

如果 batch response 数量不匹配，`LocalEmbeddingWorkerProvider` 会失败，不会静默丢弃或补齐结果。

### Q13：Worker 不可用时 Java 侧如何处理？

`RestClientLocalEmbeddingWorkerClient` 调用失败时会抛出异常。`LocalEmbeddingWorkerProvider` 会把普通异常包装成 `EmbeddingProviderException("Local embedding worker request failed", exception)`。

这样上层能明确知道是 embedding provider 调用失败，而不是得到一个空向量或错误向量。当前版本没有做自动重试、熔断或降级，这些属于后续生产级治理范围。

### Q14：为什么不现在做 Docker Compose？

因为 V2.5.2 的目标是 provider 接入，不是部署编排。Docker Compose 会引入模型缓存目录、镜像构建时间、启动顺序、health check 和多服务依赖管理。

这些都值得做，但应该单独作为 Local Worker Dockerization 版本推进，避免当前版本边界过大。

### Q15：为什么不现在做 Vector DB？

Vector DB 改变的是向量存储和召回方式，而 V2.5.2 改变的是 embedding 生成方式。两者一起做会让效果不可归因。

当前项目仍然可以基于已有 exact cosine search 跑通链路。Vector DB 可以在 V2.6 单独选型，比较 exact search 和 ANN 在 recall、latency、cost 上的差异。

### Q16：为什么不现在做 Rerank？

Rerank 改变的是 TopK 初筛后的排序阶段，不是 embedding 生成阶段。如果在 local-worker 接入时同时加入 rerank，就无法判断检索结果变化来自 embedding 模型还是 rerank。

因此 V2.8 再做 Rerank，并用 NDCG@K、MRR 等排序指标评估排序改善。

### Q17：如何把 local-worker 产品化？

第一步是把 worker Docker 化，补齐 health check、模型缓存挂载和 docker-compose 集成。第二步是增加 worker 运行状态、错误率、延迟和资源占用监控。第三步才是多 worker 调度、GPU/CPU 资源管理、模型版本管理和企业 SLA。

当前版本只提供 `local-worker` provider 接入能力，还不是生产级模型服务平台。

### Q18：企业为什么需要本地 embedding？

企业常见诉求是文档不出公网、内网部署、成本可控、合规可解释，以及接入企业自研模型。本地 embedding worker 让模型推理留在企业环境内，Java 后端仍然通过稳定的 provider abstraction 使用它。

这对私有知识库、合规行业和离线环境有价值。

### Q19：后续如何接 BGE-M3、E5、Jina 或私有模型？

如果这些模型能通过 `sentence-transformers` 或兼容封装加载，优先在 Python Worker 侧替换 `model` 配置。Java 侧仍然传入 `app.embedding.local-worker.model`，接收相同结构的 response。

需要注意的是，不同模型可能有不同 dimension，也可能属于不同向量空间。因此必须更新 `app.embedding.local-worker.dimension`，并通过 provider / model / dimension 隔离存储和检索。

### Q20：local-worker 如何和 V2.5.1 Provider Benchmark Comparison 结合？

V2.5.1 已经有 provider benchmark comparison 的思路：同一 corpus、同一 benchmark cases、同一 topKValues、同一 evaluation service，对比不同 provider。

V2.5.2 接入 `local-worker` 后，后续可以手工启动 Python Worker，把 candidate provider 配成 local-worker，再用同一套 benchmark 对比 mock、openai 和 local-worker。但这应该作为手工 benchmark 或专门测试，不进入默认 CI，也不能在没有实际结果时声称指标提升。
