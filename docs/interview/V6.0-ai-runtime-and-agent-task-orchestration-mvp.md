# V6.0 AI Runtime & Agent Task Orchestration MVP 面试文档

## 1. 阶段摘要

V6.0 的阶段名称是：

```text
V6.0 AI Runtime & Agent Task Orchestration MVP
```

中文可以表述为：

```text
真实 AI Runtime 接入与 Agent 任务编排 MVP
```

这个阶段的核心目标是：把项目从“知识库问答系统”推进为“可以执行 AI 任务的编排系统”。

在 V5.0 之前，系统已经具备：

```text
文档上传；
异步文档摄入；
文档事件时间线；
文档处理分析；
文档软删除；
文档重新索引；
Collection 知识库分组；
Scoped Retrieval 范围检索；
RAG Answer API；
citations 引用来源；
rerank；
hybrid retrieval；
中文友好 Web UI。
```

这些能力让系统已经成为一个比较完整的 AI Knowledge Assistant。

但项目名称是 AI Task Orchestrator，它最终不应该只停留在“问答”，而应该支持：

```text
用户提交一个任务；
系统检索相关知识；
调用大模型；
生成结构化任务结果；
记录任务状态；
记录执行步骤；
记录引用来源；
记录模型调用信息；
失败可追踪。
```

V6.0 就是这个转折点。

它把系统从：

```text
AI Knowledge Assistant / 知识库问答系统
```

推进到：

```text
AI Task Orchestrator / AI 任务编排系统
```

## 2. V6.0 的总体架构

V6.0 明确采用 Java + Python 协作架构。

整体分工是：

```text
Java / Spring Boot：企业级后端主系统和任务编排；
Python Worker：真实 AI Runtime 能力。
```

### Java 负责

```text
API；
MySQL；
RabbitMQ；
任务状态；
任务事件；
RAG orchestration；
Collection scope；
Document lifecycle filter；
citations；
错误响应；
Web UI；
manual；
测试。
```

### Python 负责

```text
embedding；
LLM generation；
后续 rerank；
后续 agent tool execution。
```

也就是说：

```text
Java 是 orchestrator；
Python 是 model runtime worker。
```

这种分工更接近真实 AI 应用工程，而不是把所有模型能力都硬塞进 Java。

## 4. 为什么要做 V6.0

V3.1 到 V5.0 主要解决的是知识库系统能力：

```text
文档怎么进来；
文档怎么处理；
文档怎么删除；
文档怎么重建索引；
文档怎么分组；
Ask 怎么按范围检索。
```

但这仍然更像：

```text
RAG Knowledge Base Backend
```

还不是真正的：

```text
AI Task Orchestrator
```

AI Task Orchestrator 的关键不是“用户问一句，系统答一句”，而是：

```text
用户提交一个目标；
系统创建任务；
后台异步执行；
检索相关知识；
调用模型；
形成结果；
记录过程；
失败可排查。
```

所以 V6.0 引入 Agent Task。

它让项目从即时问答进入任务执行：

```text
Ask：即时问题回答；
Agent Task：异步任务执行和结构化结果生成。
```

这也是从 AI 应用 demo 走向 AI workflow / Agent Infra 的关键一步。

## 5. 核心设计一：Java + Python AI Runtime 分工

### 5.1 为什么不直接用 Java 调模型

Java 可以调用 HTTP API，但 AI 生态的模型运行、embedding、rerank、本地 LLM 推理主要在 Python 生态。

如果把所有模型逻辑都写在 Java 中，会遇到：

```text
模型库生态弱；
本地模型加载不方便；
HuggingFace / sentence-transformers 支持不自然；
后续 vLLM / SGLang / reranker 接入不方便；
Prompt experiment 不灵活。
```

所以 V6.0 选择：

```text
Java 只做模型 provider 抽象和调用编排；
Python 负责真实模型运行。
```

---

### 5.2 Java 的职责

Java 侧负责：

```text
选择 provider；
构造请求；
调用 Python Worker；
处理超时；
处理失败；
记录 metadata；
把结果写入 task / event / citation；
保证默认测试稳定。
```

Java 不直接关心模型内部如何运行。

---

### 5.3 Python 的职责

Python Worker 负责：

```text
加载 embedding model；
生成 vectors；
调用或加载 LLM；
返回 content；
返回 usage；
返回 latency；
暴露 health endpoint。
```

Python Worker 可以是本地模型，也可以封装外部 API。

## 6. 核心设计二：Provider 分层

V6.0 的一个重要工程原则是：

```text
默认 mock，真实 provider 走 profile。
```

### 6.1 Mock Provider

MockEmbeddingProvider 和 MockLlmProvider 用于：

```text
单元测试；
默认 profile；
CI；
离线开发；
不依赖模型下载；
不依赖 Python worker；
不依赖外部 API。
```

这保证：

```powershell
.\mvnw.cmd test
```

可以稳定运行。

---

### 6.2 Local Python Provider

LocalPythonEmbeddingProvider / LocalPythonLlmProvider 用于：

```text
本地真实模型验证；
manual demo；
真实 RAG 效果；
真实 LLM 生成；
后续 Agent runtime 接入。
```

它们通过 HTTP 调用 Python Worker。

但必须通过 profile 启用，例如：

```text
local-ai；
python-ai；
real-ai。
```

默认配置不能启用真实 provider。

---

### 6.3 为什么用 profile 隔离

因为真实模型会带来：

```text
模型下载；
本地算力；
网络依赖；
API key；
延迟；
不稳定；
费用。
```

这些都不能污染默认测试。

所以正确边界是：

```text
默认：mock；
manual：real provider；
test：低依赖。
```

## 11. 核心设计七：Agent Task 异步执行

Agent Task 必须异步执行，而不是请求中同步调用 LLM。

流程是：

```text
POST /agent/tasks
-> 创建 task，status = PENDING
-> 发送 MQ message
-> listener 消费
-> status = RUNNING
-> RAG retrieval
-> LLM generation
-> 保存 result / citations / metadata
-> status = COMPLETED
```

失败时：

```text
status = FAILED；
保存 errorCode；
保存 errorMessage；
保存 traceId；
记录 TASK_FAILED event。
```

为什么要异步：

```text
LLM 调用可能很慢；
检索和生成可能失败；
用户需要查看任务历史；
后续可以扩展重试、取消、队列、优先级；
符合 Task Orchestrator 定位。
```

---

## 12. 核心设计八：Agent Task RAG + LLM 执行链路

后台执行核心流程：

```text
load task
-> validate collection if provided
-> record RETRIEVAL_STARTED
-> run scoped RAG retrieval
-> apply V5.0 collection filter
-> apply V4.0 lifecycle filter
-> build citations
-> record RETRIEVAL_COMPLETED
-> build agent task prompt
-> record LLM_STARTED
-> call LlmProvider
-> record LLM_COMPLETED
-> save result
-> save model metadata
-> task COMPLETED
```

这说明 Agent Task 并不是绕开之前的 RAG 系统，而是复用并编排已有能力。

## 13. 核心设计九：Scoped Retrieval 继续生效

Agent Task 可以绑定 collectionId。

规则：

```text
collectionId 为空：使用全部文档；
collectionId 不为空：只使用指定 Collection。
```

必须继续遵守：

```text
V5.0 collection filter；
V4.0 lifecycle filter；
current active chunk filter。
```

也就是说，Agent Task 的上下文必须同时满足：

```text
属于指定 Collection，或全库模式；
document 为 ACTIVE；
chunk 为 current active；
不是 DELETED document；
不是 old / superseded chunk。
```

这避免 Agent Task 绕过知识库问答中的安全过滤。

---

## 14. 核心设计十：no-context 处理

如果当前范围没有可用上下文，比如：

```text
collection 为空；
collection 下只有 DELETED 文档；
collection 下只有 old chunks；
全库也没有 active chunks。
```

推荐行为是：

```text
task COMPLETED；
result = 当前范围内没有可用于执行任务的知识库内容；
citationCount = 0；
不调用 LLM。
```

为什么不调用 LLM：

```text
没有上下文时调用 LLM 会增加幻觉风险；
没有必要产生模型成本；
任务结果应该诚实说明上下文不足。
```

## 15. 核心设计十一：Prompt 设计

V6.0 不做复杂 agent planner，只做简单结构化任务执行。

System prompt 语义：

```text
你是一个企业知识库任务执行助手。你只能基于提供的知识库上下文完成任务。如果上下文不足，需要明确说明不足，不要编造。
```

User prompt 包含：

```text
任务目标；
知识库范围；
检索到的上下文；
citation index；
输出格式要求。
```

建议输出格式：

```text
任务结论；
关键依据；
风险 / 不确定性；
下一步建议；
引用来源。
```

明确不做：

```text
多轮 planner；
tool calling；
ReAct；
multi-agent；
self-reflection loop；
长期 memory。
```

## 17. 核心设计十三：失败处理

V6.0 需要处理多类失败。

常见错误包括：

```text
collection 不存在；
retrieval 异常；
LLM provider 异常；
Python worker unavailable；
timeout；
response empty；
serialization error。
```

建议错误码：

```text
AGENT_TASK_NOT_FOUND；
AGENT_TASK_COLLECTION_NOT_FOUND；
AGENT_TASK_RETRIEVAL_FAILED；
AGENT_TASK_LLM_FAILED；
AI_RUNTIME_UNAVAILABLE；
AI_RUNTIME_TIMEOUT；
AGENT_TASK_EXECUTION_FAILED。
```

失败时必须：

```text
task.status = FAILED；
保存 errorCode；
保存 errorMessage；
保存 traceId；
记录 TASK_FAILED event；
前端显示中文错误。
```

不能失败后只打日志而不更新任务状态。

## 20. 关键取舍

### 20.1 为什么不做复杂 Agent Planner

V6.0 的目标是 Agent Task Orchestration MVP，不是复杂 autonomous agent。

复杂 planner 会引入：

```text

多步决策；

循环执行；
```

这些会显著扩大边界。

V6.0 先做：

```text

一次检索；

保存结果；
记录事件。
```

这是合理 MVP。

---

tool calling 需要：

```text
工具注册；
工具 schema；
参数校验；
执行沙箱；
权限；
工具失败处理；

```

V6.0 先不做。

### 20.3 为什么不做 streaming

streaming 会引入：

```text
SSE；
WebSocket；
前端流式渲染；
断连处理；
partial result；
任务状态同步。
```

当前异步任务已经能满足 MVP。

---

### 20.4 为什么 no-context 不调用 LLM


Agent Task 应该诚实返回：


默认测试必须低依赖，所以使用 mock provider。

## 26. 面试高强度追问与回答

### 26.1 V6.0 和之前的 Ask 问答有什么区别？

回答：

```text
Ask 是即时问答，用户提一个问题，系统同步或近同步返回答案。Agent Task 是异步任务执行，用户提交一个目标，系统创建任务、进入队列、检索知识库、调用 LLM、保存结构化结果，并记录状态、事件、引用和模型 metadata。它更接近任务编排系统，而不是单次问答接口。
```

---

### 26.2 为什么 Java 和 Python 要分工？

回答：

```text
Java 更适合企业后端，包括 API、数据库、事务、状态机、异步任务和错误处理。Python 在 AI 生态上更强，适合 embedding、LLM、reranker、本地模型和后续工具执行。所以我让 Java 做 orchestrator，让 Python 做 model runtime worker。
```

---

### 26.3 为什么默认测试不用真实 Python Worker？

回答：

```text
真实 Python Worker 依赖模型下载、本地算力、端口、网络或 API key，会导致测试不稳定。默认测试应该验证业务逻辑，所以使用 mock provider。真实 AI Runtime 通过 local-ai profile 手工验证。
```

---

### 26.4 Agent Task 如何保证不绕过 Collection filter？

回答：

```text
Agent Task 执行时复用已有 scoped RAG retrieval。如果 task 绑定 collectionId，retrieval 会继续应用 V5.0 的 collection scope filter，只保留属于该 Collection 的候选结果。final context、citations 和 prompt 构造前都会经过统一 candidate validation。
```

---

### 26.5 Agent Task 如何保证不引用已删除文档？

回答：

```text
V4.0 已经实现 lifecycle filter，要求 document 必须 ACTIVE，chunk 必须 current active。Agent Task 复用这套 retrieval filter，所以即使 VectorStore 返回了已删除文档或旧 chunks，也不会进入 final context、citations 或 prompt。
```

---

### 26.6 no-context 为什么不调用 LLM？

回答：

```text
如果当前知识库范围没有可用上下文，调用 LLM 会增加幻觉风险和模型成本。Agent Task 的目标是基于知识库执行任务，所以 no-context 时应该直接返回“当前范围没有可用知识库内容”，而不是让模型编造。
```

---

### 26.7 为什么不做复杂 tool calling？

回答：

```text
V6.0 是 Agent Task Orchestration MVP，重点是把任务、RAG、LLM、状态和事件串起来。Tool calling 会引入工具 schema、参数校验、执行沙箱、权限、失败恢复和多轮模型决策，复杂度较高，适合作为后续阶段单独实现。
```

---

### 26.8 LLM 调用失败时怎么处理？

回答：

```text
LLM 调用失败时，任务状态会更新为 FAILED，保存 errorCode、errorMessage 和 traceId，并记录 TASK_FAILED event。前端可以展示中文错误原因和技术详情，避免任务一直卡在 RUNNING。
```

---

### 26.9 为什么 Agent Task 要异步？

回答：

```text
LLM 调用和知识库检索可能耗时较长，也可能失败。异步任务可以记录状态、事件、失败原因和历史结果，也为后续重试、取消、优先级和队列管理打基础。这符合 Task Orchestrator 的定位。
```

---

### 26.10 这个阶段体现什么工程能力？

回答：

```text
V6.0 体现的是 AI 应用编排能力。它不是简单调用一次 LLM，而是把 Java 后端、Python 模型服务、RAG 检索、Collection 范围、文档生命周期过滤、异步任务状态机、事件时间线、结果持久化和中文 UI 串成一个完整 workflow。这更接近真实 Agent Infra / AI Application Runtime。
```
