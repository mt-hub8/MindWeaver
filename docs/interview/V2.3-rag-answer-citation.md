# V2.3 RAG Answer with Citation 面试文档

## 0. 版本摘要

V2.3 实现了同步 RAG Answer baseline：用户提交 query 后，系统复用已有 document search / embedding search 能力检索 TopK chunks，将检索结果构造成 RAG context prompt，调用现有 `LlmClient` / `MockLlmClient`，并返回 answer、chunk-level citations、retrieval metadata 和 llm metadata。

这一版的定位很明确：它是把 V2.2 的“能检索相关 chunk”推进到“能基于 chunk 生成回答并返回引用来源”的第一版闭环。

必须强调的是，V2.3 不是 production-grade RAG system。当前仍然是工程 baseline：

- 不接真实 LLM；
- 不接真实 Embedding Provider；
- 不接 Vector DB；
- 不做 ANN Search；
- 不做 Hybrid Search；
- 不做 Rerank；
- 不做 Retrieval Evaluation；
- 不持久化 RAG Answer；
- 不做异步 RAG Task。

这版的价值不在于真实语义效果，而在于把 RAG 的核心工程链路先跑通：retrieve -> prompt -> generate -> citations -> metadata response。

## 1. 本版本解决的问题

V2.2 Mock Embedding & Vector Search 已经实现了检索链路：

```text
query
-> query embedding
-> cosine similarity
-> TopK chunks
```

但用户真正想要的不是一组 chunk 排名，而是：

```text
question
-> answer with sources
```

V2.3 解决的就是从 retrieval 到 answer 的缺口：

```text
query
-> retrieve TopK chunks
-> build context prompt
-> call MockLlmClient
-> return answer + citations
```

V2.2 解决的是“系统能不能从文档 chunk 中找出相关片段”。V2.3 解决的是“系统能不能把这些片段组织成上下文，交给 LLM 生成答案，并告诉用户答案依据来自哪些 chunk”。

这一步是从 RAG retrieval prototype 进入 RAG answer baseline 的关键节点。没有 V2.3，系统只能证明向量检索链路存在；有了 V2.3，系统开始具备 RAG 应用的用户可感知闭环。

## 2. 为什么现在做 V2.3

V2.3 依赖前面的几层能力：

- V2.1 完成 Document Upload & Chunking，系统有了可管理的文档和 chunk；
- V2.1.1 完成 Adaptive Chunking，chunk 有了 `headingPath`、`startOffset`、`endOffset`、`chunkStrategy` 等 metadata；
- V2.2 完成 Mock Embedding & Vector Search，系统可以根据 query 检索 TopK chunks；
- V2.2.x 完成 Production Hardening，任务系统补齐了 atomic claim、outbox、attempt、测试和 CI；
- V2.2.5 ~ V2.2.7 修正 README、API 边界、Outbox 事务边界和 metrics polish，让项目展示面更可信。

所以 V2.3 的时机是合理的：先有文档处理，再有检索，再有生成。反过来，如果一开始就做 RAG Answer，会缺少 chunk metadata、embedding search 和 citation 基础，最后很容易变成“拿一段硬编码文本问 Mock LLM”的演示，而不是一个可演进的 RAG 系统。

这一版也保持了克制：

- 先做 retrieval，再做 answer；
- 先做 baseline，再做 evaluation；
- 先用 Mock LLM 打通闭环，再接真实 provider；
- 不把真实模型、向量数据库、评测、异步 task 全塞进同一个版本。

这种顺序让每一层都有清晰职责，也方便面试时解释系统为什么是逐层演进出来的，而不是临时拼接。

## 3. 为什么选择同步 RAG Answer API

V2.3 选择同步 API：

```text
POST /rag/answer
-> retrieve
-> prompt
-> MockLlmClient
-> return answer
```

而不是直接把 RAG Answer 做成异步任务。

同步方案的优点是边界小、验收简单，能专注验证 RAG answer 逻辑本身。它不会污染当前 task 系统的数据模型，也不需要立刻引入 `taskType`、RAG payload、RAG result schema、answer persistence、citation persistence 等概念。

如果这一版直接做异步 RAG task，会牵出一串额外设计：

- `taskType` 如何区分普通 LLM task 和 RAG task；
- RAG request payload 如何存；
- RAG answer result schema 如何和普通 task result 兼容；
- RAG answer 是否持久化；
- citations 是否持久化；
- `task_attempt` 如何绑定 RAG metadata；
- output chunks 是否要和 citation 绑定；
- 失败重试时是否重新检索；
- prompt 和 retrieval evidence 是否要审计。

这些都是真实问题，但不应该在 V2.3 一次性解决。第一版先用同步 API 验证 RAG 闭环。后续接真实 LLM、真实 embedding、evaluation 之后，再将 RAG Answer task 化，接入异步 Orchestrator，会更稳。

面试中可以这样表达：

> 我没有一上来把 RAG Answer 做成异步任务，因为那会同时改变任务系统、结果结构、citation 持久化和执行审计。V2.3 先做同步 API，把 retrieve -> prompt -> generate -> citation 的业务链路跑通。等真实模型、真实检索效果和评测体系稳定后，再把它接回已有的异步任务编排底座。

## 4. API 设计

### 4.1 POST /rag/answer

接口：

```http
POST /rag/answer
```

请求 DTO：`RagAnswerRequest`

字段：

- `query`：用户问题，必填；
- `documentId`：可选，用于限定只在某个 document 内检索；
- `topK`：可选，默认 5，最大 20；
- `requestedModel`：可选，传给 `ModelRouter` 选择实际模型。

请求示例：

```json
{
  "query": "为什么这个项目要使用 Transactional Outbox？",
  "documentId": 1,
  "topK": 5,
  "requestedModel": "mock-llm"
}
```

当前 `topK` 的处理规则是：

- `topK == null` 时使用默认值 5；
- `topK <= 0` 时使用默认值 5；
- `topK > 20` 时截断为 20。

这和已有 search 接口略有边界差异：`DocumentEmbeddingService.search(...)` 对 `topK <= 0` 会返回参数错误，而 V2.3 RAG Answer 为了用户体验选择回退默认值。

### 4.2 Response

响应 DTO：`RagAnswerResponse`

字段：

- `query`：原始问题；
- `answer`：LLM 返回的回答，或无上下文时的兜底回答；
- `citations`：chunk-level citations；
- `retrieval`：检索 metadata；
- `llm`：LLM 调用 metadata。

`citations` 类型是 `List<RagCitationResponse>`，字段包括：

- `citationId`；
- `documentId`；
- `chunkId`；
- `chunkIndex`；
- `score`；
- `headingPath`；
- `startOffset`；
- `endOffset`；
- `chunkStrategy`；
- `contentPreview`。

`retrieval` 类型是 `RagRetrievalMetadataResponse`，字段包括：

- `topK`；
- `returnedCount`；
- `embeddingProvider`；
- `embeddingModel`；
- `distanceMetric`。

`llm` 类型是 `RagLlmMetadataResponse`，字段包括：

- `provider`；
- `model`；
- `promptTokenCount`；
- `completionTokenCount`；
- `totalTokenCount`；
- `latencyMs`。

响应示例：

```json
{
  "query": "为什么这个项目要使用 Transactional Outbox？",
  "answer": "Mock LLM response for prompt: ...",
  "citations": [
    {
      "citationId": 1,
      "documentId": 1,
      "chunkId": 10,
      "chunkIndex": 0,
      "score": 0.91,
      "headingPath": "V2.2.2 Reliable Dispatch",
      "startOffset": 120,
      "endOffset": 560,
      "chunkStrategy": "RECURSIVE_TEXT",
      "contentPreview": "Transactional Outbox 用于解决数据库提交和 MQ 投递之间的 dual write problem..."
    }
  ],
  "retrieval": {
    "topK": 5,
    "returnedCount": 1,
    "embeddingProvider": "mock",
    "embeddingModel": "mock-embedding-v1",
    "distanceMetric": "COSINE"
  },
  "llm": {
    "provider": "mock",
    "model": "mock-llm",
    "promptTokenCount": 100,
    "completionTokenCount": 120,
    "totalTokenCount": 220,
    "latencyMs": 5
  }
}
```

上面的数字只是字段形态示例，不代表性能指标或固定测试结果。

## 5. Citation 设计

V2.3 的 citation 是 chunk-level citation。它表示答案使用了哪些检索到的 chunk 作为上下文来源。

它不是：

- sentence-level citation；
- PDF page citation；
- answer 每句话绑定 citation；
- citation accuracy evaluation。

每个 citation 包含：

- `citationId`：从 1 开始，与 prompt 中 `[1]`、`[2]` 编号对齐；
- `documentId`：定位文档；
- `chunkId`：定位具体 chunk；
- `chunkIndex`：定位 chunk 在文档中的顺序；
- `score`：检索相关性分数；
- `headingPath`：来自 Adaptive Chunking 的标题路径；
- `startOffset` / `endOffset`：chunk 在原文中的大致位置；
- `chunkStrategy`：当前 chunk 的切分策略；
- `contentPreview`：用于前端展示和人工验收的短文本预览。

这些字段的价值不同：

- `documentId` / `chunkId` 用于稳定定位来源；
- `score` 帮助判断 citation 的相关性；
- `headingPath` 提高可读性；
- `startOffset` / `endOffset` 为后续精确溯源准备；
- `chunkStrategy` 为后续 Chunking Evaluation 准备；
- `contentPreview` 方便用户快速判断引用是否靠谱。

当前 `contentPreview` 最长截取 160 字符，避免把完整 chunk 内容塞进 response。完整内容仍然来自 document chunk，不在 citation 里重复返回超长文本。

局限也要明确：chunk-level citation 只能说明答案依据了哪些 chunk，不能证明答案每句话都被严格引用支持。要做到更强的引用质量，需要后续做 answer sentence 与 evidence span 的对齐，以及 citation accuracy evaluation。

## 6. Prompt 构造策略

`RagPromptBuilder` 负责把 query 和检索到的 chunks 构造成上下文 prompt。

当前 prompt 结构大致如下：

```text
你是一个基于给定资料回答问题的助手。

要求：
1. 只能根据 Context 中的信息回答；
2. 如果 Context 中没有答案，请回答“无法从当前资料中确定”；
3. 回答要简洁、准确；
4. 引用资料时使用 [1]、[2] 这样的编号。

Question:
{query}

Context:
[1]
documentId: {documentId}
chunkId: {chunkId}
headingPath: {headingPath}
content:
{content}

[2]
documentId: {documentId}
chunkId: {chunkId}
headingPath: {headingPath}
content:
{content}

Answer:
```

这个设计有几个目的。

第一，明确要求 LLM 只能基于 Context 回答，降低它编造外部信息的倾向。注意这只能降低 hallucination 风险，不能从根本上消除 hallucination。

第二，citation 编号必须和 response citations 对齐。Prompt 中 `[1]` 对应 response 中 `citationId = 1`，这样前端和用户能把回答里的引用编号映射回具体 chunk。

第三，prompt 中包含 `documentId`、`chunkId`、`headingPath`，这样即使 LLM 在回答中引用编号，系统侧也有完整 metadata 能解释每个编号来自哪里。

第四，空检索结果不应该让 LLM 自由发挥。当前实现中，如果 retrieved chunks 为空，`RagAnswerService` 直接返回“无法从当前资料中确定。”，不调用 LLM。这比把空 context 交给 LLM 更安全，也更符合 RAG 的“基于资料回答”原则。

## 7. 核心流程

V2.3 的核心流程是：

```text
POST /rag/answer
-> RagAnswerController 接收 RagAnswerRequest
-> RagAnswerService 校验 query / topK
-> 构造 DocumentSearchRequest
-> 调用已有 DocumentEmbeddingService.search(...)
-> 获取 TopK chunks
-> 构造 chunk-level citations
-> RagPromptBuilder 构造 context prompt
-> ModelRouter 选择模型
-> 调用 LlmClient / MockLlmClient
-> 读取 LlmResponse metadata
-> 返回 RagAnswerResponse
```

空检索结果处理：

```text
retrieved chunks 为空
-> answer = "无法从当前资料中确定。"
-> citations = []
-> retrieval.returnedCount = 0
-> llm metadata 为空
-> 不调用 LlmClient
```

这条空结果路径很重要。RAG 的核心不是“无论如何都生成一句话”，而是“基于检索证据回答”。如果没有证据，就应该明确无法确定。

- `RagAnswerServiceTest`：RAG service 编排测试。

- 没有重写 embedding search，而是复用 `DocumentEmbeddingService.search(...)`；
- 没有绕过 `LlmClient`，而是继续通过 `LlmClient.generate(...)` 调用 Mock LLM；
- 没有新增 `rag_answer` 表；
- 没有新增 `rag_citation` 表；
- 没有引入真实 LLM provider；
- 没有引入真实 embedding provider；
- 没有引入 Vector DB；
- 没有修改 outbox、attempt、atomic claim 等生产硬化逻辑。

这使 V2.3 更像是在已有检索和 LLM 抽象之上加了一层 RAG 应用服务，而不是重新开一套并行系统。

## 12. 工程权衡

### 12.1 为什么先用 Mock LLM

先用 Mock LLM 的核心原因是验证工程链路，而不是追求真实回答质量。

如果这一版直接接真实 LLM，会引入 API key、费用、限流、网络失败、模型不稳定输出、数据合规等问题。那样一旦结果不好，很难判断问题来自 prompt、retrieval、chunking、embedding，还是模型本身。

当前项目已经有 `LlmClient` 抽象和 `MockLlmClient`，V2.3 复用这层抽象，可以让测试稳定可重复。后续替换真实 OpenAI、Claude、Qwen 或本地模型时，RAG Answer 的服务编排不用推倒重来。

### 12.2 为什么不新增数据库表

V2.3 的目标是同步 RAG Answer baseline，不是 answer history / audit 系统。

如果新增 `rag_answer`、`rag_citation` 表，就要同时设计 answer 生命周期、citation schema、历史查询 API、权限边界、清理策略和评测数据关系。这会让版本膨胀。

所以这一版不新增数据库表，只返回即时结果。等后续需要 answer history、RAG 审计或 evaluation 数据沉淀时，再单独设计持久化结构更合理。

### 12.3 为什么不持久化 RAG answer

不持久化 RAG answer 是为了减小版本边界。

V2.3 关注的是：

```text
retrieve -> prompt -> generate -> citation response
```

如果同时持久化 answer，就会把 evaluation、audit、history、user permission、storage cost 混在一起。当前先验证 RAG answer 逻辑，后续如果要做可回放、可审计、可评估的 RAG Answer，再新增 answer persistence。

### 12.4 为什么不做异步 RAG task

异步 RAG task 不是不能做，而是不适合在 V2.3 做。

它会引入：

- `taskType`；
- RAG payload；
- RAG result schema；
- RAG answer persistence；
- citation persistence；
- attempt 与 RAG metadata 绑定；
- output chunk 与 citation 的关系；
- 重试时是否重新检索的问题。

这些都需要设计。第一版先用同步 API 验证 RAG answer 逻辑，后续真实模型耗时变长、answer 需要持久化时，再接入已有 Orchestrator。

### 12.5 为什么不做 sentence-level citation

当前 chunk metadata 已经能支持基础溯源：用户能知道答案参考了哪些 chunk，以及 chunk 在文档中的位置。

sentence-level citation 需要更复杂的 answer parsing、evidence alignment 和 citation quality evaluation。比如答案的一句话可能融合了多个 chunk，也可能是模型对 chunk 的概括，不一定能简单映射到一个原文 span。

所以 V2.3 先做 chunk-level citation。后续结合 Citation Accuracy Evaluation，再考虑句子级或 span 级 citation。

### 12.6 为什么不做 evaluation

V2.3 是 answer baseline，evaluation 是下一阶段能力。

RAG Evaluation 不是随手写几个断言就够了，它需要 query-evidence 数据集，需要定义相关 chunk、期望答案、引用质量标准，并计算 Recall@K、MRR、Context Precision、Citation Accuracy 等指标。

这些工作应该在 V2.4 单独展开。否则 V2.3 会同时做 answer、评测、指标和优化，边界太大。

## 14. 面试高强度追问与回答

### Q1：什么是 RAG？

RAG 是 Retrieval-Augmented Generation，也就是检索增强生成。它不是让 LLM 直接凭参数记忆回答，而是先根据用户问题从外部知识库中检索相关资料，再把这些资料作为 Context 放进 prompt，让 LLM 基于资料回答。

在这个项目里，V2.2 已经完成了 retrieval 部分：query embedding、chunk embedding、cosine similarity、TopK chunks。V2.3 则把 TopK chunks 进一步用于 generation：构造 context prompt，调用 `LlmClient`，返回 answer 和 citations。

当前实现是 baseline。它证明 RAG 的工程链路跑通了，但还不是 production-grade RAG，因为还没有真实 embedding、真实 LLM、vector DB、rerank 和 evaluation。

### Q2：V2.2 和 V2.3 的区别是什么？

V2.2 解决的是“找到相关资料”，V2.3 解决的是“基于资料生成回答”。

V2.2 的输出是 TopK chunks，主要面向检索层验证。V2.3 的输出是 answer、citations、retrieval metadata 和 llm metadata，更接近用户真正使用的 RAG 接口。

工程上，V2.3 没有重写 V2.2 的检索逻辑，而是复用 `DocumentEmbeddingService.search(...)`。这说明检索层和问答层是分层的：检索可以独立优化，RAG Answer 可以独立编排。

### Q3：为什么 V2.3 先做同步 API，而不是异步 task？

同步 API 的边界更小，可以专注验证 RAG Answer 的核心逻辑：retrieve、prompt、generate、citation。

如果一上来做异步 RAG task，会同时引入 `taskType`、payload schema、answer persistence、citation persistence、attempt metadata 扩展、重试时是否重新检索等问题。那会把 V2.3 从 RAG answer baseline 扩大成一次任务系统大改。

当前项目已有可靠异步任务底座，但不是所有能力都必须一开始接进任务系统。第一版同步 API 更适合快速验证业务闭环。后续真实 LLM 延迟增加、answer 需要持久化或需要排队调度时，再把 RAG Answer 接入 Orchestrator。

### Q4：为什么不新增 rag_answer / rag_citation 表？

因为 V2.3 不做 answer history，也不做 RAG 审计或评测数据沉淀。当前目标是同步返回 answer 和 citations。

如果新增 `rag_answer` 和 `rag_citation` 表，就需要设计更多问题：历史查询、用户权限、数据保留、citation schema、answer 与 retrieval run 的关系、evaluation 标注关系等。这些不是 V2.3 的核心目标。

项目的做法是先保持轻量：response 中返回 citations 和 metadata，但不持久化。等后续需要 RAG answer history、异步任务、评测或审计时，再设计专门的数据模型。

### Q5：为什么 citation 只做到 chunk-level？

因为当前系统的最小可靠 evidence 单元就是 document chunk。V2.1.1 已经为 chunk 保存了 `headingPath`、offset 和 strategy，V2.2 搜索返回的也是 chunk。因此 V2.3 自然先做 chunk-level citation。

sentence-level citation 需要更复杂的证据对齐。LLM 的一句回答可能来自多个 chunk，也可能是对证据的综合概括。要证明每句话都被原文支持，需要 answer parsing、span matching、citation evaluation 等能力。

所以 chunk-level citation 是合理的 baseline。它不完美，但足够支撑第一版溯源展示，并为后续 sentence-level citation 做准备。

### Q6：如何防止 LLM hallucination？

当前主要通过 prompt 约束和空检索处理降低 hallucination 风险。

Prompt 明确要求只能根据 Context 回答，如果 Context 中没有答案，就回答“无法从当前资料中确定”。同时，空检索结果时，系统不调用 LLM，而是直接返回无法确定，避免模型无依据发挥。

但这不能从根本上消除 hallucination。真实生产系统还需要更好的 retrieval、rerank、answer verification、citation evaluation，甚至让模型在回答前判断证据是否足够。V2.3 只是 baseline。

### Q7：如果检索结果不相关怎么办？

如果 TopK chunks 不相关，LLM 即使被要求基于 Context 回答，也可能给出弱相关或错误答案。这说明 RAG 的质量上限很大程度受 retrieval 影响。

当前项目的下一步 V2.4 就是 Chunking Evaluation，用 query-evidence 数据集计算 Recall@K、MRR、Context Precision 等指标。之后还可以接真实 embedding、rerank、hybrid search，提高检索质量。

在 V2.3 中，系统至少会返回 citations 和 score，便于人工检查“答案是不是基于错误 chunk”。这为后续评价和调试提供了入口。

### Q8：如果 context 太长怎么办？

当前 V2.3 通过 topK 限制控制 context 规模，默认 5，最大 20。这个做法简单但不够智能。

后续可以加入 context budget，例如按 token 预算截断；也可以先 rerank，再选择更少但更相关的 chunks；还可以对 chunk 做压缩摘要，或者针对 VIP 用户给更大的 context budget。

当前不做这些，是因为 V2.3 目标是 answer baseline，不是上下文优化系统。

### Q9：为什么不做 rerank？

Rerank 通常用于提升 TopK 的精度，比如先用 embedding 召回 50 个 chunk，再用 cross-encoder 或 LLM reranker 选出最相关的 5 个。

但 rerank 会引入新的模型、延迟、成本和评测问题。当前还没有真实 embedding，也没有 query-evidence 评测集，贸然引入 rerank 不容易判断收益。

所以 V2.3 先使用现有 exact cosine search 的 TopK。等 V2.4 有 evaluation 后，再决定是否接 rerank，会更有依据。

### Q10：为什么不做 evaluation？

Evaluation 需要独立设计，不是简单写几个单元测试。

要评估 RAG，需要 query-evidence 数据集，知道每个 query 对应哪些 chunk 是相关证据，还要定义答案质量和 citation 质量标准。然后才能计算 Recall@K、MRR、Context Precision、Citation Accuracy 等指标。

V2.3 是 answer baseline，先把 answer 和 citations 返回出来。V2.4 再专门做 evaluation，边界更清楚。

### Q11：怎么评估 RAG answer 质量？

可以分层评估。

第一层是 retrieval：看相关证据有没有被召回，比如 Recall@K、MRR、Context Precision。

第二层是 answer：看回答是否正确、是否完整、是否基于证据。可以用人工标注，也可以用 LLM-as-judge，但后者需要防止 judge 偏差。

第三层是 citation：看引用是否真的支持答案。chunk-level citation 可以先人工抽查，后续再做 sentence-level citation accuracy。

当前项目还没有这些评测指标，所以不能声称 RAG 效果提升，只能说已经具备评测前置结构。

### Q12：怎么评估 citation 质量？

citation 质量可以看几个方面。

首先是 citation relevance：引用的 chunk 是否和问题相关。其次是 support：引用内容是否真的支持答案中的主张。再次是 granularity：chunk 级引用是否太粗，如果 chunk 很长，用户仍然很难定位具体证据。

当前 V2.3 返回 `headingPath`、offset 和 preview，方便人工验收。后续如果要严肃评估 citation，需要构造答案句子到 evidence span 的标注数据，再计算 citation accuracy 或 support rate。

### Q13：真实 LLM 接入后会有什么问题？

真实 LLM 会带来延迟、费用、限流、网络失败、模型输出不稳定和数据合规问题。

工程上需要处理 API key、超时、重试、fallback、token usage、cost tracking、模型路由和错误分类。回答质量上也需要 prompt 调优、context 控制和 hallucination 检测。

好处是当前项目已有 `LlmClient` 抽象，V2.3 只依赖接口，不直接绑定 Mock LLM。后续真实 provider 可以挂在同一抽象下，而不用重写 RAG Answer 编排。

### Q14：真实 Embedding 接入后怎么选型？

需要结合语言、效果、成本、延迟、维度、输入长度和部署方式选型。

候选可以包括 OpenAI embedding、BGE、Qwen Embedding、Jina、E5、Voyage 等。中文项目尤其要关注中文 query 和文档的召回质量。

但不能只凭感觉选。后续应该基于 query-evidence 评测集，对比不同 embedding 模型的 Recall@K、MRR、Context Precision，同时记录 latency 和成本，再决定默认模型和不同用户层级的策略。

### Q15：后续如何把 RAG Answer 接入异步 Orchestrator？

可以新增 RAG task 类型，让用户提交 RAG 请求后创建 task，并通过现有 outbox、RabbitMQ、atomic claim、attempt 机制执行。

需要设计 RAG payload，包括 query、document scope、topK、requestedModel 等；还要设计 result schema，保存 answer、citations、retrieval metadata、LLM metadata。`task_attempt` 也可以记录每次 RAG 执行的 prompt、model、token、latency 和错误。

这一步适合在真实 LLM 接入、answer persistence 和 evaluation 需求明确之后做。V2.3 先同步化，是为了避免过早扩大任务系统 schema。

### Q16：如果用户是 VIP，RAG Answer 可以怎么升级？

VIP 可以在 retrieval 和 generation 两端做差异化。

检索侧可以给更大的 topK、更大的 context budget、更强的 rerank、更高质量 embedding 模型或 hybrid search。生成侧可以路由到更强模型，允许更长输出，返回更详细 citations。

系统侧还可以给 VIP 更高队列优先级、更高 rate limit、更完整审计和更细的成本统计。当前 V2.3 还没有用户层级，但 `requestedModel`、metadata 和后续 retrieval policy 都能为 VIP 策略打基础。

### Q17：如果数据量扩大 10 倍，当前方案瓶颈在哪里？

当前最大的瓶颈是 V2.2 的检索实现：MySQL TEXT 存向量，Java 内存 exact cosine scan。这适合 baseline 和小规模验证，不适合大规模生产检索。

数据量扩大后，向量反序列化、全量扫描、内存占用和排序都会变慢。后续需要引入 Vector DB 或 pgvector，使用 ANN index，并支持 metadata filter、hybrid search 和 rerank。

RAG Answer 层也会遇到 context 长度和 LLM latency 问题，需要 context budget、rerank、chunk compression 和异步化。

## 16. 本版本在项目中的位置

V2.3 是 RAG Answer baseline。它不是最终 RAG 系统，而是连接 V2.2 retrieval 和 V2.4 evaluation 的中间层。

完整路线可以概括为：

```text
Document Upload
-> Adaptive Chunking
-> Mock Embedding
-> Vector Search
-> RAG Answer with Citation
-> Retrieval Evaluation
-> Real Embedding
-> Vector DB
-> Retrieval Policy / VIP Search
```

V2.3 的意义是让系统第一次从“文档可检索”走到“问题可回答并可溯源”。它还很基础，但已经具备后续评测、模型替换、向量数据库选型和异步 RAG task 化的接口基础。
