# V0.4.1 面试文档：Flyway 数据库迁移

## 1. 本版本解决的问题

V0.4.1 解决数据库结构不可追踪、不可复现的问题。早期可以依赖 Hibernate ddl-auto 快速建表，但当项目进入多版本迭代后，表结构必须通过 Flyway migration 管理，保证本地、CI 和未来部署环境的 schema 一致。

## 2. 为什么现在做这个版本

它依赖 task 和 task_event 已经形成基础数据模型。后续会持续增加 error_message、retry 字段、timeout 字段、LLM 结果、文档和 embedding 表。如果不尽早引入 Flyway，后续 schema 变化会失控。此时不做复杂数据库发布平台，是因为 Flyway 已能满足单项目演进。

## 3. 核心需求边界

本版本引入 Flyway 脚本和 ddl-auto=validate 思路。不做自动回滚、不做多数据库兼容、不做在线 DDL 平台。旧脚本一旦进入版本历史就不随意修改，新增变化通过新版本脚本表达。

## 4. 方案选择

采用 `src/main/resources/db/migration/Vx__description.sql` 管理 schema。应用启动时 Flyway 按版本执行脚本，JPA 使用 validate 校验 Entity 与数据库是否匹配。这样 schema 变化可审计、可复现。

## 5. 替代方案与暂时不选的原因

替代方案一是继续使用 Hibernate ddl-auto=update。当前不选，因为自动改表不可审计，生产环境风险高。

替代方案二是手动执行 SQL。当前不选，因为多人开发和 CI 环境容易漏执行，无法保证一致性。

## 6. 数据库设计

本版本重点是迁移机制，不一定新增业务字段。Flyway 会维护 flyway_schema_history 表，记录 version、description、script、checksum、success 等信息。后续所有业务表变化都通过 V1、V2、V3 等脚本递增。

## 7. API 设计

本版本主要是数据库演进能力，未新增用户 API。

## 8. 核心流程

应用启动
-> Flyway 扫描 db/migration
-> 按版本执行未执行脚本
-> 记录 flyway_schema_history
-> JPA validate 校验 Entity
-> 应用启动完成

## 9. 测试方案

建议覆盖应用启动时 Flyway 能执行成功、flyway_schema_history 版本正确、JPA validate 不报错。SQL 验证可查询 flyway_schema_history。异常路径包括脚本语法错误、Entity 字段与表结构不匹配。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是团队纪律：旧 migration 不能随便改，否则 checksum 会变化并破坏已部署环境。另一个难点是 Entity 和 SQL 要同步演进，否则 validate 会在启动时暴露问题。

## 12. 本版本亮点

- 把数据库结构纳入版本控制。
- 用 ddl-auto=validate 降低自动改表风险。
- 为后续多表、多字段快速迭代提供稳定机制。
- 让本地和 CI 更容易复现数据库状态。

## 13. 简历 bullet

### 简洁版

引入 Flyway 管理数据库迁移，替代 Hibernate 自动改表，提升 schema 演进的可追踪性和可复现性。

### 强化版

将 AI 任务系统的数据库结构演进纳入 Flyway migration 体系，并结合 JPA validate 校验实体映射，避免自动 DDL 带来的不可审计变更风险。

## 14. 面试高强度追问与回答

Q1：为什么生产项目不能依赖 ddl-auto=update？

A1：它会自动修改表结构，变更不可审计，也可能在生产环境执行危险 DDL。Flyway 让每次变更都有脚本和版本。

Q2：ddl-auto=validate 有什么意义？

A2：它不改表，只校验 Entity 和数据库结构是否匹配。这样能在启动阶段发现 schema 漂移。

Q3：旧 migration 为什么不能改？

A3：旧脚本可能已经在其他环境执行过，修改会导致 checksum 不一致。正确做法是新增新的 migration。

Q4：Flyway 如何支持 CI？

A4：CI 启动测试上下文时可以自动执行 migration，保证测试数据库结构与代码一致。

Q5：如果 migration 执行失败怎么办？

A5：需要修复脚本或清理测试环境。生产环境还要有备份、灰度和回滚策略。

Q6：Flyway 和 JPA Entity 谁是准？

A6：结构变更以 migration 为准，Entity 要匹配数据库。validate 用于发现不一致。

Q7：为什么不自己维护 SQL 文档？

A7：文档不能自动执行，也不能保证环境一致。Flyway 同时提供执行和版本记录。

Q8：当前局限是什么？

A8：Flyway 解决 schema 版本管理，但不自动解决大表在线变更、回滚和跨库兼容问题。

## 15. 后续演进路线

V0.4.1 支撑 V0.7 error_message、V0.8 retry 字段、V1.x LLM 字段、V2.x 文档和 embedding 表。后续可结合 CI、测试容器和发布流程提升数据库变更安全性。
