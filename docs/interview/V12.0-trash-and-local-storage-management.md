# V12.0 Trash & Local Storage Management 面试文档

## 1. 阶段摘要

V12.0 的阶段名称是：

```text
V12.0 Trash & Local Storage Management
```

中文可以表述为：

```text
垃圾箱与本地存储管理
```

这一阶段的核心目标是：把当前“删除文档”能力升级为产品级数据生命周期管理，并为本地个人知识库应用补齐垃圾箱、恢复、永久删除、自动清理、缓存管理和本地存储统计能力。

项目当前定位是：

```text
Personal AI Knowledge Workspace
个人 AI 知识工作台
```

V12.0 解决的问题是：

```text
用户删除文档后，不应该立即等同于最终删除；
进入垃圾箱的文档应该可以恢复；
超过保留期的垃圾箱文档应该可以被自动清理；
永久删除需要清理原始文件、解析文本、chunks、vectors、cache；
系统设置中应该能看到本地存储与缓存状态；
清理缓存不能误删知识库。
```

文档生命周期从原来的简单软删除，升级为：

```text
ACTIVE
-> TRASHED
-> PURGED
```

---

## 2. 为什么需要 V12.0

在 V12.0 之前，项目已经具备了较完整的个人知识库能力：

```text
文档上传；
异步文档摄入；
文档重新索引；
知识库分组；
范围检索；
RAG Answer；
RAG 质量评分；
Agent Task；
Tool-Using Agent Workflow；
本地 Ollama；
多模型供应商配置。
```

但数据生命周期还不够产品化。

如果用户删除了一个文档，系统必须回答几个问题：

```text
它还能恢复吗？
它还会不会参与问答？
它的向量还在吗？
什么时候会被真正删除？
删除后是否会清理缓存？
本地磁盘空间如何查看？
缓存能不能安全清理？
```

V12.0 就是要把这些问题系统化解决。

---

## 3. 阶段核心目标

V12.0 完成后，系统应具备：

```text
1. 文档状态支持 ACTIVE / TRASHED / PURGED；
2. 删除文档进入垃圾箱；
3. TRASHED 文档 7 天内可恢复；
4. TRASHED 文档不参与 Ask / Retrieval / Agent；
5. 恢复后文档重新可用于问答和 Agent；
6. PURGED 文档永久删除，不可恢复；
7. 支持立即永久删除；
8. 支持过期垃圾自动清理；
9. 支持垃圾箱页面；
10. 支持系统设置中的存储统计；
11. 支持缓存管理；
12. README 和 manual 更新；
13. 默认测试继续保持低依赖。
```

## 4. 核心设计一：文档生命周期升级

V12.0 将文档生命周期升级为三个状态：

```text
ACTIVE
TRASHED
PURGED
```

### 4.1 ACTIVE

含义：

```text
正常文档，可用于知识库问答、范围检索、Agent Task 和 Tool Workflow。
```

### 4.2 TRASHED

含义：

```text
已进入垃圾箱，默认保留 7 天，可恢复。
```

TRASHED 文档：

```text
不参与 Ask；
不参与 Retrieval；
不参与 Agent Task；
不出现在普通文档列表中；
出现在垃圾箱页面中；
保留 chunks、vectors、原始文件和解析文本；
恢复时不需要重新 embedding。
```

### 4.3 PURGED

含义：

```text
已永久删除，不可恢复。
```

PURGED 文档：

```text
不参与任何检索或任务；
不出现在普通文档列表或垃圾箱中；
原始文件、解析文本、chunks、vectors、相关缓存应被清理；
可以保留 minimal audit record，但不能暴露原文内容。
```

## 5. 核心设计二：状态字段

建议在 document 表中新增或完善字段：

```text
status
deleted_at
trashed_at
purge_after
purged_at
purge_status
storage_bytes
original_file_path
extracted_text_path
```

### 5.1 purge_after

默认规则：

```text
purge_after = trashed_at + 7 days
```

也就是用户删除文档后，7 天内可以恢复，超过保留期后会被自动清理。

### 5.2 display 字段

对前端返回：

```text
status
displayStatus
trashedAt
purgeAfter
remainingRetentionDays
canRestore
canPurgeNow
disabledReason
```

中文显示：

```text
ACTIVE -> 可用于问答
TRASHED -> 已放入垃圾箱
PURGED -> 已永久删除
```

## 6. 核心设计三：删除文档进入垃圾箱

原删除接口可以继续使用：

```http
DELETE /documents/{documentId}
```

但行为从“删除 / DELETED”升级为：

```text
ACTIVE -> TRASHED
```

删除时：

```text
设置 trashedAt = now；
设置 purgeAfter = now + 7 days；
不物理删除原始文件；
不物理删除 chunks；
不物理删除 vectors；
不清理 embedding cache；
Ask / Agent / Retrieval 立即过滤该文档。
```

### 6.1 幂等规则

建议规则：

```text
ACTIVE 删除后进入 TRASHED；
TRASHED 再 delete，返回已在垃圾箱；
PURGED 再 delete，返回清晰错误；
不存在 document，返回 DOCUMENT_NOT_FOUND。
```

这样可以避免用户重复点击导致异常状态。

---

## 7. 核心设计四：恢复文档

新增 API：

```http
POST /documents/{documentId}/restore
```

行为：

```text
TRASHED -> ACTIVE
```

恢复时：

```text
不需要重新 chunk；
不需要重新 embedding；
不需要重新写 vector；
恢复后可重新用于 Ask / Retrieval / Agent。
```

原因是：

```text
TRASHED 只是生命周期状态；
chunks、vectors、原始文件仍然保留；
检索层只是过滤了 TRASHED 文档。
```

### 7.1 为什么恢复不需要重新索引

进入垃圾箱不是物理删除，向量和文本片段还在。恢复只需要把状态改回 ACTIVE，原向量索引仍然有效。

这比“删除后重新上传”更符合用户习惯，也节省时间和模型成本。

---

## 8. 核心设计五：立即永久删除

新增 API：

```http
POST /documents/{documentId}/purge
```

行为：

```text
TRASHED -> PURGED
```

永久删除时应清理：

```text
原始文件；
解析文本；
document chunks；
vector store entries；
embedding cache entries；
retrieval cache entries；
document collection mapping；
可重建缓存。
```

### 8.1 ACTIVE 是否允许直接 purge

V12.0 建议：

```text
ACTIVE 不允许直接 purge；
必须先进入 TRASHED。
```

原因是：

```text
永久删除不可恢复，应该有一个垃圾箱缓冲期；
符合用户对回收站的直觉；
减少误删风险。
```

如果未来需要管理员强制删除，可以单独设计 force purge，不放在 V12.0。

## 10. 核心设计七：清空过期垃圾

新增 API：

```http
POST /documents/trash/purge-expired
```

行为：

```text
扫描 purge_after < now 且 status = TRASHED 的文档；
逐个执行 purge；
返回清理数量和失败数量。
```

测试时应使用 mock clock 或 service 方法直接测试，不依赖真实时间等待。

## 11. 核心设计八：Retrieval / Ask / Agent 过滤 TRASHED

这是 V12.0 最关键的正确性要求之一。

TRASHED 和 PURGED 文档必须在所有上下文入口中过滤：

```text
RAG Answer；
dense retrieval；
lexical retrieval；
hybrid retrieval；
rerank candidate；
finalTopK；
citations；
prompt context；
qualityScore metricDetails；
Collection scoped retrieval；
Agent knowledge_search tool；
context_summary；
Tool-Using Agent Workflow；
Evaluation / Quality Score。
```

### 11.1 为什么过滤必须在多层做

只在 UI 层隐藏不够。

必须保证：

```text
用户看不到；
检索候选中没有；
最终 prompt 中没有；
citations 中没有；
Agent tool output 中没有。
```

这样才能避免已删除文档内容继续影响回答。

## 12. 核心设计九：永久删除清理策略

V12.0 建议抽象几个清理服务：

```text
StorageCleaner
VectorCleanupService
CacheCleanupService
```

### 12.1 Vector cleanup

如果当前 VectorStore 已支持按 documentId 删除，则执行真实删除。

如果 Qdrant delete-by-documentId 暂时没有完整实现，可以新增抽象：

```text
VectorStore.deleteByDocumentId(documentId)
```

默认 mock / in-memory 实现可以 no-op 或内存删除。

Qdrant 实现如果无法安全实现，应记录 cleanup warning，而不是静默成功。

### 12.2 Cache cleanup

清理与该文档相关的：

```text
embedding cache；
retrieval cache；
其他可重建缓存。
```

注意：

```text
清理缓存不能删除 document；
不能删除 collection；
不能删除 provider config；
不能删除仍在使用的 chunks / vectors。
```

### 12.3 文件 cleanup

如果项目已经有原始文件路径和 extracted text path，则永久删除时清理。

如果当前没有完整文件持久化路径，应安全降级：

```text
记录 warning；
不伪造已删除；
在技术详情中展示清理限制。
```

## 13. 核心设计十：自动清理 Job

新增：

```text
TrashCleanupJob
```

或：

```text
DocumentPurgeScheduler
```

建议配置：

```properties
app.trash.retention-days=7
app.trash.cleanup-enabled=true
app.trash.cleanup-cron=0 0 3 * * *
```

默认：

```text
每天凌晨 3 点执行。
```

### 13.1 Job 行为

```text
查询 purgeAfter < now 且 status = TRASHED 的文档；
逐个执行 purge；
记录成功数量；
记录失败数量；
单个失败不影响其他文档；
记录日志和事件。
```

### 13.2 为什么单个失败不影响整体

如果一个文档的 vector cleanup 失败，不应该阻塞所有其他过期文档清理。

正确做法是：

```text
失败文档记录 warning / error；
继续清理其他文档；
最终汇总成功数和失败数。
```



### 14.2 永久删除操作

点击“永久删除”必须二次确认。

```text

```

这比普通“删除”更严肃，避免用户误操作。

## 16. 核心设计十三：缓存管理

新增 API：

```http
GET /storage/summary
POST /storage/cache/embedding/clear
POST /storage/cache/retrieval/clear
POST /storage/cache/clear-all
```

### 16.1 缓存清理页面文案

必须说明：

```text
缓存可以重新生成；
清理缓存不会删除原始文档和知识库记录；
清理缓存可能导致下次处理变慢。
```

### 16.2 缓存清理的边界

清理缓存不能删除：

```text
document；
chunk；
vector；
collection；
provider config；
model provider config；
agent task。
```

缓存是可重建数据，知识库是持久化数据，两者不能混淆。

## 22. 简历 bullet（中文）

```text
- 设计并实现文档垃圾箱生命周期，将原简单删除升级为 ACTIVE -> TRASHED -> PURGED 状态机，支持 7 天内恢复、立即永久删除和过期自动清理。
```

## 23. 面试高强度追问与回答

### 23.1 为什么要把 DELETED 改成 TRASHED / PURGED？

回答：

```text
因为删除在产品里有两层语义：进入垃圾箱和永久删除。TRASHED 表示可恢复，PURGED 表示不可恢复。如果只有 DELETED，一个状态同时表达“已删除但可恢复”和“已永久清理”，语义会混乱，也不利于后续自动清理和恢复。
```

### 23.2 为什么 TRASHED 文档不直接删除 vectors？

回答：

```text
因为 TRASHED 是可恢复状态。保留 chunks 和 vectors 可以让恢复变成一个状态切换，不需要重新解析、chunk、embedding 和写入向量库，既快又省模型成本。系统只需要在 Ask / Retrieval / Agent 层过滤 TRASHED 文档即可。
```

### 23.3 PURGED 时为什么要清理 chunks、vectors 和 cache？

回答：

```text
PURGED 表示永久删除，不再可恢复。为了避免本地磁盘占用持续增长，也为了避免已永久删除内容在缓存或向量库中残留，需要清理原始文件、解析文本、chunks、vectors 和相关可重建缓存。
```

### 23.4 如果 vector cleanup 失败怎么办？

回答：

```text
不能假装成功。清理过程应该记录 cleanup warning 或 error，单个文档失败不影响其他文档清理。用户侧不能再把该文档作为可恢复文档展示，同时系统要保证 Ask / Agent 不会使用 PURGED 文档。后续可以通过 cleanup retry 修复底层残留。
```

### 23.5 为什么 ACTIVE 不允许直接 purge？

回答：

```text
因为永久删除不可恢复，风险较高。产品上应该先进入垃圾箱，再由用户确认永久删除或等待保留期自动清理。这样可以减少误操作，符合用户对回收站的直觉。
```

### 23.6 缓存清理和垃圾箱有什么区别？

回答：

```text
垃圾箱保存的是可恢复的用户数据，不能随便删除；缓存是可重建的加速数据，比如 embedding cache 或 retrieval cache。清理缓存不应该删除原始文档、chunks、vectors、collection 或 provider config，只是让后续处理可能变慢。
```

### 23.7 为什么要做存储统计？

回答：

```text
个人本地知识库会长期积累文档、文本、向量和缓存。用户需要知道本地数据占用在哪里，哪些是原始文件，哪些是缓存，哪些由 Qdrant 或数据库管理。存储统计可以帮助用户理解和管理本地空间。
```

### 23.8 为什么自动清理 Job 不依赖真实定时等待测试？

回答：

```text
定时任务测试如果依赖真实时间，会慢且不稳定。更好的方式是把清理逻辑放在 service 中，通过 mock clock 或构造过期数据直接测试 purgeExpired，scheduler 只负责触发。
```
