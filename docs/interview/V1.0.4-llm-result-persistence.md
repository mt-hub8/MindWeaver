# V1.0.4 面试文档：LLM 结果持久化

## 1. 本版本解决的问题

V1.0.4 解决 LLM 调用成功后“结果只存在执行线程里”的问题。AI 输出必须保存到数据库，否则任务 SUCCESS 后用户无法查询结果，系统也无法审计模型输出、展示最终内容或进一步拆分 output chunks。

## 2. 为什么现在做这个版本

它依赖 V1.0 的 LlmClient 和 MockLlmClient。只有模型调用已经抽象出来，才需要保存 LlmResponse 的 content 和 model。它为 V1.4 输出 chunks、后续 streaming、审计和成本分析做准备。此时不保存完整 prompt trace 和 attempt，是因为先保存最终摘要即可。

## 3. 核心需求边界

本版本新增 result_content、llm_model，并通过 markTaskSucceeded 保存结果。不做 token usage、不做 streaming、不做多 attempt 历史、不做真实 Provider。保持克制是为了先完成“任务成功后能查到输出”的闭环。

## 4. 方案选择

在 task 表保存最终 resultContent 和 llmModel。TaskExecutionService 在 LlmResponse.success=true 且任务仍 RUNNING 时调用 markTaskSucceeded，由 TaskService 完成 RUNNING -> SUCCESS、结果保存、errorMessage 清空和事件记录。

## 5. 替代方案与暂时不选的原因

替代方案一是不保存结果，只通过日志查看。当前不选，因为用户和前端无法稳定读取。

替代方案二是单独建 result 表。当前不选，因为早期只保存最终结果，放在 task 表更简单；后续 output chunks 和 attempt 会补充细节。

## 6. 数据库设计

task 表新增 result_content TEXT NULL 和 llm_model VARCHAR(100) NULL。result_content 保存完整模型输出，llm_model 保存实际返回模型。后续可扩展 llm_provider、token usage、task_output_chunk 和 task_attempt。

## 7. API 设计

GET /tasks/{taskId} 返回 resultContent 和 llmModel。POST /tasks 不变。成功任务应返回 status=SUCCESS、resultContent 不为空、llmModel 为 mock-llm 或路由后的模型。

## 8. 核心流程

TaskExecutionService 调用 LlmClient
-> LlmResponse success=true
-> 检查任务仍 RUNNING
-> TaskService.markTaskSucceeded
-> 保存 resultContent / llmModel
-> RUNNING -> SUCCESS
-> GET /tasks 返回结果

## 9. 测试方案

建议覆盖正常任务最终 SUCCESS 且 resultContent 不为空、llmModel 正确、失败任务不保存成功结果、重试成功后清空旧 errorMessage。SQL 可查询 task.result_content 和 llm_model。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是保存结果前必须确认任务仍处于 RUNNING。如果任务已经被取消或超时标记 FAILED，执行线程不能再写 resultContent 并改 SUCCESS。

## 12. 本版本亮点

- 完成 LLM 调用结果查询闭环。
- 成功状态和输出结果在同一业务方法中保存。
- 清空旧 errorMessage，避免重试成功后残留失败信息。
- 为 output chunks 和 attempt 记录打基础。

## 13. 简历 bullet

### 简洁版

实现 LLM 执行结果持久化，支持保存 resultContent 和 llmModel 并通过任务详情接口返回。

### 强化版

在异步 LLM 任务执行链路中引入结果持久化，通过 markTaskSucceeded 原子化保存模型输出、实际模型名和 SUCCESS 状态，避免取消或超时后的结果覆盖。

## 14. 面试高强度追问与回答

Q1：为什么 AI 输出必须持久化？

A1：异步任务完成后用户需要查询结果，前端展示和审计也依赖持久化输出。

Q2：为什么保存 llmModel？

A2：用户请求模型和实际执行模型可能不同，保存实际模型便于排查和统计。

Q3：为什么成功时清空 errorMessage？

A3：任务可能先失败后重试成功，如果不清空旧错误，SUCCESS 任务会显示误导信息。

Q4：为什么不单独建 result 表？

A4：当前只保存最终结果，放在 task 表简单直接。后续多片段和多 attempt 会用专门表。

Q5：失败任务保存 resultContent 吗？

A5：不保存成功输出。失败任务应保留 errorMessage 和 usage metadata，避免误判为成功。

Q6：markTaskSucceeded 为什么要走状态机？

A6：只有 RUNNING 才能 SUCCESS，终态不能被覆盖。

Q7：输出很大怎么办？

A7：TEXT 能支持一定长度，但后续需要 output chunks、对象存储或分页读取。

Q8：当前局限是什么？

A8：只保存最终摘要，不记录每次 attempt 的输出和完整 provider 原始响应。
