# V0.11 面试文档：本地基础设施与开发文档

## 1. 本版本解决的问题

V0.11 解决“项目只能在作者机器上跑”的问题。通过 Docker Compose 提供 MySQL 和 RabbitMQ，本地开发者可以快速复现基础设施；通过 README 和 docs 文档说明启动、API、结构和面试表达，让项目从代码变成可交付工程。

## 2. 为什么现在做这个版本

它依赖任务系统已经有数据库、Flyway、RabbitMQ 和 API。没有本地环境标准化，后续 LLM、文档上传、embedding 会越来越难验收。此时不容器化 Spring Boot 应用，是为了保留 IDEA 本地调试体验。

## 3. 核心需求边界

本版本新增 docker-compose.yml、application-docker.properties 和文档。不做 Dockerfile、不做 Kubernetes、不做 CI/CD、不做生产部署。保持克制是为了先解决本地开发复现。

## 4. 方案选择

Docker Compose 只启动 MySQL 和 RabbitMQ，Spring Boot 仍由本机运行并使用 docker profile 连接 localhost:3307 和 localhost:5672。文档分层：README 总览、local-dev 启动、api-examples 验收、project-structure 结构、resume-interview 表达。

## 5. 替代方案与暂时不选的原因

替代方案一是要求开发者本机安装 MySQL/RabbitMQ。当前不选，因为版本和配置难统一。

替代方案二是全量容器化应用。当前不选，因为早期频繁调试 Java 代码，IDEA 本地启动更方便。

## 6. 数据库设计

本版本未新增业务数据库结构，主要提供 MySQL 容器和 Flyway 执行环境。MySQL 数据通过 volume 持久化，端口使用 3307 避免与本机 3306 冲突。

## 7. API 设计

本版本主要是开发环境和文档能力，未新增用户 API。docs/api-examples.md 记录已有 API 的调用和验收方法。

## 8. 核心流程

开发者启动 Docker Compose
-> MySQL / RabbitMQ 就绪
-> 使用 docker profile 启动 Spring Boot
-> Flyway 执行 migration
-> 调用 API 验收任务流程
-> 查 SQL / RabbitMQ 管理台排查问题

## 9. 测试方案

建议覆盖 docker compose up 后 MySQL/RabbitMQ 容器健康、Spring Boot docker profile 启动成功、Flyway 迁移执行成功、POST /tasks 和 GET /tasks/{id} 可用。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是本地环境要足够简单，同时不污染生产配置。application-docker.properties 独立承载本地容器连接信息，避免修改默认 application.properties。

## 12. 本版本亮点

- 用 Docker Compose 标准化 MySQL/RabbitMQ。
- 保留本机启动 Spring Boot，方便调试。
- 文档覆盖启动、API、结构和面试表达。
- 把文档作为工程交付的一部分。

## 14. 面试高强度追问与回答

Q1：为什么本地环境标准化重要？

A1：依赖 MySQL 和 RabbitMQ 的系统如果环境不一致，问题很难复现。Docker Compose 降低搭建成本。

Q2：为什么 Spring Boot 不容器化？

A2：当前仍处于快速开发阶段，本机启动更方便断点调试。容器化应用可放到后续。

Q3：为什么 MySQL 用 3307？

A3：避免和开发者本机已有 MySQL 3306 冲突。

Q4：文档为什么算工程交付？

A4：代码能跑只是第一步，别人能启动、验收、理解架构，项目才具备协作价值。

Q5：application-docker.properties 有什么作用？

A5：它隔离 Docker 本地环境配置，避免污染默认配置。

Q6：RabbitMQ 管理后台有什么价值？

A6：可以观察队列、exchange、消息堆积和连接，帮助排查异步调度问题。

Q7：常见问题有哪些？

A7：端口冲突、MySQL 未启动、RabbitMQ 端口占用、Flyway 失败和 profile 未启用。

Q8：当前局限是什么？

A8：它是本地开发环境，不是生产部署方案，没有监控、密钥管理和高可用。
