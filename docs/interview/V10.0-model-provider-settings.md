# V10.0 Model Provider Settings 面试文档

## 1. 阶段摘要

V10.0 的阶段名称是：

```text
V10.0 Model Provider Settings
```

中文可以表述为：

```text
多模型供应商可视化配置
```

这一阶段的核心目标是：把原先依赖 `application-local-ai.properties`、环境变量和手工配置的模型接入方式，升级为一个可视化、可扩展、可测试、可安全保存 API Key 的模型供应商配置中心。

项目当前定位已经在 V9.0 调整为：

```text
Personal AI Knowledge Workspace
个人 AI 知识工作台
```

V10.0 的意义是让这个产品从“开发者手动改配置文件切换模型”，升级为：

```text
用户可以在模型设置页面中添加模型供应商、填写 baseUrl / API Key、测试连接、选择默认 LLM 和默认 Embedding 模型。
```

---

## 2. 为什么需要 V10.0

在 V8.0 中，项目已经接入了本地 Ollama：

```text
Java Spring Boot
  -> Python AI Runtime Worker
  -> Ollama /api/embed
  -> Ollama /api/generate
```

在 V9.0 中，项目已经完成产品化收敛，新增了模型设置入口。

但是此时模型接入仍然偏工程化：

```text
通过 application-local-ai.properties 配置；
通过环境变量配置；
通过 Python Worker 启动参数配置；
普通用户不知道如何切换模型；
API Key 无统一管理入口；
不同模型供应商没有统一抽象。
```

V10.0 要解决的问题是：

```text
模型供应商如何统一管理？
API Key 如何安全保存？
用户如何在页面里测试连接？
Ask / Agent Task 如何读取当前默认模型？
如何支持本地 Ollama 与 OpenAI-compatible API，而不把厂商逻辑写死到业务代码里？
```

---

## 3. 阶段核心目标

V10.0 完成后，用户应该可以：

```text
1. 打开“模型设置”页面；
2. 查看当前模型供应商；
3. 添加本地 Ollama Provider；
4. 添加 OpenAI-compatible Provider；
5. 使用 DeepSeek / Qwen / SiliconFlow 等 OpenAI-compatible 预设；
6. 填写 baseUrl、API Key、默认 LLM 模型、默认 Embedding 模型；
7. 点击测试连接；
8. 设置默认问答模型；
9. 设置默认向量模型；
10. Ask / Agent Task 自动使用当前默认模型配置。
```

## 4. 核心设计一：ModelProviderConfig 数据模型

V10.0 新增模型供应商配置模型：

```text
model_provider_config
```

典型字段：

```text
id
provider_type
display_name
base_url
api_key_encrypted
api_key_masked
default_llm_model
default_embedding_model
default_rerank_model
embedding_dimension
enabled
default_llm
default_embedding
last_test_status
last_test_message
last_tested_at
created_at
updated_at
```

### 4.1 provider_type 设计

建议枚举：

```text
MOCK
OLLAMA
OPENAI_COMPATIBLE
CUSTOM_OPENAI_COMPATIBLE
```

含义：

```text
MOCK：默认开发测试模型，不依赖真实外部服务；
OLLAMA：本地 Ollama 模型；
OPENAI_COMPATIBLE：OpenAI-compatible API 预设；
CUSTOM_OPENAI_COMPATIBLE：用户自定义 OpenAI-compatible endpoint。
```

V10.0 不深度接入 Gemini / Claude 特殊协议，而是优先把 OpenAI-compatible 这一类接口做稳。

原因是：

```text
DeepSeek、Qwen、SiliconFlow 等平台通常可以通过 OpenAI-compatible 风格 API 接入；
先做通用协议比给每个厂商写一套业务逻辑更可扩展；
Gemini / Claude 特殊协议可以后续单独扩展。
```

## 5. 核心设计二：API Key 安全保存

V10.0 的重要安全边界是：

```text
API Key 可以被用户提交，但不能明文返回前端，不能明文写日志。
```

典型流程：

```text
用户在页面填写 API Key
-> 后端接收 request DTO
-> SecretService 加密
-> 保存 api_key_encrypted
-> 生成 api_key_masked
-> response DTO 只返回 masked key
```

### 5.1 为什么不能明文返回

API Key 是访问模型供应商的凭据。如果 GET 接口直接返回明文，前端页面、浏览器插件、日志、截图都可能导致泄露。

因此 V10.0 要求：

```text
request DTO 可以包含 apiKey；
response DTO 不能包含 apiKey 明文；
response DTO 只能包含 apiKeyMasked；
日志不能打印明文 key；
编辑 provider 时，如果 apiKey 为空，不覆盖原有 key。
```

### 5.2 加密方案

V10.0 可以采用基础本地加密：

```text
AES-GCM；
master key 来自配置项或环境变量；
测试环境使用 dummy key；
真实 key 不提交 Git。
```

后续可扩展为：

```text
Windows Credential Manager；
macOS Keychain；
Linux Secret Service。
```

这一阶段的重点是建立安全边界，而不是一步做到操作系统级凭据管理。

## 6. 核心设计三：Provider 预设

V10.0 支持 Provider 预设，降低配置门槛。

建议预设：

```text
本地 Ollama；
OpenAI-compatible 自定义；
DeepSeek compatible；
Qwen / DashScope compatible；
SiliconFlow compatible。
```

### 6.1 Ollama 预设

典型配置：

```text
provider_type = OLLAMA
displayName = 本地 Ollama
baseUrl = http://localhost:11434
defaultEmbeddingModel = qwen3-embedding:0.6b
defaultLlmModel = qwen2.5:7b
```

Ollama 不需要 API Key。

### 6.2 OpenAI-compatible 预设

典型字段：

```text
baseUrl
apiKey
defaultLlmModel
defaultEmbeddingModel
embeddingDimension
```

DeepSeek / Qwen / SiliconFlow 在 V10.0 里作为 compatible 预设，不深度写入业务服务。

这可以避免代码债：

```text
业务层只知道 ModelProviderConfig；
Provider Client 层负责适配不同 endpoint；
RAG / Agent 不直接写具体厂商判断。
```

## 7. 核心设计四：连接测试

V10.0 新增连接测试能力：

```http
POST /model-providers/{id}/test
```

测试逻辑：

```text
MOCK：直接返回 mock success；
OLLAMA：测试本地 Ollama /api/tags 或 Python Worker 状态；
OPENAI_COMPATIBLE：检查 baseUrl 和 API Key 是否完整，可选调用 /v1/models；
CUSTOM_OPENAI_COMPATIBLE：同上。
```

测试响应：

```text
status: SUCCESS / FAILED / SKIPPED
message
providerType
testedAt
latencyMs
```

注意：

```text
只有用户主动点击“测试连接”时，才允许调用真实外部 API；
默认单元测试不能调用真实外部 API；
测试失败不能导致系统崩溃；
错误要中文友好。
```

## 8. 核心设计五：默认模型选择

V10.0 支持两个默认选择：

```text
default_llm
default_embedding
```

规则：

```text
同一时间只能有一个 default LLM provider；
同一时间只能有一个 default Embedding provider；
disabled provider 不能设为默认；
如果没有数据库配置，回退到 application properties / mock；
default profile 仍然 mock。
```

这部分通常由 `ModelProviderSelectionService` 或类似服务负责。

### 8.1 为什么 LLM 和 Embedding 要分开默认

因为一个用户可能希望：

```text
LLM 用云端强模型；
Embedding 用本地模型；
或者 LLM 用本地 Ollama；
Embedding 用更适合中文检索的 API 模型。
```

LLM 和 Embedding 的成本、速度、质量、隐私属性都不同，因此要分开配置。

## 9. 核心设计六：Embedding 模型切换风险提示

V10.0 必须提示：

```text
切换 Embedding 模型后，已有文档可能需要重新索引。
```

原因：

```text
不同 embedding 模型的向量维度可能不同；
同一批文档如果用不同模型生成向量，向量空间不一致；
旧索引与新查询向量可能不兼容；
Qdrant collection 维度可能固定。
```

例如：

```text
mock embedding dimension = 384
qwen3-embedding:0.6b dimension = 1024
```

切换后如果不重新索引，可能出现：

```text
vector dimension mismatch；
检索质量下降；
新旧向量空间混用。
```

V10.0 不自动重建索引，只提示风险。自动重建应放到后续更明确的存储管理或索引管理阶段。

## 16. 面试高强度追问与回答

### 16.1 为什么 V10.0 不直接把所有主流模型都接完？

回答：

```text
因为直接接所有厂商会导致 Provider 逻辑散落在业务代码里。V10.0 的重点不是数量，而是建立统一的 ModelProviderConfig、API Key 安全保存、连接测试和默认模型选择机制。先把 Ollama 和 OpenAI-compatible 做稳，后续 Gemini / Claude 等特殊协议可以在同一抽象下扩展。
```

---

### 16.2 API Key 为什么不能明文返回？

回答：

```text
API Key 是访问模型服务的凭据，如果 GET 接口明文返回，浏览器、日志、截图、插件都有泄露风险。因此我把请求 DTO 和响应 DTO 分开，提交时允许传入 apiKey，保存时加密，查询时只返回 masked key，并且日志中不打印明文。
```

---

### 16.3 为什么 LLM 和 Embedding 要分别设置默认 Provider？

回答：

```text
因为 LLM 和 Embedding 的使用场景不同。LLM 负责生成回答，Embedding 负责构建和检索向量空间。用户可能希望用本地 embedding 保护隐私，同时用云端强模型生成报告；也可能反过来。分开配置可以提高灵活性，也能避免切换生成模型时影响索引空间。
```

---

### 16.4 切换 Embedding 模型为什么要提示重新索引？

回答：

```text
不同 Embedding 模型可能有不同维度，也处在不同向量空间。旧文档如果用模型 A 建索引，查询却用模型 B 生成向量，检索质量会下降，甚至出现 dimension mismatch。因此切换默认 Embedding 模型时必须提示用户已有文档可能需要重新索引。
```

---

### 16.5 为什么默认测试不能调用真实外部 API？

回答：

```text
真实外部 API 会带来不稳定性、成本、网络依赖和密钥依赖。默认测试应该验证业务逻辑、配置逻辑和安全边界，而不是依赖第三方服务是否可用。真实 Provider 连接测试应该作为用户主动操作或 manual verification。
```

---

### 16.6 DeepSeek / Qwen / SiliconFlow 为什么作为 OpenAI-compatible 预设？

回答：

```text
因为很多模型平台兼容 OpenAI 风格 API。把它们作为 compatible 预设，可以复用同一套 baseUrl、API Key、model name、test connection 和 provider selection 逻辑，避免为每个厂商重复写业务代码。
```
