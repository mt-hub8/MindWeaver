# 1. STAR 总览

## S：Situation，背景

项目已经完成了前面多个 RAG 主线阶段：

```text
V12 Trash & Local Storage Management
V13 Batch Ingestion & Notification
V14 Knowledge Base Health Report
V15 Structured Chunking & Hybrid Retrieval Pipeline
```

也就是说，系统已经支持：

```text
单文件上传；
批量导入；
知识库分组；
结构化 chunk；
metadata pre-filter；
hybrid retrieval；
知识库体检报告；
跨集合污染检测；
错误版本污染检测。
```

但是当进入真实的万级文档知识库场景后，仅仅“查询时按 collection_id 过滤”还不够。因为污染可能发生在更底层的向量写入和维护阶段：

```text
1. 同一个文档重复上传，可能写入重复向量；
2. 批量导入失败重试，可能重复 insert；
3. reindex 后旧 generation 向量可能仍然残留；
4. purge 后原文档删除了，但 vector 仍在；
5. embedding 模型切换后，不同模型 / 维度的向量可能混在一起；
6. collection A 的 chunk 可能错误写入 collection B 的 vector payload；
7. chunk count 和 vector count 对不上；
8. 知识库体检发现跨集合污染，但无法判断是检索 filter 问题，还是向量库底层已经污染。
```

## T：Task，任务

V16.0 的任务是建立向量索引层的硬约束：

```text
1. 每条 vector 有稳定、可预测、可幂等的 vector_id；
2. 每条 vector payload 必须包含 collection_id、document_id、chunk_uid、embedding_model、dimension、generation、status 等关键字段；
3. 向量写入使用 upsert，而不是随机 insert；
4. reindex 使用 generation 代际隔离，失败不能破坏旧索引；
5. trash / restore / purge 必须同步 vector 状态或清理 vector；
6. batch retry 不得产生重复向量；
7. 支持 vector cleanup 和 audit；
8. 支持 collection 级污染审计；
9. 输出 Vector Index Health 页面，让用户知道当前向量库是否重复、残留、缺失、串库。
```

## A：Action，行动

V16.0 设计并实现以下模块：

```text
VectorIdentityService
VectorPayloadBuilder
VectorStore 增强接口
IdempotentVectorUpsertService
VectorNamespaceGuard
VectorGenerationService
DocumentLifecycleVectorSync
BatchRetryVectorDedup
VectorCleanupService
VectorConsistencyAuditService
CollectionPollutionAuditService
Vector Index Health 页面
```

## R：Result，结果

V16.0 完成后，系统可以做到：

```text
1. 同一 chunk 重复处理不会重复写 vector；
2. batch retry 不会产生重复向量；
3. collection A 不会写入 collection B 的向量；
4. reindex 失败不破坏旧 active generation；
5. old generation 不再参与检索；
6. PURGED 文档不会残留可检索向量；
7. chunk count 和 vector count 可以对账；
8. 系统能输出向量索引健康报告；
9. V14 的 CrossCollectionLeakRate 可以进一步定位到底是检索层污染还是向量存储层污染。
```

# 2. 遇到的问题：为什么 V16.0 必须做

## 2.1 之前的 collection scoped retrieval 解决了什么

之前的策略主要是：

```text
用户选择 collection
  ↓
检索时加 collection_id filter
  ↓
只返回该 collection 下的 chunk
```

这解决的是：

```text
查询阶段的隔离。
```

但是没有解决：

```text
写入阶段是否已经污染；
重复导入是否重复写入；
reindex 后旧向量是否残留；
purge 后 vector 是否还在；
collection_id 是否真的写入 payload；
同一 chunk 是否有多个 vector_id；
embedding 模型切换后是否混用旧向量。
```

所以 V16.0 要把隔离从“查询时过滤”前移到：

```text
写入、更新、删除、重建、审计。
```

## 2.2 如果不做 V16，会出现什么后果

```text
1. 向量数量持续膨胀；
2. 同一个 chunk 多份 vector 影响检索排序；
3. 旧 generation 的 vector 仍然被召回；
4. 文档已删除但向量残留，造成幽灵引用；
5. 不同 collection 的 vector 混入同一查询上下文；
6. V14 知识库体检只能发现污染，但无法修复底层污染；
7. 用户以为分组已经隔离，实际上向量库里已经混了。
```

# 3. 参考了什么技术思路

## 3.1 Qdrant：payload-based multitenancy

Qdrant 的多租户文档建议很多场景使用：

```text
single collection + payload-based partitioning
```

也就是用 payload 字段区分 tenant / group，而不是为每个 tenant 都建独立物理 collection。

映射到 MindWeaver：

```text
tenant_id  ≈  collection_id
```

工程启发：

```text
1. 每条 vector payload 必须有 collection_id；
2. search / delete / audit 都必须带 collection_id filter；
3. collection_id 不是 UI 字段，而是向量库隔离字段；
4. 常用 payload filter 字段需要考虑索引。
```

## 3.2 Pinecone：namespace per tenant

Pinecone 的 multitenancy 文档展示了用 namespace 实现 tenant 隔离。

工程启发：

```text
每个 collection 一个 namespace 是更强隔离；
但是对本地个人知识库来说，第一版用 payload filter 更轻；
后续如果某些 collection 很大，可以演进成 namespace 级物理隔离。
```

V16 的选择：

```text
第一版：共享向量索引 + collection_id payload + deterministic vector_id + audit
后续可选：大 collection 独立 namespace
```

## 3.3 Weaviate：tenant shard isolation

Weaviate 的 multi-tenancy 设计强调 tenant 之间隔离，一个 tenant 的数据对另一个 tenant 不可见。

工程启发：

```text
隔离应该是系统不变量，不是页面选择项。
```

对应到 V16：

```text
任何 collection scoped query，final context / citation / prompt 里都不能出现其他 collection 的 vector；
任何 collection cleanup 只能影响该 collection；
任何 collection reindex 只能构建该 collection 的 generation。
```

## 3.4 Milvus：primary field、partition key、upsert

Milvus 中 primary field 用于唯一标识实体，upsert 可以基于主键进行更新式写入。

工程启发：

```text
vector_id 应该像 primary key 一样稳定；
重复处理同一 chunk 应该 upsert，而不是 insert random id；
collection_id 可以理解为 partition key 语义。
```

对应到 V16：

```text
stableVectorKey = collection_id + document_id + chunk_uid + embedding_model + embedding_dimension
vectorId = stableVectorKey + generation
```

## 3.5 Weaviate deterministic UUID

Weaviate 支持 deterministic UUID，即由稳定业务字段生成确定性 ID。

工程启发：

```text
同一对象重复导入时应生成同一个 ID；
这样更新可以覆盖，而不是新增重复对象。
```

对应到 V16：

```text
不允许随机 vector_id；
vector_id 必须由 collection_id、document_id、chunk_uid、embedding_model、dimension、generation 稳定生成。
```

## 3.6 Curator：multi-tenant vector database indexing

Curator 论文研究多租户向量数据库中的索引问题，讨论了：

```text
共享索引 + metadata filter；
每 tenant 独立索引；
如何在内存成本和查询性能之间权衡。
```

工程启发：

```text
向量隔离有两类路线：
1. 共享索引 + payload filter，轻量但依赖过滤正确性；
2. 独立索引 / namespace，隔离强但资源成本更高。
```

V16 的工程选择：

```text
本地个人知识库第一版选择共享索引 + collection_id payload filter + audit。
原因是更轻、更容易测试、不会引入复杂部署。
```

# 4. 核心优化一：Vector Identity

## 4.1 问题

很多系统在写向量时直接使用随机 ID：

```text
insert(random_uuid, vector, payload)
```

这会导致：

```text
同一个 chunk 重复处理，每次都是新向量；
批量导入 retry 产生重复 vector；
reindex 失败后残留多份旧向量；
无法判断两个 vector 是否代表同一个业务 chunk。
```

## 4.2 优化动作

新增：

```text
VectorIdentityService
```

设计两个层次的身份。

### stableVectorKey

表示：

```text
同一个 collection、同一个 document、同一个 chunk、同一个 embedding 模型和维度。
```

建议公式：

```text
stable_vector_key = sha256(
  collection_id
  + document_id
  + chunk_uid
  + embedding_model
  + embedding_dimension
)
```

### vectorId

表示：

```text
某一个 generation 中的具体 vector。
```

建议公式：

```text
vector_id = sha256(
  stable_vector_key
  + generation
)
```

或等价：

```text
vector_id = sha256(
  collection_id
  + document_id
  + chunk_uid
  + embedding_model
  + embedding_dimension
  + generation
)
```

## 4.3 为什么 vector_id 要包含这些字段

| 字段 | 作用 |
|---|---|
| collection_id | 防止不同知识库分组互相覆盖 |
| document_id | 定位文档 |
| chunk_uid | 定位稳定 chunk |
| embedding_model | 防止不同模型向量混写 |
| embedding_dimension | 防止不同维度混写 |
| generation | 支持重新索引代际隔离 |

## 4.4 工程权衡

### 为什么不只用 chunk_id

因为：

```text
同一个 chunk 在不同 collection 中可能有不同语义边界；
同一个 chunk 使用不同 embedding 模型会产生不同向量；
同一个 chunk 在不同 generation 里可能需要并存一段时间。
```

### 为什么不直接用 document_id + chunk_index

因为 chunk_index 可能随着重新切分变化。

更稳定的方式是：

```text
chunk_uid = content_hash + section_path + normalized position
```

如果 chunk pipeline 已经有 chunk_uid，则直接复用。

### 为什么 generation 单独加入 vector_id

因为 reindex 时需要：

```text
generation=1 继续服务；
generation=2 构建中；
generation=2 校验通过后切换 active；
generation=1 retired 后清理。
```

# 5. 核心优化二：Vector Payload 标准化

## 5.1 问题

如果 payload 不完整，就算查询时想过滤也做不到。

例如 payload 缺少：

```text
collection_id
status
generation
embedding_model
dimension
```

系统就无法判断：

```text
这个 vector 属于哪个 collection；
是否来自旧 generation；
是否是 TRASHED 文档；
是否使用旧 embedding 模型；
是否维度不匹配。
```

## 5.2 优化动作

新增：

```text
VectorPayloadBuilder
```

标准 payload：

```text
vector_id
stable_vector_key
collection_id
document_id
chunk_id
chunk_uid
document_generation
chunk_generation
vector_generation
embedding_model
embedding_dimension
content_hash
metadata_hash
doc_type
version
source
status
section_path
chunk_type
created_at
updated_at
```

## 5.3 必填字段

```text
collection_id
document_id
chunk_uid
embedding_model
embedding_dimension
vector_generation
status
```

## 5.4 工程规则

```text
payload 构建逻辑集中在 VectorPayloadBuilder；
不能散落在 ingestion / reindex / batch retry 各处；
payload.collection_id 必须和 document.collection_id、chunk.collection_id 一致；
payload.embedding_dimension 必须等于真实 vector.length；
payload.status 必须和文档生命周期一致。
```

## 5.5 高频过滤字段

建议预留 payload index：

```text
collection_id

status

embedding_model

```

# 6. 核心优化三：Idempotent Upsert 幂等写入

## 6.1 问题

如果所有写入都是 insert：

```text
insert(random_id)
```

那么：

```text
单文件重试会重复写；
批量重试会重复写；
reindex 中断后再跑会重复写；
相同 chunk 会出现多份向量。
```

## 6.2 优化动作

新增：

```text
IdempotentVectorUpsertService
```

流程：

```text
1. 输入 collectionId / documentId / chunk / embeddingVector / embeddingModel / dimension / generation；
2. VectorIdentityService 生成 stableVectorKey 和 vectorId；
3. VectorPayloadBuilder 构造 payload；
4. VectorNamespaceGuard 校验隔离；
5. 调用 vectorStore.upsert；
6. 返回 VectorUpsertResult。
```

## 6.3 VectorUpsertResult

```text
vectorId
stableVectorKey
operation
collectionId
documentId
chunkId
generation
embeddingModel
embeddingDimension
status
warningMessages
```

operation：

```text
CREATED
UPDATED
SKIPPED
FAILED
```

## 6.4 预期行为

```text
同一 vectorId 重复 upsert -> UPDATED，不增加 vector count；
同一 stableVectorKey 同 generation -> 覆盖；
不同 generation -> 新 vectorId；
不同 embedding_model -> 新 stableVectorKey；
不同 embedding_dimension -> 新 stableVectorKey。
```

# 7. 核心优化四：Vector Namespace Guard

## 7.1 问题

向量污染经常不是检索阶段才发生，而是写入阶段就已经错了。

例如：

```text
document.collectionId = A
chunk.collectionId = A
payload.collectionId = B
request.collectionId = A
```

如果不拦截，collection B 会被污染。

## 7.2 优化动作

新增：

```text
VectorNamespaceGuard
```

校验规则：

```text
document.collectionId == chunk.collectionId == payload.collectionId == request.collectionId
chunk.documentId == payload.documentId == request.documentId
payload.embedding_dimension == vector.length
generation 必须合法
PURGED 不允许写 active vector
embedding_model 不得为空
```

失败时：

```text
阻止 upsert；
返回 VECTOR_NAMESPACE_VIOLATION；
记录 audit event。
```

错误码：

```text
VECTOR_IDENTITY_INVALID
VECTOR_PAYLOAD_INVALID
VECTOR_NAMESPACE_VIOLATION
VECTOR_DIMENSION_MISMATCH
VECTOR_COLLECTION_MISMATCH
VECTOR_DOCUMENT_MISMATCH
VECTOR_GENERATION_INVALID
VECTOR_UPSERT_FAILED
```

## 7.3 效果

```text
把污染拦在写入前；
避免后续靠 audit 才发现；
让 collection 隔离成为系统不变量。
```

# 8. 核心优化五：Vector Generation 代际隔离

## 8.1 问题

reindex 是污染高发场景：

```text
旧索引还在；
新索引写了一半；
中途失败；
新旧 chunks 混合；
旧 vectors 没清；
检索时同时搜到 generation 1 和 generation 2。
```

## 8.2 优化动作

新增表：

```text
vector_index_generation
```

字段：

```text
id
collection_id
document_id nullable
generation
status
embedding_model
embedding_dimension
chunking_strategy
created_at
activated_at
completed_at
retired_at
summary_message
```

status：

```text
BUILDING
ACTIVE
RETIRED
FAILED
```

## 8.3 Reindex 流程

```text
1. 创建 generation = current + 1，status=BUILDING；
2. 新 chunk / vector 写入 newGeneration；
3. 校验 chunk count / vector count；
4. 校验 collection_id；
5. 校验 embedding_model / dimension；
6. newGeneration -> ACTIVE；
7. oldGeneration -> RETIRED；
8. cleanup oldGeneration。
```

## 8.4 工程权衡

### 为什么不直接覆盖旧向量

因为直接覆盖会导致：

```text
构建失败时没有可用旧索引；
用户查询遇到半新半旧结果；
无法回滚。
```

### 为什么 BUILDING 不参与检索

因为 BUILDING 还没校验完成，可能缺失向量或 metadata 错误。

# 9. 核心优化六：Document Lifecycle 与 Vector 同步

## 9.1 状态语义

```text
ACTIVE：可检索，vector payload status=ACTIVE
TRASHED：默认不可检索，payload status=TRASHED
RESTORE：恢复为 ACTIVE，不需要重新 embedding
PURGED：永久删除，必须删除 vector
DEPRECATED：默认不参与 latest 查询
```

## 9.2 关键规则

```text
trash 后 vector 不应默认可检索；
restore 后 vector 可恢复为 ACTIVE；
purge 后必须 deleteByDocumentId；
TRASHED / PURGED 不得出现在 finalTopK、citation、prompt context；
cleanup failure 必须记录 warning；
purge 不能假装成功。
```

## 9.3 工程权衡

### TRASHED 是否直接删除 vector

V16 推荐：

```text
TRASHED 不立即删除 vector，只更新 status=TRASHED；
PURGED 才物理删除 vector。
```

原因：

```text
用户可能恢复；
恢复不应重新 embedding；
只要检索默认过滤 TRASHED，就不会污染结果。
```

# 10. 核心优化七：Batch Retry 防重复

## 10.1 问题

V13 批量导入有失败重试能力，如果 vector_id 不稳定，会产生重复向量：

```text
第 1 次处理：insert random_id_1
失败重试：insert random_id_2
再次重试：insert random_id_3
```

结果：

```text
一个 chunk 三条 vector。
```

## 10.2 优化动作

V16 规则：

```text
batch item retry 复用同一 documentId / chunkUid / generation；
retryCount 不参与 vector identity；
同一 chunk 已经写入时，再次 upsert 更新或跳过；
duplicate file skip 不写 vector；
USE_EXISTING 不写新 vector；
IMPORT_ANYWAY 创建新 documentId，因此允许生成新 vector。
```

## 10.3 验收标准

```text
同一 batch item retry 三次，vector count 不增加；
processing failed 后 retry 成功，只存在一份最终 vector；
duplicate file skip 不写 vector；
IMPORT_ANYWAY 新 document 写新 vector，且 collection_id 正确。
```

# 11. 核心优化八：Vector Cleanup

## 11.1 问题

向量库长期运行会产生：

```text
orphan vector；
retired generation；
purged residue；
wrong collection vector；
missing vector；
duplicate vector。
```

V16 需要提供有范围的安全清理。

## 11.2 VectorCleanupService

支持：

```text
cleanupDocumentVectors(collectionId, documentId)
cleanupCollectionVectors(collectionId)
cleanupGenerationVectors(collectionId, generation)
cleanupRetiredGenerations(collectionId)
cleanupPurgedDocumentResidue()
cleanupOrphanVectors()
```

返回：

```text
requestedCount
deletedCount
failedCount
skippedCount
warnings
errors
```

## 11.3 安全规则

```text
所有 cleanup 必须有 scope；
禁止无条件全库删除；
删除失败要记录；
单个失败不影响其他 cleanup；
cleanup 结果进入 audit report。
```

## 12.3 审计指标

### VectorDuplicateRate（重复向量率）

定义：

```text
同一 stableVectorKey 对应多个 ACTIVE vector 的比例。
```

### VectorOrphanRate（孤儿向量率）

定义：

```text
vector 存在，但 document / chunk 不存在。
```

### VectorMissingRate（缺失向量率）

定义：

```text
ACTIVE chunk 存在，但对应 vector 不存在。
```

### CrossCollectionVectorLeakRate（跨集合向量污染率）

定义：

```text
collection audit 范围内，payload.collection_id 不属于目标 collection 的向量比例。
```

### WrongGenerationVectorRate（错误代际向量率）

定义：

```text
检索可见范围中，vector_generation != active_generation 的比例。
```

### ModelDimensionMismatchRate（模型维度不一致率）

定义：

```text
payload.embedding_dimension 与实际 vector length 不一致，或 active index 混入不兼容维度。
```

### TrashedVectorVisibleRate（垃圾箱向量可见率）

定义：

```text
TRASHED 文档对应 vector 仍在默认检索可见范围的比例。
```

### PurgedVectorResidueRate（永久删除后向量残留率）

定义：

```text
PURGED document 仍残留 vector 的比例。
```

# 17. 工程权衡总结

## 17.1 共享索引 + payload filter vs 每 collection 一个 namespace

### 方案 A：共享索引 + payload filter

优点：

```text
实现轻；
本地部署简单；
collection 多时不需要建大量物理索引；
测试更容易；
符合 Qdrant payload-based multitenancy 的思路。
```

缺点：

```text
依赖 filter 正确性；
需要 audit 防污染；
极大 collection 下性能可能受影响。
```

### 方案 B：每 collection 一个 namespace / index

优点：

```text
隔离更强；
清理更直接；
跨 collection 污染概率低。
```

缺点：

```text
实现复杂；
资源成本高；
本地部署更重；
collection 多时管理复杂。
```

V16 选择：

```text
第一版选择共享索引 + collection_id payload filter + deterministic vector_id + audit。
```

后续演进：

```text
大 collection 可升级为独立 namespace。
```

## 17.2 为什么不强依赖 Qdrant

因为项目默认测试必须低依赖：

```text
不能要求开发者安装 Qdrant 才能跑单元测试；
mock / in-memory vector store 必须能覆盖核心逻辑；
真实 Qdrant 作为 local-ai / manual verification。
```

## 17.3 为什么 TRASHED 不立即删除 vector

原因：

```text
用户可能恢复；
恢复不应重新 embedding；
只要检索默认过滤 TRASHED，就不会污染结果。
```

PURGED 才物理删除。

## 17.4 为什么 audit 不自动 cleanup

因为 audit 是诊断，cleanup 是破坏性操作。

必须分开：

```text
audit：只读，不改数据；
cleanup：用户确认后执行。
```

这符合安全原则。

## 17.5 为什么 generation 是必要的

如果没有 generation，reindex 会出现：

```text
旧向量和新向量混合；
构建失败后无法回滚；
清理旧索引没有范围；
无法判断当前检索应该使用哪一代向量。
```

generation 让 reindex 变成可控过程。

# 19. 面试高强度追问与回答

## 19.1 V16 和之前的 collection scoped retrieval 有什么区别？

回答：

```text
之前的 collection scoped retrieval 主要解决查询阶段过滤，也就是 search 时带 collection_id。V16 解决的是向量索引存储层的问题，包括写入时 collection_id 是否正确、vector_id 是否稳定、重复导入是否产生重复向量、reindex 后旧 generation 是否残留、purge 后向量是否清理，以及 collection 级污染审计。
```

## 19.2 为什么 vector_id 不能随机生成？

回答：

```text
随机 vector_id 会导致同一个 chunk 每次处理都生成新向量，批量 retry、reindex、重复上传都会产生重复数据。稳定 vector_id 可以让写入变成 upsert，同一 chunk 同一 generation 重复写入时覆盖旧值，而不是新增脏向量。
```

## 19.3 为什么 vector_id 要包含 collection_id？

回答：

```text
因为同一个文档或 chunk 可能出现在不同知识库分组中，如果 vector_id 不包含 collection_id，不同 collection 的向量可能互相覆盖或无法隔离。collection_id 放入 vector identity 后，每个 collection 的向量身份天然隔离。
```

## 19.4 为什么 embedding_model 和 embedding_dimension 也要进入 identity？

回答：

```text
不同 embedding 模型生成的向量空间不同，即使文本相同，也不能混用。dimension 也必须匹配实际向量长度，否则检索和索引都会出问题。因此 embedding_model 和 embedding_dimension 必须进入 stableVectorKey。
```

## 19.5 为什么需要 generation？

回答：

```text
generation 用于重新索引的代际隔离。reindex 时新 generation 在 BUILDING 状态构建，旧 ACTIVE generation 继续服务。新 generation 校验通过后再切换 ACTIVE，旧 generation 变 RETIRED。这样可以避免重建失败导致索引半新半旧，也方便清理旧向量。
```

## 19.6 TRASHED 文档的向量为什么不直接删除？

回答：

```text
因为垃圾箱语义允许恢复。如果 TRASHED 时直接删除向量，restore 后需要重新 embedding，成本高且慢。V16 选择 TRASHED 时更新 payload status 并默认过滤，PURGED 时才物理删除 vector。
```

## 19.7 如何防止 batch retry 重复写 vector？

回答：

```text
batch retry 复用同一 documentId、chunkUid、embeddingModel、dimension 和 generation，因此生成相同 vectorId。再次写入时调用 upsert，更新或跳过已有 vector，而不是 insert 新 ID。retryCount 不参与 vector identity。
```

## 19.8 audit 和 cleanup 为什么分开？

回答：

```text
audit 是只读诊断，负责发现重复、孤儿、缺失、跨集合污染和残留；cleanup 是破坏性操作，需要明确 scope 和用户确认。分开可以避免系统在诊断过程中误删数据。
```

## 19.9 CrossCollectionLeakRate 和 CrossCollectionVectorLeakRate 有什么区别？

回答：

```text
CrossCollectionLeakRate 是 V14 的检索结果层指标，衡量 TopK 或 final context 中有没有其他 collection 的 chunk。CrossCollectionVectorLeakRate 是 V16 的向量存储层指标，衡量某个 collection 的向量范围中是否混入其他 collection 的 vector。前者看结果，后者看底层数据。
```

## 19.10 为什么不每个 collection 一个独立向量库？

回答：

```text
每个 collection 一个独立 namespace 或 index 隔离更强，但本地个人知识库第一版会增加部署和管理复杂度。V16 先采用共享索引 + collection_id payload filter + deterministic vector_id + audit 的方式，保持轻量。后续当某个 collection 特别大时，可以升级为独立 namespace。
```

## 19.11 如果底层向量库不支持 deleteByFilter 怎么办？

回答：

```text
VectorStore 抽象会显式标记 unsupported，不能假装成功。mock / in-memory 必须支持测试。真实向量库如果暂时不支持某些能力，可以通过 scan + deleteByVectorId 降级实现，或者返回 warning 并在 manual 中说明限制。
```

## 19.12 如何证明 V16 有效果？

回答：

```text
可以通过三类结果证明：第一，同一 batch item retry 多次后 vector count 不增加；第二，purge 后 audit 不再发现 purged residue；第三，collection pollution audit 的 CrossCollectionVectorLeakRate 下降到 0，并且 V14 检索层 CrossCollectionLeakRate 同步下降。
```

# 21. 最终总结

V16.0 可以这样总结：

```text
V16.0 的核心目标是防止向量库底层污染。虽然系统之前已经有 collection scoped retrieval，但那主要解决查询阶段过滤。真实万级文档场景下，污染更可能发生在向量写入、批量 retry、reindex、trash/purge 和模型切换过程中。

因此我设计了稳定 vector identity，通过 collection_id、document_id、chunk_uid、embedding_model、embedding_dimension 和 generation 生成 vector_id，并将写入方式改为幂等 upsert。同时，我标准化了 vector payload，确保每条向量都带有 collection、document、chunk、generation、模型、维度和状态信息。

为了防止重建索引造成新旧向量混召回，我引入 vector generation 代际隔离；为了防止删除和垃圾箱状态不同步，我让 trash、restore、purge 与 vector status / cleanup 联动；为了发现长期运行后的脏数据，我实现了 vector consistency audit 和 collection pollution audit，能够检测重复向量、孤儿向量、缺失向量、跨集合污染、错误 generation 和永久删除残留。

最终，V16.0 让 RAG 系统不仅能在检索阶段过滤 collection，还能从向量写入和维护阶段保证单个知识库分组不被污染。
```
