# V1.2 面试文档：LLM usage metadata

## 1. 本版本解决的问题

V1.2 解决 LLM 调用只有结果、没有调用成本和性能元数据的问题。AI 平台必须知道 provider、token 数、latency，才能做成本统计、模型路由、SLA 诊断和用户分层。

## 2. 为什么现在做这个版本

它依赖 V1.0 LlmClient 和 V1.0.4 结果持久化。只有调用链路稳定后，才有必要记录 usage。它为 V1.3 Model Router、后续真实 Provider、VIP 策略和成本分析做准备。此时不接真实 tokenizer，是因为仍使用 Mock Provider。

## 3. 核心需求边界

本版本让 MockLlmClient 返回 provider、promptTokenCount、completionTokenCount、totalTokenCount、latencyMs，并保存到 task 表。不做真实 tokenizer、不做真实计费、不做成本 dashboard、不做多 attempt 聚合。

## 4. 方案选择

MockLlmClient 使用简单长度估算 token，provider 固定 mock，model 使用请求模型，latency 用本地耗时。TaskService.saveLlmMetadata 保存元数据，不改变任务状态，不写 task_event。

## 5. 替代方案与暂时不选的原因

替代方案一是不记录 usage。当前不选，因为后续模型路由和成本分析缺少依据。

替代方案二是接真实 tokenizer。当前不选，因为 Mock 阶段目标是工程链路，不是精确计费。

## 6. 数据库设计

task 表新增 llm_provider、prompt_token_count、completion_token_count、total_token_count、llm_latency_ms。字段保存本次最终调用摘要。后续 task_attempt 可保存每次 retry 的详细 metadata。

## 7. API 设计

GET /tasks/{taskId} 返回 llmProvider、promptTokenCount、completionTokenCount、totalTokenCount、llmLatencyMs。创建接口不变。

## 8. 核心流程

TaskExecutionService 调用 LlmClient
-> LlmResponse 返回 usage
-> saveLlmMetadata 保存 provider/model/token/latency
-> 根据 success 进入 SUCCESS 或 retry/FAILED
-> 查询接口返回 usage

## 9. 测试方案

建议覆盖成功和失败响应都返回 usage、token 字段不为空、latency 不为空、空 prompt 失败也有 provider。API 验收可创建 normal task 后查询 usage 字段。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是 usage 保存不能改变任务状态。元数据是 LLM 调用事实，状态变化仍由 markTaskSucceeded/markTaskRetryPending 控制，职责要分离。

## 12. 本版本亮点

- 引入 LLM 调用元数据。
- 成功和失败都记录 usage baseline。
- 为 Model Router 和成本统计准备数据。
- 明确 Mock token 是估算，不冒充真实计费。

## 13. 简历 bullet

### 简洁版

实现 LLM usage metadata 记录，支持 provider、token count 和 latency 字段在任务详情中返回。

### 强化版

为 Mock LLM 执行链路补齐 usage metadata，持久化 provider、prompt/completion token、total token 和 latency，为后续模型路由、成本分析和可观测性建设提供基础数据。

## 14. 面试高强度追问与回答

Q1：为什么要记录 token？

A1：token 决定模型调用成本和上下文预算，是 AI 平台资源治理的核心指标。

Q2：为什么要记录 latency？

A2：latency 影响用户体验和 SLA，也能用于模型路由和异常诊断。

Q3：Mock token 准确吗？

A3：不准确，它只是工程链路估算。真实 Provider 接入后应使用 provider 返回或 tokenizer 计算。

Q4：为什么失败也记录 usage？

A4：失败调用也可能消耗资源和时间，排查和成本分析需要记录。

Q5：saveLlmMetadata 为什么不写事件？

A5：它不是生命周期变化，只是调用元数据更新，保持 task_event 聚焦状态变化。

Q6：task 表保存 usage 是否足够？

A6：早期足够保存最终摘要。多次 retry 的完整历史需要 task_attempt。

Q7：如何支持用户分层？

A7：可以基于 token、latency、model 统计不同用户的用量，后续结合 quota 和 VIP 策略。

Q8：当前局限是什么？

A8：没有真实 tokenizer、成本单价和 dashboard，也没有多 Provider 对比。

## 15. 后续演进路线

V1.2 支撑 V1.3 Model Router、V2.2.3 task_attempt、真实 Provider usage 和后续成本统计、VIP quota。
