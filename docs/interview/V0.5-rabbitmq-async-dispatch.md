# V0.5 面试文档：RabbitMQ 异步任务投递

## 1. 本版本解决的问题

V0.5 解决 HTTP 请求和长耗时任务执行耦合的问题。POST /tasks 负责创建任务，RabbitMQ 负责把 taskId 投递给后续 Consumer。这样用户请求可以快速返回，任务执行可以异步进行，为 AI/LLM 长任务提供调度基础。

## 2. 为什么现在做这个版本

它依赖任务创建、状态查询、状态机和基础数据库迁移。没有这些能力，MQ 消息即使发出去也缺少可追踪任务。它为 V0.6 Consumer 执行、V0.8 重试、V0.9 幂等和后续 Outbox 打基础。此时不做复杂 Worker 集群，是因为先跑通单队列异步链路更重要。

## 3. 核心需求边界

本版本只做任务创建后的 RabbitMQ 投递，不做复杂 ack/nack 策略、不做死信队列、不做延迟队列、不做 Outbox。保持克制是为了先验证异步调度形态。

## 4. 方案选择

采用 RabbitMQ 作为任务调度消息中间件。消息体只需要包含 taskId，Consumer 收到后再查询数据库获取完整任务。这样消息保持轻量，避免 prompt 或大字段在 MQ 中重复传输。

## 5. 替代方案与暂时不选的原因

替代方案一是同步执行。当前不选，因为长任务会阻塞 HTTP 请求。

替代方案二是本地线程池。当前不选，因为线程池和应用进程强绑定，不利于后续多实例 Worker、重试和消息可观测。

## 6. 数据库设计

本版本未新增数据库结构，主要复用 task 表和 task_event 表。后续 V2.2.x 会通过 task_outbox 表解决 DB 与 MQ 的 dual write problem。

## 7. API 设计

POST /tasks 的外部契约保持不变：创建任务并返回 taskId/status。内部新增 MQ 投递动作，但客户端不需要知道 RabbitMQ 细节。

## 8. 核心流程

用户 POST /tasks
-> TaskService 保存 task
-> 写 TASK_CREATED
-> TaskDispatchProducer 发送 taskId 到 RabbitMQ
-> 返回 PENDING
-> 后续 Consumer 消费

## 9. 测试方案

建议覆盖 producer 是否发送正确 taskId、RabbitMQ 配置是否创建 exchange/queue/binding、创建任务后队列中可消费消息。异常路径包括 RabbitMQ 不可用、发送失败、重复发送。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是 MQ 引入后系统从单事务变成跨资源协作。早期可以直接发送 MQ，但后续必须面对 DB 成功 MQ 失败、MQ 成功 DB 回滚等一致性问题。

## 12. 本版本亮点

- 解耦任务创建和执行。
- 为长耗时 AI 任务建立异步调度通道。
- 消息体轻量，只传 taskId。
- 为后续 Consumer、重试、幂等和 Outbox 铺路。

## 13. 简历 bullet

### 简洁版

引入 RabbitMQ 实现任务异步投递，将任务创建与执行解耦，提升长耗时 AI 任务的可扩展性。

### 强化版

基于 RabbitMQ 设计 AI 任务异步调度链路，通过轻量 taskId 消息连接 HTTP 创建端和后续 Worker 执行端，为重试、幂等消费和可靠派发奠定基础。

## 14. 面试高强度追问与回答

Q1：为什么不用同步执行？

A1：AI 任务可能耗时较长，同步执行容易超时，也限制并发能力。MQ 让请求快速返回，执行异步进行。

Q2：为什么消息里只放 taskId？

A2：taskId 足以让 Consumer 查询完整任务。这样消息小、稳定，也避免 prompt 等大字段在 MQ 中扩散。

Q3：RabbitMQ 的价值是什么？

A3：它提供队列缓冲、生产者消费者解耦和后续多 Worker 扩展能力。任务高峰时可以先排队再消费。

Q4：MQ 发送失败怎么办？

A4：早期版本可能会导致任务停留 PENDING。后续通过 Transactional Outbox 解决 DB 与 MQ 的一致性问题。

Q5：为什么不用 Kafka？

A5：当前是任务调度场景，RabbitMQ 更轻量直接。Kafka 更适合高吞吐日志流，不是早期必要复杂度。

Q6：MQ 会不会重复投递？

A6：会，所以后续必须做 Consumer 幂等。V0.5 先接入 MQ，V0.9 再补重复消费控制。

Q7：队列堆积怎么办？

A7：可以增加 Consumer 实例、限流、监控队列长度和拆分优先级队列。当前版本先验证基本链路。

Q8：当前局限是什么？

A8：直接发送 MQ 还存在 dual write problem，没有 Outbox 和生产级失败补偿。

## 15. 后续演进路线

V0.5 支撑 V0.6 Consumer 模拟执行、V0.8 retry 重新投递、V0.9 幂等消费和 V2.2.x Transactional Outbox。
