# V15.0 Structured Chunking & Hybrid Retrieval Pipeline 面试文档

## 0. 阶段定位

阶段名称：

```text
V15.0 Structured Chunking & Hybrid Retrieval Pipeline
```

中文名称：

```text
结构化切分与混合检索主链路优化
```

V15.0 的核心定位是：

```text
V14.0 负责发现知识库检索质量问题；
V15.0 负责把这些问题修复到 RAG 检索主链路里。
```

V14.0 的知识库体检报告可以诊断：Recall@K（召回率）低、MRR@K（平均倒数排名）低、ContextPrecision@K（上下文精确率）低、CrossCollectionLeakRate（跨集合污染率）高、WrongVersionLeakRate（错误版本污染率）高、CitationAccuracy（引用准确率）低、Faithfulness（忠实性）低、metadata（元数据）缺失、chunk（文本片段）切分质量不稳定。

V15.0 的目标是将这些诊断建议落到工程实现中：结构化 chunk、section_path（章节路径）、parent / adjacent chunk（父级 / 相邻片段）、metadata 增强、collection / version / doc_type / source / status 预过滤、BM25 / keyword 检索基础版、Vector + BM25 hybrid search（混合检索）、RRF（Reciprocal Rank Fusion，倒数排名融合）排序、Reranker（重排序器）接口抽象、parent context（父级上下文）回填、重新索引、Retrieval Diagnostics（检索诊断）增强，以及与 V14 知识库体检报告联动。

一句话面试表达：

```text
V15.0 不是再做一个 RAG Demo，而是把 RAG 主链路从“普通 chunk + 向量检索”升级为“结构化 chunk + 元数据过滤 + Hybrid Search + RRF 融合 + Rerank 抽象 + 上下文回填”的可诊断检索系统，用来解决万级中文文档下向量混杂、跨集合污染、版本混召回和专有名词召回不稳定的问题。
```

---

## 1. 为什么需要 V15.0

普通 RAG 系统常见链路是：

```text
文档解析 -> 固定长度 chunk -> embedding -> 向量库 -> vector topK -> LLM
```

这个链路在小规模文档下能跑通，但在万级中文文档场景下会暴露问题：不同项目语义相似，向量检索容易跨项目召回；不同版本文档内容相近，容易新旧版本混召回；技术文档中类名、接口名、配置项、文件路径只靠向量检索不稳定；固定长度切分可能切断标题、段落、表格和代码块；chunk 缺少 section_path，单独看上下文不完整；TopK 中混入无关 chunk，LLM 被噪声带偏；用户也不知道当前回答到底用了哪些过滤条件、召回源和排序策略。

V15.0 的意义是把 V14 诊断出来的问题转化为检索链路层面的修复能力。

## 3. 核心设计一：Chunk Schema 升级

普通 chunk 只有 document_id、chunk_text、embedding，不足以支持万级文档下的过滤、诊断、回填和排序。因此 V15.0 将 chunk 升级为结构化 chunk。

建议字段：

```text
id
document_id
chunk_index
chunk_uid
parent_chunk_id
previous_chunk_id
next_chunk_id
section_path
section_title
heading_level
chunk_type
content_hash
normalized_content_hash
token_count
char_count
language
doc_type
collection_id
version
source
tags_json
status
created_at
updated_at
```

chunk_type 枚举：

```text
TEXT              普通文本
TITLE             标题
SECTION_SUMMARY   章节摘要
TABLE             表格
CODE_BLOCK        代码块
LIST              列表
QUOTE             引用
METADATA          元数据
UNKNOWN           未知
```

结构化 chunk 的价值：支持 section_path 检索加权、title / heading boost、parent chunk 回填、previous / next chunk 回填、table / code block 保护、metadata pre-filter、CrossCollectionLeakRate / WrongVersionLeakRate 计算和更准确的 citation。

## 4. 核心设计二：结构化切分策略

ChunkingStrategy：

```text
FIXED_SIZE                     固定长度切分
PARAGRAPH                      段落切分
STRUCTURE_AWARE                结构感知切分
STRUCTURE_AWARE_WITH_OVERLAP   结构感知 + overlap
```

默认配置建议：

```properties
app.chunking.strategy=STRUCTURE_AWARE_WITH_OVERLAP
app.chunking.max-chars=1200
app.chunking.min-chars=200
app.chunking.overlap-chars=120
app.chunking.preserve-code-block=true
app.chunking.preserve-table-block=true
app.chunking.include-section-path=true
```

切分规则：优先按标题层级切分；标题路径写入 section_path；段落不应被随意从中间截断；代码块尽量整体保留；表格尽量整体保留；chunk 太短时可与相邻段落合并；chunk 太长时按段落二次切分；每个 chunk 都带 document title 和 section_path metadata；previous_chunk_id / next_chunk_id 尽量建立；parent_chunk_id 可指向章节级 parent chunk 或 section summary。

工程权衡：V15.0 不做复杂 OCR、复杂表格结构还原、代码 AST 解析。V15.0 的主线是检索质量提升，第一版优先支持 Markdown / HTML / TXT / 已有解析文本；PDF 如果只能解析纯文本，就先生成基础 section_path。

## 5. 核心设计三：Metadata 增强

万级文档混杂的根因之一是“语义相似但业务域不同”。不同项目、不同版本、不同文档类型可能讨论相似概念，向量空间中很接近。解决方式不是只调 embedding，而是必须使用 metadata 缩小检索域。

必须进入 chunk 的 metadata：

```text
collection_id
document_id
doc_type
version
source
status
tags
section_path
created_at
updated_at
```

doc_type：

```text
INTERVIEW_DOC
MANUAL
README
API_DOC
DESIGN_DOC
REQUIREMENT_DOC
MEETING_NOTE
CODE_DOC
PAPER
OTHER
```

status：

```text
ACTIVE
DRAFT
DEPRECATED
TRASHED
PURGED
```

默认检索规则：TRASHED / PURGED 永远不能进入检索；DEPRECATED 默认不进入“最新方案”查询；用户指定版本时，优先 version filter；用户选择 collection 时，必须 collection filter；metadata 不能只在 UI 展示，必须进入 retrieval pipeline。

## 6. 核心设计四：Metadata Pre-filter

RetrievalFilter 对象：

```text
collectionId
docType
version
source
status
tags
createdAfter
createdBefore
includeDeprecated
includeDraft
includeTrashed
```

默认规则：includeTrashed=false；includeDeprecated=false；includeDraft=false；如果用户选择 collection，则必须 collectionId filter；如果 query metadata 中解析出 version，则必须 version filter。

filter 必须应用到：Vector retrieval、BM25 / keyword retrieval、Hybrid retrieval、Rerank candidate、finalTopK、citations、prompt context。

工程权衡：如果底层向量库支持 payload filter，应优先使用底层过滤。如果暂时不支持，则应用层二次过滤，但必须在 diagnostics 中标记 `filterMode=APPLICATION_SIDE`。应用层过滤可以保证正确性，但效率不如底层过滤，后续应迁移到 vector store payload filter。

## 7. 核心设计五：BM25 / Keyword Retrieval 基础版

向量检索适合语义问题，但不稳定处理类名、接口名、配置项、文件路径、版本号、内部代号、方法名、错误码。例如：

```text
LocalPythonLlmProvider 是做什么的？
application-local-ai.properties 里配置了什么？
V10.0 支持哪些 provider_type？
```

这些问题需要关键词检索。

KeywordRetriever 抽象：

```text
KeywordRetriever.search(query, filter, topK)
```

第一版允许数据库 LIKE、简单 inverted index、in-memory mock BM25、简单关键词打分。不强制引入 Elasticsearch、Lucene、Tantivy。原因是当前项目是本地个人知识库，V15 的目标是补齐抽象和主链路，不要过早引入重型搜索服务。

Keyword scoring 第一版可以做：exact match 加分；title / section_path 命中加权更高；chunk text 命中加权较低；多关键词命中加分；版本号命中加分；配置项 / 类名命中加分。

## 8. 核心设计六：Hybrid Retrieval 主链路

HybridRetrievalService 输入：

```text
query
filter
vectorTopK
keywordTopK
finalTopK
fusionStrategy
```

fusionStrategy：

```text
RRF
WEIGHTED_SCORE
VECTOR_ONLY
KEYWORD_ONLY
```

默认：RRF。

主流程：

```text
1. Vector retriever 召回 vectorTopK；
2. Keyword retriever 召回 keywordTopK；
3. 合并候选；
4. 使用 RRF 排序；
5. 去重；
6. 可选 rerank；
7. parent / adjacent context 回填；
8. finalTopK；
9. 输出 RetrievalDiagnostics。
```

关键约束：同一个 chunk 被 vector 和 keyword 同时召回时合并分数；finalTopK 不允许包含 TRASHED / PURGED；finalTopK 不允许违反 collection / version filter；Rerank 不能把被 filter 掉的 chunk 加回来；Parent context 回填不能突破 filter。

## 9. 核心设计七：RRF 融合

BM25 分数和 vector similarity 分数不是同一尺度。直接加权会遇到分数范围不同、不同 query 分数分布不同、归一化不稳定等问题。RRF 只依赖 rank，不直接依赖原始分数。

公式：

```text
score = sum(1 / (k + rank_i))
```

默认：

```text
k = 60
```

工程价值：实现简单；稳定；适合第一版 hybrid fusion；不强依赖底层搜索引擎分数尺度；能融合 vector rank 和 keyword rank。

## 10. 核心设计八：Reranker 接口抽象

Reranker 接口：

```text
rerank(query, candidates, topN)
```

V15 默认实现：

```text
NoopReranker
SimpleHeuristicReranker
```

SimpleHeuristicReranker 依据：query keyword overlap；title / section_path match；metadata match；exact version match；exact config key / class name match；original retrieval score。

输出字段：

```text
rerankScore
rerankReason
rankBefore
rankAfter
```

工程权衡：V15 不接真实 cross-encoder。真实 reranker 会引入额外模型部署、推理延迟和测试依赖。当前阶段先把抽象、开关、诊断和测试跑通。后续可接 bge-reranker、cross-encoder、LLM reranker、semantic ranker。

## 11. 核心设计九：Parent / Adjacent Chunk 回填

检索命中的 chunk 可能太碎，只召回一个段落但缺少标题、前置条件或相邻解释。所以需要上下文扩展。

ContextExpansionStrategy：

```text
NONE
ADJACENT
PARENT
PARENT_AND_ADJACENT
```

默认：ADJACENT。

配置：

```properties
app.retrieval.context-expansion=ADJACENT
app.retrieval.max-expanded-chunks=2
app.retrieval.max-context-chars=6000
```

约束：回填不能突破 collection filter；不能突破 version filter；不能包含 TRASHED / PURGED；回填内容标记 `retrievalSource=PARENT_CONTEXT`；回填内容不参与原始 Recall@K 造假；评测时区分 retrievedTopK、expandedChunks、finalContextChunks。

vectorHitCount
keywordHitCount
overlapCount

## 13. 核心设计十一：Reindex 能力

升级 chunk schema 和 metadata 后，旧索引不具备 section_path、chunk_type、version、doc_type、previous / next chunk、keyword index、updated vector payload。因此必须支持重新索引。

API：

```http
POST /documents/{documentId}/reindex
POST /collections/{collectionId}/reindex
POST /retrieval/reindex
```

请求参数：

```text
chunkingStrategy
includeDeprecated
forceRechunk
forceReembed
forceKeywordIndex
forceMetadataRefresh
```

行为：重新解析 / 重新切分；刷新 chunk metadata；重新生成 embedding；更新 keyword index；更新 vector payload；记录 reindex event；不处理 TRASHED / PURGED；失败时不破坏旧索引。

工程上应尽量采用 generation / atomic finalization：新索引构建完成后再切换，失败时保留旧索引继续服务。

## 14. 与 V14 Knowledge Health 的联动

V14 诊断问题，V15 修复问题：collection filter 降低 CrossCollectionLeakRate；version filter 降低 WrongVersionLeakRate；rerank / metadata filter 提升 ContextPrecision；RRF / rerank 提升 MRR；hybrid search 提升 Recall。

V14 run compare 应能比较：

```text
Vector Only
vs
Hybrid + RRF
vs
Hybrid + Rerank
vs
Hybrid + Rerank + Context Expansion
```

典型结果：

```text
Recall@10 +12%
MRR@10 +18%
CrossCollectionLeakRate -9%
WrongVersionLeakRate -15%
Latency +320ms
```

诊断：检索质量提升明显，但延迟上升，需要根据场景选择是否启用 rerank。

## 17. 工程权衡总结

### 17.1 为什么先做结构化 chunk，而不是直接换 embedding 模型

换 embedding 可能提升召回，但不能解决 collection 混杂、version 混杂、doc_type 缺失、section_path 缺失、代码块被切断、旧文档参与检索等问题。结构化 chunk 和 metadata 是更底层的治理能力。

### 17.2 为什么不引入 Elasticsearch

Elasticsearch 能做 BM25，但当前项目是本地个人知识库，直接引入会增加安装成本、部署复杂度、测试依赖和本地运行门槛。V15 先抽象 KeywordRetriever，用简单实现跑通链路，后续可以替换为 Lucene、Elasticsearch 或 Tantivy。

### 17.3 为什么 parent chunk 回填不能算入原始 Recall

因为它不是检索命中的结果，而是上下文扩展。如果把回填 chunk 算入 Recall，会导致指标虚高。所以必须区分 retrievedTopK、expandedChunks、finalContextChunks。

### 17.4 为什么 filter 必须贯穿全链路

如果只在 vector retrieval 前 filter，但 keyword retrieval、rerank、context expansion 没有 filter，错误 collection 或错误版本仍然会回到 final context。所以 filter 要贯穿 vector、keyword、hybrid merge、rerank、context expansion、finalTopK、citations、prompt。

## 18. 面试高强度追问与回答

### 18.1 V15 和 V14 的关系是什么？

回答：V14 是知识库体检报告，负责发现问题，比如 Recall 低、ContextPrecision 低、跨集合污染高、错误版本污染高。V15 是检索主链路优化，负责解决这些问题，通过结构化 chunk、metadata pre-filter、Hybrid Search、RRF、rerank 和上下文回填降低污染并提升召回排序质量。

### 18.2 为什么固定长度 chunk 不适合万级文档？

回答：固定长度 chunk 会打断标题、段落、代码块和表格，也丢失章节路径。万级文档下，很多 chunk 单独看语义不完整，向量检索容易把语义相似但上下文不同的内容召回。结构化切分可以保留 section_path、chunk_type、parent/adjacent 关系，让检索和引用更稳定。

### 18.3 为什么 metadata 比单纯 embedding 更重要？

回答：embedding 只能表达语义相似，不能表达业务边界。不同项目、不同版本、不同文档类型可能语义相似，但不能混召回。collection_id、version、doc_type、status 这类 metadata 可以在检索前缩小搜索域，直接降低跨集合污染和错误版本污染。

### 18.4 为什么要做 Hybrid Search？

回答：向量检索擅长语义问题，但对类名、接口名、配置项、版本号、文件路径等精确符号不稳定。BM25 / keyword 检索正好擅长这些。Hybrid Search 同时使用 vector 和 keyword，可以兼顾语义召回和精确词召回。

### 18.5 为什么用 RRF 融合，而不是直接加权分数？

回答：vector similarity 和 BM25 score 不在同一尺度，直接加权需要复杂归一化，容易不稳定。RRF 只基于 rank，用 1/(k+rank) 融合多个排序结果，实现简单且稳定，适合第一版 hybrid fusion。

### 18.6 为什么不直接引入 Elasticsearch？

回答：Elasticsearch 可以提供成熟 BM25，但当前项目是本地个人知识库，目标是低门槛运行。直接引入 Elasticsearch 会增加部署复杂度和测试依赖。V15 先抽象 KeywordRetriever，用简单实现跑通链路，后续可以替换为 Lucene、Elasticsearch 或 Tantivy。

### 18.7 为什么 Reranker 先做接口，不接真实模型？

回答：真实 reranker 会增加模型部署、推理延迟和测试依赖。V15 的目标是让检索链路具备 rerank 插槽，并能在诊断中看到 rankBefore / rankAfter。第一版用 NoopReranker 和 SimpleHeuristicReranker，后续可以接 bge-reranker 或 cross-encoder。

### 18.8 Parent / Adjacent Chunk 回填解决什么问题？

回答：它解决 chunk 太碎导致上下文不完整的问题。命中的 chunk 可能只有一个段落，但回答需要标题、前置条件或相邻解释。回填 previous/next 或 parent chunk 可以提升 final context 完整度，但不能突破 metadata filter，也不能算作原始 Recall。

### 18.9 如何保证单个 collection 不被其他 collection 污染？

回答：首先在 chunk metadata 中写入 collection_id；其次 RetrievalFilter 必须在 vector、keyword、hybrid、rerank、context expansion、finalTopK、citations、prompt 全链路生效；最后通过 V14 的 CrossCollectionLeakRate 持续验证指定 collection 查询时是否仍有其他 collection chunk 进入 TopK 或 final context。

### 18.10 如何处理旧版本文档污染？

回答：文档和 chunk 必须写入 version 和 status。默认查询 latest 时过滤 DEPRECATED；用户指定版本时启用 version filter；ContextExpansion 也不能回填其他版本 chunk。V14 的 WrongVersionLeakRate 可以验证优化效果。

### 18.11 重新索引失败怎么办？

回答：不能破坏旧索引。应该采用 generation 或 atomic finalization 思路：先构建新一代 chunks、embedding、keyword index 和 vector payload，全部成功后再切换 active generation。失败时记录事件，保留旧索引继续服务。

### 18.12 V15 如何证明优化有效？

回答：通过 V14 Knowledge Health run compare。比如用同一套 Gold Test Set 分别跑 Vector Only 和 Hybrid + RRF + Filter，对比 Recall@10、MRR@10、ContextPrecision、CrossCollectionLeakRate、WrongVersionLeakRate 和 latency。如果污染率下降、MRR 提升，就能量化证明优化有效。

### 18.13 为什么不把 context expansion 当成检索结果？

回答：context expansion 是上下文组装阶段，不是召回阶段。如果把回填 chunk 算进 Recall，会导致指标虚高。因此评测时要分开记录 retrievedTopK、expandedChunks 和 finalContextChunks。

### 18.14 V15 对用户有什么直接价值？

回答：用户不需要理解底层 RRF 或 reranker。他能看到的是：同一个知识库在启用结构化切分、metadata filter 和 hybrid search 后，跨集合污染下降、错误版本召回下降、引用更准确、回答更稳定。系统还能告诉他当前检索用了哪些过滤条件和为什么选中这些 chunk。

### 18.15 V15 最大风险是什么？

回答：最大风险是链路复杂度上升。为控制风险，V15 采用抽象接口和安全降级：KeywordRetriever 先简单实现，Reranker 先 mock/simple，Parent Context 缺失时记录 warning，filter 即使底层不支持也应用层二次过滤，默认测试不依赖外部服务。

## 20. 总结表达

V15.0 可以这样总结：

```text
在 V15.0 中，我将 RAG 检索主链路从普通 chunk + 向量检索升级为结构化 chunk + metadata pre-filter + hybrid retrieval + RRF fusion + reranker abstraction + context expansion 的完整链路。

这一阶段解决的是万级中文文档下最核心的问题：不同项目、不同版本、不同文档类型的 chunk 容易因为语义相似而混召回。为了解决这个问题，我在 chunk 层增加 section_path、chunk_type、parent/adjacent chunk、collection_id、version、doc_type、status 等字段，并确保这些 metadata 贯穿 vector、keyword、hybrid、rerank、context expansion 和 final context 全链路。

同时，我实现了 KeywordRetriever 抽象、HybridRetrievalService、RRF 融合、SimpleHeuristicReranker 和 RetrievalDiagnostics，让系统不仅能提升召回和排序，还能解释为什么选中了某些 chunk，是否发生跨集合污染或错误版本污染。V15 与 V14 的知识库体检报告联动，可以量化证明结构化切分和混合检索是否真的降低污染、提升 MRR 和 ContextPrecision。
```
