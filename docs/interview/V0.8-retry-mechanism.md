# V0.8 面试文档：自动重试

## 1. 本版本解决的问题

V0.8 解决临时失败不应立即终结任务的问题。AI 任务可能因网络抖动、模型暂时不可用或下游短暂错误失败，系统需要进入 RETRY_PENDING，延迟后重新投递执行，而不是第一次失败就 FAILED。

## 2. 为什么现在做这个版本

它依赖 V0.7 的失败处理和状态机。只有能记录失败原因，才能进一步判断是否等待重试。它为后续 Scheduler、幂等消费、Outbox 和 task_attempt 做准备。此时不做指数退避和死信队列，是为了先验证最小重试闭环。

## 3. 核心需求边界

本版本新增 RETRY_PENDING、retry_count、max_retry、next_retry_at 和 TaskRetryScheduler。不做 MQ 延迟队列、不做死信队列、不做复杂退避策略、不做分布式锁。保持克制是为了让重试语义先清晰。

## 4. 方案选择

失败时先调用 markTaskRetryPending。如果 retryCount < maxRetry，则 retryCount + 1，设置 nextRetryAt，状态 RUNNING -> RETRY_PENDING。Scheduler 扫描到期任务后重新派发，Consumer 再把 RETRY_PENDING -> RUNNING。

## 5. 替代方案与暂时不选的原因

替代方案一是失败直接 FAILED。当前不选，因为临时错误会降低任务成功率。

替代方案二是 RabbitMQ TTL / 延迟队列。当前不选，因为它会引入队列插件和 MQ 配置复杂度，早期用 DB 字段更直观。

## 6. 数据库设计

task 表新增 retry_count、max_retry、next_retry_at。retry_count 记录已进入重试等待的次数，max_retry 限制最大次数，next_retry_at 表示下一次可重试时间。后续可扩展退避策略、失败分类和 task_attempt。

## 7. API 设计

GET /tasks/{taskId} 返回 retryCount、maxRetry、nextRetryAt。没有新增用户 API，重试由执行服务和 Scheduler 内部驱动。

## 8. 核心流程

执行失败
-> markTaskRetryPending
-> RUNNING -> RETRY_PENDING
-> 设置 retryCount 和 nextRetryAt
-> TaskRetryScheduler 到期扫描
-> 重新投递任务消息
-> Consumer 再次执行
-> 耗尽后 RUNNING -> FAILED

## 9. 测试方案

建议覆盖第一次失败进入 RETRY_PENDING、retryCount 增加、nextRetryAt 设置、到期任务被 Scheduler 处理、超过 maxRetry 后 FAILED。边界包括未到期不处理、retryCount 等于 maxRetry 不再重试。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是重试不能造成重复执行。Scheduler 扫描、MQ 投递、Consumer 消费之间存在时间窗口，后续必须配合幂等和 outbox。

## 12. 本版本亮点

- 引入 RETRY_PENDING 表达等待重试状态。
- 用 retry_count / max_retry 防止无限重试。
- 用 next_retry_at 支持延迟调度。
- 为后续可靠派发和执行审计奠定基础。

## 13. 简历 bullet

### 简洁版

设计任务自动重试机制，支持 RETRY_PENDING、retryCount、maxRetry 和 nextRetryAt，实现失败后的延迟重试。

### 强化版

为异步 AI 任务引入可控重试状态机，通过 retry_count、max_retry、next_retry_at 和 RetryScheduler 实现失败后的延迟重试，并避免无限重试风险。

## 14. 面试高强度追问与回答

Q1：为什么失败不一定直接 FAILED？

A1：很多失败是暂时性的，直接 FAILED 会降低成功率。重试给系统恢复机会。

Q2：为什么要 RETRY_PENDING？

A2：它明确表示任务失败但还没终结，正在等待下一次执行。比直接停在 FAILED 更能表达业务语义。

Q3：为什么限制 maxRetry？

A3：无限重试会浪费资源并制造队列堆积。maxRetry 是保护系统的上限。

Q4：nextRetryAt 有什么价值？

A4：它让重试可延迟，避免失败后立即打爆下游，也便于 Scheduler 扫描。

Q5：为什么不用 MQ 延迟队列？

A5：早期 DB 调度更容易验证和调试。延迟队列适合后续生产化优化。

Q6：Scheduler 多实例有什么风险？

A6：多个实例可能同时扫描同一任务并重复投递。后续需要 reserve、outbox 或分布式协调。

Q7：重试成功后旧 errorMessage 怎么办？

A7：后续成功处理应清空 errorMessage 或保留 attempt 历史，避免最终 SUCCESS 还显示旧错误。

Q8：当前局限是什么？

A8：固定延迟和简单扫描不够生产级，缺少指数退避、失败分类和死信处理。

## 15. 后续演进路线

V0.8 支撑 V0.9 幂等消费、V2.2.2 retry 派发走 outbox 和 V2.2.3 task_attempt 记录每次重试执行。
