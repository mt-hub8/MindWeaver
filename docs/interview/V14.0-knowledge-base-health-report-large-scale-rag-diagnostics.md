# V14.0 Knowledge Base Health Report & Large-Scale RAG Diagnostics 面试文档

## 0. 阶段定位

阶段名称：

```text
V14.0 Knowledge Base Health Report & Large-Scale RAG Diagnostics
```

中文名称：

```text
知识库体检报告与万级文档 RAG 诊断
```

这一阶段的目标不是再做一个普通 RAG 问答页面，而是把项目升级为：

```text
知识库质量诊断系统 / RAG 体检系统
```

普通 RAG 产品的问题是：用户只知道“回答不准”，但不知道原因。V14.0 要把“回答不准”拆解成可定位、可解释、可优化的问题：

```text
回答不准
  ↓
定位是召回问题、排序问题、上下文污染问题、版本混杂问题、引用问题，还是生成不忠实问题
  ↓
给出 chunk 切分、metadata 补全、collection 过滤、hybrid search、rerank、version filter、prompt 约束等优化建议
  ↓
用户优化知识库
  ↓
再次运行评测，比较优化前后分数
```

最终能力是：

```text
用户上传万级中文文档后，系统能输出知识库健康评分、主要扣分项、指标明细、失败 case、污染来源和优化建议。
```

一句话面试表达：

```text
我做的不只是 RAG 问答，而是知识库质量诊断闭环。系统可以在万级中文文档、多 collection、多版本、多文档类型混合场景下，通过 Recall@K、MRR@K、NDCG@K、ContextPrecision@K、CrossCollectionLeakRate、WrongVersionLeakRate、CitationAccuracy、Faithfulness 等指标，定位回答不准到底是召回问题、排序问题、上下文污染问题、版本混杂问题，还是生成不忠实问题。
```

## 1. 为什么要做这个阶段

当前项目已经有：

```text
文档上传；
批量导入；
知识库分组；
范围检索；
RAG Answer；
单次 RAG Quality Score；
垃圾箱；
模型 Provider 配置；
Agent Task；
Tool Workflow；
Ollama local-ai runtime。
```

但万级中文文档场景下，真正的难点不是“向量库能不能存”，而是：

```text
1. 检索空间太大，TopK 容易被噪声占满；
2. 不同项目 / 业务线 / 版本 / 文档类型的 chunk 语义相似，容易混召回；
3. 类名、接口名、配置项、版本号、路径等精确符号，纯向量检索不稳定；
4. chunk 太碎时上下文不完整，chunk 太大时噪声太多；
5. 用户不知道应该优化 chunk、metadata、collection、embedding、BM25、rerank 还是 prompt；
6. 没有 Gold Test Set 时，RAG 调参容易变成主观感觉。
```

所以 V14.0 的核心不是“再加一层检索”，而是建立一个完整闭环：

```text
测试集准备
  ↓
构建 / 选择知识库索引
  ↓
执行多检索策略
  ↓
记录召回结果
  ↓
计算检索指标
  ↓
可选执行 RAG 生成
  ↓
计算生成指标
  ↓
综合评分
  ↓
诊断原因
  ↓
生成优化建议
  ↓
优化前后对比
```

## 2. 证据来源与设计依据

V14.0 的设计结合了主流搜索 / RAG 工程实践、公开论文和本项目的产品化抽象。

### 2.1 Azure AI Search：Hybrid Search 与 RRF

证据点：

```text
Azure AI Search 官方文档把 Hybrid Search 描述为同时执行 full-text search 和 vector search；
full-text search 使用 BM25，vector search 使用 HNSW / eKNN 等向量检索；
结果通过 RRF（Reciprocal Rank Fusion，倒数排名融合）合并。
```

对本项目的启发：

```text
Vector Only 不够；
BM25 适合接口名、类名、配置项、版本号、文件路径；
Vector 适合语义相似；
RRF 适合作为第一版融合排序方式。
```

### 2.2 Qdrant：Metadata Filtering、Payload Index、Hybrid Query

证据点：

```text
Qdrant 支持 payload filtering；
Qdrant 文档说明向量检索结合过滤时，只有 vector index 不够，还需要 payload index 加速过滤字段；
Qdrant 也支持 dense / sparse / fusion 等 hybrid query 思路。
```

对本项目的启发：

```text
collection_id、version、doc_type、source、tags 不应该只是展示字段；
它们应该成为 metadata / payload filter；
常用过滤字段应该建索引；
否则万级文档下只能全库向量搜完再应用层过滤，性能和准确性都不理想。
```

### 2.3 Ragas 与 ARES：RAG 组件级评估

证据点：

```text
Ragas 将 RAG 评估拆成 Faithfulness、Answer Relevancy、Context Recall、Context Precision 等指标；
ARES 将 RAG 评估拆成 context relevance、answer faithfulness、answer relevance 等维度。
```

对本项目的启发：

```text
RAG 评估不能只看最终答案；
检索层和生成层必须分开；
检索对了不等于回答对了；
回答像样不等于引用可靠。
```

### 2.4 中文检索与 RAG Benchmark

证据点：

```text
T2Ranking 是大规模中文 passage ranking benchmark，包含 30 万以上 query 和 200 万以上 passage，并提供细粒度相关性标注；
DuReader-Retrieval 是中文 passage retrieval benchmark，包含 9 万以上 query 和 800 万以上 passage，来自商业搜索引擎场景；
CRUD-RAG 是中文 RAG 综合 benchmark，覆盖 Create / Read / Update / Delete 多类 RAG 应用，并分析 retriever、context length、knowledge base construction、LLM 等组件。
```

对本项目的启发：

```text
T2Ranking / DuReader-Retrieval 可用于中文检索和排序能力验证；
CRUD-RAG 可作为中文 RAG 综合评测参考；
但最能体现项目价值的是自建 Gold Test Set，因为它能覆盖你的项目版本、collection、配置项、接口名、旧版本污染等真实问题。
```

### 2.5 本项目自己的产品化抽象

公开资料不会直接给出“知识库健康评分 100 分固定公式”。V14.0 中的以下内容属于本项目基于工程实践做出的产品化设计：

```text
RAG Quality Score 100 分综合评分；
CrossCollectionLeakRate（跨集合污染率）；
WrongVersionLeakRate（错误版本污染率）；
精准 / 全面 / 平衡 / 生成可信四种评分模式；
Veto Rule 强制降级规则；
指标异常到优化建议映射；
知识库体检报告页面。
```

面试时要谨慎表达：

```text
这些不是行业固定标准，而是结合公开 RAG 评估实践和本项目知识库场景设计的产品化评分体系。后续需要通过 Gold Test Set 和人工偏好持续校准权重。
```

## 4. Evaluation Dataset 与 Gold Test Set 设计

### 4.1 为什么 Gold Test Set 是核心

公开 benchmark 只能验证通用能力，但不能覆盖你的项目真实问题，例如：

```text
V10.0 文档和 V11.0 文档混召回；
模型设置文档和 RAG 评分文档语义相似；
同一个项目不同版本文档混召回；
接口名、类名、配置项必须精确命中；
用户问“最新方案”时 deprecated 文档不应该出现；
用户问 V10.0 时 V9 / V11 / V12 不应该混入。
```

所以 V14.0 的核心是支持自建 Gold Test Set。

### 4.5 工程取舍

允许字段不完整：

```text
没有 expected_chunk_ids：Recall@K 不能严格计算；
没有 expected_answer_points：AnswerCoverage 不能严格计算；
没有 metadata_filter：版本污染、collection 污染只能部分判断。
```

必须遵守：

```text
缺失字段不能伪造成有效指标；
指标要有 available / unavailableReason；
评分解释要说明哪些指标不可用。
```

## 5. Retrieval Strategy Runner 设计

### 5.1 支持策略

```text
VECTOR_ONLY
BM25_ONLY
VECTOR_WITH_METADATA_FILTER
HYBRID
HYBRID_RRF
HYBRID_RRF_RERANK
HYBRID_RRF_RERANK_PARENT_CONTEXT
```

中文：

```text
纯向量检索；
BM25 关键词检索；
向量 + 元数据过滤；
混合检索；
混合检索 + RRF 融合；
混合检索 + RRF + 重排序；
混合检索 + RRF + 重排序 + 父级上下文回填。
```

### 5.2 策略说明

#### VECTOR_ONLY

优点：

```text
语义相似能力强；
适合自然语言问答。
```

缺点：

```text
接口名、类名、配置项、版本号等精确符号不稳定；
不同项目类似概念容易混召回。
```

#### BM25_ONLY

优点：

```text
精确词匹配强；
适合类名、接口名、配置项、文件路径、版本号。
```

缺点：

```text
不擅长语义改写；
用户问法和文档表达不一致时召回差。
```

#### VECTOR_WITH_METADATA_FILTER

优点：

```text
先缩小搜索域；
降低跨集合污染和错误版本污染。
```

关键：

```text
collection_id、version、doc_type、status、source 必须可靠。
```

#### HYBRID

做法：

```text
Vector topK + BM25 topK
```

先合并候选，不一定马上精排。

#### HYBRID_RRF

RRF 公式：

```text
score = sum(1 / (k + rank_i))
```

默认：

```text
k = 60
```

优点：

```text
不需要直接比较 BM25 score 和 vector similarity score；
只依赖排名；
实现简单、稳定。
```

#### HYBRID_RRF_RERANK

V14 不强制接真实 reranker，只做接口：

```text
Reranker.rerank(query, candidates)
```

默认实现：

```text
MockReranker；
SimpleHeuristicReranker。
```

后续可以接：

```text
bge-reranker；
cross-encoder；
LLM reranker；
semantic ranker。
```

#### PARENT_CONTEXT

解决 chunk 太碎问题：

```text
如果命中 chunk 只有局部上下文，就回填父章节或相邻 chunk。
```

V14 的安全降级：

```text
有 parentChunkId：回填 parent；
无 parentChunkId：回填 adjacent chunk；
都没有：记录 unsupported warning。
```

## 7. 检索指标设计

### 7.1 Recall@K（召回率）

中文含义：

```text
正确 chunk 被找回来多少。
```

公式：

```text
Recall@K = retrieved_expected_chunks / total_expected_chunks
```

低分说明：

```text
正确内容根本没进候选。
```

优化建议：

```text
优化 chunk 切分；
补充标题、摘要、关键词；
换 embedding 模型；
增加 query rewrite；
增加 BM25 / hybrid search。
```

### 7.2 HitRate@K（命中率）

中文含义：

```text
TopK 中是否至少有一个正确 chunk。
```

公式：

```text
HitRate@K = 1 if any expected chunk/doc in TopK else 0
```

用途：

```text
快速判断是否命中答案来源。
```

### 7.3 MRR@K（平均倒数排名）

中文含义：

```text
第一个正确结果排得靠不靠前。
```

公式：

```text
MRR@K = 1 / rank_of_first_relevant
```

低分说明：

```text
正确内容找到了，但排得太靠后。
```

优化建议：

```text
加 rerank；
优化排序权重；
减少相似噪声 chunk；
使用 RRF 融合排序。
```

### 7.4 NDCG@K（归一化折损累计增益）

中文含义：

```text
多个相关结果的整体排序质量。
```

适合：

```text
多证据问题；
expected_rank 或 relevance grade 存在的 case。
```

第一版如果没有多级相关性，可以用二值 relevance：

```text
expected 命中 = 1
未命中 = 0
```

### 7.5 ContextPrecision@K（上下文精确率）

中文含义：

```text
TopK 中相关 chunk 占比。
```

公式：

```text
ContextPrecision@K = relevant_chunks_in_topK / K
```

低分说明：

```text
TopK 混入太多无关内容。
```

优化建议：

```text
降低 TopK；
加 metadata filter；
加 rerank；
优化 chunk 粒度；
做上下文压缩。
```

### 7.6 CrossCollectionLeakRate（跨集合污染率）

中文含义：

```text
指定 collection 查询时，TopK 中来自错误 collection 的 chunk 占比。
```

公式：

```text
CrossCollectionLeakRate = wrong_collection_chunks / topK_chunks
```

这是本项目特色指标。

高分说明：

```text
其他知识库分组的内容进入上下文；
LLM 可能被错误项目、错误业务线带偏。
```

优化建议：

```text
强制 collection_id filter；
重新划分知识库；
检查 document-collection mapping；
检查 rerank / finalTopK 前是否丢失过滤条件。
```

### 7.7 WrongVersionLeakRate（错误版本污染率）

中文含义：

```text
指定版本查询时，TopK 中来自错误版本的 chunk 占比。
```

公式：

```text
WrongVersionLeakRate = wrong_version_chunks / topK_chunks
```

高分说明：

```text
新旧版本混召回；
用户问 V10.0，系统引用 V11/V12；
用户问最新方案，系统引用 deprecated 文档。
```

优化建议：

```text
给文档增加 version 字段；
给文档增加 status: active / deprecated；
默认过滤 deprecated；
指定版本问题强制 version filter。
```

## 8. 生成指标设计

### 8.1 AnswerCoverage（答案覆盖率）

中文含义：

```text
回答覆盖 expected_answer_points 的比例。
```

公式：

```text
AnswerCoverage = covered_points / total_expected_answer_points
```

第一版实现：

```text
关键词 / 短语包含匹配；
不声称是完美语义判断。
```

### 8.2 CitationAccuracy（引用准确率）

中文含义：

```text
引用是否真的来自 expected docs / chunks，并能支持答案。
```

基础计算：

```text
expected citation count / total citation count
```

扣分情况：

```text
answer_must_cite = true 但 citations 为空；
citation 来自 negative_doc_ids；
citation 来自 wrong collection；
citation 来自 wrong version。
```

### 8.3 Faithfulness（忠实性）

中文含义：

```text
答案是否被上下文支持。
```

V14 默认启发式：

```text
answer 非空且有 citation，加分；
无上下文却生成确定性答案，低分；
answer 明显超出引用上下文，扣分；
必须标记 heuristic = true。
```

不能说这是严格事实一致性评估。

### 8.4 RefusalAccuracy（拒答准确率）

中文含义：

```text
无答案时是否正确拒答。
```

适用：

```text
query_type = NO_ANSWER
```

正确行为：

```text
回答“不知道 / 上下文不足 / 未找到依据”。
```

错误行为：

```text
无上下文但生成确定性答案。
```

### 8.5 AnswerRelevance（回答相关性）

```text

```

第一版启发式：

```text

answer 非空；
```

## 9. RAG Quality Score 100 分综合评分

### 9.1 三层评分对象

```text
QueryQualityScore：单条 case 评分
DatasetQualityScore：单个评测集 / run 评分
KnowledgeBaseHealthScore：知识库健康评分
```

### 9.2 统一评分公式

```text
totalScore = Σ(metric_score_i * weight_i)
```

其中：

```text
Recall、MRR、NDCG、ContextPrecision、Faithfulness、CitationAccuracy 越高越好；
CrossCollectionLeakRate、WrongVersionLeakRate 越低越好，需要转换为 1 - rate；
最终归一化为 0-100。
```

### 9.3 分数等级

```text
90-100  优秀
75-89   良好
60-74   一般
0-59    较差
UNKNOWN 暂无评分
```

### 9.4 四种评分模式

#### 精准模式

适合：

```text
企业制度、合同、接口文档、财务文档、合规文档。
```

特点：

```text
宁可少召回，也不能乱召回。
```

权重：

```text
ContextPrecision@K       30
CrossCollectionLeakRate  20
WrongVersionLeakRate     15
MRR@K                    15
Recall@K                 10
CitationAccuracy         10
```

工程含义：

```text
最怕错误上下文进入 LLM；
跨集合污染和错误版本污染要重罚；
Recall 重要，但不是第一优先级。
```

#### 全面模式

适合：

```text
研究资料、学习资料、论文库、竞品分析、多文档总结。
```

特点：

```text
先尽量找全，再排序。
```

权重：

```text
Recall@K                 35
NDCG@K                   20
MRR@K                    15
ContextPrecision@K       15
CrossCollectionLeakRate  10
Faithfulness              5
```

#### 平衡模式

默认模式：

```text
Recall@K                 25
ContextPrecision@K       20
MRR@K                    15
NDCG@K                   15
CrossCollectionLeakRate  10
WrongVersionLeakRate      5
Faithfulness              5
CitationAccuracy          5
```

#### 生成可信模式

适合最终问答质量评估：

```text
Faithfulness             25
CitationAccuracy         20
AnswerCoverage           20
RefusalAccuracy          15
ContextPrecision@K       10
Recall@K                 10
```

## 10. Veto Rule 强制降级

### 10.1 为什么需要 Veto Rule

加权平均可能掩盖严重问题。

例如：

```text
Recall 很高；
但 CrossCollectionLeakRate 也很高；
平均分可能仍然不错。
```

这会误导用户。

### 10.2 规则

```text
CrossCollectionLeakRate > 30%  -> 最终分数最高 70
WrongVersionLeakRate > 25%     -> 最终分数最高 75
Faithfulness < 0.5             -> 最终分数最高 65
Recall@10 < 0.4                -> 最终分数最高 60
CitationAccuracy < 0.5         -> 最终分数最高 70
```

每条规则输出：

```text
ruleCode
title
description
originalScore
cappedScore
severity
```


```text

```

### 11.3 诊断表

| 指标异常 | 说明什么 | 优化建议 |
|---|---|---|
| Recall@K 低 | 正确内容没找回来 | 优化 chunk 切分、补标题/摘要/关键词、换 embedding、加 query rewrite、加 BM25 |
| MRR@K 低 | 找到了但排序靠后 | 加 rerank、优化排序权重、减少相似噪声 chunk、使用 RRF |
| NDCG@K 低 | 多个相关结果排序不理想 | 区分强弱相关、加 reranker、调整 fusion 权重 |
| ContextPrecision@K 低 | TopK 混入无关内容 | 降低 TopK、加 metadata filter、加 rerank、优化 chunk 粒度 |
| CrossCollectionLeakRate 高 | 跨知识库污染 | 强制 collection_id filter、重新划分知识库 |
| WrongVersionLeakRate 高 | 新旧版本混召回 | 增加 version/status 字段、过滤 deprecated |
| CitationAccuracy 低 | 引用不支持答案 | 加 citation 校验、只允许基于命中 chunk 回答 |
| Faithfulness 低 | LLM 脱离上下文发挥 | 强化 prompt、降低生成自由度、增加拒答规则 |
| RefusalAccuracy 低 | 无答案时胡编 | 增加 negative query 测试集、上下文不足时拒答 |
| Metadata 缺失 | 无法过滤 | 补充 collection_id、version、doc_type、source、section_path |

## 15. 工程权衡

### 15.1 为什么不默认做 LLM Judge

LLM Judge 可以更准确评估 faithfulness，但默认不做：

```text
会引入额外模型成本；
本地 qwen2.5:7b judge 稳定性有限；
外部 API 会破坏默认低依赖测试；
judge 本身也可能错；
V14 的核心是先把检索指标、污染指标和诊断闭环跑通。
```

后续可以做：

```text
可选 Judge Provider；
只在用户主动开启时调用；
结果标记 judgeModel；
保留人工校验入口。
```

### 15.2 为什么不做 OCR / 表格复杂还原

OCR 和复杂表格结构还原是文档理解阶段的大坑：

```text
需要图像模型；
需要表格结构识别；
需要坐标和版面信息；
需要大量异常处理。
```

V14 聚焦：

```text
RAG 评测和检索诊断。
```

不是文档解析平台。

### 15.3 为什么不做自动调参

自动调参涉及：

```text
不同 topK；
不同 chunk size；
不同 rerank topN；
不同 retrieval strategy；
多轮实验；
资源消耗；
推荐策略。
```

V14 先做：

```text
评测结果 + 人可读建议。
```

后续再做自动实验和参数推荐。

### 15.4 为什么不做完整 Reranker

真实 reranker 需要模型、延迟、成本和部署。

V14 做：

```text
Reranker 接口抽象；
Mock / simple reranker；
评测链路支持 rerank strategy。
```

后续可接：

```text
bge-reranker；
cross-encoder；
LLM reranker。
```

### 15.5 为什么污染指标是重点

普通指标看：

```text
有没有命中 expected chunk。
```

但你的真实问题是：

```text
有没有混入错误 collection；
有没有混入错误 version。
```

所以必须加入：

```text
CrossCollectionLeakRate；
WrongVersionLeakRate。
```

这两个指标直接衡量：

```text
上下文是否来自正确业务域；
答案是否被旧版本或错误项目污染。
```

## 19. 面试高强度追问与回答

### 19.1 V14 和 V11 的区别是什么？

回答：

```text
V11 是单次 RAG Answer 的质量评分，关注一次问答的回答质量、引用质量和简单诊断。V14 是知识库级别的体检系统，关注整个知识库在一组 Gold Test Set 上的检索表现、跨集合污染、错误版本污染、多策略对比、评分模式和优化前后指标变化。
```

### 19.2 为什么万级文档下向量检索会混杂？

回答：

```text
因为语义相似不等于业务域正确。不同项目、不同版本、不同文档类型可能讨论相似概念，向量检索会把语义相近的 chunk 放到一起。如果没有 collection_id、version、doc_type 等 metadata filter，LLM 可能拿到错误项目或旧版本上下文。
```

### 19.3 为什么要做 CrossCollectionLeakRate？

回答：

```text
普通 Recall 只告诉我有没有召回正确内容，但不告诉我是否混入错误 collection。对于多项目知识库，错误 collection 的 chunk 即使语义相关，也会污染上下文。所以我设计 CrossCollectionLeakRate 衡量指定 collection 查询时，TopK 中有多少来自非目标 collection。
```

### 19.4 为什么要做 WrongVersionLeakRate？

回答：

```text
项目文档经常有多个版本。用户问 V10.0 或问最新方案时，如果系统召回 V9 或 deprecated 文档，就会导致回答过时。WrongVersionLeakRate 专门衡量错误版本文档进入 TopK 的比例。
```

### 19.5 为什么不只看最终答案？

回答：

```text
最终答案混合了检索、上下文组装、prompt 和 LLM 生成多个环节。如果只看答案，不知道问题出在哪里。V14 把评估拆成检索指标、生成指标和污染指标，可以定位是召回、排序、上下文污染、引用还是生成问题。
```

### 19.6 为什么要支持 Vector Only / BM25 / Hybrid 对比？

回答：

```text
不同 query 类型适合不同检索方式。向量检索擅长语义相似，BM25 擅长类名、接口名、配置项、版本号等精确匹配。Hybrid 可以同时利用两者。只有做策略对比，才能量化证明哪种策略对当前知识库有效。
```

### 19.7 为什么使用 RRF？

回答：

```text
BM25 分数和 vector similarity 分数不在同一尺度，直接加权不稳定。RRF 只依赖各自的排名，用 1 / (k + rank) 融合多个排序结果，简单、稳定，也被 Azure AI Search 等 hybrid search 实践采用。
```

### 19.8 为什么不默认接 cross-encoder reranker？

回答：

```text
真实 reranker 会引入额外模型、延迟和部署成本。V14 的目标是先把评测链路、策略抽象和指标诊断跑通，所以先做 Reranker 接口和 mock/simple 实现，后续可以接 bge-reranker 或 cross-encoder。
```

### 19.9 为什么缺失指标不能伪造？

回答：

```text
如果没有 expected_chunk_ids，就不能声称计算了严格 Recall@K；如果没有 expected_answer_points，就不能声称计算了严格 AnswerCoverage。评测系统必须诚实，否则会给用户虚假的确定性。V14 对不可计算指标标记 UNKNOWN，并在报告中说明原因。
```

### 19.10 为什么要有 Veto Rule？

回答：

```text
加权平均可能掩盖严重问题。例如 Recall 很高但跨集合污染率也很高，平均分可能仍然不错。但对业务来说，错误 collection 进入上下文是严重问题。所以 Veto Rule 会对 CrossCollectionLeakRate、WrongVersionLeakRate、Faithfulness 等关键风险设置分数上限。
```

### 19.11 评分权重依据是什么？

回答：

```text
权重不是行业固定标准，而是基于不同业务场景的错误成本设计。合同、合规、接口文档错误召回成本高，所以精准模式更重视 ContextPrecision、跨集合隔离和版本隔离；研究和学习场景遗漏证据成本高，所以全面模式更重视 Recall 和 NDCG；普通个人知识库用平衡模式；最终回答可信度评估用生成可信模式。后续可以通过 Gold Test Set 和人工偏好校准权重。
```

### 19.12 公开 benchmark 为什么不够？

回答：

```text
T2Ranking 和 DuReader-Retrieval 能测中文检索和排序基础能力，CRUD-RAG 能测中文 RAG 综合能力。但它们无法覆盖我项目中的 V10/V11/V12 文档版本混杂、collection 选择、模型配置文档、接口类问题等真实场景。所以公开 benchmark 是参考，自建 Gold Test Set 才是核心。
```

### 19.13 没有 Gold Test Set 怎么办？

回答：

```text
没有 Gold Test Set 时，系统可以先做启发式诊断，比如 metadata 完整度、cross collection 过滤情况、citation 缺失、no-context 比例、TopK score 分布等。但严格的 Recall、MRR、NDCG 必须依赖 expected docs 或 expected chunks。系统会提示用户补充 Gold Test Set，以获得更可靠评分。
```

### 19.14 V14 如何指导用户实际优化知识库？

回答：

```text
每个指标异常都会映射到具体建议。例如 Recall 低建议优化 chunk 切分、补标题摘要关键词、换 embedding 或加 BM25；ContextPrecision 低建议加 metadata filter、降低 TopK 或加 rerank；CrossCollectionLeakRate 高建议强制 collection_id filter；WrongVersionLeakRate 高建议补 version/status 字段并过滤 deprecated 文档。
```

### 19.15 为什么这比普通 RAG Demo 更有面试价值？

回答：


## 21. 总结表达

```text
在 V14.0 中，我把项目从普通 RAG 问答升级为知识库质量诊断系统。这个阶段新增了评测集管理、Gold Test Set 导入、多检索策略对比、检索指标计算、生成指标基础计算和知识库健康评分页面。

针对万级中文文档容易出现的向量内容混杂问题，我设计了 CrossCollectionLeakRate 和 WrongVersionLeakRate 两个特色指标，用于衡量跨知识库污染和错误版本污染。同时，系统支持精准、全面、平衡、生成可信四种评分模式，并通过 Veto Rule 防止严重污染被平均分掩盖。

最终，系统不再只告诉用户“回答不准”，而是能定位到底是 Recall 低、MRR 低、ContextPrecision 低、跨集合污染、版本混杂、引用不准确，还是 LLM 没有忠实使用上下文，并给出具体的知识库优化建议。这使 RAG 从玄学调参变成可观测、可评测、可诊断、可迭代的工程系统。
```

## 22. 参考资料与证据来源

> 这些来源用于支撑 V14.0 的设计方向。具体权重表、100 分评分和 CrossCollectionLeakRate / WrongVersionLeakRate 是本项目基于这些工程实践做出的产品化设计，不是公开资料中的固定行业标准。

### 22.1 Microsoft Azure AI Search：Hybrid Search / RRF

证据点：

```text
Hybrid search 结合 full-text search 和 vector search；
BM25 与 vector ranking 可以并行；
RRF 可用于合并多个排序结果。
```

用于支撑：

```text
Vector + BM25 + RRF 的 hybrid retrieval pipeline。
```

### 22.2 Qdrant：Filtering / Payload Index / Hybrid Query

证据点：

```text
Qdrant 支持 payload filtering；
向量检索结合过滤时需要 payload index；
dense / sparse / fusion 支持 hybrid query 思路。
```

用于支撑：

```text
collection_id、version、doc_type、source、tags 要进入 metadata / payload filter；
常用过滤字段需要索引。
```

### 22.3 Ragas Metrics

证据点：

```text
Ragas 提供 Faithfulness、Answer Relevancy、Context Recall、Context Precision 等组件级 RAG 指标。
```

用于支撑：

```text
检索层和生成层分开评估。
```

### 22.4 ARES

证据点：

```text
ARES 将 RAG 评估拆成 context relevance、answer faithfulness、answer relevance 等维度。
```

用于支撑：

```text
上下文相关性、回答忠实性、回答相关性需要分开判断。
```

### 22.5 T2Ranking

证据点：

```text
大规模中文 passage ranking benchmark；
包含 30 万以上 query 和 200 万以上 passage；
有细粒度相关性标注。
```

用于支撑：

```text
中文 passage ranking / reranking 能力需要专门评测。
```

### 22.6 DuReader-Retrieval

证据点：

```text
大规模中文 passage retrieval benchmark；
包含 9 万以上 query 和 800 万以上 passage；
来自商业搜索引擎场景。
```

用于支撑：

```text
中文 query 和 passage 表达不一致是现实问题；
dense retrieval 需要在中文真实检索场景验证。
```

### 22.7 CRUD-RAG

证据点：

```text
中文 RAG 综合 benchmark；
覆盖 Create / Read / Update / Delete 多类 RAG 应用；
分析 retriever、context length、knowledge base construction、LLM 等组件。
```

用于支撑：

```text
RAG 评测不能只看 QA；
知识库构建和 retriever 本身也需要进入评测。
```
