# V0.4 面试文档：task_event 事件表

## 1. 本版本解决的问题

V0.4 解决“task 表只有当前状态，无法知道历史发生了什么”的问题。task 表适合保存当前状态，而 task_event 用来保存 TASK_CREATED、STATUS_CHANGED 等历史事件，形成任务 Timeline，便于审计、排障和面试中解释可观测性设计。

## 2. 为什么现在做这个版本

它依赖任务创建、查询和状态机。后续 Consumer、失败处理、重试、取消、超时都会产生状态变化，如果不提前记录事件，问题发生后只能看到最终状态，看不到过程。此时不做完整 Event Sourcing，是为了避免过度复杂化。

## 3. 核心需求边界

本版本新增事件记录能力，记录任务创建和状态变化。不做事件回放、不做消息总线、不做复杂审计系统。不把 task_event 当作主状态来源，主状态仍由 task 表承载。

## 4. 方案选择

采用 task 表保存当前状态，task_event 表保存历史事件。事件包含 taskId、eventType、fromStatus、toStatus、message、createdAt。这样查询当前状态很快，排查历史时也有完整轨迹。

## 5. 替代方案与暂时不选的原因

替代方案一是只保存 task 当前状态。当前不选，因为失败后无法知道经历了几次重试、何时进入 RUNNING、是谁触发取消。

替代方案二是完整 Event Sourcing，以事件重建状态。当前不选，因为复杂度高，查询当前状态不如 task 表直接。

## 6. 数据库设计

新增 task_event 表。核心字段包括 id、task_id、event_type、from_status、to_status、message、created_at。task_id 用于关联 task，event_type 区分 TASK_CREATED 和 STATUS_CHANGED。后续可按 task_id + created_at 建索引，支持 Timeline 查询。

## 7. API 设计

本版本主要是内部事件记录能力，未新增用户 API。事件可通过 SQL 验证，后续可以扩展 GET /tasks/{taskId}/events 用于前端 Timeline。

## 8. 核心流程

创建任务
-> 保存 task
-> 写 TASK_CREATED

状态变化
-> 状态机校验
-> 更新 task.status
-> 写 STATUS_CHANGED
-> 后续查询事件 Timeline

## 9. 测试方案

建议覆盖创建任务写 TASK_CREATED、状态更新写 STATUS_CHANGED、fromStatus/toStatus 正确、非法状态流转不写事件。SQL 验证可查询 task_event 按 created_at 排序。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是事件必须和状态变化保持一致。如果 task 更新成功但事件没写，审计会断裂；如果非法流转也写事件，会制造假历史。后续 Outbox 阶段还会进一步处理跨资源一致性。

## 12. 本版本亮点

- 区分当前状态和历史过程。
- 为失败排查、重试分析和取消审计提供 Timeline。
- 不引入完整 Event Sourcing，保持实现轻量。
- 为后续 Agent Trace / Execution Trace 打基础。

## 13. 简历 bullet

### 简洁版

新增 task_event 事件表记录任务创建和状态变化，支持任务生命周期审计与排障。

### 强化版

在异步任务系统中引入当前状态表与历史事件表分离设计，通过 task_event 保存 TASK_CREATED / STATUS_CHANGED 轨迹，为重试、取消、超时和后续执行 Trace 提供可观测基础。

## 14. 面试高强度追问与回答

Q1：为什么不能只靠 task 当前状态？

A1：当前状态只能告诉我们结果，不能解释过程。排查失败任务时，需要知道它何时开始、失败几次、是否重试或取消。

Q2：task_event 和日志有什么区别？

A2：日志是非结构化或半结构化运行信息，task_event 是业务数据，可查询、可审计、可和 taskId 精确关联。

Q3：为什么不做完整 Event Sourcing？

A3：当前只需要审计历史，不需要用事件重建所有状态。保留 task 当前状态能简化查询和实现。

Q4：事件写入失败怎么办？

A4：早期版本可以和 task 更新放在同一事务内。后续跨 MQ 投递时，需要 Outbox 解决一致性问题。

Q5：fromStatus/toStatus 为什么重要？

A5：它们能直接表达状态流转路径，比如 RUNNING -> FAILED 或 RETRY_PENDING -> RUNNING，比只看 message 更可靠。

Q6：task_event 会不会无限增长？

A6：会随着任务量增长。后续可以按时间归档、按 tenant 分区或建立冷热数据策略。

Q7：task_event 能用于前端吗？

A7：可以扩展 Timeline API，展示任务从创建到完成的完整过程。

Q8：当前局限是什么？

A8：它记录生命周期事件，但还不是每次执行 attempt 的完整审计。后续 task_attempt 会补充执行级证据链。

## 15. 后续演进路线

V0.4 支撑 V0.7 失败事件、V0.8 重试事件、V0.10 取消/超时事件和 V2.2.x task_attempt。后续可扩展为任务 Timeline API 或 Agent 执行 Trace。
