# V0.3 面试文档：任务状态机

## 1. 本版本解决的问题

V0.3 解决任务生命周期“不受约束会乱跳”的问题。异步任务会经历 PENDING、RUNNING、SUCCESS、FAILED、CANCELLED 等状态，如果没有状态机，接口或执行线程可能把 SUCCESS 改回 RUNNING，或把 FAILED 重新改成 SUCCESS，导致数据脏乱。状态机为任务流转建立统一规则。

## 2. 为什么现在做这个版本

它依赖 V0.1 的任务模型和 V0.2 的查询能力。后续 RabbitMQ、Consumer、失败处理、重试、取消、超时都会修改状态，因此必须先建立流转约束。此时不做复杂工作流引擎，是因为当前任务生命周期有限，用轻量状态机更清晰。

## 3. 核心需求边界

本版本定义任务状态和合法流转，主要包括 PENDING、RUNNING、SUCCESS、FAILED、CANCELLED，后续扩展 RETRY_PENDING。不做 BPMN、不做复杂审批流、不做多阶段 DAG。保持克制是为了让状态流转直接服务异步执行。

## 4. 方案选择

采用 TaskStatus 枚举表达有限状态，用 TaskStateMachine 判断 fromStatus -> toStatus 是否允许。终态 SUCCESS / FAILED / CANCELLED 不允许继续流转。状态更新统一经过 Service 校验，避免 Controller 或 Worker 绕过规则。

## 5. 替代方案与暂时不选的原因

替代方案一是在各个方法中写 if 判断。当前不选，因为规则会分散，后续重试和取消会让维护成本变高。

替代方案二是引入工作流引擎。当前不选，因为项目还处在任务编排底座阶段，状态数量有限，轻量状态机更易理解和测试。

## 6. 数据库设计

本版本主要复用 task.status 字段，未必新增独立表。status 保存当前状态，后续 task_event 会保存状态变化历史。状态字段建议使用枚举字符串，便于排查和 SQL 阅读。

## 7. API 设计

如果项目保留 PATCH /tasks/{taskId}/status，该接口用于手动验证状态机。Request 包含 status 和 message。非法流转应返回 400。该接口不是核心业务入口，主要用于开发验收和状态机测试。

## 8. 核心流程

状态更新请求
-> TaskService 读取当前状态
-> TaskStateMachine 校验流转
-> 合法则更新 task.status
-> 非法则返回 400
-> 后续写入事件表

## 9. 测试方案

建议覆盖 PENDING -> RUNNING、RUNNING -> SUCCESS、RUNNING -> FAILED、RUNNING -> CANCELLED、终态不能继续流转、非法 SUCCESS -> RUNNING。后续扩展 RETRY_PENDING 时应补充 RUNNING -> RETRY_PENDING、RETRY_PENDING -> RUNNING。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点在于所有状态修改必须走同一规则入口。异步系统里状态可能来自 Controller、Consumer、Scheduler、取消接口和超时扫描器，如果有一个入口绕过状态机，就会产生脏状态。

## 12. 本版本亮点

- 把任务生命周期显式建模，降低隐式状态判断风险。
- 用终态约束防止 SUCCESS / FAILED / CANCELLED 被重新执行。
- 为重试、取消、超时提供统一约束基础。
- 状态机逻辑容易单元测试和面试讲解。

## 13. 简历 bullet

### 简洁版

设计任务状态机，约束 PENDING、RUNNING、SUCCESS、FAILED、CANCELLED 等状态流转，防止异步任务非法更新。

### 强化版

为异步 AI 任务系统建立轻量状态机，将生命周期流转集中校验，避免多入口状态更新导致 SUCCESS 回退、FAILED 重跑等脏状态问题，并支撑后续重试、取消和超时机制。

## 14. 面试高强度追问与回答

Q1：为什么状态机是异步任务系统核心？

A1：异步任务有多个修改状态的来源。状态机把合法流转集中起来，保证不同入口遵守同一生命周期规则。

Q2：为什么终态不能继续流转？

A2：SUCCESS、FAILED、CANCELLED 代表任务已结束。如果允许继续流转，会导致重复执行、审计混乱和结果覆盖。

Q3：为什么不用数据库约束实现状态机？

A3：数据库可以约束枚举值，但很难表达 from -> to 的业务规则。状态机放在业务层更清晰，必要时再配合数据库条件更新。

Q4：RETRY_PENDING 为什么后续才加？

A4：重试语义需要 retry_count、next_retry_at 和 Scheduler 支撑。先定义基础状态，再增量扩展重试状态，边界更清楚。

Q5：PATCH 状态接口安全吗？

A5：它适合开发测试状态机，生产中应限制权限或移除。真正业务入口应由 Service 和 Worker 控制状态流转。

Q6：状态机如何和 MQ 幂等结合？

A6：Consumer 收到重复消息时，只有 PENDING/RETRY_PENDING 可进入 RUNNING，终态或 RUNNING 会被拒绝，从而避免重复执行。

Q7：状态机如何测试？

A7：用枚举组合测试合法和非法流转，尤其关注终态、重复 RUNNING、重试和取消路径。

Q8：当前方案局限是什么？

A8：它是轻量状态机，不支持复杂 DAG、人工审批或多步骤工作流。但对当前 AI 任务编排底座已经足够。
