# V9.0 Local Personal Knowledge Workspace 面试文档

## 1. 阶段摘要

V9.0 的阶段名称是：

```text
V9.0 Local Personal Knowledge Workspace
```

中文可以表述为：

```text
本地个人 AI 知识工作台产品化收敛
```

这一阶段的核心目标不是继续堆 Agent 功能，而是把前面已经实现的 RAG、知识库、Collection、Agent Task、Tool Workflow、Ollama 本地模型接入等能力，重新收敛成一个用户能够理解、能够启动、能够配置、能够使用的本地个人 AI 产品。

项目定位从：

```text
企业级 AI Task Orchestrator
```

调整为：

```text
Personal AI Knowledge Workspace
个人 AI 知识工作台
```

一句话定位：

```text
一个本地优先的个人 AI 应用，用户可以上传自己的资料，配置本地或外部模型，构建个人知识库，并让 AI 基于自己的知识执行问答、总结、报告生成和任务编排。
```

V9.0 不是新增底层算法阶段，而是产品化收敛阶段。它解决的是：

```text
项目功能已经很多，但普通用户打开后不知道从哪里开始；
README 更像工程记录，不像产品主文档；
页面暴露过多 eventType / metadata / traceId 等技术细节；
用户不知道当前模型是什么、数据在哪里、如何启动本地 AI Runtime；
项目名称和定位仍然偏“企业级”，不符合当前个人本地 AI 应用方向。
```

---

## 2. 为什么要做 V9.0

在 V3.0 到 V8.0 之间，项目已经具备了大量能力：

```text
文档上传；
异步文档摄入；
文档处理事件时间线；
知识库分组；
范围检索；
RAG Answer；
引用来源；
文档软删除；
文档重新索引；
Agent Task；
Tool-Using Agent Workflow；
Python AI Runtime Worker；
Ollama 本地 embedding 与 LLM。
```

但是功能堆叠到一定程度后，会出现新的问题：

```text
功能很多，但产品主线不清楚；
用户不知道第一步该做什么；
页面像工程调试后台，而不是个人 AI 工具；
README 难以让第一次进入仓库的人快速理解项目；
模型配置、本地运行、数据目录、技术详情之间缺少统一解释。
```

所以 V9.0 的目标是把项目从：

```text
功能很多的工程项目
```

推进为：

```text
有明确定位、有产品入口、有中文文档、有本地启动路径的个人 AI 知识工作台。
```

## 3. V9.0 的核心变化

V9.0 主要包含七类收敛动作：

```text
1. 产品名称与定位调整；
2. 首页信息架构重构；
3. 简洁中文 UI 与技术详情折叠；
4. 模型设置中心基础版；
5. 系统设置与本地数据目录展示；
6. Windows 本地启动脚本基础版；
7. README 中文产品化系统重写与 manual 文档补齐。
```

这些动作的共同目标是：

```text
降低普通用户理解和使用门槛；
让项目在 GitHub 上更像一个产品；
让面试官能快速理解项目价值；
为后续多模型 Provider、RAG 评分、垃圾箱、批量上传、Skill 系统打基础。
```

## 4. 核心设计一：产品重新定位

### 4.1 为什么不再强调“企业级”

此前项目定位偏向：

```text
企业级 AI Task Orchestrator
```

这个定位容易把后续路线带向：

```text
多租户；
权限；
workspace；
团队协作；
审计；
计费；
生产监控；
SaaS 部署。
```

这些能力确实有工程价值，但不是当前最适合的产品主线。

当前更适合的定位是：

```text
个人本地 AI 知识库 + 可配置模型 + Agent 任务编排工具
```

也就是：

```text
Personal AI Knowledge Workspace
个人 AI 知识工作台
```

这样项目的用户更加明确：

```text
想在本地处理私人文档的人；
想构建个人知识库的人；
想体验 RAG / Agent 工作流的开发者；
想学习 AI 应用后端和 Agent Infra 的人。
```

---

### 4.2 新产品名称

用户可见产品名称统一为：

```text
Personal AI Knowledge Workspace
个人 AI 知识工作台
```

README、首页、导航、manual 都围绕这个名称展开。

但在技术说明中，可以保留：

```text
项目底层包含 AI Task Orchestration、RAG、Agent Workflow 等工程能力。
```

也就是说：

```text
产品定位是个人 AI 知识工作台；
工程能力包含任务编排、RAG、Agent Workflow。
```



## 9. 核心设计六：Windows 本地启动脚本基础版

V9.0 新增：

```text
scripts/windows/
```

包含：

```text
check-env.ps1；
start-local.ps1；
stop-local.ps1；
README.md。
```

### 9.1 check-env.ps1

用于检查本地环境：

```text
Java 是否可用；
Python 是否可用；
Ollama 是否可用；
qwen3-embedding:0.6b 是否存在；
qwen2.5:7b 或 qwen2.5:3b 是否存在；
8001 端口是否占用；
8080 端口是否占用；
11434 端口是否可访问。
```

输出必须中文友好：

```text
Java：已检测到；
Python：未检测到；
Ollama：未启动；
模型 qwen3-embedding:0.6b：已安装；
端口 8001：可用 / 已占用。
```

### 9.2 start-local.ps1

用于基础启动：

```text
提示用户确认 Ollama 已启动；
启动 Python Worker；
启动 Spring Boot local-ai profile；
打开浏览器 http://localhost:8080。
```

本阶段不做复杂守护进程，也不强行控制 Ollama 生命周期。

### 9.3 stop-local.ps1

用于基础停止：

```text
提示用户手动关闭对应窗口；
或安全停止本脚本启动的进程。
```

不能误杀用户其他 Java / Python / Ollama 进程。

### 9.4 scripts/windows/README.md

说明：

```text
如何检查环境；
如何启动；
如何停止；
端口被占用怎么办；
模型没下载怎么办；
常见问题。
```

这一步让项目从开发者手动启动多个服务，逐步接近“一键本地使用”。

## 12. 关键取舍

### 12.1 为什么 V9.0 不做完整 API key 管理

API key 管理涉及：

```text
加密存储；
操作系统凭据；
云模型 Provider；
测试连接；
错误处理；
默认模型选择；
安全展示；
不要泄露 key。
```

这是一个完整阶段，适合 V10.0。

V9.0 只先建立“模型设置”入口和当前运行状态展示。

---

### 12.2 为什么不做登录系统

当前产品定位是本地个人应用，默认运行在用户本机。

本地单用户模式不一定需要登录。

登录系统会引入：

```text
用户表；
密码 hash；
session / JWT；
权限；
文档归属；
API key 归属；
workspace；
多用户隔离。
```

这不属于 V9.0。

---

### 12.3 为什么不做垃圾箱自动清理

垃圾箱需要设计：

```text
ACTIVE -> TRASHED -> PURGED；
恢复；
7 天后清理；
物理文件移动；
向量清理；
缓存清理；
后台 job。
```

这应该放到后续 V12.0。

V9.0 只在系统设置中做规划说明。

---

### 12.4 为什么不做批量上传

批量上传涉及：

```text
batch 表；
并发控制；
失败重试；
批量进度；
通知；
文件级去重；
文本级去重；
chunk 级去重。
```

这应该放到 V13.0。

---

### 12.5 为什么不做 Skill 和多 Agent

Skill 和多 Agent 都很吸引人，但会引入：

```text
Skill Manifest；
执行权限；
自定义 prompt；
工具范围；
多 Agent 状态；
角色记忆；
成本控制；
循环控制。
```

必须等产品基础、模型配置、RAG 质量和任务控制更稳定后再做。

## 18. 面试高强度追问与回答

### 18.1 为什么这个阶段要改项目定位？

回答：

```text
因为项目已经从单纯的任务编排后端演进出了文档上传、个人知识库、RAG 问答、模型接入、Agent Task 和 Tool Workflow 等能力。继续强调“企业级”会把路线带向权限、多租户、workspace 和 SaaS，但当前更适合做本地优先的个人 AI 知识工作台，让普通用户能上传资料、配置模型、构建知识库并执行 AI 任务。
```

---

### 18.2 V9.0 为什么不继续做 Agent 功能？

回答：

```text
因为前面已经完成了 Agent Task 和 Tool Workflow，继续堆多 Agent 或 Skill 会增加复杂度，但用户仍然可能不知道怎么启动和使用。V9.0 的目标是产品化收敛，先把项目名称、首页、README、模型设置、本地数据目录和启动脚本整理清楚，让已有功能真正可用。
```

---

### 18.3 为什么要把技术详情折叠？

回答：

```text
eventType、metadata、traceId、provider、dimension、tokens 对开发者有用，但普通用户默认看到这些会很困惑。所以 V9.0 采用分层展示：默认展示中文状态、结果、引用和下一步操作；需要调试时再展开技术详情。这让同一个页面同时服务普通用户和开发者。
```

---

### 18.4 模型设置中心为什么只做基础版？

回答：

```text
完整模型 Provider 设置涉及 API key 加密存储、云模型配置、连接测试、模型选择和安全展示，适合作为独立阶段。V9.0 先做当前运行状态展示和本地 Ollama / Python Worker 连接测试，建立模型配置入口，不把功能范围扩大。
```

---

### 18.5 为什么不立刻把 MySQL 换成 SQLite？

回答：

```text
存储替换是底层架构变化，会影响文档、任务、事件、Collection、向量索引和测试。V9.0 的目标是产品化收敛，不是存储迁移。当前只先规划本地数据目录，后续如果要做 local single-user mode，再单独处理 SQLite / embedded queue / 本地向量库。
```

---

### 18.6 README 为什么要重写？

回答：

```text
README 是项目的第一入口。之前的 README 更像工程记录，不能让第一次进入仓库的人快速理解产品。V9.0 把 README 重写成中文产品主文档，说明项目是什么、适合谁、怎么启动、怎么接 Ollama、怎么上传文档和创建任务、Java 和 Python 怎么分工、当前不做什么以及后续路线。
```

---

### 18.7 为什么不做登录？

回答：

```text
当前定位是本地个人应用，默认在用户本机使用。登录系统主要用于多用户、云端、workspace 和权限隔离，这会显著扩大项目边界。所以 V9.0 不做登录，后续如果要做云端或多用户版本，再单独设计用户系统。
```

---

### 18.8 Windows 启动脚本的价值是什么？

回答：

```text
项目已经涉及 Spring Boot、Python Worker、Ollama 和本地模型，如果每次都手动启动，会降低使用体验。Windows 脚本可以检查 Java、Python、Ollama、模型和端口状态，并启动 Python Worker 和 Spring Boot local-ai profile，降低本地体验门槛。
```

---

### 18.9 这个阶段体现什么工程能力？

回答：

```text
V9.0 体现的是产品化工程能力和边界控制能力。它没有继续盲目加功能，而是对项目定位、信息架构、页面层级、模型运行状态、本地启动方式和 README 文档做系统收敛，让已有 RAG / Agent / AI Runtime 能力成为用户可以理解和体验的产品。
```
