# V2.5 Real Embedding Provider 面试文档

## 0. 版本摘要

V2.5 将原来的 MockEmbedding 链路升级为可配置的 EmbeddingProvider 架构：保留 Mock provider 作为默认实现，同时新增 OpenAI-compatible Real Embedding Provider。它让 RAG 检索从“固定 mock embedding 实现”升级为“可配置、可扩展、可测试的 embedding provider 架构”。

本版本的边界很明确：

- 不是 Vector DB；
- 不是 Rerank；
- 不是 Hybrid Search；
- 不是 Evaluation Persistence；
- 不是本地模型 worker；
- 默认仍然使用 Mock Provider；
- 默认 test / CI 不调用外部 API；
- OpenAI-compatible provider 只通过配置开启；
- 不声称 Real Embedding 一定提升 Recall@K、Precision@K、NDCG@K、MRR 或 ContextPrecision@K。

V2.5 的重点不是“立刻证明真实 embedding 效果更好”，而是把 embedding 这一层从固定实现变成可替换的 provider 能力，并把 provider / model / dimension metadata 纳入生成、保存和搜索链路。

## 1. 本版本解决的问题

V2.2 的 Mock Embedding 已经打通了工程链路：

```text
document chunk
-> mock embedding
-> vector search
-> RAG answer
-> retrieval evaluation
```

MockEmbedding 的价值很高：

- deterministic，适合断言和回归测试；
- 免费，不产生外部 API 成本；
- 稳定，不依赖网络；
- 不需要 API key；
- 适合本地开发和 CI；
- 适合早期验证 RAG 链路是否跑通。

但 MockEmbedding 的局限也很明显：

- 不代表真实语义相似度；
- 不能真实评估 query 与 chunk 的语义相关性；
- 不适合真实企业知识库检索；
- 无法支撑严肃的 embedding model 对比；
- 无法回答“换成真实 embedding 后 retrieval metrics 是否变化”。

V2.5 解决的是 embedding 层的架构问题：

- 新增 `EmbeddingProvider` 抽象；
- 保留 `MockEmbeddingClient`；
- 新增 `OpenAiCompatibleEmbeddingProvider`；
- 通过 `EmbeddingProperties` 和 `EmbeddingProviderConfiguration` 选择 provider；
- 默认 `app.embedding.provider=mock`；
- provider / model / dimension metadata 进入 `EmbeddingResponse` 和 `document_chunk_embedding`；
- `DocumentEmbeddingService` 依赖 provider abstraction；
- search 时通过 provider / model 查询候选 embedding，并通过 query embedding dimension 过滤；
- 默认测试不依赖外部网络和 API key。

## 2. 为什么现在做 V2.5

V2.5 放在 V2.4.x 之后是有意安排的。

前置阶段已经具备：

- V2.4：`RetrievalEvaluationService` 和 `POST /evaluations/retrieval`；
- V2.4.1：Precision@K、NDCG@K 等更完整的 ranking metrics；
- V2.4.2：固定 corpus 和 benchmark JSON；
- V2.4.3：Fixed vs Adaptive Chunking comparison baseline；
- V2.4.4：Benchmark Runner / Evidence Mapper，把 `expectedEvidenceIds` 映射到运行时 `expectedChunkIds`。

这意味着现在接入 Real Embedding Provider 之后，可以用同一套 benchmark 去观察检索结果，而不是只证明“OpenAI API 能调通”。

核心表达：

```text
没有 evaluation baseline 前接真实 Embedding，只是 API integration。
有了 V2.4.x benchmark 后接入，才是可评估的 retrieval quality upgrade。
```

V2.5 本身只做 provider 接入，不做 Mock vs Real 的指标对比。Mock vs Real benchmark comparison 应该放到 V2.5.1，因为那会涉及固定数据集、配置切换、重复运行和指标对比。

## 3. 方案选择

### 方案 A：直接在 MockEmbeddingClient 里写 OpenAI 调用

优点：

- 实现最快；
- 改动少；
- 可以快速生成真实向量。

缺点：

- Mock 和 Real 耦合；
- `MockEmbeddingClient` 名字和职责失真；
- 测试不稳定；
- CI 可能依赖外部 API；
- API key、网络、成本污染核心测试；
- 后续难以扩展其他 provider；
- 难以做 provider-level evaluation。

结论：不推荐。

### 方案 B：EmbeddingProvider 抽象 + Mock Provider + Real Provider

当前采用这个方案。

做法：

- 新增 `EmbeddingProvider`；
- `MockEmbeddingClient` 实现 `EmbeddingProvider`；
- 新增 `OpenAiCompatibleEmbeddingProvider`；
- 通过 `EmbeddingProperties` 配置 provider；
- 默认 provider 为 `mock`；
- `openai` provider 只在配置开启时使用。

优点：

- 抽象清晰；
- 测试稳定；
- CI 不依赖外部 API；
- 可扩展多个 provider；
- provider / model / dimension metadata 可以统一管理；
- 后续可接 OpenAI、Spring AI、本地模型服务、企业私有 embedding 服务。

缺点：

- 需要重构 `DocumentEmbeddingService`；
- 需要处理配置、异常、metadata 和测试隔离；
- 比直接写 API 调用复杂。

结论：当前推荐方案。

### 方案 C：直接引入 Spring AI EmbeddingModel

Spring AI 提供了成熟的 `EmbeddingModel` 抽象，方向是对的。

优点：

- Spring 生态；
- 抽象成熟；
- 未来可接多个 provider。

缺点：

- 引入新框架依赖；
- 当前项目目标是训练 provider abstraction、metadata、测试隔离和 evaluation 链路；
- 直接上框架可能降低当前版本的工程表达价值；
- 当前项目已有自己的 RAG / evaluation 结构，贸然引入会扩大边界。

结论：V2.5 暂不直接引入 Spring AI，但参考它的抽象方向。后续可以通过 adapter 接入。

### 方案 D：直接接本地 sentence-transformers worker

例如 `all-MiniLM-L6-v2`、BGE、E5 等开源模型可以用于 semantic search。

优点：

- 模型本身免费；
- 可离线；
- 适合私有化；
- 有开源生态。

缺点：

- Java 直接跑模型复杂；
- 可能需要 Python FastAPI worker；
- 需要模型部署、CPU/GPU 推理、批处理、超时和资源管理；
- 会把 V2.5 从 provider 抽象拖成模型服务部署版本。

结论：后续可以做 V2.5.2 Local Embedding Worker，不放在 V2.5。

## 4. 核心架构设计

V2.5 真实代码中新增或调整的核心结构包括：

- `EmbeddingProvider`
- `EmbeddingProperties`
- `EmbeddingProviderConfiguration`
- `MockEmbeddingClient`
- `OpenAiCompatibleEmbeddingProvider`
- `EmbeddingProviderException`
- `OpenAiEmbeddingRequest`
- `OpenAiEmbeddingResponse`
- `OpenAiEmbeddingHttpClient`
- `RestClientOpenAiEmbeddingHttpClient`
- `DocumentEmbeddingService`

### EmbeddingProvider

`EmbeddingProvider` 是统一抽象，核心方法包括：

- `EmbeddingResponse embed(EmbeddingRequest request)`
- `List<EmbeddingResponse> embedBatch(List<EmbeddingRequest> requests)`
- `String provider()`
- `String model()`
- `int dimension()`

它让 `DocumentEmbeddingService` 不再依赖具体 mock 实现，而是依赖可替换的 provider。

### MockEmbeddingClient

`MockEmbeddingClient` 继续保留 deterministic mock embedding。V2.5 修改后，它同时实现：

- `EmbeddingClient`
- `EmbeddingProvider`

它仍然是默认 provider，metadata 仍然是：

- provider：`mock`
- model：`mock-embedding-v1`
- dimension：`128`
- distanceMetric：`COSINE`

保留它的原因是确保 V2.2 / V2.3 / V2.4.x 的测试和本地开发不被外部 API 影响。

### OpenAiCompatibleEmbeddingProvider

`OpenAiCompatibleEmbeddingProvider` 是新增 real provider。它的职责是：

- 调用 OpenAI-compatible embeddings endpoint；
- 支持 `embed(...)` single input；
- 支持 `embedBatch(...)` batch input；
- 使用配置中的 model；
- 解析 response 中的 `data[].embedding`；
- 返回 `EmbeddingResponse`；
- 设置 provider 为 `openai`；
- 设置 model 为配置中的 model；
- 使用 response vector size 作为返回 dimension；
- 校验 configured dimension 和 response vector size 是否匹配。

测试中不真实访问 OpenAI，而是通过 fake `OpenAiEmbeddingHttpClient` 验证 request、response 和错误处理。

### EmbeddingProperties / Configuration

`EmbeddingProperties` 使用前缀：

```text
app.embedding
```

`EmbeddingProviderConfiguration` 负责选择 active provider：

- `app.embedding.provider=mock` 或为空时使用 `MockEmbeddingClient`；
- `app.embedding.provider=openai` 时使用 `OpenAiCompatibleEmbeddingProvider`。

`activeEmbeddingProvider` 是 `@Primary` bean，业务侧注入 `EmbeddingProvider` 时拿到当前配置选择的 provider。

### DocumentEmbeddingService

`DocumentEmbeddingService` 改为依赖 `EmbeddingProvider`。

生成 embedding 时：

- 使用 active provider 生成 chunk embedding；
- 保存 provider / model / dimension / distanceMetric / vector；
- 返回 `DocumentEmbeddingResponse` 中的 provider / model / dimension。

搜索时：

- query 使用 active provider 生成 query embedding；
- 如果 request 未传 embeddingProvider / embeddingModel，则默认使用 active provider 的 provider / model；
- repository 先按 provider / model 查候选；
- 再按 query embedding dimension 过滤；
- 最后计算 cosine similarity。

## 5. 配置设计

真实配置位于 `src/main/resources/application.properties`：

```properties
app.embedding.provider=mock
app.embedding.openai.base-url=https://api.openai.com/v1
app.embedding.openai.api-key=${OPENAI_API_KEY:}
app.embedding.openai.model=text-embedding-3-small
app.embedding.openai.timeout-ms=10000
app.embedding.openai.dimension=1536
```

配置含义：

- `app.embedding.provider`：选择 active provider，目前支持 `mock` / `openai`；
- `app.embedding.openai.base-url`：OpenAI-compatible base URL；
- `app.embedding.openai.api-key`：API key，从环境变量 `OPENAI_API_KEY` 注入，默认空；
- `app.embedding.openai.model`：embedding model；
- `app.embedding.openai.timeout-ms`：连接超时；
- `app.embedding.openai.dimension`：预期维度，用于校验 response vector。

必须强调：

- 默认 provider 是 `mock`；
- 没有 API key 时默认测试不失败；
- 不应把真实 key 写进代码、配置或 Git；
- `base-url` 可配置，因此可以兼容 OpenAI-compatible 服务，而不是把 endpoint 写死。

## 6. 为什么默认仍然是 Mock Provider

默认 provider 仍然是 mock，是为了工程稳定性。

原因：

- 本地测试稳定；
- CI 稳定；
- 不需要 API key；
- 不产生外部成本；
- 不依赖网络；
- 不会因为外部服务波动导致测试失败；
- 不会泄露密钥；
- 能保留之前 V2.2 / V2.3 / V2.4.x 的回归测试稳定性。

Real Provider 的定位是：

- 手工验收；
- 真实检索效果实验；
- 后续 V2.5.1 Mock vs Real benchmark comparison；
- 企业级真实 RAG 的前置能力。

这是一种常见取舍：默认路径要稳定，真实 provider 要可配置开启。

## 7. 为什么 API key 不能写死

API key 是敏感配置，不能进入代码仓库。

必须遵守：

- 不能写进代码；
- 不能把真实值写进 `application.properties`；
- 不能提交到 Git；
- 应该通过环境变量注入；
- 默认空值不影响 mock provider；
- `openai` provider 缺少 key 时给出明确错误。

当前代码中，`OpenAiCompatibleEmbeddingProvider` 会在调用前校验 `apiKey`，如果为空或 blank，会抛出 `EmbeddingProviderException`，错误信息包含 api key 相关说明。

## 8. provider / model / dimension 为什么重要

Embedding vector 不是普通数字数组，它属于某个模型的向量空间。

不同 provider / model 之间：

- vector dimension 可能不同；
- 向量空间不同；
- 即使 dimension 相同，也不代表可以直接比较；
- 混用会导致 cosine similarity 没有语义意义；
- search 结果会被污染。

因此必须保存和使用：

- provider；
- model；
- dimension。

当前项目做法：

- `EmbeddingResponse` 返回 provider / model / dimension；
- `DocumentEmbeddingService` 保存到 `document_chunk_embedding.embedding_provider`；
- 保存 `embedding_model`；
- 保存 `vector_dimension`；
- search 通过 provider / model 查询候选；
- search 通过 query embedding dimension 过滤；
- 避免不同 embedding space 混用。

这个边界是 V2.5 最重要的工程点之一。接入真实 provider 不只是“能调 API”，还要保证搜索时不会把不同模型的向量混在一起比较。

## 9. 为什么 search 不能混用不同 provider / model / dimension

假设：

- chunk A 用 mock embedding；
- chunk B 用 `text-embedding-3-small`；
- query 用 `text-embedding-3-small`。

如果搜索时同时拿 chunk A 和 chunk B 算 cosine similarity，那么 chunk A 的向量空间和 query 不一致，结果没有意义。

即使两个模型维度相同，也不能默认混用。原因是：

- 训练数据不同；
- 训练目标不同；
- 向量分布不同；
- 归一化方式可能不同；
- 相同维度不等于相同语义空间。

所以 V2.5 的规则是：

```text
同一次 search 必须在同一个 provider / model / dimension 下比较。
```

当前实现中，provider / model 是 repository 查询条件，dimension 是 service 层过滤条件。

## 10. OpenAI-compatible Provider 设计

真实 provider 类名：

```text
OpenAiCompatibleEmbeddingProvider
```

HTTP 相关类：

- `OpenAiEmbeddingHttpClient`
- `RestClientOpenAiEmbeddingHttpClient`
- `OpenAiEmbeddingRequest`
- `OpenAiEmbeddingResponse`

Endpoint：

```text
POST {baseUrl}/embeddings
```

request 方向：

```json
{
  "model": "text-embedding-3-small",
  "input": ["text1", "text2"]
}
```

response 方向：

```json
{
  "data": [
    {
      "index": 0,
      "embedding": [0.1, 0.2]
    }
  ],
  "model": "text-embedding-3-small"
}
```

当前实现支持：

- single input：`embed(...)`；
- batch input：`embedBatch(...)`；
- response 按 `index` 排序后映射回结果；
- metadata：provider = `openai`，model = 配置 model，dimension = vector size；
- distanceMetric = `COSINE`；
- fake HTTP client 单元测试，不访问外部网络。

错误处理包括：

- apiKey 缺失；
- HTTP / client exception；
- response data 为空；
- vector 为空；
- batch response 数量和 input 数量不匹配；
- dimension mismatch。

这些错误不应该被吞掉，因为 embedding 失败通常意味着配置、外部服务、模型返回或数据格式有问题，必须明确暴露。

## 11. 为什么不直接用 Spring AI

Spring AI 的 `EmbeddingModel` 是成熟抽象，验证了 provider abstraction 这个方向。

但 V2.5 暂不直接引入 Spring AI，原因是：

- 当前项目重点是自己设计 provider 边界；
- 需要显式处理 metadata、测试隔离、错误处理和 search filtering；
- 直接使用框架容易掩盖这些工程取舍；
- 当前项目已经有自己的 RAG / evaluation 结构；
- 引入新框架会扩大版本边界。

后续可以用 adapter 方式接入 Spring AI：

```text
SpringAiEmbeddingProvider implements EmbeddingProvider
```

这样既保留当前项目的 provider abstraction，也能复用 Spring AI 的 provider 生态。

## 12. 为什么不现在做本地 embedding worker

本地开源 embedding 模型适合后续私有化和低成本路线，例如 sentence-transformers、BGE、E5 等。

但 V2.5 不做本地 embedding worker，因为它会引入另一类复杂度：

- Java 直接跑模型复杂；
- 可能需要 Python FastAPI service；
- 需要模型下载和部署；
- 需要 CPU/GPU 推理资源管理；
- 需要 batch queue；
- 需要超时、重试、健康检查；
- 需要模型版本管理。

这会让 V2.5 从“provider abstraction”变成“模型服务部署”版本。更合理的路线是后续单独做：

```text
V2.5.2 Local Embedding Worker
```

## 13. 为什么不做 Vector DB / Rerank / Hybrid Search

V2.5 的唯一变量是 embedding provider。

如果同时做 Vector DB：

- 会引入 ANN index；
- 引入索引参数；
- 引入近似召回误差；
- 引入部署复杂度；
- 难以判断变化来自 provider 还是索引。

如果同时做 Rerank：

- 会改变排序；
- 直接影响 NDCG / MRR；
- 难以判断提升来自 embedding 还是 rerank。

如果同时做 Hybrid Search：

- 会加入 BM25 / lexical retrieval；
- 增加另一个变量；
- 指标变化难以归因。

因此这些能力后置：

- V2.6 Vector DB Selection；
- V2.8 Rerank；
- 后续 Hybrid Search。

这体现的是实验纪律：一次只改变一个核心变量。

## 14. 数据库设计与兼容性

V2.5 没有新增 Flyway，也没有新增数据库表。

原因是现有 `document_chunk_embedding` 已经支持保存：

- `embedding_provider`
- `embedding_model`
- `vector_dimension`
- `distance_metric`
- `embedding_vector`

V2.5 复用这些字段，而不是新增：

- `embedding_provider` 表；
- `embedding_model` 表；
- `embedding_job` 表；
- `embedding_cache` 表；
- `vector_index` 表。

provider / model / dimension 是 embedding metadata，也是检索正确性的基础。当前阶段只需要把这些 metadata 用好，不需要过早设计 provider registry 或 embedding job persistence。

## 15. API 设计

V2.5 没有新增生产 API。

继续复用已有能力：

- document embedding generation；
- document search；
- retrieval evaluation / benchmark runner。

现有链路中：

- `EmbeddingResponse` 返回 provider / model / dimension；
- `DocumentEmbeddingResponse` 返回 embedding provider / model / dimension / distanceMetric；
- `DocumentSearchResultResponse` 返回 embeddingProvider / embeddingModel / distanceMetric；
- `RetrievalEvaluationService` 继续复用 document search。

V2.5 的目标是 provider 架构升级，不是新暴露一组 embedding management API。后续如果要做企业级 provider 管理，再考虑 provider 管理 API、成本统计和 dashboard。

## 18. 工程权衡

### 18.1 为什么不直接写死 OpenAI API

直接写死会让代码短期变快，但长期变差：

- provider 耦合；
- 测试不稳定；
- 后续难扩展；
- API key 风险；
- 无法做 provider 对比；
- 无法保留 mock 回归测试路径。

V2.5 选择 provider abstraction，是为了让真实 provider 成为可插拔能力，而不是把 OpenAI 调用硬塞进现有 mock 类。

### 18.2 为什么保留 Mock

Mock 是工程稳定器：

- deterministic；
- 无成本；
- 无外部依赖；
- 无密钥；
- 适合 CI；
- 适合回归测试；
- 能保证 V2.2 / V2.3 / V2.4.x 不被真实服务波动影响。

真实 provider 应该是可选能力，而不是默认测试路径。

### 18.3 为什么 Real Provider 通过配置开启

通过配置开启的好处：

- 不影响默认环境；
- 可手工验收；
- 可按环境切换；
- 不让外部 API 影响主测试；
- 适合未来多 provider 扩展。

当前配置方式：

```properties
app.embedding.provider=openai
app.embedding.openai.api-key=${OPENAI_API_KEY:}
```

### 18.4 为什么 metadata 是核心

embedding vector 必须绑定 provider / model / dimension。

没有 metadata，系统无法判断：

- query vector 属于哪个 provider；
- chunk vector 属于哪个 provider；
- 两者是否能比较；
- vector dimension 是否匹配；
- 检索结果是否被跨模型污染。

metadata 是检索正确性的工程保障，而不是展示字段。

### 18.5 为什么不做 Vector DB / Rerank

V2.5 只改变 embedding provider。

Vector DB、Rerank、Hybrid Search 都会改变 retrieval behavior。如果同时做，就无法判断指标变化来自哪里。先把 provider 抽象稳定，再用 benchmark 做 Mock vs Real，再进入 Vector DB 和 Rerank，是更可解释的演进路线。

## 24. 面试高强度追问与回答

### Q1：为什么不能直接在 MockEmbeddingClient 里写 OpenAI 调用？

因为 `MockEmbeddingClient` 的职责是 deterministic mock embedding，它承担的是本地开发和 CI 稳定性。如果直接把 OpenAI 调用塞进去，mock 和 real 会混在一个类里，测试会受网络、API key、费用和外部服务稳定性影响。更严重的是，后续要接其他 provider 时会继续往这个类里堆分支，最后变成一个难维护的 God class。V2.5 选择 `EmbeddingProvider` 抽象，是为了把 mock、OpenAI-compatible provider、未来本地模型或企业私有 provider 都放到统一接口下。

### Q2：为什么要抽象 EmbeddingProvider？

Embedding 是 RAG pipeline 的基础能力，但 provider 会变化。今天是 mock 和 OpenAI-compatible endpoint，后续可能是 Spring AI、本地 sentence-transformers worker、企业私有模型服务。抽象 `EmbeddingProvider` 后，业务侧的 `DocumentEmbeddingService` 只依赖 `embed`、`embedBatch`、`provider`、`model`、`dimension`，不关心底层怎么调用。这样可以保证 provider 可替换、测试可隔离、metadata 可统一管理。

### Q3：为什么 MockEmbeddingClient 还要保留？

Mock provider 是回归测试的稳定基线。真实 embedding API 会涉及网络、密钥、费用、限流和服务可用性，不能让 CI 依赖这些变量。MockEmbeddingClient deterministic、免费、不需要外部服务，适合验证 RAG 链路、evaluation harness 和基础 search 行为。因此 V2.5 是新增真实 provider，而不是替换掉 mock。

### Q4：为什么默认 provider 仍然是 mock？

默认 mock 是为了让项目开箱可测。任何开发者 clone 项目后，不需要 API key、不需要联网、不需要付费，就能跑通 `.\mvnw.cmd test`。真实 provider 通过配置开启，这样本地、CI、手工验收和生产-like 环境可以分开。默认路径稳定，真实路径可选，是更稳的工程设计。

### Q5：为什么 CI 不能调用真实 embedding API？

CI 调外部 API 会引入不确定性：网络抖动、服务限流、API key 配置、账号额度、费用、模型返回变化都可能导致测试失败。单元测试应该验证代码逻辑，不应该依赖第三方服务可用性。V2.5 的 OpenAI provider 测试使用 fake HTTP client，验证 request 构造、response 解析和异常处理；真实服务调用留给手工验收或单独的 external test。

### Q6：API key 如何管理？

API key 通过环境变量 `OPENAI_API_KEY` 注入到 `app.embedding.openai.api-key=${OPENAI_API_KEY:}`。代码和配置里不写真实 key，默认空值不影响 mock provider。如果配置 `app.embedding.provider=openai` 但 key 为空，`OpenAiCompatibleEmbeddingProvider` 会抛出明确的 `EmbeddingProviderException`。这样既避免泄露，也能在配置错误时快速定位问题。

### Q7：OpenAI-compatible provider 如何配置？

核心配置是 `app.embedding.provider=openai`，再配 `app.embedding.openai.base-url`、`api-key`、`model`、`timeout-ms` 和 `dimension`。其中 `base-url` 默认是 `https://api.openai.com/v1`，实际请求路径是 `/embeddings`。因为 baseUrl 可配置，所以也能对接 OpenAI-compatible 服务，而不是只能连接官方 endpoint。

### Q8：provider / model / dimension 为什么重要？

embedding vector 属于特定模型的向量空间。provider 表示来源，model 表示具体模型，dimension 表示向量长度。不同模型的向量不能随便比较，否则 cosine similarity 没有语义意义。V2.5 把这些 metadata 保存到 `document_chunk_embedding`，并在 search 时过滤，目的是避免跨 provider、跨 model、跨 dimension 的混用。

### Q9：为什么不同 embedding model 的 vector 不能混用？

不同 embedding model 的训练数据、训练目标、向量分布和归一化方式可能不同。即使它们都输出 1536 维向量，也不代表这些维度有相同含义。query 用模型 A，chunk 用模型 B，再计算 cosine similarity，本质上是在两个不同空间里硬算距离，结果不可解释。

### Q10：如果两个 embedding model 的维度相同，能不能混用？

默认不能。dimension 相同只是数学上可以计算点积或 cosine，不代表语义空间兼容。只有在明确知道两个模型共享同一向量空间或经过对齐时，才可以考虑混用。当前项目没有做这种空间对齐，所以规则是 provider / model / dimension 都要一致或至少明确过滤。

### Q11：search 如何避免混用不同 provider / model / dimension？

`DocumentEmbeddingService.search(...)` 先确定 provider 和 model。如果 request 没传，就使用 active `EmbeddingProvider` 的 provider / model。repository 查询时按 provider / model 过滤候选 embedding；然后 service 再用 query embedding 的 dimension 过滤候选。只有通过这些过滤的 embedding 才会参与 cosine similarity。

### Q12：为什么不直接使用 Spring AI？

Spring AI 的方向是成熟的，但 V2.5 的目标是自己掌握 provider 抽象、metadata、测试隔离和 search filtering。直接引入 Spring AI 会带来新依赖，也可能掩盖当前版本想展示的工程边界。后续可以写 `SpringAiEmbeddingProvider implements EmbeddingProvider`，用 adapter 接入，而不是现在整体迁移。

### Q13：为什么不现在做本地 sentence-transformers worker？

本地模型 worker 是另一条路线，涉及模型部署、Python service、推理资源、batch、超时、健康检查和模型版本管理。它的价值很高，尤其适合私有化，但它不是 V2.5 的核心。V2.5 先把 provider abstraction 建好，后续 V2.5.2 再接 Local Embedding Worker 更清晰。

### Q14：为什么不现在做 Vector DB？

Vector DB 会引入索引、ANN、召回近似误差、部署和运维成本。V2.5 只想改变 embedding provider，如果同时引入 Vector DB，后续指标变化无法归因。应该先完成 Real Embedding Provider，再用 benchmark 对比，之后 V2.6 再评估 exact cosine search 与 Vector DB 的差异。

### Q15：为什么不现在做 Rerank？

Rerank 会改变排序结果，直接影响 NDCG@K 和 MRR。如果 V2.5 同时接 embedding 和 rerank，就无法判断排序变化来自 embedding 还是 rerank。Rerank 应该在基础 retrieval 和 evaluation 稳定后单独引入，并用 NDCG@K / MRR 评估它是否改善排序。

### Q16：OpenAI provider 失败如何处理？

当前 provider 会对常见错误显式失败：apiKey 缺失、client exception、response data 为空、vector 为空、batch 数量不匹配、dimension mismatch。错误使用 `EmbeddingProviderException` 表达，不吞异常。这样在手工验收或未来生产环境中，可以快速区分配置问题、外部服务问题和返回格式问题。

### Q17：batch embedding 要注意什么？

batch embedding 要保证输入顺序和返回结果顺序对齐。OpenAI-compatible response 中有 `index`，当前实现会按 index 排序后转换为 `EmbeddingResponse`。还要校验 response 数量和 input 数量一致，否则就可能把文本和向量错配。batch 还需要考虑超时、请求大小、成本和部分失败处理，这些可以后续继续增强。

### Q18：V2.4.x benchmark runner 如何复用到 V2.5？

V2.4.x 已经有 fixed corpus、benchmark JSON、expectedEvidenceIds -> expectedChunkIds mapper 和 `RetrievalEvaluationService`。V2.5 接入 real provider 后，可以用同一套 benchmark 在不同 provider 配置下运行。这样 Mock vs Real 的差异可以通过 Recall@K、Precision@K、NDCG@K、MRR、ContextPrecision@K 观察，而不是靠主观感觉。

### Q19：如果 Real Embedding 指标没有提升，说明什么？

这不一定说明 Real Embedding 没价值。可能是 benchmark 太小、query 太直接、corpus 不够真实、chunking 策略不合适、topK 不合适，或者 mock scorer 在该 seed 上刚好表现不错。也可能说明真实模型和当前数据领域不匹配。工程上应该分析 per-case result，而不是只看 summary，也不能预设 Real 一定更好。

### Q20：企业场景如何选择 embedding provider？

企业要综合看质量、成本、延迟、合规和部署方式。如果数据可以出公网，可以选 SaaS embedding provider；如果数据敏感，可以选私有化模型服务；如果成本敏感，可以选择小模型或本地模型；如果质量要求高，可以用 benchmark 对比多个模型。关键是用企业自己的 query-evidence benchmark 评估，而不是只看通用榜单。

### Q21：embedding 成本如何控制？

可以从几个方向控制：batch embedding、避免重复生成、后续引入 embedding cache、按文档变更增量更新、限制最大文档大小、选择不同模型档位、按用户 plan 控制 topK 和模型、记录 token / 字符消耗。V2.5 还没有实现成本 tracking，但 provider abstraction 为后续接入 usage metadata 和 cost metrics 留出了位置。

### Q22：如何防止 API key 泄露？

不要把 key 写进代码、配置或测试数据；通过环境变量或 secret manager 注入；日志中不要打印 Authorization header；CI secret 权限要最小化；默认测试不用真实 key；错误信息不要包含 key 值。当前项目使用 `${OPENAI_API_KEY:}`，并且默认 provider 是 mock，可以避免无意中在 CI 调用真实 API。

### Q23：后续如何接本地私有 embedding 服务？

可以新增 `LocalEmbeddingProvider` 或 `HttpEmbeddingProvider`，实现同一个 `EmbeddingProvider` 接口。它可以调用本地 Python FastAPI worker，返回 vector、model、dimension。这样 `DocumentEmbeddingService` 不需要变化，仍然保存 provider / model / dimension，并用相同 search 和 evaluation 逻辑验证效果。

### Q24：如果 provider 返回的 dimension 和配置不一致怎么办？

当前 `OpenAiCompatibleEmbeddingProvider` 会校验 vector size。如果配置了 dimension 且大于 0，response vector size 必须匹配，否则抛出 `EmbeddingProviderException`。这是必要的 fail-fast 设计，因为 dimension mismatch 往往意味着模型配置错、endpoint 返回不符合预期，继续保存会污染后续 search。

### Q25：为什么 V2.5 不写真实指标提升？

因为 V2.5 只实现 provider 架构和 OpenAI-compatible provider，不做 Mock vs Real benchmark comparison。没有运行固定 benchmark、没有对比结果、没有统计，就不能写 Recall@K、NDCG@K 或其他指标提升。面试中诚实区分“能力接入”和“效果评估”很重要，这也是 V2.5 和 V2.5.1 的边界。
