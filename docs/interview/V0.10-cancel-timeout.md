# V0.10 面试文档：取消与超时

## 1. 本版本解决的问题

V0.10 解决用户需要停止任务、系统不能让任务无限 RUNNING 的问题。AI 任务可能被用户取消，也可能因执行线程卡住而超时。取消和超时必须和执行线程协作，避免任务被取消或标记失败后又被执行线程写回 SUCCESS。

## 2. 为什么现在做这个版本

它依赖状态机、Consumer 执行、重试和幂等。只有任务已经异步执行，取消和超时才有意义。它为后续长耗时 LLM 调用和用户体验做准备。此时不做线程强杀，是因为 Java 线程强制中断风险高。

## 3. 核心需求边界

本版本支持 POST /tasks/{taskId}/cancel，PENDING / RETRY_PENDING 可直接取消，RUNNING 通过协作式检查取消。新增 timeout_seconds、timeout_at 和 TaskTimeoutScheduler。不做 Future.cancel、不做强制 kill、不做分布式超时协调。

## 4. 方案选择

取消通过状态机流转到 CANCELLED，并清空 nextRetryAt。RUNNING 执行期间 TaskExecutionService 定期检查 isTaskCancelled。超时由 Scheduler 扫描 RUNNING 且 timeoutAt 到期的任务，标记 FAILED，并要求执行线程写 SUCCESS 前再检查任务仍是 RUNNING。

## 5. 替代方案与暂时不选的原因

替代方案一是强制中断线程。当前不选，因为可能造成资源未释放和状态不一致。

替代方案二是不做超时，只依赖执行线程返回。当前不选，因为任务可能无限卡住，影响用户体验和资源。

## 6. 数据库设计

task 表新增 timeout_seconds 和 timeout_at。timeout_seconds 表示最大执行秒数，timeout_at 在进入 RUNNING 时设置。取消时清空 nextRetryAt，避免 RETRY_PENDING 被 Scheduler 再次派发。

## 7. API 设计

Method: POST

Path: /tasks/{taskId}/cancel

Response: 返回 TaskDetailResponse，状态变为 CANCELLED。SUCCESS / FAILED 等终态取消应返回 400。

## 8. 核心流程

取消流程：
用户 POST cancel
-> TaskService 校验状态
-> 状态改 CANCELLED
-> 执行线程检查到取消后 return

超时流程：
任务进入 RUNNING 设置 timeoutAt
-> TaskTimeoutScheduler 扫描到期 RUNNING
-> markTaskTimedOut 改 FAILED
-> 执行线程写 SUCCESS 前发现不再 RUNNING 并 return

## 9. 测试方案

建议覆盖 PENDING/RETRY_PENDING/RUNNING 取消、SUCCESS/FAILED 取消失败、超时后 RUNNING -> FAILED、执行线程不覆盖 CANCELLED/FAILED。手动验收可创建任务后立即取消或手动修改 timeout_at 为过去时间。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是取消/超时和执行线程并发。Scheduler 或用户可能先把任务改为 CANCELLED/FAILED，执行线程随后成功返回，如果不检查 isTaskRunning，就会覆盖终态。

## 12. 本版本亮点

- 支持等待中和执行中任务取消。
- 用协作式检查避免线程强杀风险。
- 增加超时扫描，防止任务无限 RUNNING。
- 执行线程写结果前二次校验，避免并发覆盖。

## 13. 简历 bullet

### 简洁版

实现任务取消与超时机制，支持 PENDING/RETRY_PENDING/RUNNING 取消和 RUNNING 超时失败处理。

### 强化版

为异步 AI 任务系统设计协作式取消和超时扫描机制，通过执行线程周期检查与写终态前校验，避免取消或超时任务被后续执行结果覆盖。

## 14. 面试高强度追问与回答

Q1：为什么 RUNNING 取消是协作式？

A1：强制 kill 线程风险高，协作式取消让执行逻辑在安全点检查状态并退出。

Q2：为什么取消后要清空 nextRetryAt？

A2：否则 RETRY_PENDING 任务取消后仍可能被 RetryScheduler 扫描并重新派发。

Q3：timeoutAt 什么时候设置？

A3：任务真正进入 RUNNING 时设置，避免还未执行的任务被误判超时。

Q4：超时为什么标记 FAILED 而不是 CANCELLED？

A4：超时是系统判定执行失败，CANCELLED 是用户主动取消，语义不同。

Q5：执行线程为什么写 SUCCESS 前要检查 RUNNING？

A5：因为任务可能已被取消或超时标记 FAILED，不检查会覆盖终态。

Q6：为什么不实现 Future.cancel？

A6：它需要线程池管理和中断配合，当前阶段复杂度高且不一定安全。

Q7：取消和重试如何配合？

A7：取消 RETRY_PENDING 后清空 nextRetryAt，使任务不再被重试 Scheduler 派发。

Q8：当前局限是什么？

A8：取消检查依赖执行逻辑主动轮询，真实外部 LLM 调用中还需要请求超时和 provider cancel 能力。
