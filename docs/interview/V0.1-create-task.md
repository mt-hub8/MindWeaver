# V0.1 面试文档：创建 AI 任务

## 1. 本版本解决的问题

V0.1 解决的是“AI 长任务如何先被系统接住”的问题。用户提交 prompt 后，系统不直接同步执行，而是先把任务作为一条可追踪的数据记录保存下来，返回 taskId 和初始状态 PENDING。它的系统价值在于建立任务生命周期的起点，为后续异步调度、状态查询、失败处理、重试、取消、超时和 LLM 执行提供统一入口。

## 2. 为什么现在做这个版本

创建任务是所有后续能力的前置条件。没有 task 记录，就无法查询状态、写事件、投递 MQ、执行 LLM 或做 RAG。V0.1 不提前做 MQ / LLM / RAG / 用户系统，是为了先验证最小闭环：请求能进来，核心数据能落库，客户端能拿到任务标识。V0.1 越克制，后续扩展越自然，因为每个能力都可以围绕 taskId 增量接入。

## 3. 核心需求边界

本版本只做 POST /tasks、prompt 入库、初始状态 PENDING、返回 taskId 和 status。不做异步执行、不做 RabbitMQ、不做真实 LLM、不做 RAG、不做多用户权限。保持克制的原因是先明确任务模型和 API 契约，避免在基础对象还不稳定时引入复杂执行链路。

## 4. 方案选择

核心方案是用 task 表保存当前任务记录，prompt 作为原始用户输入，status 初始化为 PENDING。PENDING 表示任务已经被系统接收，但尚未被执行者处理。创建接口返回 taskId，而不是返回最终结果，体现出异步任务系统的基本模型：提交和执行解耦。

## 5. 替代方案与暂时不选的原因

替代方案一是同步执行并直接返回结果。当前不选，因为 AI 任务可能耗时长、失败率高，同步 HTTP 请求容易超时，也不利于后续重试。

替代方案二是创建时直接调用 MQ 或 LLM。当前不选，因为任务基础模型还未沉淀，过早接入执行侧会让排错边界变复杂。

## 6. 数据库设计

本版本涉及 task 表的基础字段，核心包括 id、prompt、status、created_at、updated_at。id 是任务追踪主键，prompt 保存用户输入，status 保存当前生命周期状态。后续可以扩展 error_message、retry_count、timeout_at、result_content、requested_model 等字段。

## 7. API 设计

Method: POST

Path: /tasks

Request:

```json
{
  "prompt": "normal task"
}
```

Response 包含 taskId 和 status，status 初始为 PENDING。设计上创建接口只表示“任务已接收”，不承诺立即执行完成。

## 8. 核心流程

用户请求
-> TaskController 接收 CreateTaskRequest
-> TaskService 创建 TaskEntity
-> Repository 保存 task
-> 返回 taskId 和 PENDING

## 9. 测试方案

建议覆盖 CreateTaskRequest 入参、prompt 正常保存、status 初始化为 PENDING、返回 taskId、空 prompt 的边界处理。API 手动测试可调用 POST /tasks 后查询数据库 task 表。异常路径应关注非法请求体和数据库保存失败。

## 11. 工程难点

难点不是代码复杂度，而是边界克制。V0.1 必须避免把执行、调度、模型调用混入创建动作，否则后续很难拆分职责。另一个难点是状态命名要能支撑未来扩展，PENDING 必须准确表达“已创建但未执行”。

## 12. 本版本亮点

- 用 taskId 建立异步任务系统的追踪锚点。
- 用 PENDING 明确任务生命周期起点。
- 创建接口只负责接收任务，为后续 MQ、状态机、事件表留出清晰边界。
- 没有过早引入 LLM/RAG，降低早期系统复杂度。

## 13. 简历 bullet

### 简洁版

实现 AI 任务创建接口，支持 prompt 入库、任务状态初始化和 taskId 返回，为异步执行链路建立基础任务模型。

### 强化版

从最小闭环切入设计 AI 任务编排系统，以 task 表和 PENDING 初始状态建立任务生命周期入口，为后续 RabbitMQ 调度、状态机、重试和 LLM 执行提供统一追踪基础。

## 14. 面试高强度追问与回答

Q1：为什么创建任务不直接执行？

A1：因为 AI 任务可能耗时较长，同步执行会阻塞 HTTP 请求并增加超时风险。先创建任务再异步执行，可以让请求快速返回 taskId，也让后续重试、取消、状态查询成为可能。

Q2：为什么初始状态是 PENDING？

A2：PENDING 表示任务已被系统接收但还未被执行者 claim。这个状态能自然衔接后续 RUNNING、SUCCESS、FAILED 等状态，也能区分“未开始”和“执行中”。

Q3：为什么先做最小闭环？

A3：最小闭环能验证领域模型和 API 契约。只有任务创建稳定后，后续 MQ、Consumer、LLM、RAG 才能以 taskId 为中心逐步扩展。

Q4：如果 prompt 很长怎么办？

A4：当前阶段先保存原始 prompt。后续可以增加长度限制、TEXT 字段、输入校验、文件化输入或 prompt 引用表，避免单表字段过大。

Q5：如果支持多用户怎么扩展？

A5：可以在 task 表增加 user_id / tenant_id，并在查询和取消接口中加入权限校验。V0.1 不提前做多用户，是为了不把权限系统混入任务模型验证。

Q6：为什么不先做 RAG？

A6：RAG 依赖文档、chunk、embedding、检索和 prompt 拼接。如果没有任务模型，RAG 的执行结果和失败状态无法被可靠追踪。

Q7：创建接口返回结果还是返回 taskId？

A7：异步系统应该返回 taskId。最终结果通过查询接口获取，这样可以支持长耗时执行和前端轮询。

Q8：V0.1 的局限是什么？

A8：它只能创建任务，不能执行、查询完整生命周期或处理失败。但这个局限是刻意保留的，用来保证基础模型简单稳定。

## 15. 后续演进路线

V0.1 支撑 V0.2 查询任务、V0.3 状态机、V0.5 RabbitMQ 投递和 V1.x LLM 执行。后续所有能力都围绕 taskId 进行状态更新、事件记录和结果持久化。
