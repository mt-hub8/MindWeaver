# V0.6 面试文档：Consumer 模拟执行

## 1. 本版本解决的问题

V0.6 解决“消息被投递后谁来执行任务”的问题。Consumer 从 RabbitMQ 收到 taskId 后调用 TaskExecutionService，模拟任务执行，并完成 PENDING -> RUNNING -> SUCCESS 的状态流转。它让异步链路从投递走到执行闭环。

## 2. 为什么现在做这个版本

它依赖 V0.5 RabbitMQ 投递和前面的状态机。没有 Consumer，MQ 中的消息不会产生业务效果。此时不接真实 LLM，是为了先验证 Worker、状态更新和执行服务边界，为 V1.0 LLM Client 做准备。

## 3. 核心需求边界

本版本只做 Consumer 接收 taskId、TaskExecutionService 模拟执行、成功后更新状态。不做真实模型调用、不做重试、不做失败持久化、不做并发幂等。保持克制是为了先跑通执行主路径。

## 4. 方案选择

Consumer 只负责接收消息和委托执行，不直接写复杂业务。TaskExecutionService 承载执行流程，TaskService 负责状态更新。这样 MQ 层、执行层、状态层职责分离。

## 5. 替代方案与暂时不选的原因

替代方案一是 Consumer 直接更新所有业务状态。当前不选，因为会让消息层和业务层耦合。

替代方案二是直接接真实 LLM。当前不选，因为外部 API 会引入网络、费用、密钥和失败复杂度，不利于先验证异步执行骨架。

## 6. 数据库设计

本版本未新增数据库结构，主要复用 task.status 和 task_event。执行成功会更新 task 当前状态，并可记录 STATUS_CHANGED。

## 7. API 设计

本版本主要是内部执行能力，未新增用户 API。外部仍通过 POST /tasks 创建任务，通过 GET /tasks/{taskId} 查询状态。

## 8. 核心流程

RabbitMQ 消息
-> TaskDispatchConsumer 收到 taskId
-> TaskExecutionService 执行
-> TaskService 更新 PENDING -> RUNNING
-> 模拟 sleep / 处理
-> RUNNING -> SUCCESS
-> 查询接口看到 SUCCESS

## 9. 测试方案

建议覆盖 Consumer 收到消息后调用 TaskExecutionService、正常任务最终 SUCCESS、状态事件顺序正确。异常路径暂可建议覆盖，后续 V0.7 专门处理失败。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是职责分层。Consumer 不应该变成业务大杂烩，执行流程也不能绕过 TaskService，否则状态机和事件记录会失效。

## 12. 本版本亮点

- 跑通 MQ 到 Worker 的异步执行闭环。
- 把消息消费和任务执行解耦。
- 用 Mock Execution 降低外部依赖。
- 为 LLM Client 接入预留执行服务边界。

## 13. 简历 bullet

### 简洁版

实现 RabbitMQ Consumer 与 TaskExecutionService，跑通任务从 PENDING 到 RUNNING 再到 SUCCESS 的异步执行闭环。

### 强化版

设计 Consumer 轻量委托与执行服务分层，通过 Mock Execution 验证异步任务执行主链路，为后续 LLM 调用、失败处理、重试和幂等控制奠定基础。

## 14. 面试高强度追问与回答

Q1：为什么 Consumer 不直接写复杂业务？

A1：Consumer 是消息入口，直接写复杂业务会导致 MQ 层和业务规则耦合。委托给 TaskExecutionService 更易测试和扩展。

Q2：为什么用 Mock Execution？

A2：Mock 能避免真实 LLM 的网络和成本干扰，先验证执行链路、状态流转和异步调度。

Q3：PENDING -> RUNNING 为什么必要？

A3：RUNNING 表示任务已被执行者接手。它让查询接口能区分等待中和执行中，也为取消、超时和幂等提供依据。

Q4：执行成功后为什么写 SUCCESS？

A4：SUCCESS 是终态，表示任务已完成，后续重复消息不应再执行。

Q5：如果 Consumer 崩溃怎么办？

A5：早期版本先不深挖 ack/nack 和重投递。后续需要通过 RabbitMQ 机制、幂等和 Outbox 提升可靠性。

Q6：如何为真实 LLM 做准备？

A6：TaskExecutionService 已经是执行入口，后续只需把模拟执行替换为 LlmClient 调用，而不是改 Consumer。

Q7：Mock Execution 有什么局限？

A7：它不能代表真实模型延迟、错误类型和 token usage，但足以验证系统控制流。

Q8：如何手动验收？

A8：启动 RabbitMQ 和应用，POST /tasks 创建任务，等待 Consumer 消费，再 GET /tasks/{id} 确认状态变为 SUCCESS。

## 15. 后续演进路线
