# V3.5 Ingestion Performance Summary & Failure Analytics 面试文档

## 1. 版本摘要

V3.5 的版本名称是：

```text
V3.5 Ingestion Performance Summary & Failure Analytics
```

中文可以表述为：

```text
文档摄入性能汇总与失败分析
```

这个版本的核心目标是：基于 V3.4 的 ingestion event timeline（摄入事件时间线），对整个文档摄入系统做系统级统计分析。

V3.4 解决的是单个任务的问题：

```text
某个任务经历了哪些步骤？
每一步耗时多久？
哪一步失败？
失败原因是什么？
是否发生过重试？
```

V3.5 解决的是系统整体问题：

```text
最近有多少文档摄入任务？
成功率是多少？
失败率是多少？
平均处理耗时是多少？
哪个阶段最慢？
最常见失败原因是什么？
最近失败任务有哪些？
哪些任务处理特别慢？
```

也就是说，V3.5 把项目从“单任务调试追踪”升级到“系统级摄入质量分析”。

---

## 2. 为什么现在做 V3.5

前面几个版本已经形成了完整链路：

```text
V3.1：产品化重置，让用户理解系统能做什么；
V3.2：上传文档到提问，打通 Upload-to-Ask 产品闭环；
V3.3：异步文档摄入，让上传任务可排队、可追踪、可重试；
V3.4：摄入事件时间线，让单个任务的每一步可观察、可调试；
```

到了 V3.5，系统已经拥有足够的数据基础：

```text
document_ingestion_task；
document_ingestion_event；
status；
step；
durationMs；
errorCode；
errorMessage；
retryCount；
createdAt；
completedAt。
```

这些数据不应该只用于查看单个任务。更有价值的是聚合分析：

```text
最近 24 小时系统处理得怎么样？
embedding 是不是主要耗时瓶颈？
PDF 无可提取文本是不是最常见失败原因？
最近有哪些失败任务需要处理？
哪些任务特别慢？
```

这就是从 debug trace（调试追踪）到 analytics（分析）。

## 3. 本版本解决的问题

### 3.1 解决只能单个任务排查的问题

V3.4 的事件时间线适合点进某一个任务查看细节。但如果系统有几十个任务，逐个点开排查效率很低。

V3.5 通过 analytics summary（分析汇总）直接给出整体视图：

```text
总任务数；
成功数；
失败数；
待处理数；
处理中数；
成功率；
失败率；
平均耗时；
阶段耗时；
失败原因；
慢任务。
```

这让开发者可以快速判断系统运行质量。

### 3.2 解决性能瓶颈不清的问题

文档摄入包括多个阶段：

```text
文档切分；
生成文档向量；
写入知识库索引。
```

如果只看总耗时，无法知道具体慢在哪里。V3.5 基于 event timeline 中的 durationMs 聚合阶段平均耗时：

```text
文档切分平均耗时；
生成文档向量平均耗时；
写入知识库索引平均耗时。
```

如果结果显示“生成文档向量平均耗时明显最高”，说明优化方向可能是 embedding provider、batch embedding、embedding cache、模型调用并发或 worker 性能。

### 3.3 解决失败原因不可汇总的问题

单个失败任务只能看到单次 errorCode。V3.5 基于 TASK_FAILED event 聚合 top failure reasons（常见失败原因）：

```text
PDF_NO_EXTRACTABLE_TEXT：PDF 没有可提取文字；
EMBEDDING_PROVIDER_ERROR：生成文档向量失败；
VECTOR_STORE_WRITE_ERROR：写入知识库索引失败；
UNKNOWN_ERROR：未知错误。
```

如果大部分失败来自 PDF 无可提取文本，下一步应该优化用户提示或考虑 OCR。如果大部分失败来自 embedding provider，说明模型服务稳定性需要优先处理。

### 3.4 解决慢任务不可发现的问题

有些任务虽然成功了，但耗时很长。V3.5 返回 slowTasks（处理较慢的任务），并计算 bottleneckStage（主要耗时阶段）。

例如：

```text
product-manual.pdf，总耗时 18.2 秒，主要耗时阶段：生成文档向量；
api-doc.md，总耗时 12.7 秒，主要耗时阶段：写入知识库索引。
```

这样可以定位性能异常任务。

## 4. 核心设计

### 4.1 IngestionAnalyticsService

V3.5 新增核心服务：

```text
IngestionAnalyticsService
```

它负责：

```text
读取 document_ingestion_task；
读取 document_ingestion_event；
按时间窗口过滤任务；
统计任务数量；
计算成功率 / 失败率；
计算平均总耗时；
聚合阶段平均耗时；
聚合常见失败原因；
查询最近失败任务；
查询慢任务列表；
生成 analytics summary DTO。
```

重要原则：聚合逻辑不要写在 Controller 里。Controller 只负责接收参数和返回结果，Analytics Service 才是分析逻辑的核心。

### 4.2 不新增 analytics summary 表

V3.5 不新增统计表，不做定时聚合。它直接基于已有表实时查询：

```text
document_ingestion_task；
document_ingestion_event。
```

原因是：当前数据量小，实时查询足够；可以避免引入定时任务、summary 表和原始事件表的数据一致性问题，也保持版本边界克制。

后续如果任务量变大，可以再做 daily summary table（每日汇总表）、scheduled aggregation（定时聚合）或 metrics export（指标导出）。

### 4.3 时间窗口设计

V3.5 支持简单时间窗口：

```text
24h：最近 24 小时；
7d：最近 7 天；
30d：最近 30 天；
all：全部时间。
```

默认是 24h。

API 示例：

```http
GET /documents/ingestions/analytics?window=24h
```

时间筛选优先基于 task.createdAt。这样可以让用户快速切换观察范围，但不引入复杂日期选择器。

### 4.4 Analytics API

新增：

```http

```

返回内容包括：


## 5. 关键指标设计

### 5.1 总任务数

```text
totalTasks
```

表示当前时间窗口内所有 ingestion tasks，是成功率、失败率计算的分母。

### 5.2 成功任务数

```text
completedTasks
```

表示当前窗口内状态为 COMPLETED 的任务数量。

### 5.3 失败任务数

```text
failedTasks
```

表示当前窗口内状态为 FAILED 的任务数量。

### 5.4 成功率

```text
successRate = completedTasks / totalTasks
```

如果 totalTasks = 0，successRate = 0，前端显示“暂无数据”，不能显示 NaN 或 Infinity。

### 5.5 失败率

```text
failureRate = failedTasks / totalTasks
```

同样需要处理 totalTasks = 0 的情况。

### 5.6 平均总耗时

优先计算：

```text
completedAt - createdAt
```

只统计 COMPLETED 任务。不把 FAILED、PROCESSING、PENDING 任务纳入平均总耗时。

如果没有 completed tasks：

```text
averageTotalDurationMs = null
```

前端显示“暂无完成任务”。

### 5.7 阶段平均耗时

基于 event 表中的 completed events 聚合：

```text
CHUNKING_COMPLETED -> 文档切分；
EMBEDDING_COMPLETED -> 生成文档向量；
VECTOR_WRITE_COMPLETED -> 写入知识库索引。
```

每个阶段返回：

```text
stage；
displayName；
averageDurationMs；
sampleCount。
```

如果某阶段没有样本，则 sampleCount = 0，averageDurationMs = null，前端显示“暂无数据”。

### 5.8 常见失败原因

基于 TASK_FAILED event 的 errorCode 聚合。

规则：

```text
按 errorCode 分组；
按 count 降序；
最多返回前 5 个；
errorCode 为空时归为 UNKNOWN_ERROR。
```

中文映射示例：

```text
PDF_NO_EXTRACTABLE_TEXT -> PDF 没有可提取文字；
EMBEDDING_PROVIDER_ERROR -> 生成文档向量失败；
VECTOR_STORE_WRITE_ERROR -> 写入知识库索引失败；
UNKNOWN_ERROR -> 未知错误。
```

### 5.9 最近失败任务

recentFailures 返回最近失败任务，建议最多 5 条。

字段包括：

```text
taskId；
documentId；
filename；
errorCode；
errorMessage；
displayMessage；
failedAt；
retryCount。
```

用途是让用户和开发者快速定位最近失败。

### 5.10 慢任务列表

slowTasks 返回耗时最长的任务，建议最多 5 条。

字段包括：

```text
taskId；
documentId；
filename；
totalDurationMs；
bottleneckStage；
bottleneckDisplayName；
createdAt；
completedAt。
```

bottleneckStage 的计算方式：读取该 task 的阶段 completed events，从 CHUNKING、EMBEDDING、VECTOR_WRITING 中选择 durationMs 最大的阶段。

如果无法判断，则返回 null，前端显示“暂无法判断主要耗时阶段”。

## 7. 关键取舍

### 7.1 为什么不接 Prometheus / Grafana

Prometheus / Grafana 是系统级监控工具，适合长期指标采集和图表化展示。

但 V3.5 的目标是基于业务任务数据做轻量分析。当前系统已经有 task 和 event 表，直接查询即可。

接 Prometheus / Grafana 会引入指标导出、Prometheus 配置、Grafana dashboard、部署复杂度和额外运维成本，超出了 V3.5 的边界。

### 7.2 为什么不接 OpenTelemetry

OpenTelemetry 适合分布式 trace。但 V3.5 是文档摄入业务分析，不是跨服务链路追踪。

当前更重要的是：

```text
文档处理成功率；
失败率；
阶段耗时；
失败原因。
```

这些更适合从业务事件表聚合。

### 7.3 为什么不做定时聚合表

定时聚合表适合大数据量场景，但当前项目数据量不大，实时查询 task 和 event 表即可。

新增 summary 表会带来定时任务、数据延迟和一致性问题，所以 V3.5 暂不做。

### 7.4 为什么不做复杂图表

本版本核心是分析能力，不是可视化大屏。

卡片和表格足够表达成功率、失败率、平均耗时、阶段耗时、失败原因和慢任务。

复杂图表会引入前端依赖，偏离当前项目主线。

### 7.5 为什么不做成本美元估算

成本估算需要 embedding provider price、token 统计、模型维度、调用次数、缓存命中率、向量存储成本和失败重试成本。

这属于后续 FinOps / 成本控制版本。V3.5 只做性能和失败分析。

## 13. 面试高强度追问与回答

### 13.1 V3.5 和 V3.4 有什么区别？

回答：

```text
V3.4 关注单个任务的事件时间线，回答“这个任务经历了哪些步骤、哪一步失败、每一步耗时多久”。

V3.5 基于这些事件数据做系统级聚合，回答“整个文档摄入系统运行得怎么样、成功率是多少、哪个阶段最慢、最常见失败原因是什么、最近有哪些失败任务和慢任务”。
```

### 13.2 为什么需要 IngestionAnalyticsService？

回答：

```text
因为 analytics 涉及多种聚合逻辑，包括状态统计、成功率、失败率、平均耗时、阶段耗时、失败原因聚合、慢任务识别。如果把这些逻辑写在 Controller 里，会导致 Controller 过重、难测试、难维护。

所以我把聚合逻辑集中到 IngestionAnalyticsService 中，Controller 只负责参数解析和返回响应。
```

### 13.3 成功率怎么计算？

回答：

```text
成功率等于 completedTasks / totalTasks。totalTasks 为 0 时返回 0，并且前端显示“暂无数据”，避免出现 NaN 或 Infinity。
```

### 13.4 平均总耗时怎么计算？

回答：

```text
平均总耗时只统计 COMPLETED 任务，使用 completedAt - createdAt。FAILED、PROCESSING 和 PENDING 任务不纳入平均总耗时，因为它们没有完整生命周期，纳入会导致指标失真。
```

### 13.5 阶段平均耗时来自哪里？

回答：

```text
来自 V3.4 的 ingestion event timeline。比如 CHUNKING_COMPLETED、EMBEDDING_COMPLETED、VECTOR_WRITE_COMPLETED 事件中记录了 durationMs，V3.5 基于这些事件聚合出文档切分、生成文档向量、写入知识库索引的平均耗时。
```

### 13.6 如何判断慢任务的瓶颈阶段？

回答：

```text
对某个 completed task，读取它的阶段 completed events，从 CHUNKING、EMBEDDING、VECTOR_WRITING 这些阶段中选择 durationMs 最大的阶段，作为 bottleneckStage。如果无法判断，就返回 null，前端显示“暂无法判断主要耗时阶段”。
```

### 13.7 为什么不接 Prometheus / Grafana？

回答：

```text
Prometheus / Grafana 更适合系统指标监控和长期趋势图表。但 V3.5 当前要做的是基于业务任务表和事件表的轻量分析，例如成功率、失败原因、阶段耗时和慢任务。

当前数据量小，直接查询已有表足够。接 Prometheus / Grafana 会引入部署和运维复杂度，超出版本目标。
```

### 13.8 为什么不做定时聚合表？

回答：

```text
定时聚合表适合大数据量场景，但当前项目数据量不大，实时查询 task 和 event 表即可。新增 summary 表会带来定时任务、数据延迟和一致性问题，所以 V3.5 暂不做。
```

### 13.9 为什么页面不用图表库？

回答：

```text
本版本的目标是提供可理解的分析视图，不是做复杂 dashboard。卡片和表格已经可以展示成功率、失败率、阶段耗时、失败原因和慢任务。引入 Chart.js、ECharts 或前端框架会增加项目复杂度，偏离当前后端主线。
```

### 13.10 这个版本体现什么工程能力？

回答：

```text
它体现的是业务可观测性和分析能力。前面版本记录了异步任务事件，V3.5 进一步把这些事件聚合成系统级指标，帮助判断文档处理系统是否稳定、哪里慢、哪里容易失败。

这不是简单展示数据，而是把任务状态、事件耗时和错误码转化为可决策的分析信息。
```
