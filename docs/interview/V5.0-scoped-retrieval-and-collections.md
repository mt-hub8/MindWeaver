# V5.0 Scoped Retrieval & Collections 面试文档

## 1. 阶段摘要

V5.0 的阶段名称是：

```text
V5.0 Scoped Retrieval & Collections
```

中文可以表述为：

```text
知识库范围检索与文档分组
```

这个阶段的核心目标是：把原来的“全库问答”升级为“可选择范围的知识库问答”。

在 V4.0 之前，系统已经具备：

```text
文档上传；
异步摄入；
文档软删除；
文档重新索引；
旧 chunk 过滤；
事件时间线；
摄入分析；
RAG Answer API；
citations；
rerank；
hybrid retrieval。
```

但 Ask 页面默认仍然是：

```text
全库检索
```

这会带来一个真实产品问题：

```text
不同主题、不同项目、不同用途的文档混在一起时，RAG 可能召回不属于当前问题范围的内容。
```

例如用户只想问：

```text
项目 A 文档；
产品手册；
面试资料；
技术方案；
某一组 API 文档。
```

如果系统默认从所有文档中检索，就可能出现上下文污染：

```text
答案引用了不相关项目的文档；
相似术语来自错误业务域；
hybrid search 命中了其他分组里的关键词；
rerank 把范围外文档排到前面。
```

V5.0 通过引入 Collection（知识库分组）解决这个问题。

用户可以：

```text
创建知识库分组；
把文档加入分组；
从分组移出文档；
在 Ask 页面选择“全部文档”或某个分组；
系统只在指定范围内检索；
citations 只来自该范围内的文档。
```

## 2. V5.0 包含的内部 Phase

V5.0 作为一个大阶段一次性完成，内部包含三个 Phase。

### Phase 1：Collection Model & Document Assignment

目标：

```text
新增 Collection 知识库分组模型，并支持文档加入 / 移出分组。
```

核心能力：

```text
Collection 表；
document_collection 关系表；
POST /collections；
GET /collections；
GET /collections/{collectionId}；
POST /collections/{collectionId}/documents/{documentId}；
DELETE /collections/{collectionId}/documents/{documentId}；
Documents API 展示文档所属分组。
```

---

### Phase 2：Scoped Ask / Retrieval Filter

目标：

```text
Ask / RAG 支持按 collectionId 限定检索范围。
```

核心能力：

```text
/rag/answers 支持可选 collectionId；
不传 collectionId 保持全库检索；
传 collectionId 时只检索该分组下文档；
collection filter 与 V4.0 lifecycle filter 同时生效；
dense / lexical / hybrid / rerank / citations / prompt 都不能绕过 scope filter。
```

---

### Phase 3：Collection UI + Evaluation + Manual 收敛

目标：

```text
把分组管理、问答范围选择、Documents 分组归属、首页入口和 manual 文档统一收敛。
```

核心能力：

```text
/collections.html；
Documents 页面加入 / 移出分组；
Ask 页面选择问答范围；
首页知识库分组入口；
scoped retrieval manual；
阶段级回归测试。
```

## 3. 为什么要做 V5.0

一个真实知识库问答系统通常不会永远只有“全库问答”。

当文档数量变多后，用户会自然需要范围隔离：

```text
只问某个项目；
只问某个部门；
只问某类资料；
只问某几份文档组成的知识库。
```

如果没有范围控制，RAG 系统容易出现：

```text
召回范围过大；
无关文档进入上下文；
答案引用错误来源；
同名概念跨业务污染；
用户无法控制回答依据。
```

V5.0 的核心价值是：

```text
让用户能控制 RAG 的知识边界。
```

这一步让系统从：

```text
上传文档后统一进入一个大知识库
```

升级为：

```text
文档可以被组织成多个知识库分组，并按分组提问。
```

## 4. 核心设计一：Collection 知识库分组

### 4.1 Collection 的含义

Collection 在本项目中不是权限空间，也不是 workspace。

它的定位是：

```text
知识库分组。
```

一个 Collection 表示一组可一起检索和问答的文档。

例如：

```text
项目 A 文档；
产品手册；
技术方案；
面试资料；
API 文档；
论文资料。
```

---

### 4.2 Collection 数据模型

V5.0 新增 Collection 模型，核心字段通常包括：

```text
id；
name；
description；
status；
createdAt；
updatedAt。
```

其中：

```text
name：分组名称；
description：分组说明；
status：预留分组状态，当前可以简单处理。
```

前端中文展示：

```text
知识库分组；
分组名称；
分组说明；
文档数量；
可用于问答的文档。
```

---

### 4.3 Document-Collection 关系

文档和 Collection 是多对多关系：

```text
一个 document 可以加入多个 collection；
一个 collection 可以包含多个 document。
```

因此 V5.0 新增关系表：

```text
document_collection
```

核心字段：

```text
collection_id；
document_id；
created_at。
```

这个设计支持：

```text
同一份文档既属于“项目 A 文档”，也属于“技术方案”；
不同分组可以复用同一文档；
Ask 时按 collectionId 限定范围。
```

---

## 5. 核心设计二：Collection API

V5.0 新增 Collection 管理 API。

### 5.1 创建 Collection

```http
POST /collections
```

请求示例：

```json
{
  "name": "项目 A 文档",
  "description": "项目 A 的需求、设计和接口说明"
}
```

响应包含：

```text
collectionId；
name；
description；
documentCount；
activeDocumentCount；
createdAt；
updatedAt。
```

---

### 5.2 查询 Collection 列表

```http
GET /collections
```

返回所有知识库分组。

每个分组包含：

```text
collectionId；
name；
description；
documentCount；
activeDocumentCount；
createdAt；
updatedAt。
```

其中：

```text
documentCount：分组内所有文档数量；
activeDocumentCount：分组内仍可用于问答的 ACTIVE 文档数量。
```

这个字段对用户很重要，因为已删除文档可以保留归属关系，但不会参与问答。

---

### 5.3 查询 Collection 详情

```http
GET /collections/{collectionId}
```

返回：

```text
分组基本信息；
分组下的文档列表；
每个文档的状态；
是否可用于问答；
是否可移出分组。
```

---

### 5.4 文档加入 Collection

```http
POST /collections/{collectionId}/documents/{documentId}
```

语义：

```text
把某个文档加入知识库分组。
```

需要处理：

```text
collection 不存在；
document 不存在；
重复加入；
已删除文档加入分组。
```

已删除文档可以保留归属，但 Ask 仍然不会使用它。

前端需要提示：

```text
已删除文档可以保留分组归属，但不会参与问答。
```

---

### 5.5 文档移出 Collection

```http
DELETE /collections/{collectionId}/documents/{documentId}
```

语义：

```text
把某个文档从知识库分组中移出。
```

移出后，该文档不再属于该 Collection。

如果 Ask 指定该 Collection，就不会再检索这个文档。

## 6. 核心设计三：Scoped Ask / 范围问答

### 6.1 请求兼容扩展

V5.0 对 `/rag/answers` 做兼容扩展。

旧请求仍然可用：

```json
{
  "query": "问题内容",
  "topK": 5
}
```

新请求可以增加可选字段：

```json
{
  "query": "问题内容",
  "topK": 5,
  "collectionId": 1
}
```

兼容规则：

```text
不传 collectionId：保持全库问答；
传 collectionId：只在指定知识库分组中问答。
```

这不破坏旧 API 调用方。

---

### 6.2 Scoped Retrieval 语义

当 collectionId 为空：

```text
全库检索，但仍过滤 DELETED document 和 old / superseded chunks。
```

当 collectionId 不为空：

```text
只允许使用该 Collection 下的 ACTIVE documents 和 current active chunks。
```

也就是说，V5.0 的 filter 需要同时满足三类条件：

```text
Collection scope：文档属于指定分组；
Lifecycle status：文档必须 ACTIVE；
Chunk validity：chunk 必须是 current active chunk。
```

---

### 6.3 为什么必须做 application-level filter

即使未来 VectorStore 支持 metadata filter，也不能完全依赖它。

因为当前系统里存在：

```text
旧向量仍保留；
已删除文档的向量仍可能被检索；
旧 generation chunks 的 vectors 仍可能存在；
collection 外文档可能被 dense search 或 lexical search 召回。
```

所以必须在应用层统一过滤：

```text
VectorStore / lexical retriever / hybrid fusion / rerank 之后；
finalTopK / citations / prompt 之前。
```

这样才能保证：

```text
collection 外文档不会进入最终回答；
已删除文档不会进入最终回答；
旧版本 chunks 不会进入最终回答。
```

---

## 7. 核心设计四：覆盖所有检索路径

V5.0 的 scoped filter 不能只作用在一个地方。

必须覆盖：

```text
dense retrieval；
lexical retrieval；
hybrid fusion；
rerank；
citations；
prompt context；
retrieval metadata final results；
no-context 判断。
```

否则会出现漏洞。

例如：

```text
dense 检索过滤了 collection；
但 lexical 检索仍然命中 collection 外关键词；
hybrid fusion 又把 collection 外文档带回来；
rerank 再把它排到前面；
citations 最终引用了错误范围文档。
```

所以 V5.0 必须在统一的 candidate validation 层处理 scope + lifecycle filter。

---

## 8. 核心设计五：空范围处理

V5.0 要处理几个 no-context 场景。

### 8.1 Collection 存在但没有文档

返回 no-context answer，不调用 LLM。

中文提示：

```text
当前分组下没有可用于问答的文档，请先添加已启用文档。
```

---

### 8.2 Collection 中只有 DELETED 文档

返回 no-context answer，不调用 LLM。

中文提示：

```text
当前分组中的文档均已删除，无法用于问答。
```

---

### 8.3 Collection 中只有 old / superseded chunks

返回 no-context answer，不调用 LLM。

中文提示：

```text
当前分组暂无可用的最新索引片段，请重新建立索引或上传文档。
```

---

## 9. 核心设计六：Retrieval Metadata

V5.0 可以在 retrieval metadata 中增加 scope 信息。

建议字段：

```text
scopeType：ALL_DOCUMENTS / COLLECTION；
collectionId；
collectionName；
filteredByCollectionCount；
filteredByLifecycleCount；
finalContextCount。
```

前端中文展示：

```text
当前问答范围：全部文档；
当前问答范围：项目 A 文档；
已过滤不在该分组内的候选片段；
已过滤已删除或旧版本片段。
```

这些字段让用户知道系统为什么没有使用某些候选结果，也让开发者能排查 filter 是否生效。

## 12. 关键取舍

### 12.1 为什么做 Collection，而不是 workspace

workspace 通常意味着：

```text
用户；
团队；
权限；
隔离；
计费；
审计；
多租户。
```

V5.0 当前只需要解决：

```text
知识范围选择。
```

所以选择 Collection 作为轻量文档分组，不引入权限和多租户复杂度。

---

### 12.2 为什么不做复杂标签系统

标签系统会引入：

```text
多标签组合过滤；
标签搜索；
标签层级；
标签编辑；
标签统计。
```

V5.0 只需要让用户选择一个知识库分组进行问答。

Collection 足够表达这个需求。

---

### 12.3 为什么 collectionId 是可选字段

为了兼容旧 API。

如果不传 collectionId，旧的全库问答行为保持不变。

如果传 collectionId，则进入范围检索。

这样不会破坏已有 Web UI、测试和调用方。

---

### 12.4 为什么不做权限

Collection 不是权限边界。

当前阶段没有用户系统，所以不能伪装成权限隔离。

V5.0 的边界是：

```text
检索范围控制，不是访问权限控制。
```

---

### 12.5 为什么仍然需要 lifecycle filter

Collection 只表示文档归属，不表示文档一定可用。

一个文档可能属于某个 Collection，但它已经被删除，或者它的 chunk 已经过期。

所以 scope filter 必须和 V4.0 lifecycle filter 叠加：

```text
属于 Collection；
document ACTIVE；
chunk current active。
```

三者都满足才可以进入回答上下文。

## 18. 面试高强度追问与回答

### 18.1 为什么要做 Collection？

回答：

```text
因为全库问答在文档变多后容易产生上下文污染。用户很多时候只想问某个项目、某类资料或某组文档。Collection 让用户可以把文档组织成知识库分组，并在 Ask 时选择检索范围，避免无关文档进入回答上下文。
```

---

### 18.2 Collection 和 workspace 有什么区别？

回答：

```text
Collection 在当前项目里只是知识库分组，用于控制检索范围。workspace 通常涉及用户、团队、权限、多租户和隔离。V5.0 不做权限系统，所以我没有把 Collection 设计成 workspace，而是保持为轻量的文档分组。
```

---

### 18.3 为什么 `/rag/answers` 的 collectionId 是可选字段？

回答：

```text
为了保持兼容性。旧请求不传 collectionId 时仍然执行全库问答。新请求传 collectionId 时进入指定分组问答。这样不会破坏已有 UI、测试或 API 调用方。
```

---

### 18.4 如果 VectorStore 返回 collection 外文档怎么办？

回答：

```text
不能信任 VectorStore 一定按 collection 过滤。系统必须在应用层做 candidate validation。VectorStore 返回 candidates 后，retrieval service 会根据 documentId / chunkId 回查关系和状态，只保留属于指定 collection、document ACTIVE、chunk current active 的候选。
```

---

### 18.5 scope filter 和 lifecycle filter 有什么区别？

回答：

```text
scope filter 判断文档是否属于当前问答范围，比如是否属于指定 Collection。lifecycle filter 判断文档和 chunk 是否当前有效，比如 document 是否 ACTIVE、chunk 是否 current active。

一个文档属于 Collection，不代表它可用；它可能已经被删除。一个 chunk 属于文档，也不代表它是当前版本；它可能是旧 generation。所以两个 filter 必须叠加。
```

---

### 18.6 为什么要覆盖 hybrid 和 rerank？

回答：

```text
因为系统已经支持 dense、lexical、hybrid fusion 和 rerank。如果 collection filter 只在 dense 阶段生效，lexical 或 hybrid 仍可能把范围外文档带回来。因此必须在统一 candidate validation 层保证所有路径最终进入 citations 和 prompt 前都经过 scope + lifecycle 校验。
```

---

### 18.7 collection 为空时为什么不调用 LLM？

回答：

```text
如果 collection 下没有可用文档，就没有可靠上下文。继续调用 LLM 会增加幻觉风险，也浪费成本。所以系统应该返回 no-context answer，并提示用户先添加已启用文档。
```

---

### 18.8 DELETED 文档如果仍在 Collection 里怎么办？

回答：

```text
可以保留它的分组关系，用于历史展示。但 Ask 时不会使用它，因为 V4.0 lifecycle filter 会过滤 DELETED document。页面也会提示“已删除文档不会参与问答”。
```

---

### 18.9 为什么不做权限？

回答：

```text
权限需要用户系统、角色、ACL、多租户和安全边界。V5.0 的目标是知识范围控制，不是访问控制。如果把权限混进来，阶段边界会过大。所以当前 Collection 只是检索范围，不是权限隔离。
```

---

### 18.10 这个阶段体现什么工程能力？

回答：

```text
V5.0 体现的是 RAG 检索范围控制和多维过滤一致性。系统不仅要知道怎么检索，还要知道“在哪些文档范围内检索”，并且要保证 collection scope、document lifecycle、chunk generation 三个维度同时生效。这是 RAG 产品从 demo 走向真实知识库应用的重要能力。
```

## 21. 总结表达

这个阶段可以用一段话总结：

```text
在 V5.0 中，我把 AI Knowledge Assistant 从全库问答升级为支持知识库分组和范围检索的系统。系统新增 Collection 模型和 document_collection 多对多关系，支持创建知识库分组、将文档加入 / 移出分组，并在 Ask 页面选择“全部文档”或某个分组作为问答范围。

在 RAG Answer API 中，我通过可选 collectionId 保持旧请求兼容：不传 collectionId 时仍然全库问答，传入 collectionId 时只在指定 Collection 中检索。检索链路会在 dense、lexical、hybrid、rerank、citations 和 prompt 构造前统一做 application-level filter，确保候选结果同时满足“属于指定 Collection、document 为 ACTIVE、chunk 为 current active”。这样即使 VectorStore 返回 collection 外文档、已删除文档或旧版本 chunks，也不会进入最终回答。

V5.0 还新增了中文友好的 Collections 页面、Documents 分组归属管理、Ask 问答范围选择和 scoped retrieval manual，使用户可以低门槛理解和控制知识库问答范围，避免不同主题文档互相污染。
```
