# V1.0 面试文档：LLM 抽象与 MockLlmClient

## 1. 本版本解决的问题

V1.0 解决任务执行逻辑和具体模型调用强耦合的问题。通过 LlmClient、LlmRequest、LlmResponse 和 MockLlmClient，TaskExecutionService 不再只做纯模拟 sleep，而是通过统一接口调用 LLM Provider 的抽象实现。

## 2. 为什么现在做这个版本

它依赖阶段 0 的可靠异步任务系统。只有任务调度、状态、失败、重试、取消和超时具备基础能力后，接入 LLM 才有可靠承载。此时不接真实 OpenAI/Claude/Qwen，是为了先验证接口抽象和执行链路。

## 3. 核心需求边界

本版本定义 LlmClient 抽象和 MockLlmClient，并让 TaskExecutionService 调用它。不做真实外部 API、不写 API Key、不做 Prompt Template、不做 Token Usage、不做 Streaming/RAG/Agent。

## 4. 方案选择

LlmClient 定义 generate(LlmRequest) -> LlmResponse。MockLlmClient 作为本地 Provider，prompt 包含 fail/失败时返回失败，否则返回模拟 content。TaskExecutionService 只依赖接口，不关心具体 Provider。

## 5. 替代方案与暂时不选的原因

替代方案一是在 TaskExecutionService 直接写模型调用。当前不选，因为会把执行编排和 Provider 细节耦合。

替代方案二是直接接真实 LLM。当前不选，因为会引入密钥、成本、网络失败和合规问题，影响基础链路验证。

## 6. 数据库设计

本版本未新增数据库结构，主要复用 task 表。后续 V1.0.4 会增加 result_content 和 llm_model 保存结果。

## 7. API 设计

本版本主要是内部执行能力，未新增用户 API。外部仍通过 POST /tasks 创建任务，通过 GET /tasks/{id} 查看最终状态。

## 8. 核心流程

Consumer 收到 taskId
-> TaskExecutionService claim RUNNING
-> 构造 LlmRequest
-> 调用 LlmClient.generate
-> MockLlmClient 返回 LlmResponse
-> 成功 SUCCESS，失败进入重试或 FAILED

## 9. 测试方案

建议覆盖 MockLlmClient 成功响应、失败响应、空 request、空 prompt、TaskExecutionService 成功和失败分支。异常路径包括 LlmClient 抛异常后进入原有失败处理。

## 10. 实际测试结果

历史版本已在后续版本中被整体回归验证；如需补充该版本独立测试结果，应重新运行相关 API / 单元测试后补充。

## 11. 工程难点

难点是抽象边界：LlmClient 只表达模型调用，不承担任务状态、重试、取消。TaskExecutionService 编排调用和状态，TaskService 管理状态流转。

## 12. 本版本亮点

- 将模型调用抽象为 LlmClient。
- 用 MockLlmClient 本地验证成功和失败。
- 保持任务执行系统不依赖具体 Provider。
- 为真实模型接入预留接口。

## 13. 简历 bullet

### 简洁版

设计 LlmClient 抽象和 MockLlmClient，实现任务执行链路与模型 Provider 解耦。

### 强化版

在异步 AI 任务系统中抽象 LlmClient / LlmRequest / LlmResponse，并接入 MockLlmClient 验证执行链路，为后续 OpenAI、Claude、Qwen 或本地模型 Provider 扩展奠定接口基础。

## 14. 面试高强度追问与回答

Q1：为什么需要 LlmClient 抽象？

A1：不同 Provider 的 API、错误和响应格式不同。统一接口让执行服务不依赖具体实现。

Q2：为什么先用 Mock LLM？

A2：Mock 不需要密钥和网络，能稳定验证编排逻辑、失败处理和重试。

Q3：MockLlmClient 有什么意义？

A3：它不是为了语义效果，而是为了验证工程链路，尤其是成功/失败分支。

Q4：为什么 TaskExecutionService 不直接调 OpenAI？

A4：直接调用会把 Provider 细节写死，后续换模型或多 Provider 会很难。

Q5：LlmResponse 为什么需要 success 字段？

A5：Provider 可能返回业务失败而不是抛异常，success 能统一表达调用结果。

Q6：异常和失败响应怎么区分？

A6：异常是调用过程异常，失败响应是 Provider 明确返回失败。两者都进入任务失败处理。

Q7：后续如何接真实 Provider？

A7：新增实现类，实现 LlmClient，处理认证、请求、响应映射和错误转换。

Q8：当前局限是什么？

A8：Mock 不代表真实模型效果和延迟，也没有 token usage、streaming 和成本统计。
